「オブジェクト指向」という言葉、あなたは自分の言葉で自信を持って説明できますか？
…正直に言うと、これ、めちゃくちゃ難しいですよね。

ネットで調べてみると、大体こんな説明が出てきます。
> **「オブジェクト指向（Object-Oriented）とは」**
> データと処理を「オブジェクト（モノ）」としてまとめ、それらを組み合わせてシステムを構築する設計手法のこと。現実世界の概念をプログラムに落とし込みやすく、再利用性やメンテナンス性に優れているため、現代の開発では主流となっています。


### 「…で、結局どういうこと？」
この説明を読んで、「なるほど、完全に理解した！」と言える人がどれくらいいるでしょうか。おそらく、多くの人が「わかったような、わからないような…」というモヤモヤを抱えるはずです。

抽象的すぎて、いまいちピンときませんよね。

### COBOLからJavaへ。6年越しの「気づき」
実は私、入社から **6年間ずっと「COBOL」** という言語を扱ってきました。
COBOLはいわゆる「非オブジェクト指向（手続き型）」の代表格のような言語です。

その後、心機一転して **「Java」** というオブジェクト指向言語の世界に飛び込みました。

「手続き型」のど真ん中にいた人間が、「オブジェクト指向」の壁にぶつかり、悩み抜いた結果。ようやく自分なりの答えが見えてきたんです。


## オブジェクト指向とは？
6年間COBOL（手続き型）にどっぷり浸かった私が、Java（オブジェクト指向）に触れて気づいた **「決定的な違い」** を、自分なりの言葉で紐解いてみます。

### 最小単位が「機能」か、それとも「モノ」か？
まず、誰かに「オブジェクト指向を一言で言うと？」と聞かれたら、私はこう答えます。

**「プログラムを組み立てる部品の単位が『モノ』である」**

これだけだと少し抽象的ですね。長年親しんできたCOBOLなどの「手続き型言語」と比較すると、その差がはっきりします。

| 言語タイプ | プログラムの構成単位 | 考え方のベース |
| :--- | :--- | :--- |
| **手続き型 (COBOL/C等)** | **機能（処理）** | 「何をどう処理するか」という手順 |
| **オブジェクト指向 (Java等)** | **モノ（オブジェクト）** | 「誰が（何が）どう動くか」という主語 |

手続き型が「計算する」「印字する」といった **処理（アクション）** を積み上げるのに対し、オブジェクト指向はまず「顧客」「注文」「ペン」といった **実体（モノ）** を作り、そこにデータと動きを持たせるイメージです。今の段階では、**そもそもの設計思想が異なる**ということだけ押さえておきましょう。

## オブジェクト指向で実現できること
では次に、**「オブジェクト指向を使うと、具体的に何がどう便利になるのか？」** という核心部分に迫ります。
ここを理解すると、プログラミングがただの「作業」から、スマートな「設計」へと進化します。

### オブジェクト指向の真骨頂は「概念」を扱えること
オブジェクト指向を導入する最大のメリット。それは一言で言うと、システムの中に **「概念（抽象）」** を持ち込めることにあります。

手続き型の世界（COBOLなど）では、「見た目は似ているけど、細かい処理が違うから共通化できない……」と泣く泣く諦めていた場面が多々ありました。しかし、オブジェクト指向なら **「概念」というフィルター** を通すことで、バラバラだったものを「同じ仲間」として扱えるようになります。

![](https://storage.googleapis.com/zenn-user-upload/00e4fd0d3842-20260416.png =580x)

形や色が違っても、抽象化することで「三角形である」という共通点を見つけ出し、ひとまとめにする。この **「同一視する力」** こそが、オブジェクト指向の真骨頂なんです。


### 現場で役立つ「概念」の2大パターン
では、具体的にどうやって「概念」を使いこなせばいいのでしょうか？
私が実務で試行錯誤した結果、たどり着いたのが **「真逆の思想を持つ2つのパターン」** です。
「真逆の思想を持つ2つのパターン」という言葉は、技術書に載っているような堅苦しい理論ではなく、現場で感じた「私なり」の分類です。

![](https://storage.googleapis.com/zenn-user-upload/c0f7f338a962-20260413.png)

### パターン1：相手を「意識しない」自由（疎結合パターン）
1つ目は、**「使う側が、相手の正体を詳しく知らなくても済むようにする」** ためのパターンです。

![](https://storage.googleapis.com/zenn-user-upload/26ccb834b382-20260415.png =400x)

ここに、道具を使って文字を書く **「Person（人）クラス」** があるとします。

概念を使わない場合：
「ボールペンを使う処理」と「鉛筆を使う処理」を別々に書かなければなりません。もし将来「万年筆」が追加されたら、Personクラスの中身をわざわざ書き直すハメになります。これではPersonさんは、新しい道具が出るたびに勉強し直さなければならず、大変です。

概念を使う場合：
ここで、**「Writable（書けるもの）」** という概念を登場させます！

Personさんは、相手がボールペンだろうが鉛筆だろうが、「あ、これは『書けるもの（Writable）』だな」とさえ分かれば、それを使って作業ができます。

ここがメリット！
新しく「万年筆」が登場しても、それが「Writable（書けるもの）」のルールに従ってさえいれば、Personクラスを修正する必要は一切ありません。

このように、**「相手を詳しく知らなくても仕事ができる」** という、風通しの良い関係（疎結合）が作れるのです。

さあ、コードで見てみましょう！
「理屈はわかったけど、実際にどう書くの？」という方のために、簡単なソースコードを用意しました。

>**【仕様】**
>
>**ペンと鉛筆があります。それぞれで以下の処理を行ってください。**
>**①芯を新しくする（UpdateCore）**
ペンの場合：「ペンの芯を変えます」と表示
鉛筆の場合：「鉛筆を削ります」と表示
>
>**②書類を作成する（MakeDocument）**
「＊＊＊　書類を作成します　＊＊＊」と表示
ペンの場合：「ペンで書きます」と表示
鉛筆の場合：「鉛筆で書きます」と表示

>
>**【実行結果】**
>```

>＊＊＊　書類を作成しました　＊＊＊
>

>＊＊＊　書類を作成します　＊＊＊

>＊＊＊　書類を作成しました　＊＊＊
>```
>
>**【実装】**
>
>```java:Writable.java
>public interface Writable { // 筆記用具（書けるもの）
>    public void write();
>    public void updateCore();
>}
>```
>```java:Pen.java
>public class Pen implements Writable{
>    @Override
>    public void write() {
>        System.out.println("ペンで書きます");
>    } 
>    @Override
>    public void updateCore() {
>        System.out.println("ペンの芯を変えます");
>    }
>}
>```
>```java:Pencil.java
>public class Pencil implements Writable{
>    @Override
>    public void write() {
>        System.out.println("鉛筆で書きます");
>    }
>    @Override
>    public void updateCore() {
>        System.out.println("鉛筆を削ります");
>    }
>}
>```
>```java:Person.java
>public class Person {
>    public void makeDocument(Writable writable) { 
>        // 最重要！：ペンか鉛筆を受け取っているが、「筆記用具」と意識している
>        System.out.println("＊＊＊　書類を作成します　＊＊＊");

### パターン2：お作法を「強制する」安心（密結合パターン）
2つ目は、**「決まった手順（お作法）を守らせる」** ためのパターンです。

![](https://storage.googleapis.com/zenn-user-upload/48d3c5ffdfeb-20260415.png =450x)

プログラミングをしていると、「やり方は自由でいいんだけど、この順番だけは絶対に飛ばさないで！」という場面がよくあります。

**たとえば：「書類作成」のワークフロー**
あなたが「書類を作成するシステム」を作っているとしましょう。どんな種類の書類であっても、必ず以下の手順を踏まなければならないというルールがあるとします。

プロセス1： 書式をチェックする
プロセス2： 承認印の枠を作る
詳細プロセス： その書類固有の中身を書く

もし、これらをバラバラに実装していたらどうなるでしょうか？
ある人は「プロセス2」を作り忘れ、またある人は「詳細プロセス」を先にやってしまうかもしれません。これではシステムとして大混乱です。

**「型（テンプレート）」で強制する**
ここで、オブジェクト指向の「概念」が威力を発揮します。
「書類作成を行うもの」という共通の枠組み（テンプレート）をあらかじめ作っておくのです。

親（概念）の言い分：
「書類を作る流れは『1→2→3』って決めておいたよ。具体的な中身だけ、君たちの好きなように書いてね。ただし、順番は勝手に変えられないよ！」

子（実体）の役割：
「了解です！僕は『中身を書く』ところだけに集中しますね。全体の流れは親分に任せます！」

そして、親と子がガチっと合わさることで、一つの機能が完成します。

ここがポイント！
全体の流れ（お作法）を親がガッチリ握っておくことで、「プロセス2を実行し忘れた」というような人為的ミスを物理的に防げます。誰が書いても、システムとして **「正しい振る舞い」** が保証される。これがこのパターンの最大の安心感です。

さあ、コードで見てみましょう！
（コードを見るのが苦手な方は、ここは読み飛ばしてOKです！）

以下の仕様で、どんな実行結果になるかイメージしながら眺めてみてください。


>**仕様**
>
>顧客ごとに「売上明細伝票」プログラムを作成する。
売上明細伝票は顧客ごとにバラバラのレイアウトであるが、以下については共通している。
①ヘッダーを出す（＊＊ヘッダー＊＊）
②顧客固有の処理①を行う
③顧客固有の明細を繰り返し出す（デフォルト３回）
　※繰り返し回数を変える場合、顧客固有の処理①で変更する
④顧客固有の処理②を行う
⑤フッターを出す（＊＊フッター＊＊）
　※今回は簡潔にするために固定文字列を出力することとする
>
>
>**実行結果**
>
>```
>＊＊ヘッダー＊＊
><<<　独自処理１やります　>>>
><<<　明細処理やります　>>>
><<<　明細処理やります　>>>
><<<　明細処理やります　>>>
><<<　明細処理やります　>>>
><<<　明細処理やります　>>>
><<<　独自処理２やります　>>>
>＊＊フッター＊＊
>```
>
>**実装**
>
>```java:Template.java
>public abstract class Template {
>    protected int iterationNumber = 3;
>	
>    public void makeDocument() { // 処理の手順を実装
>        System.out.println("＊＊ヘッダー＊＊");
>        originalProcess01();
>        for (int i=0; i<iterationNumber; i++) {
>            specificsProcess();
>        }
>        originalProcess02();
>        System.out.println("＊＊フッター＊＊");
>    }
>	
>    protected abstract void originalProcess01();
>    protected abstract void originalProcess02();
>    protected abstract void specificsProcess();
>}
>```
>```java:Document01.java
>public class Document01 extends Template{
>    @Override
>    protected void originalProcess01() {
>        super.iterationNumber = 5;
>        System.out.println("<<<　独自処理１やります　>>>");
>    }
>    @Override
>    protected void originalProcess02() {
>        System.out.println("<<<　独自処理２やります　>>>");
>    }
>    @Override
>    protected void specificsProcess() {
>        System.out.println("<<<　明細処理やります　>>>");
>    }
>}

>        Template document = new Document01();
>        document.makeDocument();
>    }
>}

## オブジェクト指向がシステム開発のメインストリームになった理由
なぜ、これほどまでにオブジェクト指向が「開発の当たり前」になったのでしょうか？

結論から言うと、**「Webシステムの普及が、オブジェクト指向を決定的なメインストリームに押し上げた」**。私はそう確信しています。

今回は、Web開発とオブジェクト指向の意外なほど密接な関係について、少しマニアックに、でも分かりやすく紐解いてみます。


### 1. Webの「面倒くさい」を隠蔽した魔法
初期のWeb開発（CGIなど）は、飛んできたリクエストを解析して、1行ずつHTMLを書き出す…といった「手続き」をすべて自力で書くスタイルが主流でした。

しかし、Webシステムを作っていると、あることに気づきます。**「あれ、これ前回も同じようなこと書いたな？」** という「お決まりのパターン」が山ほどあるんです（ただし、パターンが一緒なだけで、全く同じ処理ではないので「共通化」はできないことに注意して下さい）。

* データベースと繋ぐ処理
* ユーザーが入れたデータのチェック（バリデーション）
* 「このURLならこの処理へ」という交通整理

ここで登場したのが **「Webアプリケーションフレームワーク」** です。
これらは、前回お話しした「密結合パターン（お作法を強制する）」の親玉のような存在です。

フレームワークが面倒な土台を「部品（スーパークラス）」として用意してくれたおかげで、開発者は **「そのシステム固有の差分（サブクラス）」だけを書けばいい** という革命が起きたのです。


### 2. 「学んだ」のではなく「使わされていた」？
面白い事実があります。実は、現場のWebエンジニアの多くが「オブジェクト指向とは何か」を完璧に語れるわけではありません。

それなのに、なぜみんなオブジェクト指向で書けるのか？
それは、**「フレームワークを使っていたら、いつの間にかオブジェクト指向を強制されていたから」** です。

* **JavaのSpring**
* **Ruby on Rails**
* **PHPのLaravel**

これらの有名なツールは、すべてオブジェクト指向の思想でガチガチに固められています。私たちは「Webサイトを効率よく作りたい」という一心でこれらのツールを使い、その結果、**無意識のうちにオブジェクト指向という巨大なレールの上を走らされていた**、というのが実態に近いでしょう。

### 3. 複雑すぎる世界を救った「枠組み」
Webシステムが「単なる掲示板」から「巨大なECサイトやSNS」へと進化するにつれ、コードの量は爆発的に増えました。

もしオブジェクト指向（そしてそれを体現したフレームワーク）がなかったら…と想像すると、ゾッとします。おそらくWeb開発の世界は、複雑に絡まり合った「スパゲッティコード」の海に沈んで、今のような便利なサービスは誕生していなかったはずです。

オブジェクト指向は、ただの「おしゃれな書き方」ではありません。爆発的に増える **「システムの複雑さ」を管理し、破綻させないための唯一の防波堤** だったのです。
