# 前提知識
SQLにおいて、実務上のキモとなるのが **「テーブルの結合（JOIN）」** です。

## 1. なぜ結合が必要なのか
データベースのテーブルは基本的に **「正規化」** という手法を用いて設計されています。これは、データを効率よく、かつ矛盾が発生しないように、あえて**複数のテーブル**に切り分けて管理するためです。

そのため、アプリケーション開発で作成するSQLの多くは、一つのテーブルだけで完結することはありません。複数のテーブルから必要なデータをかき集め、加工・編集した上で、効率よく取り出すスキルが求められます。

![](https://storage.googleapis.com/zenn-user-upload/49bf2917e85c-20260320.png)



## 2. 学習の第一歩
結合を正しく使いこなすためには、まず **「内部結合（INNER JOIN）」と「外部結合（OUTER JOIN）」の違いを明確に理解すること**が、最初にして最大のステップとなります。

# 問題10-1（内部結合） *Lv2*
### OEスキーマの各テーブルを使用して、指定された条件に合うデータを取得してください。（順不同）

**抽出条件**
  - `ORDER_ID` が 「2454」 である注文データ

**取得する項目（カラム）**
以下の4つの項目を表示するようにクエリを作成してください。
* ORDER_DATE
  - `ORDERS` テーブルの `ORDER_DATE` を使用します。表示形式は **「YYYY-MM-DD HH24:MI:SS」** としてください。

* CUSTOMER_NAME
  - `CUSTOMERS` テーブルの `CUST_FIRST_NAME` と `CUST_LAST_NAME` を結合して表示してください。姓名の間には **半角スペース** を入れるようにしましょう。
  
* PRODUCT_NAME
  - `PRODUCT_INFORMATION` テーブルから取得してください。
  
* QUANTITY
   - `ORDER_ITEMS` テーブルから取得してください。


## 期待する結果
| ORDER_DATE          | CUSTOMER_NAME  | PRODUCT_NAME    | QUANTITY | 
| ------------------- | -------------- | --------------- | -------- | 
| 2007-10-02 16:49:34 | Manisha Taylor | PS 12V /P       | 3        | 
| 2007-10-02 16:49:34 | Manisha Taylor | MB - S600       | 0        | 
| 2007-10-02 16:49:34 | Manisha Taylor | Video Card /E32 | 12       | 
| 2007-10-02 16:49:34 | Manisha Taylor | KB 101/ES       | 120      | 
| 2007-10-02 16:49:34 | Manisha Taylor | Resin           | 18       | 
| 2007-10-02 16:49:34 | Manisha Taylor | Screws <B.32.P> | 16       | 
| 2007-10-02 16:49:34 | Manisha Taylor | Screws <S.32.S> | 13       | 


## 解答例
```sql:例1：INNER JOINを使用
SELECT
    TO_CHAR(o.order_date, 'YYYY-MM-DD HH24:MI:SS')  AS "ORDER_DATE",
    c.cust_first_name || ' ' || c.cust_last_name  AS "CUSTOMER_NAME",
    p.product_name,
    oi.quantity
FROM
    oe.order_items                   oi
    INNER JOIN oe.orders              o 
       ON oi.order_id = o.order_id
    INNER JOIN oe.customers           c 
       ON o.customer_id = c.customer_id
    INNER JOIN oe.product_information p 
       ON oi.product_id = p.product_id
WHERE o.order_id = 2454
```
```sql:例2：使用するテーブルをカンマで区切る方法
SELECT
    TO_CHAR(o.order_date, 'YYYY-MM-DD HH24:MI:SS')  AS "ORDER_DATE",
    c.cust_first_name || ' ' || c.cust_last_name  AS "CUSTOMER_NAME",
    p.product_name,
    oi.quantity
FROM
    oe.order_items         oi,
    oe.orders              o,
    oe.customers           c,
    oe.product_information p
WHERE
        o.order_id = 2454
    AND oi.order_id = o.order_id
    AND o.customer_id = c.customer_id
    AND oi.product_id = p.product_id
```
```sql:例3：USINGを使用（例1の派生）
SELECT
    TO_CHAR(order_date, 'YYYY-MM-DD HH24:MI:SS')  AS "ORDER_DATE",
    c.cust_first_name || ' ' || c.cust_last_name  AS "CUSTOMER_NAME",
    p.product_name,
    oi.quantity
FROM
    oe.order_items                   oi
    INNER JOIN oe.orders              o 
       USING(order_id)
    INNER JOIN oe.customers           c 
       USING(customer_id)
    INNER JOIN oe.product_information p 
       USING(product_id)
WHERE order_id = 2454
```

## 解説
内部結合（INNER JOIN）は、SQLで最も頻繁に使われる結合方法です。一言でいうと、**「複数のテーブル間で、指定した条件が両方に存在するデータのみを結びつける」**　という仕組みです。
![](https://storage.googleapis.com/zenn-user-upload/1252567be480-20260320.png)

### 1. 内部結合の仕組み
内部結合は、ベン図でいうところの「重なり合った部分（積集合）」を取り出します。
![](https://storage.googleapis.com/zenn-user-upload/ed0661145cce-20260320.png =180x)


### 2. 条件に合わないデータは「消滅」する
例として、下記２つのテーブルがあるとします。
* **社員テーブル**: 社員全員の情報が網羅されています。
* **資格テーブル**: 資格を保有している社員がどのような資格を持つかの情報が存在します（**資格がない社員のデータはありません**）。

これらを「社員ID」などで内部結合すると、**資格を持っていない社員の情報は、検索結果から消えてしまいます。**

内部結合は、**「結合する両方のテーブルに存在するデータのみ」** を抽出する仕組みです。そのため、片方の「資格テーブル」にデータが存在しない社員は、結合の条件から漏れてしまい、出力対象外となってしまうのです。

##### 解決策：全員を表示したいときは？

「資格の有無にかかわらず、まずは社員全員をリストアップしたい（資格がなければ「資格なし」と出したいなど）」という場合には、**外部結合（OUTER JOIN）** を使用します。


### 3. 例1（例3）、例2どちらで書くべきか？
結論から申し上げますと、現代のシステム開発においては例1の **`INNER JOIN`（ANSI規格）を使うのが圧倒的に推奨**されます。
「テーブルをカンマで列挙する書き方（例2）」は古いスタイル（Oracle独自規格など）であり、現在はメンテナンス性やミスの防止という観点から、避けるべき「レガシーな書き方」とされています。

#### なぜ `INNER JOIN` の方がいいのか？

主な理由は、**「結合の条件」と「データの絞り込み」をはっきりと分離できるから**です。

| 項目 | `INNER JOIN` (推奨) | カンマ区切り (非推奨) |
| :--- | :--- | :--- |
| **可読性** | 結合条件（`ON`）が明確で読みやすい | `WHERE` 句に条件が混ざって複雑になる |
| **安全性** | 結合条件を書き忘れるとエラーになる | 書き忘れると意図しない膨大なデータが出る（直積） |
| **柔軟性** | `LEFT JOIN` などへの書き換えが容易 | 外部結合（`(+)`など）の書き方が特殊で難しい |
| **標準化** | 世界標準（ANSI）なので、どのDBでも動く | 古いOracleや特定の環境に依存しがち |



### 4. USING句の使用（例3）
通常、テーブル同士を結合（JOIN）する際は `ON` 句を使って「どの列とどの列を結びつけるか」を指定します。しかし、**結合する列名が両方のテーブルで全く同じ**場合、`USING` 句を使うことでより簡潔に記述できます。

#### ① 基本的な使い方
例えば、`EMP`（従業員）テーブルと `DEPT`（部門）テーブルを部門番号（`DEPTNO`）で結合する場合を考えてみましょう。

**【ON句による標準的な結合】**
```sql
SELECT e.ENAME, d.DNAME
FROM EMP e
JOIN DEPT d ON e.DEPTNO = d.DEPTNO;
```

**【USING句による結合】**
```sql
SELECT ENAME, DNAME
FROM EMP
JOIN DEPT USING (DEPTNO);
```

#### ⓶　USING句のメリット
* **可読性の向上:** `e.DEPTNO = d.DEPTNO` と繰り返す必要がなく、直感的に「DEPTNOで繋いでいる」ことがわかります。
* **列の重複解消:** `SELECT *` を実行した際、`ON` 句では結合キーが2つ表示されますが、`USING` 句では**1つの列として統合**されて出力されます。

### **ORA-00918: 列の定義が未確定です（column ambiguously defined）**

SQL初心者が必ずといっていいほど遭遇する「ORA-00918」エラーについて説明します。
<br>

### 1. エラーの本質：何が起きているのか？
複数のテーブルを結合（JOIN）して検索する際、それぞれのテーブルに同名の列が存在することがあります（例：`ID`、`NAME`、`CREATED_AT` など）。
SQLの中でその列名を単独で書くと、Oracleは「どのテーブルのデータを取り出せばいいのか」を判断できず、処理をストップさせます。
<br>
### 2. 具体的な発生パターンと修正例

#### ダメな例（NG）

`EMPLOYEES`テーブルと`DEPARTMENTS`テーブルの両方に `DEPT_ID` という列がある場合：

```sql
SELECT DEPT_ID, EMP_NAME, DEPT_NAME
FROM EMPLOYEES E
JOIN DEPARTMENTS D ON E.DEPT_ID = D.DEPT_ID;
```

> **エラーの理由:** `SELECT`句にある `DEPT_ID` が、`EMPLOYEES`のものか `DEPARTMENTS`のものか区別がつきません。

<br>
#### 修正案：テーブル名（または別名）で修飾する

どのテーブルの列かを明示します。通常は、短く書くために**テーブル別名（エイリアス）**を使います。

```sql
SELECT E.DEPT_ID, E.EMP_NAME, D.DEPT_NAME
FROM EMPLOYEES E
JOIN DEPARTMENTS D ON E.DEPT_ID = D.DEPT_ID;
```

<br>

### 3. 注意点と陥りやすいポイント

#### ① WHERE句やORDER BY句での漏れ

`SELECT`句だけ直しても、他の場所で指定が漏れているとエラーになります。

* `WHERE DEPT_ID = 10` → `WHERE E.DEPT_ID = 10`
* `ORDER BY DEPT_ID` → `ORDER BY E.DEPT_ID`

<br>

#### ② USING句を使用する場合の特殊ルール

`JOIN`で `ON` ではなく `USING` を使う場合、**逆にテーブル修飾をしてはいけない**という特殊なルールがあります。

* **NG:** `SELECT E.DEPT_ID FROM EMPLOYEES E JOIN DEPARTMENTS D USING (DEPT_ID);`
* **OK:** `SELECT DEPT_ID FROM EMPLOYEES E JOIN DEPARTMENTS D USING (DEPT_ID);`

> `USING` を使うと、Oracle内部でその列が「結合キー」として1つに統合されるため、修飾すると逆に「未確定（あるいは不適切）」とみなされることがあります。

<br>

#### ③ インラインビュー（サブクエリ）での列名重複

サブクエリの中で同名の列を複数SELECTしていると、外側のクエリでそのサブクエリを呼んだ時に ORA-00918 が発生します。

```sql
-- サブクエリ内で名前が重複している例
SELECT * FROM (
    SELECT E.ID, D.ID FROM EMPLOYEES E, DEPARTMENTS D
) -- ここで「ID」という名前の列が2つできてしまう

```

この場合は、サブクエリ内で `D.ID AS DEPT_ID` のように別名をつけて重複を回避する必要があります。

<br>

### 4. トラブルを防ぐためのベストプラクティス

1. **常にエイリアス（別名）を使う**：
単一テーブルのクエリであっても、将来的にJOINが増える可能性があるため、最初から `T1.COLUMN` のようにテーブル別名をつけて書く癖をつけると安全です。
2. **SELECT * を避ける**：
`SELECT *` を使うと、意図せず同名の列を複数取得してしまい、後の処理（特にビューの作成時など）で混乱を招きます。
3. **共通列の命名規則**：
可能であれば、システム設計時に「どのテーブルのIDか」がすぐわかる名前（`EMP_ID` と `DEPT_ID` など）にするのが理想的です。
:::

# 問題10-2（外部結合１） *Lv2*
### HRスキーマ`COUNTRIES`テーブルより、`COUNTRY_ID`を全て取得してください。あわせて、SHスキーマ `COUNTRIES` テーブルより`COUNTRY_ID` に対応する `COUNTRY_NAME` を取得してください。

```sql:例2：使用するテーブルをカンマで区切る方法
    hr.countries h,
    sh.countries s
WHERE
    h.country_id = s.country_iso_code (+)
```

## 解説
外部結合（OUTER JOIN）を一言でいうと、**「片方のテーブルのデータはすべて残し、もう片方に一致するデータがあればくっつける」** という結合方法です。
内部結合（INNER JOIN）では「両方にデータがないと消えてしまう」のに対し、外部結合は **「データがなくても行を消さない」** のが最大の特徴です。


### 1. ３つの外部結合
外部結合（OUTER JOIN）には、**「どのテーブルを主役（全件表示）にするか」** によって3つの種類があります。

#### ① 左外部結合 (LEFT OUTER JOIN)　　※当問題
実務で最もよく使われる、いわば「王道」の結合です。**左側に書いたテーブルのデータはすべて表示** し、右側のテーブルに一致するデータがあればくっつけます。
**※先に登場したテーブルを「左」、後に登場したテーブルを「右」と表現します。**
![](https://storage.googleapis.com/zenn-user-upload/04905b5770ed-20260320.png)
![](https://storage.googleapis.com/zenn-user-upload/fb2f5f198011-20260321.png =180x)

#### ② 右外部結合 (RIGHT OUTER JOIN)
LEFT JOINの逆で、**右側に書いたテーブルのデータをすべて表示**します。
![](https://storage.googleapis.com/zenn-user-upload/42c0a9755b69-20260321.png)
![](https://storage.googleapis.com/zenn-user-upload/a8e03dd27c4a-20260321.png =180x)

#### ③ 完全外部結合 (FULL OUTER JOIN)
左と右、**両方のテーブルのデータをすべて表示**します。
![](https://storage.googleapis.com/zenn-user-upload/e422a1ec896c-20260321.png)
![](https://storage.googleapis.com/zenn-user-upload/f2a7bf46cf82-20260321.png =180x)

#### 3種類をクイック比較

| 種類 | 主役（全件残る方） | 一致しない箇所の表示 |
| :--- | :--- | :--- |
| **LEFT JOIN** | 左テーブル| 右側の列が **NULL** になる |
| **RIGHT JOIN** | 右テーブル | 左側の列が **NULL** になる |
| **FULL JOIN** | **両方**のテーブル | 相手がいない列はすべて **NULL** |

<br>

### 2. 外部結合の「超」重要な注意点
#### ① WHERE句による「内部結合化」現象
これが最も多い失敗です。`LEFT JOIN` をした後に、**右側のテーブルのカラムに対して `WHERE` 句で条件を指定する**と、せっかく残した行が消えてしまいます。

* **❌ 失敗例**:
  ```sql
  SELECT * FROM employees e
  LEFT JOIN departments d ON e.dept_id = d.dept_id
  WHERE d.location = '東京'; -- ここでNULLの行がすべて捨てられる！
  ```
  `location` が `NULL` の行（部署なし社員）は「東京」ではないため、結果から除外されます。これでは `INNER JOIN` と同じ結果になってしまいます。
* **⭕ 対策**:
  結合相手を絞り込みたい場合は、`WHERE` 句ではなく、**`ON` 句の中に条件を書く**のが正解です。

#### ② NULLの扱いに注意が必要
外部結合の結果には必ず `NULL` が含まれる可能性があります。
* 集計関数（`COUNT` など）を使う際、`COUNT(*)` は全行数えますが、`COUNT(d.dept_id)` は `NULL` を無視します。
* 画面表示などで `NULL` を出したくない場合は、`NVL`（Oracle）や `COALESCE`（標準SQL）を使って「部署なし」などの代替文字に置き換える工夫が必要です。

### 3. 実装上のポイント
#### ① 「OUTER」は任意
`LEFT OUTER JOIN` の`OUTER`は省略可能。`LEFT JOIN`と書く場合が多いです。 

#### ② 「(+)」で外部結合が実現可能（例2）
* `(+)`はOracle独自の結合演算子であります。
* **「データが不足している（空っぽかもしれない）側の列」** に付けます。
「こちら側にはデータがないかもしれないけれど、もう一方のテーブルに合わせて無理やり空行（NULL）を作って結合してね」というサインだと考えると分かりやすいと思います。
* Oracle公式でもANSI標準構文（LEFT/RIGHT JOIN）の使用を推奨しています。非推奨の書き方ですが、昔から頻繁に使用されてきたため、現在でもよく見かける書き方です。
* `FULL JOIN`は直接実現できません。実現するとしたら、`LEFT JOIN`と`RIGHT JOIN`した結果を`UNION`するなとが必要です。

<br>

:::message
### 右外部結合(RIGHT JOIN)って必要？
数学的・論理的には、**`RIGHT JOIN` でできることはすべて `LEFT JOIN` で書けます。** テーブル A と B があるとき、`A RIGHT JOIN B` は `B LEFT JOIN A` と全く同じ結果を返します。そのため、多くのエンジニアは「常に `LEFT JOIN` を使う」というマイルールを持っており、`RIGHT JOIN` を一生書かない人も珍しくありません。私もまだ実践では一度も使ったことはありません。

では、なぜわざわざ `RIGHT JOIN` が存在するのか、あえて使うとしたらどんな時でしょうか？

<br>
### 1. 複雑な「結合の連鎖」を壊したくない時
これが唯一と言ってもいい、実戦的な使い道です。

例えば、すでに 3 つのテーブルを結合しているクエリに、さらにもう 1 つテーブルを足したい場面を想像してください。

```sql
SELECT *
FROM customers c
JOIN orders o ON c.id = o.customer_id
JOIN details d ON o.id = d.order_id
-- ここで「商品マスター(products)」を足したい。
-- ただし、注文されていない商品も含めてすべて表示したい。
```

ここで `LEFT JOIN` にこだわると、`products` を一番左（`FROM` の直後）に持ってくるために、**クエリ全体を書き換える**必要が出てきます。

しかし、`RIGHT JOIN` を使えば、今の流れを維持したまま最後に「右側（新しく足す方）を優先してね」と付け加えるだけで済みます。

```sql
...
RIGHT JOIN products p ON d.product_id = p.id;
```
このように、**「左から右へ流れるロジックの組み立てを崩したくない」** という時に役立ちます。（それでも個人的には `LEFT JOIN` で書き直しちゃうと思います…）
<br>

### 2. 言語としての「対称性（シンメトリー）」

SQL は「自然言語（英語）」に近い形を目指して作られた言語です。
「左があるなら、右があってもいいじゃないか」という、いわばデザイン上の**対称性**のために用意されています。数学のプラスとマイナス、右足と左足のような関係ですね。
<br>

### 3. 分析の視点を変えた時
「注文(A)に紐づく顧客(B)」という視点でクエリを書いていたけれど、途中で「いや、主役は顧客(B)全員だ」と思い直した時、テーブルの順序を入れ替える手間を省いて `RIGHT` に書き換える…という「手抜きのショートカット」として使われることもあります。

<br>
### なぜ `RIGHT JOIN`でなく、`LEFT JOIN` が推奨されるのか？
それでもなお、世の中の解説や現場のコーディング規約で `LEFT JOIN` が推奨されるのには理由があります。

* **人間の認知特性**: 私たちは（特に横書き文化では）左から右へ情報を処理します。「メインのテーブルが左にあり、そこに情報を付け足していく」という `LEFT JOIN` の流れの方が、直感的に理解しやすいのです。
* **読みやすさの統一**: `LEFT` と `RIGHT` が 1 つのクエリに混在すると、どちらのテーブルが主軸なのか一瞬で判断できなくなり、バグ（結合漏れ）の原因になります。
:::


:::message
### とりあえず「外部結合」
SQL初心者に内部結合、外部結合の概念を教えた後、非常に多いのが、内部結合で済むところ、**全て外部結合を使用する**パターンです。
「とりあえず LEFT JOIN にしておけばデータが消えないから安心」というのは、SQL初心者が必ず通る道なのかもしれません。

### 1. なぜ「とりあえず外部結合」にしてしまうのか？（心理的背景）

初心者が外部結合を選んでしまう最大の理由は **「データの消失への恐怖」** です。

* **「消えるのが怖い」:** 内部結合（INNER JOIN）を使うと、結合相手がいない行が結果から消えてしまいます。これを見て「計算が合わなくなるのでは？」と不安になります。
* **「仕様が不明確」:** テーブル間の関係性（親に対して子が必ず存在するかどうか）を把握しきれていないため、「念のため」すべて残そうとします。
* **「左側が主役という直感」:** 「このテーブルのデータを全部見たい」という直感に、LEFT JOIN の挙動がマッチしすぎてしまうのです。
<br>
### 2. 外部結合を多用することによる 3 つのデメリット

「とりあえず」で外部結合を使うと、以下のような実害が発生します。

| デメリット | 内容 |
| :--- | :--- |
| **品質の隠蔽** | 本来「結合相手が必ずいるはず」のデータに不備（ゴミデータ）があっても、NULLとして表示されるため**異常に気づけません**。 |
| **集計ミス** | `COUNT` や `AVG` を使う際、意図しない NULL が混じることで**計算結果が狂う**原因になります。 |
| **パフォーマンス低下** | オプティマイザ（Oracleの実行計画を立てる司令塔）の選択肢を狭めてしまい、大規模データでは**処理が劇的に遅くなる**ことがあります。 

<br>

### 3. 【対策】INNER か OUTER かを見極める「魔法の質問」

資料には、判断基準として以下のフローチャートのようなチェックリストを載せると親切です。

> **Q. 結合相手がいないレコードは「表示されないのが正しい」ですか？**
> * **はい（いないなら無価値）** $\rightarrow$ **INNER JOIN**
>     * 例：注文詳細のない注文、社員のいない部署など。
> * **いいえ（いなくても存在自体を表示したい）** $\rightarrow$ **OUTER JOIN**
>     * 例：売上が0円の商品一覧、まだ配属先が決まっていない新入社員など。

<br>

### 5. まとめ
 **「外部結合は『優しさ』ではなく『例外処理』である」**
 すべてを救おうとするのではなく、データの意味（ビジネスルール）に従って、潔く切り捨てる INNER JOIN を使いこなせるようになりましょう。
 
:::

# 問題10-3（外部結合２） *Lv2*
### HRスキーマ`COUNTRIES`テーブルと、SHスキーマ`COUNTRIES`テーブルを比較し、どちらかにしか存在しない国を抽出してください。
* HRスキーマに存在しない場合、「HRになし」、SHスキーマに存在しない場合、「SHになし」を表示する。
*  HRスキーマの `COUNTRY_ID` と SHスキーマの `COUNTRY_ISO_CODE` を比較対象とする。

## 期待する結果
| COUNTRY | STATUS   | 
| ------- | -------- | 
| BE      | SHになし | 
| ML      | SHになし | 
| ZM      | SHになし | 
| CL      | HRになし | 
| ES      | HRになし | 
| HU      | HRになし | 
| IE      | HRになし | 
| MY      | HRになし | 
| NZ      | HRになし | 
| PL      | HRになし | 
| RO      | HRになし | 
| SA      | HRになし | 
| SE      | HRになし | 
| TH      | HRになし | 
| TR      | HRになし | 
| ZA      | HRになし | 


## 解答例
```sql:例1：UNION ALLを使用
WITH unmatch_data AS (
    SELECT
        h.country_id       AS "HR",
        s.country_iso_code AS "SH"
    FROM
        hr.countries h
        FULL OUTER JOIN sh.countries s 
          ON h.country_id = s.country_iso_code
    ORDER BY
        h.country_id,
        s.country_iso_code
)
SELECT
    hr        AS "COUNTRY",
    'SHになし' AS "STATUS"
FROM
    unmatch_data
WHERE
    sh IS NULL
UNION ALL
SELECT
    sh        AS "COUNTRY",
    'HRになし' AS "STATUS"
FROM
    unmatch_data
WHERE
    hr IS NULL
```
```sql:例2：CASE文を使用
WITH unmatch_data AS (
    SELECT
        h.country_id       AS "HR",
        s.country_iso_code AS "SH"
    FROM
        hr.countries h
        FULL OUTER JOIN sh.countries s 
          ON h.country_id = s.country_iso_code
    ORDER BY
        h.country_id,
        s.country_iso_code
)
SELECT
    CASE
        WHEN hr IS NULL THEN sh 
        ELSE hr END AS "COUNTRY",
    CASE
        WHEN hr IS NULL THEN 'HRになし' 
        ELSE 'SHになし' END AS "STATUS"
FROM
    unmatch_data ud
WHERE
     hr IS NULL
  OR sh IS NULL
```

## 解説
### 1. FULL JOINとは？
FULL JOINは、**「左側のテーブル」と「右側のテーブル」の両方のデータをすべて取得する**結合方法です。

* **一致するデータがある場合：** 通常の結合（INNER JOIN）と同様、1つの行に統合されます。
* **一致するデータがない場合：** 相手側のカラムは **NULL** で埋められ、データ自体は抽出結果に残ります。

### 2. 基本的な構文
```sql
SELECT 
    A.ID, A.NAME, B.SCORE
FROM 
    TABLE_A A
FULL OUTER JOIN 
    TABLE_B B ON A.ID = B.ID;
```
※ `OUTER` は省略可能ですが、明示的に書くことで「一致しない行も含む」意図が伝わりやすくなります。

### 3. 結合のコスト
FULL JOINは内部的に「LEFT JOIN ＋ RIGHT JOIN ＋ 重複排除（UNION）」に近い動きをするため、データ量が多いと非常に重い処理になります。インデックスの設計が重要です。

# 問題10-4（自己結合1） *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、従業員（EMPLOYEE）の氏名と従業員の上司（MANAGER）の氏名を取得してください。（順不同）

## 期待する結果
| EMPLOYEE_ID | EMPLOYEE_NAME   | MANAGER_ID | MANAGER_NAME    | 
| ----------- | --------------- | ---------- | --------------- | 
| 100         | Steven King     |            |                 | 
| 101         | Neena Yang      | 100        | Steven King     | 
| 102         | Lex Garcia      | 100        | Steven King     | 
| 103         | Alexander James | 102        | Lex Garcia      | 
| 104         | Bruce Miller    | 103        | Alexander James | 
| 105         | David Williams  | 103        | Alexander James | 
| 106         | Valli Jackson   | 103        | Alexander James | 
| 107         | Diana Nguyen    | 103        | Alexander James | 
| 108         | Nancy Gruenberg | 101        | Neena Yang      | 
| 109         | Daniel Faviet   | 108        | Nancy Gruenberg | 
| 110         | John Chen       | 108        | Nancy Gruenberg | 


## 解答例
```sql
SELECT
    emp.employee_id,
    emp.first_name || ' ' || emp.last_name AS employee_name,
    emp.manager_id,
    mng.first_name || ' ' || mng.last_name AS manager_name
FROM
    hr.employees emp
    LEFT JOIN hr.employees mng 
      ON emp.manager_id = mng.employee_id
WHERE
    emp.employee_id <= 110
ORDER BY
    emp.employee_id
```

## 解説
SQLにおける**自己結合（Self-Join）** は、その名の通り「同じテーブル同士を結合する」手法です。「なぜ同じテーブルをくっつける必要があるの？」と最初は不思議に思うかもしれませんが、組織の上下関係などを扱う際には欠かせないテクニックです。

### 1. 自己結合のイメージ
同じテーブルを2つの別々のテーブルがあるかのように見立てて結合します。SQL文の中では、同じテーブル名が2回登場するため、**必ず「別名（エイリアス）」をつけて区別** する必要があります。

### 2. 具体例：上司の名前を取得する
「社員テーブル（Employees）」を簡潔に表すと、以下のようになります。

| 社員ID (ID) | 名前 (Name) | 上司ID (Manager_ID) |
| --- | --- | --- |
| 1 | 佐藤（社長） | NULL |
| 2 | 鈴木 | 1 |
| 3 | 高橋 | 1 |

「社員名と、その上司の名前を並べて表示したい」場合、自分のテーブルの中に上司のデータも入っているため、自己結合を使います。

```sql
SELECT
    E1.Name AS 社員名,
    E2.Name AS 上司名
FROM
    Employees E1  -- 1つ目を「部下側」とみなす
LEFT JOIN
    Employees E2  -- 2つ目を「上司側」とみなす
ON
    E1.Manager_ID = E2.ID; -- 部下の上司IDが、上司側の社員IDと一致するもの

```
### 3. 自己結合の主な使い道

* **階層構造の管理**: 上記のような「上司と部下」「親カテゴリーと子カテゴリー」などの関係。
* **重複データのチェック**: 同じテーブル内で、同じメールアドレスを持つ別々のIDを探す場合など。
* **ランキングや前後比較**: 同じ商品の「今日の価格」と「昨日の価格」を比較する場合。

### 4. 注意点と落とし穴
##### ① 別名（Alias）は必須
同じテーブル名で結合するため、`FROM Employees, Employees` と書くとSQLはどちらの列を指しているのか判断できずエラーになります。必ず `A` や `B` 、あるいは `Staff` と `Manager` のように別名をつけてください。

#### ② 結合の種類の選択（INNER vs LEFT）
先ほどの例で、`INNER JOIN` を使うと、**上司がいない「社長（佐藤さん）」が結果から消えてしまいます。**
最上位のデータ（NULLを含む行）も表示したい場合は、必ず **`LEFT OUTER JOIN`** を使うのが鉄則です。

#### ③ データの増殖（デカルト積）の危険
結合条件（`ON`句）を間違えたり、条件が緩すぎたりすると、行数が「テーブルの行数 × テーブルの行数」に膨れ上がり、サーバーに大きな負荷をかけることがあります（1,000行のテーブルなら100万行に！）。実行前に条件をしっかり確認しましょう。

#### ④ 同じペアの重複
例えば「同じ趣味を持つ人同士をペアにする」自己結合をすると、「佐藤さんと鈴木さん」「鈴木さんと佐藤さん」という重複した結果や、「佐藤さんと佐藤さん（自分自身）」という結果が出てしまいます。

> **対策:** `ON A.ID < B.ID` のように、IDの大小比較を条件に加えることで、重複や自分自身との結合を防げます。

# 【完全版】問題10-5（自己結合2） *Lv3*
### OEスキーマ`ORDERS`テーブルより同一年月（`ORDER_DATE`の年月）に複数の注文をした人と注文を取得してください。（順不同）

*　**`CUST_NAME`は`CUSTOMERS`テーブルより取得してください**

## 期待する結果
| CUSTOMER_ID | CUST_NAME          | ORDER_ID | ORDER_DATE                  | 
| ----------- | ------------------ | -------- | --------------------------- | 
| 105         | Matthias MacGraw   | 2358     | 2008-01-08T17:03:12.654278Z | 
| 105         | Matthias MacGraw   | 2356     | 2008-01-26T09:22:41.934562Z | 
| 148         | Gustav Steenburgen | 2451     | 2007-12-17T17:03:52.562632Z | 
| 148         | Gustav Steenburgen | 2386     | 2007-12-06T12:22:34.225609Z | 


## 解答例
```sql:例2：同一年月で注文が一致しない→複数
SELECT
    o1.customer_id,
    c.cust_first_name || ' ' || c.cust_last_name AS cust_name ,
    o1.order_id,
    o1.order_date
FROM
         oe.orders o1
    INNER JOIN oe.orders o2 
       ON o1.customer_id = o2.customer_id
      AND o1.order_id <> o2.order_id
      AND extract(YEAR FROM o1.order_date) = extract(YEAR FROM o2.order_date)
      AND extract(MONTH FROM o1.order_date) = extract(MONTH FROM o2.order_date)
    INNER JOIN oe.customers c 
       ON o1.customer_id = c.customer_id
```
```sql:例2：COUNTして2以上→複数
WITH o1 AS (
    SELECT
        customer_id,
        extract(YEAR FROM order_date)  AS yr,
        extract(MONTH FROM order_date) AS mt,
        COUNT(*)
    FROM
        oe.orders
    GROUP BY
        customer_id,
        extract(YEAR FROM order_date),
        extract(MONTH FROM order_date)
    HAVING
        COUNT(*) > 1
)
SELECT
    o2.customer_id,
    c.cust_first_name || ' ' || c.cust_last_name AS cust_name,
    o2.order_id,
    o2.order_date
FROM
         o1
    INNER JOIN oe.orders    o2 
       ON o1.customer_id = o2.customer_id
      AND o1.yr = extract(YEAR FROM o2.order_date)
      AND o1.mt = extract(MONTH FROM o2.order_date)
    INNER JOIN oe.customers c 
       ON o2.customer_id = c.customer_id
```
```sql:例3：分析関数を使用（最もスマート）
SELECT
    customer_id,
    cust_name,
    order_id,
    order_date
FROM
    (
        SELECT
            o.customer_id,
            c.cust_first_name || ' ' || c.cust_last_name  AS cust_name,
            o.order_id,
            o.order_date,
            COUNT(*) OVER(PARTITION BY o.customer_id,
                  TRUNC(o.order_date, 'mm')) AS monthly_order_count
        FROM
                 oe.orders o
            INNER JOIN oe.customers c 
               ON o.customer_id = c.customer_id
    )
WHERE
    monthly_order_count > 1; -- 2回以上注文した人だけ抽出
```


## 解説
例1は **「同じ顧客が、同じ年月に、別の注文IDで活動しているか」** を自己結合で紐解いた解答例です。「複数の注文」を `order_id <> o2.order_id`（自分以外の注文が存在するか）で表現した点は、SQLの集合論的な考え方が表れた解き方です。教育の参考として取り上げました。実践的な観点では、例3をお勧めします。

### 1. クエリの解説：なぜこれで「複数注文」が特定できるのか
このクエリは、1つの注文（o1）に対して、「同じ月の、自分じゃない注文（o2）」を相棒として探してくる構造になっています。

* **`o1.customer_id = o2.customer_id`**: まず「同じ人」に絞り込む。
* **`o1.order_id <> o2.order_id`**: ここがポイントです。もしその月に1回しか注文していなければ、相棒（o2）が見つからないため、INNER JOINの結果から消えます。2回以上注文している人だけが生き残ります。
* **`EXTRACT(YEAR/MONTH...)`**: `order_date`の年月単位に一致させています。


### 2. 実務での注意点
#### ① パフォーマンス（関数の使用）
`ON` 句の中で `EXTRACT(YEAR FROM o1.order_date)` のようにカラムに関数を通すと、**`order_date` に貼られた通常のインデックスが使えなくなります。**
* **影響:** 注文データが数百万件ある場合、このクエリは非常に低速になります。

### ② 比較のコスト
`<>`（不一致）による自己結合は、内部的に「全件 vs 全件」に近い比較を強いることがあり、計算コストが高くなりがちです。

# 問題10-6（クロス結合） *Lv3*
### HRスキーマ`COUNTRIES`テーブルの`REGION_ID` = 20のデータについて、2020年および2021年のデータを記録できるような新たなテーブルを作成したい。その元となるデータを作成してください。（順不同）

## 期待する結果
| YEAR | NAME                     | DATA | 
| ---- | ------------------------ | ---- | 
| 2020 | Argentina                |      | 
| 2020 | Brazil                   |      | 
| 2020 | Canada                   |      | 
| 2020 | Mexico                   |      | 
| 2020 | United States of America |      | 
| 2021 | Argentina                |      | 
| 2021 | Brazil                   |      | 
| 2021 | Canada                   |      | 
| 2021 | Mexico                   |      | 
| 2021 | United States of America |      | 

## 解答例
```sql
WITH t_year AS (
    SELECT 2020 AS "YEAR" FROM dual
    UNION ALL
    SELECT 2021 AS "year" FROM dual
)
SELECT
    t."YEAR",
    c.country_name AS "NAME",
    null           AS "DATA"
FROM
    hr.countries c
    CROSS JOIN t_year t
WHERE
    region_id = 20
```

## 解説
クロス結合（CROSS JOIN）は、一言で言うと **「総当たり戦」** です。
2つのテーブルのすべての行を組み合わせて、考えられるすべてのパターンを作り出す結合方法です。
![](https://storage.googleapis.com/zenn-user-upload/7ae6f5777a1e-20260324.png =300x)



### 1. クロス結合（デカルト積）の仕組み
片方のテーブルの行数が $N$、もう片方のテーブルの行数が $M$ の場合、結果は $N \times M$ 行になります。
![](https://storage.googleapis.com/zenn-user-upload/aee281e869d7-20260324.png =200x)

#### 具体例：商品 × サイズ
「Tシャツ」と「パーカー」という商品があり、それぞれに「S」「M」「L」のサイズ展開がある場合を考えてみましょう。

| 商品 (Products) |
| :--- |
| Tシャツ |
| パーカー |

| サイズ (Sizes) |
| :--- |
| S |
| M |
| L |

**SQL文：**
```sql
SELECT p.name, s.size_name
FROM products p
CROSS JOIN sizes s;
```

**結果（2 × 3 = 6行）：**
| name | size_name |
| :--- | :--- |
| Tシャツ | S |
| Tシャツ | M |
| Tシャツ | L |
| パーカー | S |
| パーカー | M |
| パーカー | L |

### 2. いつ使うのか？（主なユースケース）

「結合条件がないなんて、いつ使うの？」と思うかもしれませんが、実務では以下のような場面で重宝します。

* **全パターンのマスタ作成:** 上記のように、商品と属性（サイズ、色など）の全組み合わせを作るとき。
* **テストデータの生成:** 少ない行数から大量のダミーデータを作りたいとき。
* **カレンダー表示:** 「日付マスタ」と「全店舗マスタ」をクロス結合して、売上がゼロの日も含めた全店舗のレポート枠を作るとき。

### 3. 注意事項：ここが落とし穴！

クロス結合は強力ですが、一歩間違えるとシステムをダウンさせる破壊力を持っています。

#### ① 「データ爆発」によるパフォーマンス劣化
行数が掛け算で増えるため、大きなテーブル同士を結合すると一瞬でメモリやディスクを圧迫します。
* 1,000行 $\times$ 1,000行 = **1,000,000行**（まだ大丈夫）
* 100,000行 $\times$ 100,000行 = **10,000,000,000行（100億行！）** $\rightarrow$ **フリーズ確定**

#### ② 意図しないクロス結合（旧形式の罠）
昔の書き方（カンマ区切り）を使っていると、結合条件（WHERE句）を書き忘れただけで勝手にクロス結合になってしまいます。

```sql
-- 危険な書き方（WHEREを忘れると意図せずクロス結合になる）
SELECT * FROM employees, departments; 

-- 安全な書き方（意図的な場合は明示的に書く）
SELECT * FROM employees CROSS JOIN departments;
```
> **初心者のためのアドバイス：**
> 23aiを含む現代のSQLでは、必ず `CROSS JOIN` と明示的に書きましょう。そうすることで、後からコードを読む人が「あ、これはわざと総当たりにしているんだな」と理解できます。

#### ③ 23aiでの「結合制限」
Oracle 23aiでは、非常に巨大なデカルト積が発生しそうなクエリに対して、最適化（オプティマイザ）がブレーキをかけようとしますが、基本的には開発者が責任を持って行数を制御する必要があります。

## 解答例
```sql
SELECT
    e1.employee_id,
    e1.first_name,
    e1.last_name,
    e2.employee_id AS dup_id,
    e2.first_name  AS dup_f_name,
    e2.last_name   AS dup_l_name
FROM
         hr.employees e1
    INNER JOIN hr.employees e2 
            ON e1.first_name = e2.first_name -- 名前が同じ
           AND e1.employee_id < e2.employee_id; -- 重複表示を防ぐ工夫
```

## 解説
**「同じ名前を持つ別の人」を特定する、自己結合（Self Join）の書き方**です。

### 1. クエリの解説
このクエリのポイントは、1つのテーブルを「2つの役割」に分けて扱っている点にあります。

#### ステップ1：テーブルのエイリアス（e1, e2）
同じ `hr.employees` テーブルに `e1` と `e2` という別名を付けることで、あたかも「2つの異なる名簿」があるかのように比較させています。

#### ステップ2：名前の一致（e1.first_name = e2.first_name）
これが結合のメイン条件です。これにより、名前が同じペアがすべて抽出されます。

#### ステップ3：IDによる比較（e1.employee_id < e2.employee_id）
ここが最もテクニカルな部分です。この不等号には2つの重要な役割があります。
1.  **自分自身を除外する**: `e1.id = e2.id` のケースを弾きます。
2.  **重複（鏡写し）を防ぐ**: 
    もし単に `e1.id != e2.id` と書くと、「AさんとBさん」「BさんとAさん」という、左右を入れ替えただけの同じペアが2行出てきてしまいます。`<` を使うことで、必ず **「IDが小さい方を左、大きい方を右」** という1パターンだけに絞り込めます。


### 2. 注意点

#### ① インデックスの効果
このクエリを数万件、数十万件のデータに対して実行する場合、`first_name` にインデックスが貼られていないと、**「入れ子ループ結合（Nested Loop Join）」** が発生し、処理が非常に重くなります。
* **対策:** `CREATE INDEX idx_emp_fname ON employees(first_name);` の有無を確認しましょう。

#### ② 大文字・小文字の区別
Oracleのデフォルト設定では `'John'` と `'john'` は別人として扱われます。
* **対策:** `UPPER(e1.first_name) = UPPER(e2.first_name)` とすることで、入力ミスによる重複も拾えるようになります。

#### ③ 3人以上の重複がある場合
もし「David」という名前の人が3人（ID: 10, 20, 30）いた場合、結果はどうなるでしょうか？
* (10, 20), (10, 30), (20, 30) の **3行** が出力されます。

「誰と誰が重複しているか」を全パターン出すならこれで正解ですが、単に「重複している人のリスト」が欲しいだけなら、`GROUP BY` を使った方がスッキリする場合もあります。

# 【完全版】問題10-8（外部結合と自己結合） *Lv3*
### HRスキーマ`EMPLOYEES`テーブルの`FIRST_NAME`が同じデータを下記ルールに従い、取得してください。（順不同）
* **重複数を取得する。自分以外で１人一致する場合は１、２人一致する場合は２とする（列名：DUPLICATION）。**
* **状態を取得する。重複ありの場合、「重複あり」、重複なしの場合、「ユニーク」とする（列名：STATUS）。**

## 解答例
```sql:例1：自己結合で取得
SELECT
    e1.employee_id,
    e1.first_name,
    e1.last_name,
    COUNT(e2.employee_id) AS "DUPLICATION",
    CASE
        WHEN COUNT(e2.employee_id) > 0 THEN
            '重複あり'
        ELSE
            'ユニーク'
    END                   AS "STATUS"
FROM
    hr.employees e1
    LEFT OUTER JOIN hr.employees e2 
                 ON e1.first_name  =  e2.first_name
                AND e1.employee_id <> e2.employee_id -- 自分自身を除外
WHERE
    e1.employee_id BETWEEN 150 AND 165
GROUP BY
    e1.employee_id,
    e1.first_name,
    e1.last_name
ORDER BY
    "DUPLICATION" DESC;
```
```sql:例2：分析関数を使用（最もスマート）
WITH dup AS (
    SELECT
        employee_id,
        first_name,
        last_name,
        COUNT(*)
        OVER(PARTITION BY first_name) - 1 AS "DUPLICATION",
        CASE
            WHEN count(*)
                 OVER(PARTITION BY first_name) > 1 THEN
                '重複あり'
            ELSE
                'ユニーク'
        END                               AS "STATUS"
    FROM
        hr.employees
)
SELECT
    *
FROM
    dup
WHERE
    employee_id BETWEEN 150 AND 165
ORDER BY
    "DUPLICATION" DESC
```

## 解説
例1のクエリーは、**「自分と同じ名前を持つ“他人”が何人いるか」** を一人ひとりの視点からカウントし、その結果をもとにステータスを判定するというロジックです。
「外部結合（LEFT OUTER JOIN）」の性質を利用して、「重複がない人（0人）」も結果に残している点がポイントです。
結合の教育の参考として取り上げましたが、実践的な観点では、例2をお勧めします。

### 1. ロジックのポイント
このクエリの最大の特徴は、**「1対多（または0）」の関係をあえて作り出している**点です。

#### ① なぜ LEFT OUTER JOIN なのか？
もしこれを `INNER JOIN` にしてしまうと、同じ名前の人がいない（ユニークな）人は結合相手が見つからず、**結果から消滅してしまいます。** `LEFT OUTER JOIN` を使うことで、相手がいなくても「自分」の行は維持され、`COUNT(E2.EMPLOYEE_ID)` が `0` として計算されるようになります。

#### ② `<>` による自己除外
`E1.EMPLOYEE_ID <> E2.EMPLOYEE_ID` がないと、全ての人が自分自身とマッチしてしまい、全員が「重複あり（1人以上）」になってしまいます。この一文が「他人を探す」という定義を完成させています。

#### ③ 集計の単位
`GROUP BY E1.EMPLOYEE_ID, E1.FIRST_NAME, E1.LAST_NAME` とすることで、社員一人ひとりに対して「相棒（E2）」が何人ついたかを集計しています。



### 2. 注意点：パフォーマンス（計算量）
このクエリは、名前が重複している人が多ければ多いほど、結合時に行数が一時的に膨れ上がります。
* 1,000人の「John」がいた場合、Johnの行だけで **1,000 $\times$ 999 = 999,000行** の中間データが発生します。
* 全社員が数万人規模になると、インデックス（`FIRST_NAME`）がない場合、非常に時間がかかるクエリになります。

# 【完全版】問題10-9（部分外部結合） *Lv3*
### COスキーマ`ORDERS`テーブルおよび`ORDER_ITEMS`テーブルより、注文年月（`ORDER_TMS`）、商品ごとの、注文数を取得してください。ただし、注文の存在しない月には注文数0としてください。取得対象は商品IDが8または37としてください。（年月の昇順）

## 解答例
```sql
WITH full_year_month AS ( -- ORDERSテーブルに存在するすべての年月
    SELECT DISTINCT
        TO_NUMBER(TO_CHAR(
            TRUNC(o.order_tms, 'MM'),
            'YYYYMM'
        )) ot
    FROM
        co.orders o
), order_pr AS ( -- 年月、商品ごとの注文数を取得
    SELECT
        TO_NUMBER(TO_CHAR(
            TRUNC(o.order_tms, 'MM'),
            'YYYYMM'
        ))               ot,
        oi.product_id,
        SUM(oi.quantity) sm
    FROM
             co.orders o
        INNER JOIN co.order_items oi 
           ON o.order_id = oi.order_id
    GROUP BY
        trunc(o.order_tms, 'mm'),
        oi.product_id
)
SELECT
    f.ot,
    o.product_id,
    COALESCE(o.sm, 0) AS sm
FROM
    full_year_month f
    LEFT JOIN order_pr        o 
    PARTITION BY ( o.product_id )  -- 注文が存在しない場合、0件と表示するためのレコード作成
      ON f.ot = o.ot
WHERE o.product_id IN (8, 37)
ORDER BY
    o.product_id,
    f.ot
```

## 解説
このクエリは、`Partitioned Outer Join` を使用して、**「売上が一度もなかった月」を無視せず、0としてレポートに表示させる**ためのスマートな書き方です。

### 1. このクエリが解決している問題：データの「歯抜け」
通常、SQLで `GROUP BY` をすると「データが存在する月」しか結果に出てきません。
* **商品ID 8**: 1月、2月、4月に売上がある（3月はデータなし）
* **商品ID 37**: 2月、3月に売上がある（1月、4月はデータなし）

これをそのまま出力すると、商品ごとに月がバラバラになり、比較やグラフ作成が困難になります。


### 2. クエリの構造とPartitioned Outer Joinの役割

#### ① 下準備（WITH句）
* **`full_year_month`**: カレンダーの役割です。システム内に存在する「全ての年月」のマスターリストを作っています。
* **`order_pr`**: 実際の商品別・月別の注文実績です。ここには「注文がなかった月」の行は存在しません。

#### ② 魔法の結合（Partitioned Outer Join部分）
```sql
FROM
    full_year_month f
    LEFT JOIN order_pr o 
    PARTITION BY ( o.product_id ) -- ★ここが肝
      ON f.ot = o.ot
```
ここでは以下の処理が内部で行われています。
1.  実績データ（`order_pr`）を **`product_id` ごとに透明な壁で区切り**、小さなグループを作ります。
2.  それぞれのグループに対して、カレンダー（`full_year_month`）を**強制的に全行ぶつけます。**
3.  結果、商品8に対しても、商品37に対しても、**「その商品専用の全12ヶ月分の枠」** が生成されます。

![](https://storage.googleapis.com/zenn-user-upload/34e1ccde38d1-20260327.png)


### 3. この書き方のメリット

#### 1. データの「密度」を均一にできる（データ・デンシフィケーション）
すべての商品に対して、共通の「年月」軸でデータが並びます。これにより、Excelに貼り付けてそのまま折れ線グラフにしたり、前月比を計算したりするのが非常に簡単になります。

#### 2. `COALESCE` や `NVL` が活きる
行自体が存在しない場合、`NVL` を使っても 0 にはできません。Partitioned Outer Joinによって「値が空（NULL）の行」が作られるため、初めて `COALESCE(o.sm, 0)` が機能し、綺麗なレポートになります。

#### 3. クエリがシンプルになる
この機能を使わない場合、あらかじめ「全商品 × 全年月」の巨大な掛け算テーブル（直積）を自前で作ってから実績をぶつける必要がありますが、`PARTITION BY` を使うとその手間をDB側が自動で引き受けてくれます。
