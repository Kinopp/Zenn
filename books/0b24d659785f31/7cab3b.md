# 前提知識
MODEL句は、Oracle Database（10g以降）に搭載されている、非常に強力かつユニークな機能です。
一言でいうと、**「SQLの中に、Excel（スプレッドシート）のような計算機能を持ち込む」** ための構文です。
通常のSQLは「行」単位で処理を進めますが、MODEL句を使うと、データを **「多次元配列（セルの集まり）」** として扱い、任意のセルに対して複雑な数式（ルール）を適用できるようになります。
![](https://storage.googleapis.com/zenn-user-upload/4d964f5d8c83-20260323.png)



## 1. 基本コンセプト：データを「Excelのシート」として捉える

### イメージ
通常のクエリ結果は単なるテーブルですが、MODEL句を適用すると、それが**Excelのワークシート**になります。

* **行のラベル**: Excelの行番号（1, 2, 3...）にあたるものが、MODEL句の「ディメンション（Dimension）」です。
* **列のラベル**: Excelの列番号（A, B, C...）にあたるものが、MODEL句の「メジャー（Measure）」です。
* **数式**: `C1 = A1 + B1` のようなExcelの数式を、SQL内に「ルール（Rule）」として記述します。

これにより、通常のWindow関数（OVER句）などでは記述が極めて困難な、**「行をまたいだ相互依存関係のある計算」** が、スッキリと（Excelの数式のように）書けるようになります。


## 2. 必須の3つの要素（MODEL句の構成）

MODEL句を構成する際、データをどう「Excelシート」にするかを定義する3つの主要なキーワードがあります。

| キーワード | 役割 | Excelでのイメージ |
| :--- | :--- | :--- |
| **DIMENSION BY** | 配列の「添字（インデックス）」となる列を指定 | 行ラベル（例: `年度, 地区`） |
| **MEASURES** | 計算対象となる「値（セルの中身）」となる列を指定 | 列ラベル（例: `売上, 利益`） |
| **RULES** | セルに対する計算式（ルール）を記述 | セルに入る数式（例: `売上['2025', '東京'] = ...`） |

# 問題15-1（MODEL句初歩の初歩）　 *Lv3*
### SHスキーマ`SALES`テーブルの売上額合計（`AMOUNT_SOLD`）を年月ごとに集計し、2022年10月から2022年12月の値を元に、下記ルールに従い、2023年1月から2023年3月の売上額を予測してください。（`TIME_ID`の昇順）
* **2023年1月：2022年10月の売上額 × 1.1（小数点第３位切り捨て）**
* **2023年2月：2022年11月の売上額 × 1.1（小数点第３位切り捨て）**
* **2023年3月：2022年11月の売上額 × 1.1（小数点第３位切り捨て）**


## 期待する結果
| TM     | SM         | 
| ------ | ---------- | 
| 202210 | 2449167.91 | 
| 202211 | 2474687.5  | 
| 202212 | 2547042.11 | 
| 202301 | 2694084.7  | 
| 202302 | 2722156.25 | 
| 202303 | 2801746.32 | 

## 解答例
```sql
SELECT
    tm,
    sm
FROM
    (
        SELECT
            TO_NUMBER(to_char(time_id, 'yyyymm')) AS tm,
            SUM(amount_sold)                      AS sm
        FROM
            sh.sales
        WHERE
            time_id BETWEEN DATE '2022-10-01' AND DATE '2022-12-31'
        GROUP BY
            tm
    )
MODEL
    DIMENSION BY ( tm )
    MEASURES ( sm )
    RULES (
        sm[202301]= TRUNC(sm[202210]* 1.1, 2),
        sm[202302]= TRUNC(sm[202211]* 1.1, 2),
        sm[202303]= TRUNC(sm[202212]* 1.1, 2)
    )
ORDER BY
    tm
```

## 解説
Oracle Database独自の強力な機能である **`MODEL` 句** を使用して、既存の集計データから「未来の予測データ」を動的に生成する例です。

### 1. `MODEL` 句の使い方（構造の理解）
このクエリにおける `MODEL` 句は、以下の3つの要素で「SQLの結果セットを一時的なスプレッドシート（Excel）」のように定義しています。

1.  **`DIMENSION BY ( tm )`【行の座標】**:
    * Excelの「行番号」に相当します。ここでは「年月（`tm`）」を座標として指定しています。これにより、`sm[202210]` のように特定の月のデータをピンポイントで指し示すことが可能になります。
2.  **`MEASURES ( sm )`【セルの値】**:
    * Excelの「セルの中身」に相当します。計算や書き換えの対象となる数値（売上合計 `sm`）を指定します。
3.  **`RULES`【計算式】**:
    * Excelの「数式バー」に相当します。
    * **`sm[202301] = ...`**: 元のデータ（`FROM` 句の結果）には存在しない「202301」という座標（行）を、このルールによって **その場で作り出しています**。
    * 右辺では `sm[202210]`（実績値）を参照し、1.1倍（10%増）して代入しています。


### 2. `MODEL` 句を使うメリット
通常の `SELECT` 文や `UNION ALL` を使った手法と比較して、以下のようなメリットがあります。

##### ① 存在しない行を「捏造」できる
通常のSQLでは、テーブルにないデータ（未来の月など）を表示するには `UNION ALL` でダミー行を足すなどの手間が必要です。`MODEL` 句なら、**ルールを書くだけで新しい行が自動的に生成（UPSERT）される** ため、クエリが非常にシンプルになります。

#### ② セル単位の「自由な参照」が可能
通常のSQLは「行」単位で処理しますが、`MODEL` 句は **「セル」単位** で処理します。
* 「3ヶ月前の値を参照する」
* 「特定の月の値だけを2倍にする」
といった、**特定のピンポイントな座標へのアクセス** が Excel 感覚で記述できます。

#### ③ 複雑なビジネスロジックをSQLに完結させられる
本来、このような「実績ベースの予測シミュレーション」は、一度データをアプリケーション（JavaやPython）側へ取り出したり、Excelに貼り付けたりして計算することが多いです。
`MODEL` 句を使えば、**データベース内で高速に計算を完結させ、結果だけをアプリに返す** ことができるため、システム全体のパフォーマンスと開発効率が向上します。

### 3. 注意点
* 右辺で参照しているデータ（例：`202210`）が `FROM` 句の結果に含まれていない場合、計算結果は `NULL` になり、新しい行（`202301`）も生成されません。確実にデータがあることを確認するか、`NVL` 関数を活用しましょう。

# 【完全版】問題15-2（未来の繰り返し予測） *Lv4*
### 【15-1の応用】SHスキーマ`SALES`テーブルの売上額合計（`AMOUNT_SOLD`）を年月ごとに集計し、2022年10月から2022年12月の値を元に、下記ルールに従い、2023年1月から2023年12月の売上額を予測してください。（`TIME_ID`の昇順）
* **2023年`X`月：2023年`X`月 の 3ヶ月前 の売上額 × 1.1（小数点第３位切り捨て）**


## 期待する結果
| TM     | SM         | 
| ------ | ---------- | 
| 202210 | 2449167.91 | 
| 202211 | 2474687.5  | 
| 202212 | 2547042.11 | 
| 202301 | 2694084.7  | 
| 202302 | 2722156.25 | 
| 202303 | 2801746.32 | 
| 202304 | 2963493.17 | 
| 202305 | 2994371.87 | 
| 202306 | 3081920.95 | 
| 202307 | 3259842.48 | 
| 202308 | 3293809.05 | 
| 202309 | 3390113.04 | 
| 202310 | 3585826.72 | 
| 202311 | 3623189.95 | 
| 202312 | 3729124.34 | 

## 解答例
```sql:例1：MODEL句を使用
SELECT
    TO_NUMBER(to_char(
        ADD_MONTHS(DATE '2022-10-01', seq - 1),
        'yyyymm'
    )) AS tm,
    sm
FROM
    (
        SELECT
            ROW_NUMBER()
            OVER(
                ORDER BY
                    TO_NUMBER(to_char(time_id, 'yyyymm'))
            )                                     AS seq,
            TO_NUMBER(TO_CHAR(time_id, 'yyyymm')) AS tm,
            SUM(amount_sold)                      AS sm
        FROM
            sh.sales
        WHERE
            time_id BETWEEN DATE '2022-10-01' AND DATE '2022-12-31'
        GROUP BY
            tm
    )
MODEL 
    DIMENSION BY ( seq )
    MEASURES ( sm,
               tm )
    RULES ITERATE(12) (
        sm[iteration_number + 4]= TRUNC(sm[iteration_number + 1]* 1.1, 2)
    )
ORDER BY
    seq
```
```sql:例2：再帰的WITH句を使用
WITH 
-- 1. 実績データを準備
 base_data AS (
    SELECT
        ROW_NUMBER()
        OVER(
            ORDER BY
                TO_NUMBER(to_char(time_id, 'yyyymm'))
        )                                     AS seq,
        TO_NUMBER(to_char(time_id, 'yyyymm')) AS tm,
        SUM(amount_sold)                      AS sm
    FROM
        sh.sales
    WHERE
        time_id BETWEEN DATE '2022-10-01' AND DATE '2022-12-31'
    GROUP BY
        TO_NUMBER(to_char(time_id, 'yyyymm'))
),
-- 2. 再帰計算（ここがループ本体）
 prediction (
    seq,
    tm,
    sm
) AS (
    -- 実績データ：最初の手順（実績3ヶ月分を入れる）
    SELECT
        seq,
        tm,
        sm
    FROM
        base_data
    UNION ALL
    -- 予測データ（再帰部）：一期前のデータを使って次を作る
    -- seq=1(202210) から seq=4(202301) を作る
    SELECT
        p.seq + 3,
        TO_NUMBER(to_char(
            add_months(TO_DATE(p.tm
                               || '01', 'yyyymmdd'), 3),
            'yyyymm'
        )),
        TRUNC(p.sm * 1.1, 2)
    FROM
        prediction p
    WHERE
        p.seq <= 12 -- 15回目（12ヶ月分）まで繰り返す
)
SELECT
    tm,
    sm
FROM
    prediction
ORDER BY
    seq;
```

## 解説
例1のクエリの特徴は、**「過去3ヶ月の実績をもとに、未来の売上を1年分（12回）ループ計算で予測する」** ことにあります。

### 1. `ROW_NUMBER()` を使用した工夫

このクエリで `ROW_NUMBER()` を使っている理由は、**`MODEL` 句の「座標（インデックス）」を扱いやすい連番に変換するため** です。
サブクエリーの結果は以下の通りであり、これを元にループ計算を行います。
| SEQ | TM     | SM         | 
| --- | ------ | ---------- | 
| 1   | 202210 | 2449167.91 | 
| 2   | 202211 | 2474687.5  | 
| 3   | 202212 | 2547042.11 | 

#### なぜ `tm`（年月）をそのまま使わないのか？
`tm`（202210, 202211...）をそのまま座標にすると、年をまたぐ際に「202212 の次は 202301」となり、数値として連続しません（202213〜202300が欠番になる）。
これでは `MODEL` 句の中で `[tm + 1]` といった単純な算術計算で「翌月」を指すことができません。

#### `seq` を使うメリット
* **連続性の確保**: `ROW_NUMBER()` により、月がどうあれ必ず `1, 2, 3` という綺麗な連番になります。特にループ計算を行う際は、連番を作り出してでも利用する方が、すんなりロジックを組むことが出来ると思います。
* **相対参照の簡略化**: `sm[iteration_number + 4]` のように、「4行目（新しい月）の値は、1行目（3ヶ月前）の値を参照する」といった、スプレッドシートのような直感的な指定が可能になります。


### 2. `ITERATE` の使い方
`ITERATE(12)` は、`RULES` 内に記述された計算を **12回繰り返す** という命令です。

#### ループの動き
* **`iteration_number`**: 0から始まるカウンタです（1回目は0、2回目は1...）。
* **計算式の展開**:
    * **1回目 (i=0)**: `sm[4] = sm[1] * 1.1` （2023年1月の予測 ＝ 2022年10月の実績 $\times$ 1.1）
    * **2回目 (i=1)**: `sm[5] = sm[2] * 1.1` （2023年2月の予測 ＝ 2022年11月の実績 $\times$ 1.1）
    * **10回目 (i=9)**: `sm[13] = sm[10] * 1.1` （※注：10番目は以前のループで作られた「予測値」を元にさらに計算されます）

#### 予測の連鎖
このクエリの面白い点は、最初の3回は「実績」を元に計算しますが、それ以降は **「自分がループで作った過去の予測値」を元に、さらに次の予測を作る（再帰的計算）** という動きを自動で行っている点です。

### 3. 外側のSELECTでの「年月復元」

`MODEL` 句の中で `seq` 4番以降の `tm`（年月）は計算されていないため、そのままでは `NULL` になってしまいます。そこで、一番外側の `SELECT` で以下の工夫をしています。

```sql
TO_NUMBER(to_char(ADD_MONTHS(DATE '2022-10-01', seq - 1), 'yyyymm')) AS tm
```
* **ロジック**: 「開始日（2022-10-01）」に対し、`seq - 1` ヶ月分を足すことで、`seq=1` なら 202210、`seq=4` なら 202301 というように、**連番から正しい年月を逆算（復元）** しています。これにより、`MODEL` 句内で複雑な年月計算をする手間を省いています。

### 4. `MODEL` 句（例1） vs `再帰的WITH句`（例2） の比較
再帰的WITH句を使用しても同じ結果が得られます。設計思想や使い勝手には大きな違いがあるので、その点は押さえておいた方がいいと思います。

| 比較項目 | MODEL 句 | 再帰的WITH句 (Recursive CTE) |
| :--- | :--- | :--- |
| **得意分野** | **複雑なセル参照・数式計算** | **階層構造・単純な連続処理** |
| **思考プロセス** | Excelのように「座標」で考える | プログラムの「再帰関数」で考える |
| **可読性** | 数式が並ぶため、計算ルールが追いやすい | 構造は標準的だが、結合条件が複雑になりがち |
| **柔軟性** | 複数の行を同時に参照しやすい（sm[1]+sm[2]等） | 基本的に「一行前」を参照するのが限界 |
| **パフォーマンス** | メモリ上で配列を作るため、計算は高速 | 行を1つずつ結合して増やすため、オーバーヘッドがある |

#### MODEL 句を選ぶべきケース
* **多次元の参照が必要なとき**: 「3ヶ月前の実績と、1ヶ月前の予測の平均をとる」といった、複雑な座標移動を伴う計算。
* **既存の行を「書き換える」必要があるとき**: 予測だけでなく、実績値そのものを特定のルールで補正する場合。

#### 再帰的WITH句を選ぶべきケース
* **標準SQLを重視するとき**: `MODEL` 句はOracle独自の機能ですが、再帰的WITH句は他の主要RDBMS（PostgreSQL, SQL Server等）でも動きます。
* **「次々に新しい行を生む」ことが主目的のとき**: 終了条件が明確な単純なループ（カレンダー作成など）にはこちらがスッキリします。

# 【完全版】問題15-3（完全離脱者の特定）  *Lv4*
### **「売上が連続して減少した後、ついに0（購入なし）になった」**　という「完全離脱（Churn）」した人を下記の条件により特定してください。（順不同）

**対象: SHスキーマ`sales`テーブルの顧客ごと、四半期（Quarter）ごとの売上合計。**

**条件:**
**第2四半期(Q2)が第1四半期(Q1)より減少している。**
**第3四半期(Q3)が第2四半期(Q2)よりさらに減少している。**
**第4四半期(Q4)の売上が0である。**

**件数が多いので、`cust_id`が71または100または525の人を表示対象としてください。**

**第4四半期(Q4)に`STATUS`として、下記の条件で値を設定してください。**
* **売上がQ1>Q2>Q3, Q4=0の場合、「CHURNED」**
* **売上が上記以外で、Q4=0の場合、「INACTIVE」**
* **上記以外の場合、「ACTIVE」**

## 期待する結果
| CUST_ID | QTR | AMOUNT  | STATUS   | 
| ------- | --- | ------- | -------- | 
| 71      | 1   | 160.37  |          | 
| 71      | 2   | 39.9    |          | 
| 71      | 3   | 162.34  |          | 
| 71      | 4   | 12.58   | ACTIVE   | 
| 100     | 1   | 210.95  |          | 
| 100     | 2   | 81.35   |          | 
| 100     | 4   |         | INACTIVE | 
| 525     | 1   | 4849.34 |          | 
| 525     | 2   | 4405.46 |          | 
| 525     | 3   | 486.18  |          | 
| 525     | 4   |         | CHURNED  | 

## 解答例
```sql
WITH user_quarterly_sales AS (
    -- 顧客・四半期ごとの売上合計を算出
    SELECT
        cust_id,
        TO_CHAR(time_id, 'q') AS qtr,
        SUM(amount_sold)      AS amount
    FROM
        sh.sales
    WHERE
        time_id BETWEEN DATE '2021-01-01' AND DATE '2021-12-31'
        AND cust_id IN ( 71, 100, 525 )
    GROUP BY
        cust_id,
        TO_CHAR(time_id, 'q')
)
SELECT
    *
FROM
    user_quarterly_sales
MODEL
    PARTITION BY ( cust_id ) 
    DIMENSION BY ( qtr )
    MEASURES ( 
        amount, 
        CAST(null AS VARCHAR2(20)) AS status )
    RULES (
        -- Q4の時点で「減少→減少→0」のパターンを判定
        status['4']=
            CASE
                WHEN amount['1']> amount['2']  -- Q1 > Q2 (減少)
                     AND amount['2']> amount['3']  -- Q2 > Q3 (さらに減少)
                     AND amount['4']IS NULL        -- Q4で購入なし(データがない=NULL)
                      THEN
                    'CHURNED'
                WHEN amount['4']IS NULL THEN
                    'INACTIVE'
                ELSE
                    'ACTIVE'
            END
    )
ORDER BY
    cust_id,
    qtr;
```

## 解説
「顧客が離反（CHURN）したかどうか」を、四半期ごとの売上推移から判定しているクエリーです。

### 1. クエリの構造解説

#### ① WITH句（準備）
`sh.sales` テーブルから、特定の顧客（71, 100, 525）の2021年における「四半期ごとの売上合計」を算出しています。これが `MODEL` 句に渡される「生のデータ」になります。

#### ② MODEL句（Excel化）
データを多次元配列（グリッド）として定義します。
* **`PARTITION BY (cust_id)`**: 顧客ごとに計算の範囲を区切ります（顧客Aのデータが顧客Bの計算に混ざらないようにします）。
* **`DIMENSION BY (qtr)`**: 横軸（行を特定するキー）を四半期（'1'〜'4'）にします。
* **`MEASURES`**: 操作したい「セルの値」です。売上の `amount` と、空の `status` 列を定義しています。

#### ③ RULES（計算式・判定）
ここがExcelの数式に相当します。`status['4']`（第4四半期のステータス）というセルに対して、以下のルールを適用しています。

* **CHURNED（離反）**: 第1四半期 > 第2四半期 > 第3四半期 と売上が減り続け、かつ第4四半期のデータが存在しない（`NULL`）場合。
* **INACTIVE（休眠）**: 第4四半期に購入がない場合。
* **ACTIVE（継続）**: それ以外（第4四半期に購入がある）。

### 2. このクエリのポイント
通常、SQLで「前の行の値」を参照するには `LAG` 関数などを使いますが、2つ前、3つ前と比較して複雑な条件を書くと非常に読みづらくなります。
`MODEL` 句を使えば、**`amount['1']`（第1四半期の値）のように、位置を指定して直接参照できる**ため、時系列分析や複雑なビジネスロジックを直感的に記述できます。

# 【完全版】問題15-4（減算ループ）  *Lv4*
### COスキーマ`ORDERS`テーブルには2021年2月から2022年4月までの注文データが格納されています。また、`INVENTORY`テーブルには現在（2022年4月時点）の在庫情報が格納されています。商品ごとに月平均の注文数を算出し、在庫がいつマイナスになるかシミュレーションしてください。表示対象は`PRODUCT_ID`が19または35となります。（商品ID、年月順）

## 期待する結果
| OT     | PRODUCT_ID | SUM_IVT | 
| ------ | ---------- | ------- | 
| 202204 | 19         | 70      | 
| 202205 | 19         | 50      | 
| 202206 | 19         | 30      | 
| 202207 | 19         | 10      |

## 解答例
```sql
WITH order_sum AS ( -- 商品ごとの注文数合計
    SELECT
        oi.product_id,
        SUM(oi.quantity) sum_ord
    FROM
             co.orders o
        INNER JOIN co.order_items oi ON o.order_id = oi.order_id
    GROUP BY
        oi.product_id
), inv_sum AS ( -- 商品ごとの在庫数合計
    SELECT
        i.product_id,
        SUM(i.product_inventory) sum_ivt
    FROM
        co.inventory i
    GROUP BY
        i.product_id
), inv_data AS ( -- 現在の商品ごとの在庫数、１ヶ月の注文数
    SELECT
        1     nm,
        i.product_id,
        o.sum_ord,
        round(o.sum_ord / (MONTHS_BETWEEN(DATE '2022-04-01', DATE '2021-02-01') + 1)) AS avg_ord,
        i.sum_ivt
    FROM
             order_sum o
        INNER JOIN inv_sum i ON o.product_id = i.product_id
    WHERE
        o.product_id in (19, 35)
)
SELECT
    to_char(
                add_months(DATE '2022-04-01', nm - 1),
        'yyyymm'
   ) ot,
    product_id,
    sum_ivt
FROM
    inv_data
MODEL
    PARTITION BY ( product_id ) DIMENSION BY ( nm )
    MEASURES ( sum_ivt,
               avg_ord )
    RULES ITERATE(20) UNTIL(sum_ivt[iteration_number + 1]< 0) (
        -- 次月の在庫 = 今月の在庫 - 平均注文数
        sum_ivt[iteration_number + 2]= sum_ivt[iteration_number +1]- avg_ord[1]
    )
ORDER BY
    product_id,
    nm
```

## 解説
このクエリは、Oracle SQLの強力な機能である**MODEL句**を使い、**「現在の在庫が、過去の平均的な注文ペースで売れていった場合、いつ底をつくか」をシミュレーション（バーンダウン分析）** するものです。
一言で言うと、**「SQLの中にExcelの数式を持ち込んで、未来のデータを自動生成している」** 非常に高度な処理です。

### 1. `ITERATE` の高度な使い方

`ITERATE` は `RULES` の直後に記述します。

```sql
RULES ITERATE (最大反復回数) [ UNTIL (終了条件) ] (
    -- ここに繰り返したい計算ルールを書く
    [セルの指定] = [計算式]
)
```

#### 主要な要素
* **`ITERATE(n)`**: ループを最大 $n$ 回実行します。無限ループを防ぐための「安全装置」の役割も果たします。
* **`UNTIL (条件)`**: （任意設定）反復の途中で特定の条件（例：在庫が0以下になった、目標値に達したなど）を満たした場合、ループを終了させます。
* **`iteration_number`**: システム変数です。**0から始まり**、ループするたびに 1, 2, 3... とカウントアップされます。これを使って「今何回目の処理か」を数式内で参照できます。


### 2. `ITERATE` の動き：再帰的な計算

`ITERATE` の最大の強みは、**「前のループで計算した結果を、次のループの入力として使える」** 点にあります。これを「再帰的参照」と呼びます。

#### 具体的な動作イメージ（在庫シミュレーションの場合）
前回のクエリにあった以下のルールを例にします。

```sql
RULES ITERATE(20) (
    sum_ivt[iteration_number + 2] = sum_ivt[iteration_number + 1] - avg_ord[1]
)
```

1.  **1回目のループ (`iteration_number` = 0)**:
    * `sum_ivt[2]` = `sum_ivt[1]` - `avg_ord[1]`
    * （2ヶ月目の在庫 ＝ 1ヶ月目の在庫 － 平均注文数）を計算。
2.  **2回目のループ (`iteration_number` = 1)**:
    * `sum_ivt[3]` = `sum_ivt[2]` - `avg_ord[1]`
    * （3ヶ月目の在庫 ＝ **先ほど計算した2ヶ月目の在庫** － 平均注文数）を計算。
3.  これを $n$ 回、または `UNTIL` 条件まで繰り返します。

### 3. クエリの構造解説
このクエリは、大きく分けて「現状把握」と「未来予測」の2段階で構成されています。

#### ① 準備段階（WITH句）
* **`order_sum` / `inv_sum`**: 過去の注文実績の合計と、現在の在庫合計を商品ごとに算出します。
* **`inv_data`**: シミュレーションの「初期値」を計算します。
    * **`avg_ord`**: 2021年2月〜2022年4月（約15ヶ月間）の総注文数を月数で割り、**「1ヶ月あたりの平均注文数（ペース）」**を出しています。
    * **`nm = 1`**: これをシミュレーション開始の「1ヶ月目」のフラグとして使います。

#### ② 未来予測段階（MODEL句）
ここで「未来の行」を計算によって作り出します。

* **`PARTITION BY (product_id)`**: 商品ごとに独立して計算します。
* **`DIMENSION BY (nm)`**: 「○ヶ月目」という時系列を軸（インデックス）にします。
* **`MEASURES (sum_ivt, avg_ord)`**: 操作対象となる数値（在庫数と平均注文数）です。
* **`RULES ITERATE(20) UNTIL(...)`**:
    * 最大20回（20ヶ月分）計算を繰り返します。
    * **`UNTIL(sum_ivt[...] < 0)`**: 在庫がマイナス（欠品）になったら、そこで計算をストップします。
* **計算式**:
    > **`翌月の在庫 = 今月の在庫 - 毎月の平均注文数`**
    
    この数式を繰り返し適用することで、1行しかなかったデータから、在庫が尽きるまでの未来の行が次々と生成されます。

### 4. ポイント

#### 1. SQLで「無いデータ」を作っている
通常のSQLは「テーブルにある行」しか操作できません。しかし、`MODEL`句を使うと、計算ルールに基づいて**存在しない未来の行を動的に作り出す**ことができます。

#### 2. ビジネスロジックの直感的な記述
「在庫 = 前月の在庫 - 消費量」というExcelのような再帰的なロジックを、そのままSQLで表現できるのが強みです。これを通常のSQL（Window関数など）でやろうとすると、非常に複雑な再帰クエリ（Recursive CTE）を書く必要があります。

#### 3. 可視化への適性
結果として「年月」と「在庫残高」がセットで出力されるため、BIツールなどに繋げば、そのまま **「在庫枯渇予測グラフ」** を描画できます。
