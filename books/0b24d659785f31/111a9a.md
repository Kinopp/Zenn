# 前提知識
「行パターンマッチング（MATCH_RECOGNIZE）」は、Oracle Database（特に12c以降、そして最新の23ai）における **「最強の武器」** のひとつです。
一言でいうと、**「正規表現を、文字列ではなく『行の並び』に対して実行する機能」** です。単一の行を見るのではなく、「株価が下がってから上がった（V字回復）」や「ログイン失敗が3回続いた後に成功した」といった **「データの流れ（シーケンス）」** をSQLでスマートに抽出できます。

## 1. 基本コンセプト：点ではなく「線」で捉える
従来のSQL（GROUP BYやWindow関数）は、集合や特定の範囲を集計するのは得意ですが、「Aの次にBが起こり、その後にCが起きた」という**時系列のパターン**を探すのは非常に困難でした。

行パターンマッチングを使うと、複雑な自己結合（Self-Join）を繰り返すことなく、直感的にパターンを定義できます。

![](https://storage.googleapis.com/zenn-user-upload/a360eb123a42-20260323.png)



## 2. 必須の5つの要素（MATCH_RECOGNIZE句）
これだけは押さえておきたい主要なキーワードを整理します。

| キーワード | 役割 | イメージ |
| :--- | :--- | :--- |
| **PARTITION BY** | データをグループ分けする | どの「部署」や「ユーザー」ごとに探すか？ |
| **ORDER BY** | 行を並べる（必須） | 時系列順（日付など）に並べないと意味がない |
| **MEASURES** | 出力したい値を定義する | パターンの開始日や、変化率の計算結果など |
| **PATTERN** | 探したいパターンの形（必須） | 正規表現のように `(UP+ DOWN+)` と書く |
| **DEFINE** | パターンの各変数を定義する（必須） | 「UP」とは「前の行より価格が高いこと」と定義 |

# 問題16-1（行パターンマッチング初歩の初歩１）  *Lv3*
### MATCH_RECOGNIZE句を使用して、HRスキーマ`COUNTRIES`テーブルのうち、`REGION_ID`が「20」のデータを取得してください。（順不同）

## 期待する結果
| REGION_ID | COUNTRY_ID | COUNTRY_NAME             | 
| --------- | ---------- | ------------------------ | 
| 20        | CA         | Canada                   | 
| 20        | BR         | Brazil                   | 
| 20        | AR         | Argentina                | 
| 20        | US         | United States of America | 
| 20        | MX         | Mexico                   | 

## 解答例
```sql
SELECT
    *
FROM
    hr.countries 
    MATCH_RECOGNIZE (
        ORDER BY
            region_id
        ALL ROWS PER MATCH
        PATTERN ( 
            group20+ 
        )
        DEFINE
            group20 AS region_id = 20
    )
```

## 解説
行パターンマッチング（`MATCH_RECOGNIZE`）を一言でいうと、**「正規表現の行バージョン」**　です。
通常、SQLは「1行ずつ」チェックするか、「グループ全体」をまとめますが、パターンマッチングは **「行が並んでいる順番（時系列など）」に意味を見出す** ための機能です。
私は、この機能を使いこなせるようになると、今までSQLで実現するのを諦めていたロジックの多くがSQLで実現できるようになると考えています。

### 解答例の概要
まず、最初にお断りすると、この問題を`MATCH_RECOGNIZE`を使って解決しようと思う人はいないはずです。`WHERE region_id = 20`とすればいいだけですので。ここでは、理解を目的にあえて`MATCH_RECOGNIZE`を使って解決しています。

このクエリを一言でいうと、**「`region_id` が 20 である国々を、ひとまとめの『パターン（塊）』として見つけ出し、その該当する行をすべて表示する」** という処理をしています。

通常の `WHERE region_id = 20` と結果は同じですが、内部的には「連続するデータの塊」を探しに行くという**行パターンマッチング**の動きをしています。


### クエリの解剖図

| 句 | 役割 |
| :--- | :--- |
| **`ORDER BY region_id`** | データを `region_id` 順に並べます。パターンマッチングは「並び順」が命なので、最初に整列させます。 |
| **`ALL ROWS PER MATCH`** | パターンに一致した行を**すべて**表示します（デフォルトの `ONE ROW PER MATCH` だと、集計されて1行になってしまいます）。 |
| **`PATTERN (group20+)`** | 探したい形を指定します。`+` は「1回以上の繰り返し」を意味します。つまり **「`group20` という条件に合う行が1つ以上連続している箇所」** を探します。 |
| **`DEFINE group20 AS ...`** | `group20` という記号の意味を定義します。ここでは「`region_id` が 20 であること」を条件にしています。 |


### このクエリを実行するとどうなるか？

1.  まず、全データが `region_id` ごとに並びます（10, 10, 20, 20, 20, 30... のような順）。
2.  `MATCH_RECOGNIZE` が上から順に見ていき、`region_id = 20` が始まった瞬間に「パターンの開始」と判定します。
3.  `region_id = 20` が続く限り、それを一つの「マッチ（塊）」として保持します。
4.  `region_id` が 20 以外になった時点でマッチが終了し、見つかった 20 の行をすべて出力します。


### なぜ `MATCH_RECOGNIZE` を使うのか？
- `PATTERN`の正規表現と`DEFINE`の式の組み合わせで、高度な条件を実現できるため、SQLで表現できる幅が広がります。
- `MATCH_RECOGNIZE` を使うと、様々な　**「高度な分析用のおまけ」** を簡単に追加できます。

# 問題16-2（行パターンマッチング初歩の初歩２）   *Lv3*
### MATCH_RECOGNIZE句を使用して、HRスキーマ`COUNTRIES`テーブルのうち、`REGION_ID`が「20」の件数を取得してください。

## 期待する結果
| REGION_ID | CNT | 
| --------- | --- | 
| 20        | 5   | 

## 解答例
```sql
SELECT
    *
FROM
    hr.countries 
    MATCH_RECOGNIZE (
        ORDER BY
            region_id
        MEASURES
           region_id      AS region_id,
           COUNT(*)       AS cnt
        ONE ROW PER MATCH
        PATTERN ( 
            group20+ 
        )
        DEFINE
            group20 AS region_id = 20
    )
```

## 解説
このクエリは、**「条件に一致した塊（パターン）を1行に集約して集計する」** という処理を行っています。
前回のクエリが「該当する行をすべて見せる」ものだったのに対し、今回のクエリは **「該当する塊が何件あったかというサマリー（要約）」** を作るものです。


### 前回のクエリとの違い

最大の変更点は以下の2箇所です。

#### 1. ONE ROW PER MATCH（1つのマッチにつき1行）
前回は `ALL ROWS PER MATCH` でしたが、今回は `ONE ROW PER MATCH` です。これにより、`region_id = 20` が連続している行が何行あろうとも、結果としては **1つの塊（マッチ）につき1行だけ** が出力されます。
いわば、パターンマッチング版の `GROUP BY` のような動きです。

#### 2. MEASURES（出力項目の定義）
集約して1行にする際、何を表示するかをここで決めます。
* `region_id AS region_id`：その塊の `region_id` を表示。
* `COUNT(*) AS cnt`：その塊の中に**何行（何カ国）含まれていたか**をカウント。

### 各句の役割
| 句 | 役割 |
| :--- | :--- |
| **`MEASURES`** | 集計結果として出したい項目を作る（ここではIDと件数）。 |
| **`ONE ROW PER MATCH`** | マッチした範囲をギュッと1行にまとめる。 |
| **`PATTERN (group20+)`** | `region_id = 20` が1回以上続く箇所を探す。 |
| **`DEFINE`** | `group20` が「`region_id = 20` である」ことを定義。 |

# 【完全版】問題16-3（上昇・下降パターンの認識1） *Lv4*
### SHスキーマ`SALES`テーブルを使用し、月ごとの売上が「前月より高い」状態が3回以上続いた期間を特定してください。

![](https://storage.googleapis.com/zenn-user-upload/8a0b20c7cfb9-20260319.png)

## 期待する結果
| START_MONTH | END_MONTH | DURATION | 
| ----------- | --------- | -------- | 
| 2019-06     | 2019-10   | 5        | 
| 2020-07     | 2020-09   | 3        | 
| 2020-12     | 2021-02   | 3        | 
| 2022-10     | 2022-12   | 3        | 

## 解答例
```sql
SELECT *
FROM (
    SELECT 
        TO_CHAR(time_id, 'YYYY-MM') AS month,
        SUM(amount_sold) AS total_rev
    FROM sh.sales
    GROUP BY TO_CHAR(time_id, 'YYYY-MM')
)
MATCH_RECOGNIZE (
    ORDER BY month
    MEASURES 
        FIRST(up.month) AS start_month,
        LAST(up.month) AS end_month,
        COUNT(up.month) AS duration
    ONE ROW PER MATCH
    AFTER MATCH SKIP TO LAST up
    -- パターン定義：最初の行(STRT)の後に、上昇(UP)が3回以上続く
    PATTERN (strt up{3,})
    DEFINE 
        up AS up.total_rev > prev(up.total_rev)
)
ORDER BY start_month;
```

## 解説
このクエリは、 **「売上が3ヶ月以上連続で上昇している期間（トレンド）を抽出する」** という高度な分析を行っています。

### クエリの解説
このクエリは大きく分けて2つのステップで構成されています。

#### 1. インライン・ビュー（内側のSELECT文）
まず、分析の土台となるデータを用意しています。
* `SH.SALES` テーブル（Oracleの標準サンプルスキーマ）から、月ごとの売上合計（`TOTAL_REV`）を計算しています。
* `TO_CHAR(TIME_ID, 'YYYY-MM')` を使うことで、日次データを月次データに集約しています。

#### 2. MATCH_RECOGNIZE（外側のパターンマッチング）
ここがこのクエリの核心です。時系列データの中から特定の「動き（パターン）」を探し出しています。

| 句 | 役割 |
| :--- | :--- |
| **`ORDER BY MONTH`** | データを時系列順に並べます。パターンを探すための前提条件です。 |
| **`MEASURES`** | 見つかったパターンに対して、どんな値を表示するか定義します（開始月、終了月、継続期間）。 |
| **`PATTERN (STRT UP{3,})`** | **「最初の1件（STRT）の後に、上昇（UP）が3回以上続く」** というルールを定義しています。 |
| **`DEFINE UP AS...`** | 「上昇（UP）」とは何かを定義しています。**「現在の行の売上が、前の行（PREV）の売上より大きい」** 場合を指します。 |


### ポイント

#### 1. 「PREV関数」の魔法
通常、SQLで「前の行」と比較するには`LAG`関数などを使う必要がありますが、`MATCH_RECOGNIZE`の中では`PREV()`という直感的な関数が使えます。これにより、「売上が上がり続けている」という条件が非常にシンプルに記述できています。

#### 2. `UP{3,}` の意味
これは正規表現に似た書き方です。
* `UP`: 1回の上昇
* `UP{3,}`: 3回**以上**の上昇
つまり、4ヶ月連続（開始月 + 上昇3回）で売上が上がったケースだけを抽出しています。

### 注意点
#### ① データの「欠落」に注意
`SH.SALES` スキーマには、売上がゼロの月が含まれていない場合があります。
* **注意点:** もし「1月、2月、4月」とデータが飛んでいても、このクエリはそれらを「連続」とみなしてしまいます。
* **対策:** 厳密な分析を行う場合は、あらかじめ「外部結合」などを使用して、売上がゼロの月も行として存在させる処理が必要です。

# 【完全版】問題16-4（上昇・下降パターンの認識2） *Lv4*
### HRスキーマ`EMPLOYEES`テーブルより、部門ごとに採用順（hire_date順）にデータを見たとき、「新しく入った従業員の初任給が、直前に採用された従業員よりも高い状態が3回以上連続した期間」を特定してください。

## 期待する結果
| DEPARTMENT_ID | START_HIRE_DATE | END_HIRE_DATE | CONSECUTIVE_INCREASES | START_SALARY | END_SALARY | 
| ------------- | --------------- | ------------- | --------------------- | ------------ | ---------- | 
| 50            | 2017-04-10      | 2017-11-16    | 4                     | 2100         | 5800       | 
| 50            | 2017-12-12      | 2018-02-03    | 4                     | 2400         | 2800       | 
| 80            | 2015-12-15      | 2016-03-23    | 3                     | 7500         | 10000      | 
| 80            | 2018-01-04      | 2018-01-29    | 3                     | 6200         | 10500      | 

## 解答例
```sql
SELECT
    department_id,
    start_hire_date,
    end_hire_date,
    consecutive_increases,
    start_salary,
    end_salary
FROM hr.EMPLOYEES
    
    MATCH_RECOGNIZE (
    PARTITION BY department_id
    ORDER BY hire_date
    MEASURES
        FIRST(hire_date) AS start_hire_date,
        LAST(hire_date) AS end_hire_date,
        COUNT(UP.employee_id) + 1 AS consecutive_increases,
        FIRST(salary) AS start_salary,
        LAST(salary) AS end_salary
    ONE ROW PER MATCH
    -- パターン定義：任意の開始点(STRT)から、給与が前回より高い状態(UP)が2回以上連続する
    PATTERN (STRT UP{2,}) 
    DEFINE
        UP AS salary > PREV(salary)
    )
ORDER BY consecutive_increases DESC, start_hire_date
```

## 解説
今回のクエリは、**「同じ部署内で、採用順に見て初任給が3回以上連続で上がっている期間」** を特定するものです。前回の売上分析との最大の違いは、`PARTITION BY` が加わっている点です。


### クエリの解説
このクエリは、部署という「枠」の中で、時間の経過とともに「提示給与」がどう変化したかをあぶり出します。

#### 1. `PARTITION BY department_id`（部署ごとの仕切り）
ここが最重要ポイントです。全社員を混ぜるのではなく、**部署ごとに独立して**パターンを探します。
* 開発部の給与トレンドと、営業部の給与トレンドを混同しないように、「部署の壁」を作っています。

#### 2. `PATTERN (STRT UP{2,})`（3連続以上の昇給）
* `STRT`: 任意の開始行（1人目）。
* `UP{2,}`: 「前回より給与が高い人」が**2回以上連続**して現れる。
* **合計:** 1人目 + 上昇2回 ＝ **最低3人以上の連続した昇給トレンド**を見つけています。

#### 3. `MEASURES`（分析結果の算出）
見つかった「塊（マッチ）」から、以下の情報を抽出します。
* **`consecutive_increases`**: 何人連続で上がったか。
* **`start_salary` / `end_salary`**: そのトレンドの開始時と終了時の給与額。



### ポイント

### ① 「給与が同じ」場合の挙動
`DEFINE UP AS salary > PREV(salary)` となっています。
* **注意点:** もし2人連続で「給与 500,000円」だった場合、このパターンは**途切れます**（`>` なので）。
* **対策:** 「以上」を含めたい場合は `>=` を使うなど、ビジネスルールに合わせた微調整が必要です。

### ② 採用日が同じ場合の順序
`ORDER BY hire_date` を指定していますが、同じ日に2人採用されたらどうなるでしょうか？
* **注意点:** 同日の場合、内部的な処理順序が不定になり、結果が変わる可能性があります。
* **対策:** `ORDER BY hire_date, employee_id` のように、**一意になる列（主キーなど）を第2ソートキーに含める**のが、プロの書くSQLです。

```sql
select
   e.EMPLOYEE_ID
 , lpad(' ', 2*(level-1)) || level || '. ' || e.FIRST_NAME || ' ' || e.LAST_NAME as name
 , (
      select count(*)
      from HR.EMPLOYEES sub
      start with sub.MANAGER_ID = e.EMPLOYEE_ID
      connect by sub.MANAGER_ID = prior sub.EMPLOYEE_ID
   ) as subs
from HR.EMPLOYEES e
start with e.MANAGER_ID is null

```
```sql
with hierarchy as (
   select
      lvl, EMPLOYEE_ID as id,  name, rownum as rn
   from (
      select
         level as lvl, e.EMPLOYEE_ID, e.FIRST_NAME || ' ' || e.LAST_NAME as name
      from hr.employees e
      start with e.MANAGER_ID is null
      connect by e.MANAGER_ID = prior e.EMPLOYEE_ID
      order siblings by name
   )
)
select
   id
 ,  lpad(' ', 2 *(lvl -1)) || lvl || '. ' || name as name
 , subs
from hierarchy
match_recognize (
   order by rn
   measures
      strt.rn           as rn
    , strt.lvl          as lvl
    , strt.id           as id
    , strt.name         as name
    , count(higher.lvl) as subs
   one row per match
   after match skip to next row
   pattern (
      strt higher*
   )
   define
      higher as higher.lvl > strt.lvl
)
order by rn;
```
