# 問題6-1（四則演算） *Lv1*
### **HRスキーマ`EMPLOYEES`テーブルより、下記の値を算出してください。**
* **`SALARY`に`1000`を加算した値（`+`と表示）**
* **`SALARY`に`1000`を減算した値（`-`と表示）**
* **`SALARY`に`2`を乗算した値（`*`と表示）**
* **`SALARY`に`3`を除算した値（商）（`/`と表示）**
* **`SALARY`を`7`で除算した余り（`余り`と表示）**

## 期待する結果
| EMPLOYEE_ID | SALARY | +     | -     | *     | /    | 余り | 
| ----------- | ------ | ----- | ----- | ----- | ---- | ---- | 
| 100         | 24000  | 25000 | 23000 | 48000 | 8000 | 4    | 

## 解答例
```sql
SELECT
    employee_id,
    salary,
    salary + 1000     AS "+",
    salary - 1000.    AS "-",
    salary * 2        AS "*",
    salary / 3        AS "/",
    MOD(salary, 7)    AS "余り"
FROM
    hr.employees
WHERE
    employee_id = 100
```

## 解説
SQLにおける四則演算（`+`, `-`, `*`, `/`）は非常にシンプルですが、**「NULL」や「データ型」が絡むと、計算結果が予想外のものになる**という罠が潜んでいます。

### 1. 基本的な演算子
SQLでは、算数でおなじみの記号をそのまま使います。

| 演算子 | 意味 | 例 | 結果 |
| :--- | :--- | :--- | :--- |
| **`+`** | 足し算 | `100 + 50` | `150` |
| **`-`** | 引き算 | `100 - 50` | `50` |
| **`*`** | 掛け算 | `100 * 2` | `200` |
| **`/`** | 割り算 | `10 / 4` | `2.5`|
| **`MOD`** | 余り | `MOD(10, 3)` | `1` |


### 2. 演算時の3つの大きな注意点

#### ① NULLが混じると結果もNULLになる
SQLの世界では、**「不明なもの（NULL）に何を足しても引いても、結果は不明（NULL）」**になります。

* `100 + NULL` = **`NULL`**
* `100 * NULL` = **`NULL`**



**対策:** 計算する値にNULLが含まれる可能性がある場合は、`COALESCE`（または `NVL`）関数を使って **0に置き換える** のが定石です。
```sql
-- 給与 + 手当（手当がNULLなら0として計算）
SELECT salary + COALESCE(comm, 0) FROM employees;
```

#### ② ゼロ除算（Division by Zero）のエラー
数字を `0` で割ると、ほとんどのデータベースはエラーを出し、**クエリ全体が停止**してしまいます。

**対策:** 分母が `0` になる可能性がある場合は、`NULLIF` 関数を使うとスマートに回避できます。
`NULLIF(b, 0)` は、bが0のときにNULLを返します。前述の通り「NULLで割った結果はNULL」になるため、エラーを回避して結果をNULLにできます。
```sql
-- 売上 / 客数（客数が0でもエラーにならない）
SELECT sales / NULLIF(customer_count, 0) FROM daily_reports;
```

### 3. 演算の優先順位
算数と同じく、**掛け算・割り算が先、足し算・引き算が後**です。
順序を変えたい場合は、必ず **`( )` カッコ** を使います。

* `100 + 100 * 2` = `300`
* `(100 + 100) * 2` = `400`

# 問題6-2（端数処理） *Lv1*
### **HRスキーマのEMPLOYEESテーブルより、下記の値を算出してください。**
* **`SALARY`を`7`で除算し、小数点第2位以下を切り捨て（`小数点切り捨て`と表示）**
* **`SALARY`を`7`で除算し、小数点以下を切り捨て（`切り捨て`と表示）**
* **`SALARY`を`7`で除算し、小数点以下を四捨五入（`四捨五入`と表示）**
* **`SALARY`に`7`で除算し、小数点以下を切り上げ（`切り上げ`と表示）**

## 期待する結果
| EMPLOYEE_ID | SALARY | /                  | 小数点切り捨て | 切り捨て | 四捨五入 | 切り上げ | 
| ----------- | ------ | ------------------ | -------------- | -------- | -------- | -------- | 
| 100         | 24000  | 3428.5714285714300 | 3428.5         | 3428     | 3429     | 3429     | 

## 解答例
```sql
SELECT
    employee_id,
    salary,
    salary / 7           AS "/",
    TRUNC(salary / 7, 1) AS "小数点切り捨て",
    TRUNC(salary / 7)    AS "切り捨て",
    ROUND(salary / 7)    AS "四捨五入",
    CEIL(salary / 7)     AS "切り上げ"
FROM
    hr.employees
WHERE
    employee_id = 100
```

## 解説
SQLにおける数値の丸め処理には、**TRUNC**（切り捨て）、**ROUND**（四捨五入）、**CEIL**（切り上げ）の3つの関数を使用します。

### 1. 比較まとめ表

まずは、数値 `125.456` を例にした違いを見てみましょう。

| 関数 | 処理内容 | SQL例 (小数点第2位まで) | 結果 |
| --- | --- | --- | --- |
| **`ROUND`** | **四捨五入** | `ROUND(125.456, 2)` | **`125.46`** |
| **`TRUNC`** | **切り捨て** | `TRUNC(125.456, 2)` | **`125.45`** |
| **`CEIL`** | **切り上げ** | `CEIL(125.456 * 100) / 100` | **`125.46`** |

### 2. 各関数の詳細と注意点

#### ① ROUND（四捨五入）

最も一般的な丸め処理です。

* **特徴**: 指定した桁の次の数字が「5以上なら切り上げ、4以下なら切り捨て」ます。
* **注意点**: 負の数の扱い。Oracleの場合、`ROUND(-1.5)` は `-2` になります（絶対値が大きくなる方向に動きます）。

#### ② TRUNC（切り捨て）

指定した桁より右側を、文字通り「切り捨て」てゼロにします。

* **特徴**: 指定した桁数（精度）まで残し、あとは消去します。
* **注意点**: **「負の数」**　のとき、数学的な `FLOOR`（床関数）とは挙動が異なります。
* `TRUNC(-1.9)` → **`-1`** （0に近い方へ向かう）
* `FLOOR(-1.9)` → **`-2`** （より小さい整数へ向かう）

#### ③ CEIL（切り上げ）
「その数以上の最小の整数」を返します。

* **特徴**: 小数点以下が `0.00...1` でもあれば、上の整数に跳ね上がります。
* **注意点**: **「桁数指定ができない」**。
`ROUND` や `TRUNC` と違い、第2引数で「小数点第2位まで」といった指定ができません。
> **対策**: `CEIL(値 * 100) / 100` のように、一旦桁をずらしてから計算する必要があります。


### 3. 実務上の注意点

#### 負の数の計算

ここが一番のバグの温床です。

> * **`TRUNC`**: 常に **0に向かって** 進む（`-1.9` → `-1`）
> * **`CEIL`**: 常に **正の方向（右）へ** 進む（`-1.9` → `-1`）
> * **`FLOOR`**: 常に **負の方向（左）へ** 進む（`-1.9` → `-2`）

### 浮動小数点誤差
SQLにおける **浮動小数点誤差（Floating Point Error）** は、特に「お金」や「在庫」を扱うエンジニアにとって、もっとも恐ろしいバグの原因の一つです。

「コンピュータは計算が得意」と思われがちですが、実は**10進数の小数を扱うのが非常に苦手**なのです。

### 1. なぜ誤差が起きるのか？（原理）
私たちは10進数（0〜9）を使いますが、コンピュータは2進数（0と1）で考えます。
10進数の「0.1」を2進数に変換しようとすると、実は **割り切れずに無限に続く「循環小数」** になってしまいます。

$$0.1_{10} = 0.0001100110011..._2$$

コンピュータはメモリ（保存場所）が有限なので、この無限に続く数字を途中で「えいや！」と切り捨てて保存します。これが**誤差の正体**です。

<br>
### 2. SQLで起きる「怪奇現象」
浮動小数点型（`FLOAT`）を使っていると、以下のようなことが起こります。

<br>
#### ① 「0.1 + 0.2」が「0.3」にならない
多くのシステムで `0.1 + 0.2` を計算すると、内部的には `0.30000000000000004` のような値になります。

<br>
#### ② 「=（イコール）」で検索してもヒットしない
「0.3」を保存したつもりでも、実際には「0.30000000000000004」なので、以下のクエリが空振ります。
```sql
-- 保存した値が 0.3 のはずなのに、1件もヒットしない！
SELECT * FROM orders WHERE total_price = 0.3;
```
<br>

#### ③ 合計が合わない
100万件のデータを足し合わせると、0.00000001の誤差が積み重なり、最終的に数円〜数十円のズレとして現れます。

<br>

### 3. 回避策：どうすればいいのか？

実務で誤差を防ぐためのルールはシンプルです。

<br>

#### 対策1：`NUMBER`型を使う
これらは「固定小数点型」と呼ばれ、10進数のまま正確に保存するように設計されています。**お金を扱うカラムには、絶対に `FLOAT` を使ってはいけません。**

*  `NUMBER(10, 2)`（全体10桁、うち小数2桁）

<br>

#### 対策2：整数で保存する
小数を扱わず、あらかじめ「最小単位（例：円なら1円、ドルなら1セント）」を整数として保存する方法です。
* `10.50ドル` → `1050`（セント単位）として保存。
* 表示するときだけ 100 で割る。
これなら誤差は 0% です。

<br>

#### 対策3：比較するときは「範囲」で指定する
どうしても `FLOAT` 型を比較しなければならないときは、`=` を使わず、誤差を許容する書き方をします。
```sql
-- 0.3 に極めて近い値を探す
WHERE price BETWEEN 0.29999 AND 0.30001;
```
