## 期待する結果

| 	----	| 	----	| 	----	| 	----	| 
| 	10	| 	Administration	| 	200	| 	1700	| 
| 	20	| 	Marketing	| 	201	| 	1800	| 
| 	30	| 	Purchasing	| 	114	| 	1700	| 
| 	40	| 	Human Resources	| 	203	| 	2400	| 
| 	50	| 	Shipping	| 	121	| 	1500	| 
| 	60	| 	IT	| 	103	| 	1400	| 
| 	70	| 	Public Relations	| 	204	| 	2700	| 
| 	80	| 	Sales	| 	145	| 	2500	| 
| 	90	| 	Executive	| 	100	| 	1700	| 
| 	100	| 	Finance	| 	108	| 	1700	| 
| 	110	| 	Accounting	| 	205	| 	1700	|

## 解答例
```sql
SELECT
    *
FROM
    hr.departments
WHERE
    manager_id IS NOT NULL
```

## 解説
`NULL` はSQL初心者が最初につまずくポイントであり、ベテランでもうっかりバグを仕込みやすい「要注意」な領域です。

### 1. NULLの正体：「0」でも「空文字」でもなく「不明（Unknown）」
SQLの世界において、`NULL` は値ではなく **不明(Unknown)であるという「状態」** を指します。

#### 三値論理（Three-Valued Logic）
通常の論理は「真（TRUE）」か「偽（FALSE）」の2つですが、SQLは `NULL` を含むため、**「不明（Unknown）」**　を加えた3つの状態を持ちます。
> **鉄則:** `NULL` は何と比較しても（`NULL`でさえ！）結果は `UNKNOWN` になります。
> * `10 = NULL` → **UNKNOWN**
> * `NULL = NULL` → **UNKNOWN**

### 2. 基本的な判定方法：IS NULL / IS NOT NULL
`NULL` は「値」ではないため、 `=` や `<>` といった比較演算子は使えません（正確に言うと使ってもエラーになりませんが、**常に検索結果は0件**です）。専用の判定子を使います。

#### IS NULL（値がないものを探す）
```sql
WHERE 列名 IS NULL
```

#### IS NOT NULL（値があるものを探す）
```sql
WHERE 列名 IS NOT NULL
```

:::message
### **「IS NULL」ではなく「= NULL」と書くと…**
「= NULL」と書くと常に「NO DATA FOUND」（データが見つからない）になる理由は、SQLが採用している **「3値論理（Three-Valued Logic）」** という仕組みにあります。

一言でいうと、**「不明なもの（NULL）と何かを比べても、結果は『わからない（UNKNOWN）』になるから」** です。

詳しく解説します。

#### 1. SQLにおける3つの「答え」
通常のプログラミング言語では、比較の結果は「真（TRUE）」か「偽（FALSE）」の2つだけです。しかし、SQLにはNULLがあるため、結果が**3種類**存在します。
1. **TRUE（真）**: 条件に合致する。
2. **FALSE（偽）**: 条件に合致しない。
3. **UNKNOWN（不明）**: 合致するかどうかが判定できない。

`WHERE`句は、判定結果が **「TRUE」になった行だけ**を表示する仕組みになっています。

#### 2. 「= NULL」の計算プロセス
SQLで `column = NULL` と書いたとき、コンピュータの内部では次のような思考が行われます。
* **値が「10」のカラムと比較した場合：**
「10 は『何かわからないもの（NULL）』と等しいですか？」
→ 答えは **「わかりません（UNKNOWN）」** です。
* **値が「NULL」のカラムと比較した場合：**
「『何かわからないもの』は、別の『何かわからないもの』と等しいですか？」
→ 答えは やはり **「わかりません（UNKNOWN）」** です。同じNULL同士でも、中身が不明である以上「等しい」とは断言できないからです。

#### 3. なぜ「NO DATA FOUND」になるのか
上述の通り、どんな値と比較しても `column = NULL` の評価結果は常に **`UNKNOWN`** になります。
1. データが「10」でも結果は `UNKNOWN`
2. データが「NULL」でも結果は `UNKNOWN`
3. `WHERE`句は **`TRUE` の行だけ**を通す
4. したがって、**どの行も条件を通過できず、1件もヒットしない**

という結果になります。

#### 4. 正解：なぜ「IS NULL」なら動くのか

`IS NULL` は比較演算子（=）ではなく、**「その状態（NULLかどうか）を判定する専用の述語」** です。

* `IS NULL`：「これは『不明（NULL）』という状態ですか？」
→ はい、そうです（TRUE）
* `= NULL`：「これは『不明』という値と同じですか？」
→ わかりません（UNKNOWN）

このように、英語の「Is it...?（〜の状態ですか？）」と「Does it equal...?（〜と同じ値ですか？）」の違いをイメージすると分かりやすくなります。
:::

# 問題4-2（大小比較） *Lv1*
### HRスキーマ`JOBS`テーブルより`MIN_SALARY`が5000以上10000以下のデータを取得してください。（順不同）

## 期待する結果
| 	JOB_ID	| 	JOB_TITLE	| 	MIN_SALARY	| 	MAX_SALARY	| 
| 	----	| 	----	| 	----	| 	----	| 
| 	FI_MGR	| 	Finance Manager	| 	8200	| 	16000	| 
| 	AC_MGR	| 	Accounting Manager	| 	8200	| 	16000	| 
| 	SA_MAN	| 	Sales Manager	| 	10000	| 	20080	| 
| 	SA_REP	| 	Sales Representative	| 	6000	| 	12008	| 
| 	PU_MAN	| 	Purchasing Manager	| 	8000	| 	15000	| 
| 	ST_MAN	| 	Stock Manager	| 	5500	| 	8500	| 
| 	MK_MAN	| 	Marketing Manager	| 	9000	| 	15000	| 

## 解答例
```sql:例1：betweenを使用
SELECT
    *
FROM
    hr.jobs
WHERE
    min_salary BETWEEN 5000 AND 10000
```
```sql:例2：比較演算子を使用
SELECT
    *
FROM
    jobs
WHERE
        min_salary >= 5000
    AND min_salary <= 10000
```

## 解説
### A. 単純比較演算子
SQLにおける**単純比較演算子**は、`WHERE`句などで条件を指定する際に最も頻繁に使うツールです。「この値と等しいか？」「どちらが大きいか？」を判定し、データの選別を行います。

### 1. 主要な比較演算子一覧

| 演算子 | 意味 | 使用例 |
| :--- | :--- | :--- |
| **`=`** | 等しい | `salary = 5000` |
| **`<>`** または **`!=`** | 等しくない | `dept_id <> 10` |
| **`<`** | より小さい | `age < 20` |
| **`>`** | より大きい | `price > 1000` |
| **`<=`** | 以下（その値を含む） | `stock <= 5` |
| **`>=`** | 以上（その値を含む） | `grade >= 80` |


### B. BETWEEN演算子
SQLにおける **`BETWEEN`** 演算子は、ある値が「特定の範囲内にあるかどうか」を判定するために使われます。
`WHERE` 句で「100から200の間」といった条件を指定する際、不等号を2回書くよりもスッキリと記述できるのがメリットです。

### 1. 基本的な構文
```sql
列名 BETWEEN 値1 AND 値2
```
これは内部的に **`列名 >= 値1 AND 列名 <= 値2`** と全く同じ意味として処理されます。

#### 具体的な使用例
| データ型 | SQL例 | 説明 |
| :--- | :--- | :--- |
| **数値** | `WHERE salary BETWEEN 3000 AND 5000` | 給与が3,000以上5,000以下 |
| **文字列** | `WHERE name BETWEEN 'A' AND 'C'` | Aで始まる名前からCで始まる名前まで |
| **日付** | `WHERE hire_date BETWEEN '2026-01-01' AND '2026-03-31'` | 2026年1月〜3月の入社 |


### 2. 注意すべき「4つの落とし穴」

#### ① 「境界値」を含みます（閉区間）
`BETWEEN` は **指定した値を両方とも含みます**。
「100より大きく200より小さい（100と200は含まない）」という条件にしたい場合は、`BETWEEN` は使えず、`> 100 AND < 200` と書く必要があります。

#### ② 指定する順序が逆だと「0件」になる
`BETWEEN 小さい値 AND 大きい値` という順序で書かなければなりません。
* ❌ `WHERE salary BETWEEN 5000 AND 3000`
これを実行してもエラーにはなりませんが、**結果は常に0件**になります。SQLは「5000以上、かつ、3000以下」の数値を真面目に探そうとして、そんな数字は存在しないと判断するからです。

#### ③ 日付の「時刻」問題
日付型（TIMESTAMPなど）に対して `BETWEEN` を使うとき、時刻が隠れていることに注意してください。

* **例**: `BETWEEN '2026-03-01' AND '2026-03-17'`
この場合、終了日は **'2026-03-17 00:00:00'** と解釈されます。
つまり、**3月17日の日中に発生したデータ（12:00など）はヒットしません。**

> **対策**: 
> 1. `TRUNC(hire_date)` で時刻を切り落として比較する。
> 2. `BETWEEN '2026-03-01 00:00:00' AND '2026-03-17 23:59:59'` と明示する。

#### ④ NULL の挙動
比較対象の列が **NULL** の場合、`BETWEEN` の判定結果は `UNKNOWN` となり、データは抽出されません。これは他の比較演算子と同じルールです。

### 3. NOT BETWEEN（範囲外の指定）
「指定した範囲に入らないもの」を探すときは `NOT` をつけます。

# 問題4-3（否定）   *Lv1*
### **HRスキーマ`JOB_HISTORY`テーブルより`JOB_ID`が 「ST_CLERK」 でないデータを取得してください。（順不同）**

## 期待する結果
| EMPLOYEE_ID | START_DATE           | END_DATE             | JOB_ID     | DEPARTMENT_ID | 
| ----------- | -------------------- | -------------------- | ---------- | ------------- | 
| 102         | 2011-01-13T00:00:00Z | 2016-07-24T00:00:00Z | IT_PROG    | 60            | 
| 101         | 2007-09-21T00:00:00Z | 2011-10-27T00:00:00Z | AC_ACCOUNT | 110           | 
| 101         | 2011-10-28T00:00:00Z | 2015-03-15T00:00:00Z | AC_MGR     | 110           | 
| 201         | 2014-02-17T00:00:00Z | 2017-12-19T00:00:00Z | MK_REP     | 20            | 
| 200         | 2005-09-17T00:00:00Z | 2011-06-17T00:00:00Z | AD_ASST    | 90            | 
| 176         | 2016-03-24T00:00:00Z | 2016-12-31T00:00:00Z | SA_REP     | 80            | 
| 176         | 2017-01-01T00:00:00Z | 2017-12-31T00:00:00Z | SA_MAN     | 80            | 
| 200         | 2012-07-01T00:00:00Z | 2016-12-31T00:00:00Z | AC_ACCOUNT | 90            | 

## 解答例
```sql: 例1：<> を使用
SELECT
    *
FROM
    hr.job_history
WHERE
    job_id <> 'ST_CLERK'
```

```sql:例2：!= を使用
SELECT
    *
FROM
    hr.job_history
WHERE
    job_id != 'ST_CLERK'
```
```sql:例3：^= を使用
SELECT
    *
FROM
    hr.job_history
WHERE
    job_id ^= 'ST_CLERK'
```


## 解説
SQLにおける「等しくない」を表す3つの演算子は、**機能的にはすべて同じ**です。どれを使っても実行結果やパフォーマンスに差はありません。
しかし、**「標準規格」**や**「移植性（他のDBでも動くか）」** の観点で明確な違いがあります。

### 1. 比較まとめ表

| 演算子 | 名称 | 標準規格 | 移植性（他のDB） | 特徴 |
| :--- | :--- | :--- | :--- | :--- |
| **`<>`** | アングルブラケット | **ISO標準** | **最強** | どのDBでも動く。最も推奨される。 |
| **`!=`** | エクスクラメーション | 非標準 | 高い | C言語やJavaに慣れた人が好む。 |
| **`^=`** | キャレット | 非標準 | **低い** | 主にOracle。他では動かないことが多い。 |

### 2. 各演算子の詳細

#### ① `<>` (標準の「等しくない」)
SQLの最も古い標準（ISO規格）で定義されている演算子です。
* **メリット**: SQL Server, MySQL, PostgreSQL, DB2など、ほぼ全てのRDBMSで共通して使えます。
* **推奨**: 現場のコーディング規約で特に指定がない限り、**これを使うのが一番安全**です。

#### ② `!=` (プログラミング言語風)
多くのプログラミング言語（C, Java, Pythonなど）で使われる形式です。
* **メリット**: エンジニアにとって直感的で読みやすい。
* **注意**: ほとんどの主要なDBでサポートされていますが、厳密な標準ではないため、古いシステムや特殊な環境では動かないリスクが（極めて低いですが）あります。

#### ③ `^=` (Oracleなどの独自形式)
Oracleなどの特定のデータベースでサポートされている形式です。
* **メリット**: 特になし（好みの問題）。
* **デメリット**: **移植性が最悪**です。OracleからPostgreSQLやMySQLへ移行しようとした際、この演算子を使っているとすべてのクエリを書き直す羽目になります。

### 3. 共通の注意点
どの演算子を使っても、以下の **「SQLの否定の罠」** からは逃れられません。

#### 1. NULLは「等しくない」にも含まれない

```sql
-- 部署が「10以外」の人を探す
WHERE dept_id <> 10
```
この時、`dept_id` が **NULL の人はヒットしません**。
「NULLは10ではないはずだ」と思っても、SQLの世界では `NULL <> 10` は「不明（UNKNOWN）」となり、結果から除外されます。

#### 2. インデックスが効きにくい
「等しい（`=`）」検索は特定のデータへ一直線に向かえますが、「等しくない」検索は「それ以外のすべて」を探す必要があるため、**フルテーブルスキャン（全件検索）** になりがちです。
大量のデータを扱うテーブルで否定演算子を多用すると、パフォーマンスがガクッと落ちる原因になります。

# 問題4-4（いずれかに一致） *Lv1*
### **HRスキーマ`COUNTRIES`テーブルより、`REGION_ID`が40または50のデータを取得してください。（順不同）**

## 期待する結果
| COUNTRY_ID | COUNTRY_NAME | REGION_ID | 
| ---------- | ------------ | --------- | 
| AU         | Australia    | 40        | 
| EG         | Egypt        | 50        | 
| NG         | Nigeria      | 50        | 
| ZM         | Zambia       | 50        | 
| ZW         | Zimbabwe     | 50        | 

## 解答例
```sql:例1：orを使用
SELECT
    country_id,
    country_name,
    region_id
FROM
    hr.countries
WHERE
    region_id = 40 OR region_id = 50 
```
```sql:例2：inを使用
SELECT
    country_id,
    country_name,
    region_id
FROM
    hr.countries
WHERE
    region_id IN ( 40, 50 )
```
```sql:例3:anyを使用
SELECT
    country_id,
    country_name,
    region_id
FROM
    hr.countries
WHERE
    region_id = ANY( 40, 50 )
```

## 解説
SQLにおける **`IN`**, **`OR`**, **`ANY`** はどれも「複数の条件のいずれかに合致するか」を判定するものですが、得意分野が異なります。

一言でいうと、**「読みやすさの `IN`」「自由度の `OR`」「サブクエリに強い `ANY`」** です。

### 1. IN：リスト形式の決定版
同じカラムに対して「A、B、Cのどれか」と指定する場合に最も適しています。

* **特徴**: コードが短く、読みやすい。
* **得意な場面**: 決まった値のリスト（定数）や、サブクエリの結果と比較する場合。

```sql
-- 読みやすくスッキリ！
SELECT * FROM employees
WHERE department_id IN (10, 20, 30);
```
### 2. OR：自由な組み合わせの達人
`IN` と違い、**「異なるカラム」**　や　**「異なる比較演算子」**　を組み合わせることができます。

* **特徴**: どんな条件でもつなげられるが、同じカラムの比較を並べるとコードが長くなる。
* **得意な場面**: 「部署が10、または給与が5000以上」のように、バラバラな条件をまとめるとき。

```sql
-- INでは書けない「別カラム」の組み合わせ
SELECT * FROM employees
WHERE department_id = 10 
   OR salary > 5000;
```

### 3. ANY：サブクエリとの比較の達人
主にサブクエリと一緒に使い、比較演算子（`=`, `>`, `<` など）と組み合わせて使います。

* **特徴**: `= ANY` は `IN` と全く同じ動きをしますが、`> ANY`（どれか一つより大きい）といった範囲比較ができるのが強み。
* **得意な場面**: 「他の部署の誰か一人よりも給与が高い」といった、動的な比較。

```sql
-- 他の誰か一人（ANY）よりも給与が高い人を抽出
SELECT * FROM employees
WHERE salary > ANY (SELECT salary FROM employees WHERE department_id = 20);
```
### 4. 結局どれを使うべき？（使い分けガイド）

| シチュエーション | 推奨 | 理由 |
| --- | --- | --- |
| **同じカラム**を複数の値と比較する | **`IN`** | 最も読みやすく、ミスが少ない。 |
| **複数のカラム**を条件に含める | **`OR`** | `IN` では不可能なため。 |
| サブクエリの結果と **`=`** で比較する | **`IN`** | 慣習的に `IN` が使われることが多い。 |
| サブクエリの結果と **`>` や `<**` で比較する | **`ANY`** | `IN` では範囲比較ができないため。 |

**今回の場合は、「in」を使うのがよさそうですね。**


### ⚠️ 注意点：NULL の挙動
* `IN (1, 2, NULL)` のようにリストの中に NULL がある場合、一致する値があれば `TRUE` ですが、どれにも一致しない場合は `FALSE` ではなく `UNKNOWN` になります。
* 特に `NOT IN` を使う際にリストの中に NULL が1つでもあると、**結果が常に 0件になる** という恐ろしい性質があります。

# 問題4-5（複数の条件） *Lv1*
### HRスキーマ`EMPLOYEES`テーブルより、下記条件全てを満たすデータを取得してください。（順不同）
* **`JOB_ID' が 'IT_PROG'**
* **`SALARY` が　4500**
* **`MANAGER_ID` が 102でない**

## 期待する結果
| NAME           | JOB_ID  | SALARY | MANAGER_ID | 
| -------------- | ------- | ------ | ---------- | 
| Bruce Miller   | IT_PROG | 6000   | 103        | 
| David Williams | IT_PROG | 4800   | 103        | 
| Valli Jackson  | IT_PROG | 4800   | 103        | 

## 解答例
```sql
SELECT
    first_name || ' ' || last_name AS name,
    job_id,
    salary,
    manager_id
FROM
    hr.employees
WHERE
        job_id = 'IT_PROG'
    AND salary > 4500
    AND manager_id <> 102
```

## 解説
### 1. AND演算子とは？
`AND` は、複数の条件が **「すべて正しい（真である）」** 場合のみ、その行を抽出する演算子です。条件を増やせば増やすほど、検索結果の行数は少なくなっていきます（絞り込み）。

### 2. 基本的な構文
Oracleの標準的な社員テーブル（`EMP`）を例に、「部門番号が10」**かつ**「給与が3000以上」の社員を探す場合はこう書きます。

```sql
SELECT ENAME, DEPTNO, SAL
FROM EMP
WHERE DEPTNO = 10    -- 条件1
  AND SAL >= 3000;   -- 条件2
```


### 3. 初心者が絶対知っておくべき「3つの注意点」

#### ① 演算子の優先順位（ANDはORより強い）
`AND` と `OR` を混ぜて使うとき、SQLは **「ANDを先に計算する」** というルールがあります。

* **失敗例：** 「部門10 または 部門20」で、かつ「給与が2000以上」の人を探したい
    ```sql
    -- これだと「部門10の人全員」と「部門20の給与2000以上の人」が出てしまう
    WHERE DEPTNO = 10 OR DEPTNO = 20 AND SAL >= 2000
    ```
* **解決策：** 算数と同じで **カッコ `( )`** を使い、優先順位を明示します。
    ```sql
    WHERE (DEPTNO = 10 OR DEPTNO = 20) AND SAL >= 2000
    ```

#### ② NULLが含まれる場合の挙動
SQLの論理には「真」「偽」のほかに「不明（UNKNOWN/NULL）」があります。
* `TRUE AND NULL` $\rightarrow$ **UNKNOWN**（抽出されない）
* `FALSE AND NULL` $\rightarrow$ **FALSE**

「データが入っていない（NULL）」行は、AND条件に含めた時点で検索対象から外れる可能性が高いので注意が必要です。

#### ③ 条件の並び順とパフォーマンス
以前のOracleでは「重い条件（時間がかかる処理）を後ろに書く」などのコツがありましたが、現在のOracle 23aiのオプティマイザ（賢い司令塔）は非常に優秀です。
基本的には **「人間が読んで分かりやすい順番」** で書いて問題ありません。オプティマイザが自動で効率の良い順序に並び替えて実行してくれます。

### 4. Oracle 23aiでの新常識：「Boolean型」
Oracle 23aiから、 **Boolean型（TRUE / FALSE）** がサポートされました。これにより、AND条件の書き方がより直感的になります。

```sql
-- 23ai以前：フラグカラムを 'Y' などと比較
WHERE IS_ACTIVE = 'Y' AND SAL > 5000

-- 23ai以降：Boolean型のカラムならそのまま書ける
WHERE IS_ACTIVE AND SAL > 5000
```
※ `IS_ACTIVE` 自体が TRUE か FALSE を持っているため、比較演算子なしで `AND` に組み込めます。

:::message
### **「WHERE 1=1」の謎**
あるプロジェクトで見たSQLスクリプトに「WHERE 1 = 1」と書かれており、何のことか全く理解できず、「これ何ですか？」と聞たことを思い出します。

SQLで `WHERE 1=1` と書くのは、一見無意味（常に正しい条件）に見えますが、実は**プログラムで動的にSQLを組み立てる際の「定石」** として非常に重宝されています。

主な理由は、SQL文を「文字列として連結して作る」ときのコードをシンプルにするためです。
### 1. 動的SQLの組み立てを楽にするため
プログラムで「検索条件がある場合だけ `WHERE` 句を追加する」という処理を書く場面を想像してみてください。
<br>
#### * **`1=1` がない場合**

最初の条件だけは `WHERE` を使い、2つ目以降は `AND` を使う……という「条件分岐」の判定が必要になります。

```sql
-- 最初の条件だけ WHERE で始める必要がある
SELECT * FROM users WHERE name = '田中' AND age >= 20;

-- もし名前の指定がなかったら？
SELECT * FROM users WHERE age >= 20; -- 常に「最初が何か」を気にしないといけない
```
<br>

#### * `1=1` がある場合

**「常に `WHERE 1=1` から始める」** と決めておけば、その後に続く条件はすべて `AND` でつなげるだけで済みます。

```sql
SELECT * FROM users 
WHERE 1=1 
  AND name = '田中' 
  AND age >= 20;
```

プログラム側では、条件がいくつあっても「`" AND column = value"` を末尾にくっつけるだけ」で済むので、ロジックが非常にシンプルになります。

<br>

### 2. デバッグ（コメントアウト）が楽になる
SQLを直接書いてテストしているとき、特定の条件を無効化したいことがありますよね。`1=1` があれば、行頭をコメントアウトするだけで済みます。

```sql
SELECT * FROM products
WHERE 1=1
-- AND category = '家電'  <-- ここを消してもWHERE句が壊れない
   AND price <= 5000
```

もし `1=1` がないと、最初の条件を消すときに `WHERE` というキーワード自体も書き直さなければならず、手間がかかります。

<br>
私は目安として、以下の場合好んで「`WHERE 1 = 1`」を書くようにしています。
（いずれか）
- 複雑で巨大なSQLスクリプト
- 条件が３つ以上ある
- 自身が使用するテスト用スクリプト


:::

# 問題4-6（あいまい検索1） *Lv1*
### HRスキーマ`EMPLOYEES`テーブルより、`EMAIL`に「AB」の文字列が含まれるデータを取得してください。（順不同）

## 期待する結果
| NAME           | EMAIL   | 
| -------------- | ------- | 
| Ellen Abel     | EABEL   | 
| Amit Banda     | ABANDA  | 
| Alexis Bull    | ABULL   | 
| Anthony Cabrio | ACABRIO | 

## 解答例
```sql
SELECT
    first_name || ' ' || last_name AS name,
    email
FROM
    hr.employees
WHERE
    email LIKE '%AB%'
```

## 解説
特定の文字を含むデータを検索したい場合、`LIKE` 演算子を使用します。その際、**「%（パーセント）」**という記号を使い、「ここは任意の文字が入る」という指定を行います。

この `%` を置く位置によって、検索のパターンは大きく3つに分けられます。

##### 1. 前方一致（指定した文字で始まる）

文字列の**最初**が一致しているものを検索します。

* **書き方**: `列名 LIKE 'かき%'`
* **イメージ**: 「かき」で始まり、その後に何らかの文字が続く（または何も続かない）もの。
* **例**: 「かきのたね」、「かき氷」

##### 2. 中間一致（指定した文字をどこかに含む）

文字列の**どこか**に指定した文字が含まれているものを検索します。

* **書き方**: `列名 LIKE '%かき%'`
* **イメージ**: 前後に何があっても良く、とにかく「かき」という文字が入っていればヒットします。
* **例**: 「あかかきいろ（赤、柿、黄色）」、「かきくけこ」、「おかき」

##### 3. 後方一致（指定した文字で終わる）

文字列の**最後**が一致しているものを検索します。

* **書き方**: `列名 LIKE '%かき'`
* **イメージ**: 前には何があっても良く、最後が「かき」で終わるもの。
* **例**: 「おかき」、「干しがき」

# 問題4-7（あいまい検索2） *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、`EMAIL`が「S」から始まり「S」で終わる７文字のデータを取得してください。（順不同）

## 期待する結果
| NAME           | EMAIL   | 
| -------------- | ------- | 
| Susan Jacobs   | SJACOBS | 
| Stephen Stiles | SSTILES | 
| Sigal Tobias   | STOBIAS | 

## 解答例
```sql
SELECT
    first_name || ' ' || last_name AS name,
    email
FROM
    hr.employees
WHERE
    email LIKE 'S_____S'
```

## 解説
`LIKE` 演算子を使用するにあたり、「%」は「0文字以上の任意の文字列」を指しますが、**「 _ 」は「任意の1文字（ちょうど1文字）」**　を指します。


### ワイルドカード「 _ 」の使い方

「 _ 」は、検索したい文字数が決まっている場合に非常に強力な力を発揮します。

### 1. 「1文字」という制約をつける
例えば、ある列から「A」で始まる**2文字のデータ**だけを抽出したい場合、以下のように記述します。

* **書き方**: `WHERE 列名 LIKE 'A_'`
* **ヒットする例**: 「AB」、「A1」、「Aあ」
* **ヒットしない例**: 「A」（1文字しかない）、「ABC」（3文字ある）

### 2. 特定の位置にある文字を指定する
「何文字目かにこの文字が入っている」という条件を指定するのにも便利です。

* **例：2文字目が「山」である3文字のデータを検索**
* **書き方**: `WHERE 列名 LIKE '_山_'`
* **ヒットする例**: 「小山氏」、「青山君」
* **ヒットしない例**: 「山田」（2文字しかない）、「和歌山県」（5文字ある）


### 「%」と「 _ 」の比較表
使い分けを整理すると以下のようになります。

| 記号 | 意味 | 検索のイメージ |
| --- | --- | --- |
| **`%`** | **0文字以上**の任意文字 | 長さを問わない（「かき」でも「かき氷」でもOK） |
| **`_`** | **ちょうど1文字**の任意文字 | 1文字分としてカウントする（文字数を固定する） |

# 問題4-8（SELECT文での条件） *Lv2*
### HRスキーマ `COUNTRIES` テーブルより、下記条件にて、「地域」を算出してください。
* **`REGION_ID`=10または20** : 欧米
* **`REGION_ID`=30または50** : アジア・アフリカ
* **上記以外** : その他
* **表示対象は、`COUNTRY_ID`の頭文字がA～Cのもの**
  
## 期待する結果
| COUNTRY_ID | COUNTRY_NAME | REGION_ID | 地域             | 
| ---------- | ------------ | --------- | ---------------- | 
| AR         | Argentina    | 20        | 欧米             | 
| AU         | Australia    | 40        | その他           | 
| BE         | Belgium      | 10        | 欧米             | 
| BR         | Brazil       | 20        | 欧米             | 
| CA         | Canada       | 20        | 欧米             | 
| CH         | Switzerland  | 10        | 欧米             | 
| CN         | China        | 30        | アジア・アフリカ | 

## 解答例
```sql:例1：検索 CASE 式
SELECT
    country_id,
    country_name,
    region_id,
    CASE
        WHEN region_id IN ( 10, 20 ) THEN
            '欧米'
        WHEN region_id IN ( 30, 50 ) THEN
            'アジア・アフリカ'
        ELSE
            'その他'
    END AS "地域"
FROM
    hr.countries
WHERE
    country_id BETWEEN 'A' AND 'CZ'
```
```sql:例2：単純 CASE 式
SELECT
    country_id,
    country_name,
    region_id,
    CASE region_id
        WHEN 10 THEN
            '欧米'
        WHEN 20 THEN
            '欧米'
        WHEN 30 THEN
            'アジア・アフリカ'
        WHEN 50 THEN
            'アジア・アフリカ'
        ELSE
            'その他'
    END AS "地域"
FROM
    hr.countries
WHERE
    country_id BETWEEN 'A' AND 'CZ'
```

## 解説
SQLにおける**CASE式**は、プログラミング言語での「if-then-else」のような条件分岐をクエリ内で実現するための非常に強力なツールです。

「もしこの条件に当てはまるならA、そうでなければB」といったラベル付けや集計が、SELECT文の中で自由自在に行えるようになります。

### 1. CASE式の2つの書き方

CASE式には、用途に合わせて「単純CASE式」と「検索CASE式」の2種類があります。

#### ① 単純CASE式（例2）
特定の列の値を、具体的な値と比較する場合に使います。


#### ② 【推奨】検索CASE式（例1）
より柔軟な条件（不等号や複数の列の組み合わせなど）を指定できます。実務ではこちらを使うことがほとんどです。


### 2. 使用上の3つの注意点
CASE式を使いこなすために、以下のポイントには特に気をつけてください。

#### ① 「評価順序」に注意：早い者勝ち
CASE式は、**上から順番に評価され、最初に一致した条件で処理を終了します。**
もし複数の条件に当てはまる可能性がある場合は、**範囲の狭い（厳しい）条件から先に書く**のが鉄則です。

> **例**: 80点以上の人に「優」を付けたいのに、先に `WHEN score >= 0 THEN '可'` と書いてしまうと、全員が「可」になってしまいます。

#### ② 「ELSE」は省略しないのが無難
`ELSE` を省略し、どの `WHEN` 条件にも一致しなかった場合、結果は **`NULL`** になります。
意図しない `NULL` の混入を防ぐため、デフォルト値を `ELSE` で明示的に指定しておくことを強くお勧めします。

#### ③ 「データ型」を一致させる
`THEN` や `ELSE` で返す値のデータ型は、**すべて同じ（あるいは互換性がある）**　必要があります。不一致があると、実行時にエラーが発生します。

* ⭕️：すべて「文字列」を返す
* ❌：ある条件では「数値」を返し、別の条件では「日付」を返す

# 問題4-9（指定件数のデータ取得）
### HRスキーマ`EMPLOYEES`テーブルより、任意の5件のデータを取得してください。（順不同）

## 期待する結果
| EMPLOYEE_ID | FIRST_NAME | LAST_NAME | 
| ----------- | ---------- | --------- | 
| 100         | Steven     | King      | 
| 101         | Neena      | Yang      | 
| 102         | Lex        | Garcia    | 
| 103         | Alexander  | James     | 
| 104         | Bruce      | Miller    | 

※任意の５件が取得できていればOK

## 解答例
```sql:例1：ROWNUMを使用
SELECT
    employee_id,
    first_name,
    last_name
FROM
    hr.employees
WHERE
    ROWNUM <= 5
```
```sql:例2：FETCH FIRSTを使用
SELECT
    employee_id,
    first_name,
    last_name
FROM
    hr.employees
--ORDER BY     -- 通常、ORDER BYをつけますが、要件から、ここでは敢えてコメントアウトしています
--    employee_id
FETCH FIRST 5 ROWS ONLY
```

## 解説
Oracle SQLにおける **`ROWNUM`** は、問い合わせの結果として返される各行に対して、上から順番に振られる「擬似列（疑似的な番号）」です。
非常に便利ですが、**「いつ番号が振られるのか」** という内部の動きを理解していないと、意図しない結果（あるいは0件の結果）を招く、少しクセのあるツールです。

### 1. ROWNUM の基本的な使い方（例1）
主な用途は、抽出件数の制限（「最初の5件だけ欲しい」など）です。

### 2. 覚えておくべき「超」重要な注意点

#### ① `ROWNUM = 2` は1件も取れない
ここが最大の落とし穴です。「2行目だけ欲しい」と思って `WHERE ROWNUM = 2` や `WHERE ROWNUM > 2`と書いても、結果は必ず **0件** になります。

**理由:**
`ROWNUM` は、条件に合致して「抽出が確定した行」に対して **1から順番に** 割り当てられます。
1.  最初の行をチェックする。まだ確定した行がないので、この行の候補番号は `1`。
2.  `WHERE ROWNUM = 2` という条件を見る。`1 = 2` は偽なので、この行は捨てられる。
3.  次の行をチェックする。まだ確定した行がないので、この行の候補番号は**再び `1` になる**。
4.  永遠に `1` しか現れないため、条件 `2` にたどり着くことができません。

> **解決策:**
> 2行目以降が欲しい場合は、一度サブクエリで `ROWNUM` に別名を付けて確定させてから、外側で絞り込む必要があります。

#### ② `ORDER BY` と組み合わせる時の罠
`ROWNUM` は **`ORDER BY` による並べ替えの前に** 割り振られます。

```sql
-- ❌ 意図通りに動かない例
-- 「バラバラに5件取ってきてから、それを日付順に並べる」という動きになる
SELECT * FROM oe.orders 
WHERE ROWNUM <= 5 
ORDER BY order_date DESC;
```

「最新の注文5件」を取得したいなら、先に並べ替えてから番号を振る必要があります。

```sql
-- ⭕ 正しい例（サブクエリを使う）
SELECT * FROM (
    SELECT * FROM oe.orders ORDER BY order_date DESC
)
WHERE ROWNUM <= 5;
```

### 3. モダンな書き方：FETCH FIRST 句 (12c以降)（例2）
Oracle 12c以降を使っているなら、`ROWNUM` の落とし穴を気にせず書ける **`FETCH FIRST`** 句が推奨されます。こちらの方が可読性が高く、並べ替えも直感的に行えます。

# 問題4-10（指定範囲のデータ取得）
### HRスキーマ`EMPLOYEES`テーブルを`EMPLOYEE_ID`の昇順の並べて先頭から10件スキップし、5件のデータを取得してください。（順不同）

## 期待する結果
| EMPLOYEE_ID | FIRST_NAME  | LAST_NAME | 
| ----------- | ----------- | --------- | 
| 110         | John        | Chen      | 
| 111         | Ismael      | Sciarra   | 
| 112         | Jose Manuel | Urman     | 
| 113         | Luis        | Popp      | 
| 114         | Den         | Li        | 

## 解答例
```sql
SELECT
    employee_id,
    first_name,
    last_name
FROM
    hr.employees
ORDER BY
    employee_id
OFFSET 10 ROWS
FETCH NEXT 5 ROWS ONLY
```

## 解説
Oracle 12cから導入された **`FETCH FIRST`** 句（正式には行制限句：Row Limiting Clause）は、従来の `ROWNUM` を使った複雑なサブクエリを過去のものにする、非常に直感的で強力な機能です。

「最初の○件だけ取得したい」「ページング（○件目から○件表示）をしたい」といった処理が、標準SQLに近い書き方でスッキリ記述できます。

### 1. 基本的な書き方
もっともシンプルな「最初の5件を取得する」例です。

```sql
SELECT * FROM oe.orders
ORDER BY order_date DESC
FETCH FIRST 5 ROWS ONLY;
```

#### 構文のバリエーション
* **`ONLY`**: 指定した件数だけを厳密に返します。
* **`WITH TIES`**: 最後の行と同じ値を持つデータ（同着）があれば、それらもすべて含めて返します。
* **`PERCENT`**: 件数ではなく「全体の○％」という指定も可能です。


### 2. ページング処理（OFFSET）
`FETCH FIRST` の真骨頂は、**`OFFSET`** と組み合わせたページング処理です。「11件目から10件分取得」といった処理が1行で書けます（解答例参照）。


### 3. ROWNUM との決定的な違い

| 特徴 | ROWNUM | FETCH FIRST |
| :--- | :--- | :--- |
| **並べ替えとの順序** | 先に番号が振られ、後でソートされる（罠がある） | **ソートされた後**に件数が制限される（直感的） |
| **サブクエリ** | 必須（ソート後に制限する場合） | **不要**（1つのSELECT文で完結） |
| **同着の扱い** | 考慮できない | `WITH TIES` で簡単に取得可能 |
| **可読性** | 構造が複雑になりがち | 非常に読みやすく、意図が明確 |


### 4. 注意点と使いこなしのコツ

#### ① パフォーマンスの落とし穴（深いOFFSET）
`OFFSET 1000000 ROWS FETCH NEXT 10 ROWS ONLY` のように、巨大な数値をスキップさせる場合、データベースは内部的に**100万行分を読み飛ばす（スキャンする）**必要があります。
件数が多いテーブルで深いページを指定すると、後半のページほどレスポンスが悪くなる可能性があるため注意が必要です。

#### ② ORDER BY は「ほぼ必須」
`ORDER BY` を指定せずに `FETCH FIRST` を使うと、どのデータが「最初」になるかはデータベースの気分次第（非決定的）になります。常に同じ結果を得るためには、必ず一意な列（プライマリキーなど）を含めた `ORDER BY` を書くようにしましょう。


----
