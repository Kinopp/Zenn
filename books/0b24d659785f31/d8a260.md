# 問題4-1（NULL）  *Lv1*
### **HRスキーマ`DEPARTMENTS表`より、`MANAGER`が存在する（`MANAGER_ID`が`NULL`でない）データを取得してください。（順不同）**

## 期待する結果
| 	DEPARTMENT_ID	| 	DEPARTMENT_NAME	| 	MANAGER_ID	| 	LOCATION_ID	| 
| 	----	| 	----	| 	----	| 	----	| 
| 	10	| 	Administration	| 	200	| 	1700	| 
| 	20	| 	Marketing	| 	201	| 	1800	| 
| 	30	| 	Purchasing	| 	114	| 	1700	| 
| 	40	| 	Human Resources	| 	203	| 	2400	| 
| 	50	| 	Shipping	| 	121	| 	1500	| 
| 	60	| 	IT	| 	103	| 	1400	| 
| 	70	| 	Public Relations	| 	204	| 	2700	| 
| 	80	| 	Sales	| 	145	| 	2500	| 
| 	90	| 	Executive	| 	100	| 	1700	| 
| 	100	| 	Finance	| 	108	| 	1700	| 
| 	110	| 	Accounting	| 	205	| 	1700	|

## 解答例
```sql
SELECT
    *
FROM
    hr.departments
WHERE
    manager_id IS NOT NULL
```

## 解説
今回は、データベース学習において避けては通れない最重要テーマの一つ、**「NULL（ヌル）の扱い」** に関する演習です。

このクエリーは「マネージャー（管理職）が割り当てられている部門だけを抽出する」というものですが、一見シンプルに見えて、実はSQLの深いルールが隠れています。

### 動作のポイント
* **`WHERE manager_id IS NOT NULL`**: 
    `manager_id` 列に値が入っている（空ではない）行だけを抽出します。
* **`IS NOT NULL` の意味**: 
    SQLの世界では、データがない状態を `NULL` と呼びます。`IS NOT NULL` は「何らかの値が確実に存在すること」を確認する条件式です。

### 実践的な注意点：NULLの落とし穴

SQL初心者から中級者になれるかどうかの境目は、この「NULL」を正しく扱えるかどうかにあります。

#### 1. `!= NULL` と書いてはいけない
最も多い間違いが、`WHERE manager_id != NULL` と書いてしまうことです。
* **理由**: SQLにおいて `NULL` は「値」ではなく、**「未知（不明）」という状態**です。
* 未知のものと何かを比較（= や !=）しても、結果は「不明（UNKNOWN）」となり、1件もヒットしません（詳細は下記参照）。
* そのため、必ず **`IS NULL`** または **`IS NOT NULL`** という専用の述語を使う必要があります。

#### 2. 空文字との違い（Oracle特有の挙動）
ここはOracle SQLを扱う上で**非常に重要**なポイントです。
* **多くのDB（MySQL, PostgreSQLなど）**: `NULL` と `''`（長さ0の文字列）を区別します。
* **Oracle Database**: **`''`（空文字）を `NULL` として扱います。**
    * そのため、文字列のカラムに対して「未入力」を探す場合も `IS NULL` を使います。

#### 3. NULLは「計算」を破壊する
NULLが含まれるデータで計算を行うと、予想外の結果になります。
* `100 + NULL` ＝ **`NULL`**
* `salary * commission_pct`（歩合給がNULLの場合） ＝ **`NULL`**
* 集計関数（`SUM` や `AVG`）では無視されますが、四則演算では結果がすべてNULLに飲み込まれてしまいます。


### 比較まとめ：NULL判定の書き方

| 目的 | 正しい書き方 | 間違った書き方（NG） |
| :--- | :--- | :--- |
| **空のデータを探す** | `IS NULL` | `= NULL` |
| **値があるデータを探す** | `IS NOT NULL` | `!= NULL` |
| **空文字を探す(Oracle)** | `IS NULL` | `= ''` |


今回の `hr.departments` テーブルで言えば、例えば「社長直轄の部門」などはマネージャーが設定されておらず、`manager_id` が NULL になっているケースがあります。そういったデータを除外して「現場の組織図」を作りたい時に、このクエリーが役立ちます。


<br>
:::message
### **「IS NULL」ではなく「= NULL」と書くと…**
「= NULL」と書くと常に「NO DATA FOUND」（データが見つからない）になる理由は、SQLが採用している **「3値論理（Three-Valued Logic）」** という仕組みにあります。

一言でいうと、**「不明なもの（NULL）と何かを比べても、結果は『わからない（UNKNOWN）』になるから」** です。

詳しく解説します。

#### 1. SQLにおける3つの「答え」
通常のプログラミング言語では、比較の結果は「真（TRUE）」か「偽（FALSE）」の2つだけです。しかし、SQLにはNULLがあるため、結果が**3種類**存在します。
1. **TRUE（真）**: 条件に合致する。
2. **FALSE（偽）**: 条件に合致しない。
3. **UNKNOWN（不明）**: 合致するかどうかが判定できない。

`WHERE`句は、判定結果が **「TRUE」になった行だけ**を表示する仕組みになっています。

#### 2. 「= NULL」の計算プロセス
SQLで `column = NULL` と書いたとき、コンピュータの内部では次のような思考が行われます。
* **値が「10」のカラムと比較した場合：**
「10 は『何かわからないもの（NULL）』と等しいですか？」
→ 答えは **「わかりません（UNKNOWN）」** です。
* **値が「NULL」のカラムと比較した場合：**
「『何かわからないもの』は、別の『何かわからないもの』と等しいですか？」
→ 答えは やはり **「わかりません（UNKNOWN）」** です。同じNULL同士でも、中身が不明である以上「等しい」とは断言できないからです。

#### 3. なぜ「NO DATA FOUND」になるのか
上述の通り、どんな値と比較しても `column = NULL` の評価結果は常に **`UNKNOWN`** になります。
1. データが「10」でも結果は `UNKNOWN`
2. データが「NULL」でも結果は `UNKNOWN`
3. `WHERE`句は **`TRUE` の行だけ**を通す
4. したがって、**どの行も条件を通過できず、1件もヒットしない**

という結果になります。

#### 4. 正解：なぜ「IS NULL」なら動くのか

`IS NULL` は比較演算子（=）ではなく、**「その状態（NULLかどうか）を判定する専用の述語」** です。

* `IS NULL`：「これは『不明（NULL）』という状態ですか？」
→ はい、そうです（TRUE）
* `= NULL`：「これは『不明』という値と同じですか？」
→ わかりません（UNKNOWN）

このように、英語の「Is it...?（〜の状態ですか？）」と「Does it equal...?（〜と同じ値ですか？）」の違いをイメージすると分かりやすくなります。
:::

# 問題4-2（大小比較） *Lv1*
### HRスキーマ`JOBS`テーブルより`MIN_SALARY`が5000以上10000以下のデータを取得してください。（順不同）

## 期待する結果
| 	JOB_ID	| 	JOB_TITLE	| 	MIN_SALARY	| 	MAX_SALARY	| 
| 	----	| 	----	| 	----	| 	----	| 
| 	FI_MGR	| 	Finance Manager	| 	8200	| 	16000	| 
| 	AC_MGR	| 	Accounting Manager	| 	8200	| 	16000	| 
| 	SA_MAN	| 	Sales Manager	| 	10000	| 	20080	| 
| 	SA_REP	| 	Sales Representative	| 	6000	| 	12008	| 
| 	PU_MAN	| 	Purchasing Manager	| 	8000	| 	15000	| 
| 	ST_MAN	| 	Stock Manager	| 	5500	| 	8500	| 
| 	MK_MAN	| 	Marketing Manager	| 	9000	| 	15000	| 

## 解答例
```sql:例1：betweenを使用
SELECT
    *
FROM
    hr.jobs
WHERE
    min_salary BETWEEN 5000 AND 10000
```
```sql:例2：比較演算子を使用
SELECT
    *
FROM
    hr.jobs
WHERE
        min_salary >= 5000
    AND min_salary <= 10000
```

## 解説
今回は、範囲指定のフィルタリングです。`BETWEEN` を使う方法は、SQLらしく読みやすい「構文の美しさ」がありますが、実はいくつか注意すべきルールがあります。例2の比較演算子との違いを含めて解説します。

### 動作の共通点
どちらのクエリーも、`min_salary`（最低給与）が **5,000以上かつ10,000以下** のデータを抽出します。

* **ポイント**: 境界値（5,000 と 10,000）を**含みます（Inclusive）**。
    * 数学的に言うと $5000 \le x \le 10000$ の範囲です。


### 実践的な注意点

#### 1. 値の「順序」は固定
`BETWEEN` を使う際、必ず **`小さい方の値 AND 大きい方の値`** の順で書かなければなりません。
* **NG例**: `BETWEEN 10000 AND 5000`
    * Oracleはこれを「10,000以上かつ5,000以下」と解釈しようとしますが、そのような数値は存在しないため、**エラーにはなりませんが結果は0件**になります。


## 比較まとめ

| 項目 | `BETWEEN` (例1) | 比較演算子 (例2) |
| :--- | :--- | :--- |
| **可読性** | 直感的で読みやすい | やや冗長だがロジカル |
| **柔軟性** | 常に「以上・以下」に固定 | 「より大きい（>）」などの調整が自在 |
| **実行速度** | **全く同じ** | **全く同じ** |
| **おすすめ** | シンプルな数値の範囲指定 | 日付の範囲や、複雑な境界条件 |

> [!NOTE]
> **「NOT BETWEEN」も便利**
> 逆に「5,000〜10,000以外のデータが欲しい」ときは、`WHERE min_salary NOT BETWEEN 5000 AND 10000` と書くだけでOKです。これも実務の集計でよく使います。

この後は、特定のパターン（「A」で始まる名前など）を検索する `LIKE` 演算子や、複数の値を指定する `IN` 演算子などの学習に進まれますか？

# 問題4-3（否定）   *Lv1*
### **HRスキーマ`JOB_HISTORY`テーブルより`JOB_ID`が 「ST_CLERK」 でないデータを取得してください。（順不同）**

## 期待する結果
| EMPLOYEE_ID | START_DATE           | END_DATE             | JOB_ID     | DEPARTMENT_ID | 
| ----------- | -------------------- | -------------------- | ---------- | ------------- | 
| 102         | 2011-01-13T00:00:00Z | 2016-07-24T00:00:00Z | IT_PROG    | 60            | 
| 101         | 2007-09-21T00:00:00Z | 2011-10-27T00:00:00Z | AC_ACCOUNT | 110           | 
| 101         | 2011-10-28T00:00:00Z | 2015-03-15T00:00:00Z | AC_MGR     | 110           | 
| 201         | 2014-02-17T00:00:00Z | 2017-12-19T00:00:00Z | MK_REP     | 20            | 
| 200         | 2005-09-17T00:00:00Z | 2011-06-17T00:00:00Z | AD_ASST    | 90            | 
| 176         | 2016-03-24T00:00:00Z | 2016-12-31T00:00:00Z | SA_REP     | 80            | 
| 176         | 2017-01-01T00:00:00Z | 2017-12-31T00:00:00Z | SA_MAN     | 80            | 
| 200         | 2012-07-01T00:00:00Z | 2016-12-31T00:00:00Z | AC_ACCOUNT | 90            | 

## 解答例
```sql: 例1：<> を使用
SELECT
    *
FROM
    hr.job_history
WHERE
    job_id <> 'ST_CLERK'
```

```sql:例2：!= を使用
SELECT
    *
FROM
    hr.job_history
WHERE
    job_id != 'ST_CLERK'
```
```sql:例3：^= を使用
SELECT
    *
FROM
    hr.job_history
WHERE
    job_id ^= 'ST_CLERK'
```


## 解説
「〜ではない」という否定の条件です。実は、SQLには「等しくない」を表現する記号が複数存在します。

これらは**機能的にはすべて同じ**ですが、現場での「作法」や、初心者が必ずハマる「NULLの罠」について詳しく解説します。

### 3つの記号の違い
Oracle Databaseでは、以下の3つは全く同じ意味（等しくない）として処理されます。

1.  **`<>` (例1)**: **ISO/ANSI標準規格**の書き方です。最も推奨される形式で、どのデータベース（MySQL, PostgreSQL, SQL Serverなど）でも共通して使えます。
2.  **`!=` (例2)**: 多くのプログラミング言語（C, Java, Pythonなど）と同じ記号です。標準規格ではありませんが、現在ほぼすべての主要なDBでサポートされています。
3.  **`^=` (例3)**: **Oracle特有の古い書き方**です。他のデータベースでは使えないことが多く、現在ではあまり見かけません。

### 実践的な注意点：否定条件の「三重苦」

「等しくない」を使うときは、結果が予想とズレることがよくあります。以下の3点は必ず押さえておきましょう。

#### 1. NULLは「等しくない」にも含まれない
これが最大の注意点です。もし `job_id` が **NULL** の行があったとしても、これら3つのクエリーでは**抽出されません。**
* `job_id <> 'ST_CLERK'` は、「'ST_CLERK' ではない他の何かである」という条件です。
* NULLは「何かわからない状態」なので、比較自体が成立せず、結果から除外されてしまいます。
* **対策**: NULLも含めたい場合は `WHERE job_id <> 'ST_CLERK' OR job_id IS NULL` と書く必要があります。

#### 2. インデックスが効きにくい
データベースの検索を速くする「インデックス」は、一般的に「等しい（=）」の検索には強いですが、**「等しくない（<>）」の検索には弱い**という性質があります。
* 大量のデータが入っているテーブルで否定条件を多用すると、検索スピードが落ちる原因になります。
* 「これ以外」と探すよりも「これとこれ」と肯定（`IN` など）で探せないか検討するのがプロの技です。

#### 3. 表記の統一（コーディング規約）
実務プロジェクトでは「否定は `<>` を使うこと」のようにルールが決まっていることが多いです。
* 個人的に `!=` が好きでも、プロジェクトのルールに合わせるのが「チーム開発」の基本です。迷ったら、標準の **`<>`** を使っておけば間違いありません。


### 比較まとめ

| 記号 | 標準か？ | 他のDBでの利用 | 推奨度 |
| :--- | :--- | :--- | :--- |
| **`<>`** | **○ 標準** | ほぼすべてで利用可 | **◎ (最も推奨)** |
| **`!=`** | × 非標準 | ほぼすべてで利用可 | ○ (一般的) |
| **`^=`** | × 非標準 | Oracle以外では不可 | △ (非推奨) |

<br>
:::message
### 「否定」は人間の脳にも優しくない？
`WHERE job_id <> 'ST_CLERK' AND department_id <> 50` のように否定が重なると、人間はパッと見て「結局どのデータが出るの？」と混乱しやすくなります。条件が複雑になる場合は、あえて「肯定」で書けるように整理すると、バグが減ります。
:::

# 問題4-4（いずれかに一致） *Lv1*
### **HRスキーマ`COUNTRIES`テーブルより、`REGION_ID`が40または50のデータを取得してください。（順不同）**

## 期待する結果
| COUNTRY_ID | COUNTRY_NAME | REGION_ID | 
| ---------- | ------------ | --------- | 
| AU         | Australia    | 40        | 
| EG         | Egypt        | 50        | 
| NG         | Nigeria      | 50        | 
| ZM         | Zambia       | 50        | 
| ZW         | Zimbabwe     | 50        | 

## 解答例
```sql:例1：orを使用
SELECT
    country_id,
    country_name,
    region_id
FROM
    hr.countries
WHERE
    region_id = 40 OR region_id = 50 
```
```sql:例2：inを使用
SELECT
    country_id,
    country_name,
    region_id
FROM
    hr.countries
WHERE
    region_id IN ( 40, 50 )
```
```sql:例3:anyを使用
SELECT
    country_id,
    country_name,
    region_id
FROM
    hr.countries
WHERE
    region_id = ANY( 40, 50 )
```

## 解説
複数の値を指定してデータを絞り込む「多値比較」のバリエーションです。
これらはすべて同じ結果を返しますが、実務では **「どれを使うか」でコードの読みやすさとメンテナンス性が劇的に変わります。**

### クエリーの解説

#### 例1：`OR` を使用
```sql
WHERE region_id = 40 OR region_id = 50
```
* **内容**: 「地域IDが40、または50」という条件を愚直に並べた形です。
* **特徴**: 2〜3個の条件なら問題ありませんが、条件が10個に増えると `OR region_id = ...` が延々と続き、非常に読みづらくなります。

#### 例2：`IN` を使用
```sql
WHERE region_id IN ( 40, 50 )
```
* **内容**: 指定したリスト（40, 50）の中に値が含まれているかを確認します。
* **特徴**: **実務で最も推奨される書き方**です。スッキリしていて、値が増えてもカンマで区切るだけなのでメンテナンスが楽です。

#### 例3：`= ANY` を使用
```sql
WHERE region_id = ANY( 40, 50 )
```
* **内容**: リスト内の「いずれかの値と等しい」ことを示します。
* **特徴**: `IN` と全く同じ動きをしますが、実はこの `ANY` は数値の比較演算子（`>` や `<`）と組み合わせて使うときに真価を発揮する、少しマニアックな構文です。




### 実践的な注意点

#### 1. `IN` 句の最大件数制限
Oracle SQL特有の制限として、`IN` 句の中に直接書ける値の数は **1,000個まで** というルールがあります。
* 1,001個以上の値を指定しようとするとエラーになります。
* 実務で大量のIDを指定したい場合は、「サブクエリ（別のSELECT文の結果を使う）」にするか、一時テーブルを使うといった工夫が必要になります。

#### 2. `NULL` との相性
ここでも `NULL` が牙を剥きます。
* `region_id IN ( 40, 50, NULL )` と書いても、`region_id` が NULL の行は**ヒットしません。**
* 前回学んだ通り、NULLの判定は `IS NULL` でしか行えないため、リストの中に NULL を混ぜても無視される点に注意してください。

#### 3. `ANY` の真の使い所
例3の `ANY` は、単独で使うよりも「サブクエリ」や「比較演算子」とセットで使われることが多いです。
* 例：`WHERE salary > ANY (1000, 2000, 3000)`
    * これは「1000, 2000, 3000の**いずれかよりも大きい**（＝最小値の1000より大きければOK）」という意味になります。
    * 今回のような `=` との組み合わせなら、おとなしく `IN` を使うのが一般的です。


### 比較まとめ

| 構文 | 推奨度 | メ点（メリット） | 欠点（デメリット） |
| :--- | :--- | :--- | :--- |
| **`OR`** | △ | 異なるカラムを組み合わせるなら必須 | カラム名が重複して冗長になる |
| **`IN`** | **◎ (推奨)** | 簡潔。読みやすい。 | 1,000個制限がある |
| **`ANY`** | △ | サブクエリで柔軟な比較ができる | 初見で意味が分かりづらい |

# 問題4-5（複数の条件） *Lv1*
### HRスキーマ`EMPLOYEES`テーブルより、下記条件全てを満たすデータを取得してください。（順不同）
* **`JOB_ID' が 'IT_PROG'**
* **`SALARY` が　4500**
* **`MANAGER_ID` が 102でない**

## 期待する結果
| FIRST_NAME | JOB_ID  | SALARY | MANAGER_ID | 
| ---------- | ------- | ------ | ---------- | 
| Bruce      | IT_PROG | 6000   | 103        | 
| David      | IT_PROG | 4800   | 103        | 
| Valli      | IT_PROG | 4800   | 103        | 

## 解答例
```sql
SELECT
    first_name,
    job_id,
    salary,
    manager_id
FROM
    hr.employees
WHERE
        job_id = 'IT_PROG'
    AND salary > 4500
    AND manager_id <> 102
```

## 解説
複数のフィルター条件を組み合わせた、より実戦的なクエリーです。
「特定の職種」かつ「一定以上の給与」かつ「特定の上司ではない」という、実務のデータ抽出やレポート作成で非常によくあるパターンです。

このクエリーを正しく理解し、ミスなく書きこなすためのポイントを解説します。

### 条件の「絞り込み」プロセス
この `WHERE` 句は、3つの条件がすべて `AND` で結ばれています。これはベン図でいうところの「重なった部分（積集合）」を取り出していることになります。

1.  **`job_id = 'IT_PROG'`**: 最初に職種で大きく絞り込みます。
2.  **`AND salary > 4500`**: その中から、給与が4,500より高い人だけに限定します。
3.  **`AND manager_id <> 102`**: さらに、上司が102番の人を除外します。


### 実践的な注意点

#### 1. 「否定」と「NULL」の再確認
以前の例でも触れましたが、`manager_id <> 102` という条件があるため、もし **`manager_id` が NULL（上司が設定されていない）社員がいた場合、その人は結果に含まれません。**
* 「102番以外」という条件は、「102番以外の**値が入っている人**」を指すからです。
* NULLの人も含めたい場合は `(manager_id <> 102 OR manager_id IS NULL)` と書く必要があります。

#### 2. 条件を書く順番とパフォーマンス
最近の Oracle (オプティマイザ) は非常に賢いので、書く順番によって速度が変わることは少なくなりましたが、基本的には **「最も件数が絞り込めそうな条件」を先に書く** のが、SQLを読みやすくするコツです。
* 例えば、社員全体の中で `IT_PROG` が数人しかいないのであれば、最初にそれを書くことで「あ、IT部門の人を調べているんだな」と意図が伝わりやすくなります。

#### 3. 可読性を高める「インデント」
解答例にあるように、`AND` の位置を揃えて書くのは非常に良い習慣です！
```sql
WHERE
        job_id = 'IT_PROG'
    AND salary > 4500
    AND manager_id <> 102
```
このように書くと、後で「給与の条件だけ消したいな」と思った時に、1行コメントアウトするだけで済むので、メンテナンス性が格段に上がります。


<br>

:::message
### **「WHERE 1=1」の謎**
あるプロジェクトで見たSQLスクリプトに「WHERE 1 = 1」と書かれており、何のことか全く理解できず、「これ何ですか？」と聞たことを思い出します。

SQLで `WHERE 1=1` と書くのは、一見無意味（常に正しい条件）に見えますが、実は**プログラムで動的にSQLを組み立てる際の「定石」** として非常に重宝されています。

主な理由は、SQL文を「文字列として連結して作る」ときのコードをシンプルにするためです。
### 1. 動的SQLの組み立てを楽にするため
プログラムで「検索条件がある場合だけ `WHERE` 句を追加する」という処理を書く場面を想像してみてください。
<br>
#### * **`1=1` がない場合**

最初の条件だけは `WHERE` を使い、2つ目以降は `AND` を使う……という「条件分岐」の判定が必要になります。

```sql
-- 最初の条件だけ WHERE で始める必要がある
SELECT * FROM users WHERE name = '田中' AND age >= 20;

-- もし名前の指定がなかったら？
SELECT * FROM users WHERE age >= 20; -- 常に「最初が何か」を気にしないといけない
```
<br>

#### * `1=1` がある場合

**「常に `WHERE 1=1` から始める」** と決めておけば、その後に続く条件はすべて `AND` でつなげるだけで済みます。

```sql
SELECT * FROM users 
WHERE 1=1 
  AND name = '田中' 
  AND age >= 20;
```

プログラム側では、条件がいくつあっても「`" AND column = value"` を末尾にくっつけるだけ」で済むので、ロジックが非常にシンプルになります。

<br>

### 2. デバッグ（コメントアウト）が楽になる
SQLを直接書いてテストしているとき、特定の条件を無効化したいことがありますよね。`1=1` があれば、行頭をコメントアウトするだけで済みます。

```sql
SELECT * FROM products
WHERE 1=1
-- AND category = '家電'  <-- ここを消してもWHERE句が壊れない
   AND price <= 5000
```

もし `1=1` がないと、最初の条件を消すときに `WHERE` というキーワード自体も書き直さなければならず、手間がかかります。

<br>
私は目安として、以下の場合好んで「`WHERE 1 = 1`」を書くようにしています。
（いずれか）
- 複雑で巨大なSQLスクリプト
- 条件が３つ以上ある
- 自身が使用するテスト用スクリプト
:::

# 問題4-6（あいまい検索1） *Lv1*
### HRスキーマ`EMPLOYEES`テーブルより、`EMAIL`に「AB」の文字列が含まれるデータを取得してください。（順不同）

## 期待する結果
| FIRST_NAME | EMAIL   | 
| ---------- | ------- | 
| Ellen      | EABEL   | 
| Amit       | ABANDA  | 
| Alexis     | ABULL   | 
| Anthony    | ACABRIO | 

## 解答例
```sql
SELECT
    first_name,
    email
FROM
    hr.employees
WHERE
    email LIKE '%AB%'
```

## 解説
部分一致検索（あいまい検索）の基本ですね。
「正確なメールアドレスは忘れたけれど、途中に『AB』が入っていたはず」といった、あやふやな情報を頼りにデータを特定したい時に非常に強力な武器になります。

このクエリーの仕組みと、実務でエンジニアが最も警戒する「パフォーマンスの罠」について深掘り解説します。


### 1. `LIKE` 演算子とワイルドカード
`LIKE` は、特定のパターンに一致する文字列を検索するための演算子です。ここで使われている `%`（パーセント）は「ワイルドカード」と呼ばれます。

* **`%`**: **0文字以上の任意の文字列**を表します。
* **`%AB%`**: 「何かの文字（0文字でも可） + AB + 何かの文字」という意味になり、結果として**文字列のどこかに AB が含まれていればヒット**します。



### 2. 検索のバリエーション
ワイルドカードを置く位置によって、検索の意味がガラリと変わります。

* **`'AB%'`（前方一致）**: 「AB」から始まるもの（例：**AB**EL）
* **`'%AB'`（後方一致）**: 「AB」で終わるもの（例：D**AB**）
* **`'%AB%'`（部分一致）**: 「AB」をどこかに含むもの（例：C**AB**LE）


### 実践的な注意点：プロが教える「禁じ手」

実務の現場では、今回の例のような `'%...%'`（最初に `%` が来る書き方）には非常に慎重になります。

#### 1. 「フルスキャン」の恐怖
データベースには、検索を高速化する「インデックス（索引）」という仕組みがあります。辞書の索引と同じで、「Aから始まる単語」はすぐに見つけられます。
しかし、`'%AB%'` のように**最初が `%` だと、インデックスが使えません。**

* **結果**: 100万件のデータがあれば、100万件すべてを端から端までチェックする「フルテーブルスキャン」が発生し、**検索が劇的に遅くなります。**
* **対策**: 可能な限り `'AB%'`（前方一致）で検索できないか検討します。



#### 2. 大文字・小文字の区別（再確認）
Oracle SQLにおいて、データの中身は厳密に区別されます。
* `email LIKE '%AB%'` → `'abc@...'` は**ヒットしません。**
* 実務では、ユーザーがどちらを入力しても良いように、以下のように関数を組み合わせて「大文字に統一して比較」することが一般的です。
  `WHERE UPPER(email) LIKE '%AB%'`

## パターン指定まとめ

| パターン | 意味 | 検索スピード（目安） |
| :--- | :--- | :--- |
| `LIKE 'AB%'` | **前方一致**（ABで始まる） | **速い**（インデックスが効く） |
| `LIKE '%AB'` | **後方一致**（ABで終わる） | **遅い** |
| `LIKE '%AB%'` | **部分一致**（ABを含む） | **遅い** |

<br>
:::message
### **「とりあえず % で囲む」のは卒業しよう！**
開発環境の少ないデータでは一瞬で終わるクエリーも、本番環境の巨大なデータではシステム全体を止める原因になることがあります。`LIKE` を使うときは、常に「インデックスが使えるか？」をセットで考えるのが、プロのSQLエンジニアへの第一歩です。
:::

# 問題4-7（あいまい検索2） *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、`EMAIL`が「S」から始まり「S」で終わる７文字のデータを取得してください。（順不同）

## 期待する結果
| FIRST_NAME | EMAIL   | 
| ---------- | ------- | 
| Susan      | SJACOBS | 
| Stephen    | SSTILES | 
| Sigal      | STOBIAS | 

## 解答例
```sql
SELECT
    first_name,
    email
FROM
    hr.employees
WHERE
    email LIKE 'S_____S'
```

## 解説
前回は「何文字でもOK」な `%`（パーセント）でしたが、今回は **「文字数を厳密に指定する」** ための `_`（アンダースコア）の演習です。

この `_` を使いこなすと、データクレンジングや特定のフォーマット（型番や固定長のIDなど）を探す精度が劇的に上がります。解説と実務での注意点をまとめました。

### 1. アンダースコア（`_`）ワイルドカードの役割
`LIKE` 演算子において、`_` は **「任意の1文字」** を表します。

* **今回のパターン `'S_____S'`**:
    1.  先頭が **'S'** で始まる
    2.  間に任意の文字が **5文字** 入る
    3.  最後が **'S'** で終わる
* **合計文字数**: $1 (S) + 5 (\_) + 1 (S) = 7$ 文字



### 2. ヒットする例・しない例
この条件は非常に厳格です。
* **○ ヒットする**: `STEVENS`（S + 5文字 + S）
* **× ヒットしない**: `STEVES`（合計6文字なので足りない）
* **× ヒットしない**: `STEPHENSON`（Sで始まりSを含むが、文字数が多すぎる）


### 実践的な注意点

#### 1. 「数え間違い」というヒューマンエラー
実務で `_` をたくさん並べると、パッと見て何個並んでいるか分からなくなり、バグの原因になります。
* **対策**: 文字数を条件にしたい場合は、無理に `LIKE` を使わず、**`LENGTH` 関数** を組み合わせる方が読みやすくなることがあります。
    * 例：`WHERE email LIKE 'S%S' AND LENGTH(email) = 7`
    * これなら「Sで始まってSで終わる、かつ7文字」という意図が明確に伝わります。

#### 2. 固定長チャネル（CHAR型）の罠
もし検索対象の列が `VARCHAR2`（可変長）ではなく `CHAR`（固定長）の場合、末尾に「見えない空白」が入っていることがあります。
* `CHAR(10)` の列に `'STEVENS'` と入っていると、実際は `'STEVENS   '`（後ろに3つの空白）となっており、`'S_____S'` ではヒットしなくなります。Oracleを扱う際、列のデータ型を確認するのは必須の習慣です。


### ワイルドカードの使い分けまとめ

| 記号 | 意味 | 用途 |
| :--- | :--- | :--- |
| **`%`** | 0文字以上の任意の文字列 | 検索ワードが含まれているか探す時 |
| **`_`** | **正確に1文字** | 桁数やフォーマットが決まっている時 |

<br>
:::message
### **「アンダースコア自体を検索したい時は？」**
もしメールアドレスの中に実際に含まれている `_`（アンダースコア文字）を探したい場合は、そのまま書くとワイルドカードとして扱われてしまいます。その場合は `ESCAPE` 句を使って、「この後のアンダースコアはただの文字だよ！」と教えてあげる必要があります。
（例）商品名に「10%」という文字列が含まれるものを探す場合
```sql
product_name LIKE '%10@%%' ESCAPE '@'
```
:::

# 問題4-8（SELECT文での条件） *Lv2*
### HRスキーマ `COUNTRIES` テーブルより、下記条件にて、「地域」を算出してください。
* **`REGION_ID`=10または20** : 欧米
* **`REGION_ID`=30または50** : アジア・アフリカ
* **上記以外** : その他
* **表示対象は、`COUNTRY_ID`の頭文字がA～Cのもの**
  
## 期待する結果
| COUNTRY_ID | COUNTRY_NAME | REGION_ID | 地域             | 
| ---------- | ------------ | --------- | ---------------- | 
| AR         | Argentina    | 20        | 欧米             | 
| AU         | Australia    | 40        | その他           | 
| BE         | Belgium      | 10        | 欧米             | 
| BR         | Brazil       | 20        | 欧米             | 
| CA         | Canada       | 20        | 欧米             | 
| CH         | Switzerland  | 10        | 欧米             | 
| CN         | China        | 30        | アジア・アフリカ | 

## 解答例
```sql:例1：検索 CASE 式
SELECT
    country_id,
    country_name,
    region_id,
    CASE
        WHEN region_id IN ( 10, 20 ) THEN
            '欧米'
        WHEN region_id IN ( 30, 50 ) THEN
            'アジア・アフリカ'
        ELSE
            'その他'
    END AS "地域"
FROM
    hr.countries
WHERE
    country_id BETWEEN 'A' AND 'CZ'
```
```sql:例2：単純 CASE 式
SELECT
    country_id,
    country_name,
    region_id,
    CASE region_id
        WHEN 10 THEN
            '欧米'
        WHEN 20 THEN
            '欧米'
        WHEN 30 THEN
            'アジア・アフリカ'
        WHEN 50 THEN
            'アジア・アフリカ'
        ELSE
            'その他'
    END AS "地域"
FROM
    hr.countries
WHERE
    country_id BETWEEN 'A' AND 'CZ'
```

## 解説
SQLにおける「条件分岐」の真打ち、**`CASE` 式**です。
これまでは `WHERE` 句で「行」を絞り込んできましたが、`CASE` 式を使うと「列の値」そのものを条件に応じて作り変えることができます。

プログラミング言語（If文やSwitch文）に近い感覚で使える、非常に強力で楽しい機能です。2つの書き方の違いと実務上の重要ポイントを解説します。

### 1. 検索 CASE 式 (例1)
```sql
CASE 
    WHEN region_id IN (10, 20) THEN '欧米'
    ...
END
```
* **特徴**: `WHEN` の後ろに `IN` や `>`、`<`、あるいは `IS NULL` など、**自由な条件式**を書くことができます。
* **メリット**: 非常に柔軟です。「IDが10、かつ給与が5000以上ならA案」といった複雑な条件もこれ一つで書けます。

### 2. 単純 CASE 式 (例2)
```sql
CASE region_id 
    WHEN 10 THEN '欧米'
    ...
END
```
* **特徴**: 最初に評価対象のカラム（`region_id`）を指定し、`WHEN` の後には「値」だけを書きます。
* **メリット**: 構文がスッキリして読みやすいです。特定の値に対する「置き換え（マッピング）」をするだけなら、こちらの方がスマートです。


### 実践的な注意点：CASE式の「鉄則」

#### 1. 最初に一致したものが優先される
`CASE` 式は上から順番に評価されます。
もし、複数の `WHEN` 条件に当てはまるデータがあったとしても、**一番最初に `TRUE` になった `THEN` の値が返され、それ以降は無視されます。**
* 優先順位が高い（より細かい）条件を上に書くのがコツです。

#### 2. `ELSE` を省略しない
`ELSE` を書かずに、どの `WHEN` 条件にも当てはまらなかった場合、その結果は **`NULL`** になります。
* 意図しない `NULL` の発生はバグの元です。今回のように「その他」と指定するか、せめて元の値をそのまま出すなど、**「漏れ」がない設計**を心がけましょう。

#### 3. 文字列範囲の `WHERE` 句
`WHERE country_id BETWEEN 'A' AND 'CZ'` について。
* これは「辞書順」での比較です。
* `'C'` で始まる国（Canadaなど）は含まれますが、`'D'` で始まる国は含まれません。
* `'CZ'` という指定は、`'C'` から始まり、2文字目が `'Z'` 以下のものを指します。文字列の範囲指定は、直感とズレる場合があるので、実行結果をよく確認する癖をつけましょう。


### 比較まとめ

| 項目 | 検索 CASE 式 (例1) | 単純 CASE 式 (例2) |
| :--- | :--- | :--- |
| **柔軟性** | **◎ 非常に高い** | △ 特定の値との比較のみ |
| **可読性** | 条件が複雑だと煩雑になる | **◎ スッキリして読みやすい** |
| **NULL判定** | `WHEN col IS NULL` と書ける | `WHEN NULL` は正しく動かない |
| **推奨シーン** | 範囲指定や複数カラムの判定 | 単純な値の置き換え |

# 問題4-9（指定件数のデータ取得）  *Lv1*
### HRスキーマ`EMPLOYEES`テーブルより、任意の5件のデータを取得してください。（順不同）

## 期待する結果
| EMPLOYEE_ID | FIRST_NAME | LAST_NAME | 
| ----------- | ---------- | --------- | 
| 100         | Steven     | King      | 
| 101         | Neena      | Yang      | 
| 102         | Lex        | Garcia    | 
| 103         | Alexander  | James     | 
| 104         | Bruce      | Miller    | 

※任意の５件が取得できていればOK

## 解答例
```sql:例1：ROWNUMを使用
SELECT
    employee_id,
    first_name,
    last_name
FROM
    hr.employees
WHERE
    ROWNUM <= 5
```
```sql:例2：FETCH FIRSTを使用
SELECT
    employee_id,
    first_name,
    last_name
FROM
    hr.employees
--ORDER BY     -- 通常、ORDER BYをつけますが、任意の５件のため、ここでは敢えてコメントアウトしています
--    employee_id
FETCH FIRST 5 ROWS ONLY
```

## 解説
取得する行数を制限する「Top-Nクエリ」の手法です。Oracle 12c以降を使っている現場では、今まさに「古い書き方（`ROWNUM`）」から「新しい書き方（`FETCH FIRST`）」への過渡期にあります。

この2つの違いを正しく理解しておかないと、**「データが5件取れたけど、欲しかった5件じゃない！」** という悲劇が起こります。解説と注意点を見ていきましょう。

### 例1：`ROWNUM`（伝統的な擬似列）
* **仕組み**: 抽出された各行に、上から順に `1, 2, 3...` と番号を振るOracle特有の「擬似列」を利用します。
* **動作**: 条件に合うデータが見つかるたびに番号が振られ、5番目のデータが見つかった時点で検索を終了します。

### 例2：`FETCH FIRST`（モダンな標準構文）
* **仕組み**: SQL:2008標準に基づいた比較的新しい書き方です（Oracle 12cから導入）。
* **動作**: 意味としては例1と同じですが、**「何件取得したいのか」という意図がSQLの最後に書かれる**ため、英語の文章のように読みやすいのが特徴です。



### 実践的な注意点：ここが合否を分ける！

#### 1. `ORDER BY` との「順序」問題（最重要）
ここが、開発者が最もミスを犯しやすいポイントです。
* **`ROWNUM` の罠**: 
    実は、`ROWNUM` は **「並び替える前」** に番号が振られます。
    もし「給与が高い順に5人」取りたい場合、`ROWNUM <= 5 ORDER BY salary DESC` と書いても、**「適当に取った5人を、後から並び替える」** という動きになり、正しいTop 5になりません。
    ※正しくやるには「サブクエリ（入れ子構造）」にする必要があります。
* **`FETCH FIRST` の優秀さ**: 
    こちらは **「並び替えた後」** に5件取ってくれます。
    `ORDER BY salary DESC FETCH FIRST 5 ROWS ONLY` と書くだけで、正しく高額給与者Top 5が取得できます。

#### 2. `ROWNUM` の「1」の壁
`ROWNUM` は必ず「1」から始まらなければなりません。
* `WHERE ROWNUM <= 5`（1から5まで） → **OK**
* `WHERE ROWNUM = 1`（1件目） → **OK**
* `WHERE ROWNUM = 2`（2件目だけ） → **NG（1件もヒットしなくなります）**
  > [!NOTE]
  > 1件目が条件を満たさない限り、2件目という番号自体が生成されないためです。

#### 3. コメントアウトされている `ORDER BY` について
例2で `ORDER BY` がコメントアウトされていますが、実務では **行数制限をするなら `ORDER BY` は必須** と考えてください。
* データベースにとって、ソートされていないデータの並び順は「たまたまその時そうなっただけ」の不安定なものです。
* 明示的に並び替えないと、実行するたびに取得される5件の顔ぶれが変わってしまうリスクがあります。今回は要件として「**任意の**５件」であるため、あえてコメントアウトしています。


### 比較まとめ

| 項目 | `ROWNUM` (例1) | `FETCH FIRST` (例2) |
| :--- | :--- | :--- |
| **導入時期** | 昔から（レガシー） | Oracle 12c以降（モダン） |
| **可読性** | `WHERE`句に混ざるので少し複雑 | 直感的で読みやすい |
| **ソートとの相性** | 悪い（サブクエリが必要） | **非常に良い（自然に書ける）** |
| **柔軟性** | 行番号での処理に強い | ページング（○件目から○件）に強い |

# 問題4-10（指定範囲のデータ取得）  *Lv2*
### HRスキーマ`EMPLOYEES`テーブルを`EMPLOYEE_ID`の昇順の並べて先頭から10件スキップし、5件のデータを取得してください。（順不同）

## 期待する結果
| EMPLOYEE_ID | FIRST_NAME  | LAST_NAME | 
| ----------- | ----------- | --------- | 
| 110         | John        | Chen      | 
| 111         | Ismael      | Sciarra   | 
| 112         | Jose Manuel | Urman     | 
| 113         | Luis        | Popp      | 
| 114         | Den         | Li        | 

## 解答例
```sql
SELECT
    employee_id,
    first_name,
    last_name
FROM
    hr.employees
ORDER BY
    employee_id
OFFSET 10 ROWS
FETCH NEXT 5 ROWS ONLY
```

## 解説
今回はWebアプリケーションなどの「ページング処理（1ページに10件ずつ表示、など）」を支える、非常にモダンで強力な構文です。

この `OFFSET` と `FETCH` の組み合わせは、Oracle 12cから導入された比較的新しい標準SQLの書き方です。実務で正しく使うためのポイントを解説します。



### ページングの仕組み
このクエリーは、全社員の中から **「11番目から15番目までの5人」** をピンポイントで抽出しています。

1.  **`ORDER BY employee_id`**: まず、社員ID順に並べます。これがページングの「基準」になります。
2.  **`OFFSET 10 ROWS`**: 最初の10行を **読み飛ばし（スキップ）** ます。
3.  **`FETCH NEXT 5 ROWS ONLY`**: スキップした直後の位置から、**次の5行だけ**を取得します。


### 実践的な注意点

#### 1. `ORDER BY` は「必須」と考える
ページングにおいて、`ORDER BY` を省略するのは非常に危険です。
* **理由**: データベースには「元の並び順」という概念がありません。並び替えをせずに `OFFSET` を使うと、実行するたびにスキップされるデータが変わってしまう可能性があり、**「2ページ目を開いたら、1ページ目にいた人がまた出てきた」** というバグの原因になります。

#### 2. 「深いページ」ほど処理が重くなる
`OFFSET 1000000 ROWS FETCH NEXT 10 ROWS` のようなクエリー（100万件飛ばして10件取る）は、一見効率的に見えますが、実は**実行速度が低下**します。
* **理由**: データベース内部では、結局「100万件分をカウント」してから捨てるという処理が発生するためです。大規模なデータで「1万ページ目」などを表示させる場合は、別の設計（前回取得した最後のIDを条件にするなど）が必要になることもあります。

### 3. 表記のバリエーション
SQLのルール上、いくつかの書き換えが可能です。
* `ROWS` は `ROW` でもOK（1行の場合など）。
* `NEXT` は `FIRST` と書いても同じ動きをします。
* `ONLY` の代わりに `WITH TIES` と書くと、「最後の行と同じ値を持つデータも全部含める」という特殊な取得も可能です。

### 比較まとめ

| 項目 | `OFFSET / FETCH` (例) | 従来の `ROWNUM` を使ったページング |
| :--- | :--- | :--- |
| **可読性** | **◎ 直感的で非常に読みやすい** | △ サブクエリが2重になり複雑 |
| **標準化** | ○ 標準SQLに近い | × Oracle固有の書き方 |
| **柔軟性** | ○ パラメータ指定が簡単 | △ 境界値の計算ミスが起きやすい |
| **導入Ver.** | Oracle 12c 以降 | ほぼ全てのバージョン |
