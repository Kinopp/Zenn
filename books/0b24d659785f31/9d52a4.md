# 問題7-1（現在日時＜date型＞） *Lv1*
### 現在日付（DATE型）を取得してください。

## 期待する結果
| NOW                  | 
| -------------------- | 
| 2026-03-07T23:42:48Z | 

※値は実行した日時

## 解答例
```sql:例1：SYSDATEで取得
SELECT
    SYSDATE AS "NOW"
FROM
    dual
```
```sql:例2：CURRENT_DATEで取得
SELECT
    CURRENT_DATE AS "NOW"
FROM
    dual
```
```sql:FROM DUALを省略（23c以降）
SELECT
    SYSDATE AS "NOW"
```

## 解説
`SYSDATE` と `CURRENT_DATE` はどちらも「現在の日時」を返しますが、その決定的な違いは **「サーバーの時計」を見るか、「セッション（ユーザー）の時計」を見るか** にあります。一言で言うと、**タイムゾーンを考慮するかどうか** です。

#### 1. SYSDATE と CURRENT_DATE の比較

| 項目 | `SYSDATE` | `CURRENT_DATE` |
| --- | --- | --- |
| **基準** | **データベース・サーバー**の時刻 | **セッション（接続ユーザー）** の時刻 |
| **タイムゾーン** | サーバーOSの設定に従う | セッションの設定（`TIME_ZONE`）に従う |
| **用途** | ログ記録、システム管理用 | ユーザーの現地時間に合わせた表示用 |

#### 2. 具体的な違い（例）
例えば、データベースサーバーが **東京（日本）** にあり、あなたが **ニューヨーク（アメリカ）** から接続しているケースを考えてみましょう。

① **SYSDATE を実行した場合**
   - サーバーがある東京の時刻（JST）が返ります。
   - 結果：`2026-03-11 13:00:00`（日本時間）

② **CURRENT_DATE を実行した場合**
   - あなたのセッション設定（ニューヨーク時間：EST/EDT）に変換された時刻が返ります。
   - 結果：`2026-03-11 00:00:00`（ニューヨーク時間）

#### 3. 実践での使い分け
* **SYSDATE を使うべき時**
  - データの作成日時（`CREATED_AT`）を記録する時。
  - バッチ処理の開始・終了ログを記録する時。
  - サーバー側で一貫した「絶対的な時間」が必要な場合。

* **CURRENT_DATE を使うべき時**
  - グローバル展開しているシステムで、ユーザーに「あなたの今の時間は◯時ですよ」と表示したい時。
  - セッションごとの時差を吸収して計算したい場合。

:::message

  | DUMMY |


      ```sql

      ```

:::

# 問題7-2（現在日時＜TimeStamp型＞） *Lv1*
### 現在日付（TIMESTAMP型）を取得してください。

## 期待する結果
| NOW                         | 
| --------------------------- | 
| 2026-03-07T23:57:37.027035Z | 

※値は実行した日付

## 解答例
```sql:例1：SYSTIMESTAMPで取得
SELECT
    SYSTIMESTAMP AS "NOW"
FROM
    dual
```
```sql:例2：CURRENT_TIMESTAMPで取得
SELECT
    CURRENT_TIMESTAMP AS "NOW"
FROM
    dual
```
```sql:例3：LOCALTIMESTAMPで取得
SELECT
    LOCALTIMESTAMP AS "NOW"
FROM
    dual
```

## 解説
Oracleにおいて、`SYSTIMESTAMP` と `CURRENT_TIMESTAMP` はどちらも「現在の時刻」を返します。
考え方は、`SYSDATE` と `CURRENT_DATE`と同様です。
また、`LOCALTIMESTAMP` は、現在のセッション（接続環境）のタイムゾーンにおける日付と時刻を **TIMESTAMP型** で返します。

### 基本的な特徴
| 関数名 | 戻り値の型 | 基準となる場所 | タイムゾーン情報の有無 |
| :--- | :--- | :--- | :--- |
| **SYSTIMESTAMP** | `TIMESTAMP WITH TIME ZONE` | **サーバー** | あり |
| **CURRENT_TIMESTAMP** | `TIMESTAMP WITH TIME ZONE` | **セッション** | あり |
| **LOCALTIMESTAMP** | `TIMESTAMP` | **セッション** | **なし** |


SQLにおける **`DATE`** と **`TIMESTAMP`** は、どちらも「時」に関するデータを扱いますが、その **「細かさ（精度）」** と **「タイムゾーン（時差）の扱い」** に大きな違いがあります。
ざっくり言うと、**「日付だけでいいなら DATE」「秒単位や時差まで管理したいなら TIMESTAMP」** という使い分けになります。

### 1. 主な違いの比較表

データベースの種類（Oracle, MySQL, PostgreSQLなど）によって細かな仕様は異なりますが、一般的な違いは以下の通りです。

| 項目 | DATE | TIMESTAMP |
| --- | --- | --- |
| **保持する情報** | 年・月・日（・時・分・秒※） | 年・月・日・時・分・秒 **+ 小数点以下の秒** |
| **精度の最小単位** | 1秒（または1日） | **0.000001秒（マイクロ秒）単位など** |
| **タイムゾーン** | 通常は持たない | **時差情報を保持できる**（WITH TIME ZONE） |
| **主な用途** | 誕生日、入社日、祝日など | ログの記録、決済時刻、システム更新日時 |

※Oracleなど一部のDBでは `DATE` 型も秒まで保持しますが、MySQLやPostgreSQLでは「日付のみ」を指すことが多いです。


### 2. `DATE` と `TIMESTAMP`3つの違い

#### ① 精度（小数点以下の秒数）
`DATE` 型は秒単位までが限界ですが、`TIMESTAMP` は「1秒未満」の細かい時間を記録できます。

* **DATE**: `2026-03-13 09:15:00`
* **TIMESTAMP**: `2026-03-13 09:15:00.123456`

Webサービスのアクセスログや銀行の振り込み処理など、1秒間に大量のデータが発生する場合は、順序を正確に記録するために `TIMESTAMP` が必須となります。

#### ② タイムゾーン（時差）のサポート
`TIMESTAMP` 型には、**「TIMESTAMP WITH TIME ZONE」** という、世界の時差を考慮できる設定があります。

* **DATE**: 「2026/03/13」と入れたら、どこで見てもその日付。
* **TIMESTAMP (TZ付)**: 日本で「9:00」に保存したデータを、ロンドンで見ると自動的に「0:00」として表示してくれるような処理が可能です。

グローバルなサービスを作る場合は、`TIMESTAMP` を選ぶのが定石です。

#### ③ 扱える範囲（期間）
DBMSによりますが、扱える「期間」に差がある場合があります。
Oracleの場合は、どちらも「紀元前4712年〜西暦9999年」まで対応しており、扱える期間に差はありません。

### 3. どちらを使うべき？
迷ったときは、以下の基準で選んでみてください。

* **`DATE` を使うべきケース**
* 「時間」の情報が不要なとき（誕生日、記念日、契約開始日など）。
* データ容量を少しでも節約したいとき。


* **`TIMESTAMP` を使うべきケース**
* 「いつ作成・更新されたか」というシステム的なログを残すとき。
* 1秒間に何度も発生するイベントを区別したいとき。
* 世界中のユーザーが利用し、時差を考慮する必要があるとき。

> **豆知識：2038年問題**
> 古いシステムで `TIMESTAMP` を使っていると、2038年に上限値を超えて誤作動する可能性があります。最近のデータベースでは対策が進んでいますが、歴史的な背景として知っておくとよいと思います。

# 問題7-3（日付のフォーマット） *Lv1*
### HRスキーマ`JOB_HISTORY`表の全データを取得してください。その際、`START_DATE`,`END_DATE`を「YYYY-MM-DD」の形式で表現してください。（順不同）

## 期待する結果
| EMPLOYEE_ID | START_DATE | END_DATE   | 
| ----------- | ---------- | ---------- | 
| 102         | 2011-01-13 | 2016-07-24 | 
| 101         | 2007-09-21 | 2011-10-27 | 
| 101         | 2011-10-28 | 2015-03-15 | 
| 201         | 2014-02-17 | 2017-12-19 | 
| 114         | 2016-03-24 | 2017-12-31 | 
| 122         | 2017-01-01 | 2017-12-31 | 
| 200         | 2005-09-17 | 2011-06-17 | 
| 176         | 2016-03-24 | 2016-12-31 | 
| 176         | 2017-01-01 | 2017-12-31 | 
| 200         | 2012-07-01 | 2016-12-31 | 

## 解答例
```sql
SELECT
    employee_id,
    TO_CHAR(start_date, 'YYYY-MM-DD') AS start_date,
    TO_CHAR(end_date, 'YYYY-MM-DD')   AS end_date
FROM
    hr.job_history
```

## 解説
SQLにおける **`TO_CHAR`** 関数は、日付や数値を「人間が見やすい文字列」に変換するための非常に便利なツールです。
「ただの変換」だと思って油断していると、余計な空白が入ったり、言語設定のせいでエラーが出たりと、意外な落とし穴があります。

### 1. 基本的な使い方
`TO_CHAR` は大きく分けて「日付の変換」と「数値の変換」の2通りに使われます。

#### ① 日付を文字列にする（当問題）
日付データを「2026/03/13」や「13:00」といった特定のフォーマットに変換します。

```sql
-- 2026年03月13日 08:55:34 の場合
SELECT TO_CHAR(SYSDATE, 'YYYY/MM/DD HH24:MI:SS') FROM dual;
-- 結果: '2026/03/13 08:55:34'
```

#### ② 数値を文字列にする
数値をカンマ区切りにしたり、通貨記号をつけたりします。

```sql
SELECT TO_CHAR(1234567, '9,999,999') FROM dual;
-- 結果: ' 1,234,567' （※先頭に空白が入ることに注意！）
```

### 2. よく使うフォーマット要素

| 要素 | 説明 | 例 |
| --- | --- | --- |
| **`YYYY`** | 4桁の西暦 | 2026 |
| **`MM`** | 2桁の月 | 03 |
| **`DD`** | 2桁の日 | 13 |
| **`HH24` / `MI` / `SS`** | 24時間表記の時 / 分 / 秒 | 08:55:34 |
| **`DAY`** | 曜日（フルネーム） | 金曜日 / FRIDAY |
| **`9`** | 数値（値がない場合は空白） | `9,999` → `  123` |
| **`0`** | 数値（値がない場合は0埋め） | `0,999` → `0,123` |

### 3. 実務で役立つテクニック

#### 「FM」で余計な空白やゼロを消す
デフォルトでは `MM` は `03` のように2桁固定になりますが、先頭に **`FM` (Fill Mode)** をつけると、余計な「0」や空白を取り除けます。

* `TO_CHAR(hire_date, 'YYYY/MM/DD')` → `2026/03/01`
* `TO_CHAR(hire_date, 'FMYYYY/MM/DD')` → `2026/3/1` 

### 4. 知っておくべき3つの注意点（落とし穴）

#### ① 「謎の空白」問題（数値変換）
数値に対して `TO_CHAR(100, '999')` とすると、結果の先頭に1文字分の空白が入ります。これは **「符号（プラスかマイナスか）」を入れるための予約席** です。

> **対策:** 空白が邪魔な場合は `FM999` と書くか、`TRIM` 関数を使いましょう。

#### ② 言語設定（NLS設定）の影響
曜日の表示（`DAY`）などは、データベースの言語設定によって「金曜日」になったり「FRIDAY」になったりします。

> **対策:** 環境に左右されたくない場合は、第3引数で言語を指定します。
> `TO_CHAR(date, 'DAY', 'NLS_DATE_LANGUAGE = AMERICAN')`

#### ③ 大文字・小文字の区別
フォーマット指定の `MON` (大文字) と `mon` (小文字) は区別されます。

* `MON` → `MAR`
* `mon` → `mar`
* `Mon` → `Mar`

# 問題7-4（年、月、日の抽出） *Lv2*
### OEスキーマ`CUSTOMERS`テーブルより、1985年生まれの顧客を取得してください。その際、以下の通り編集してください。（順不同）
  - **`CUST_FIRST_NAME`と`CUST_LAST_NAME`を結合して`NAME`とする。**
  - **`DATE_OF_BIRTH`を「xxxx年xx月xx日」に編集し、`BIRTHDAY`とする。**

## 期待する結果
| ID  | NAME             | BIRTHDAY      | 
| --- | ---------------- | ------------- | 
| 215 | Orson Perkins    | 1985年5月24日 | 
| 253 | Sally Bogart     | 1985年1月14日 | 
| 255 | Brooke Shepherd  | 1985年2月13日 | 
| 328 | Hannah Field     | 1985年10月6日 | 
| 838 | Alfred Nicholson | 1985年1月25日 | 
| 913 | Bob Sharif       | 1985年1月10日 | 


## 解答例
```sql:例1：EXTRACTを使用
SELECT
    customer_id       AS id,
    cust_first_name || ' ' || cust_last_name AS "NAME",
    EXTRACT(YEAR FROM date_of_birth) || '年' || 
      EXTRACT(MONTH FROM date_of_birth) || '月' ||
        EXTRACT(DAY FROM date_of_birth) || '日' AS "BIRTHDAY"
FROM
    oe.customers
WHERE
    date_of_birth BETWEEN date'1985-01-01' 
                      AND date'1985-12-31'
ORDER BY
    id
```
```sql:例2：TO_CHARを使用
SELECT
    customer_id       AS id,
    cust_first_name || ' ' || cust_last_name AS "NAME",
    TO_CHAR(date_of_birth, 'YYYY') || '年' || 
      TO_CHAR(date_of_birth,'FMMM') || '月' ||
        TO_CHAR(date_of_birth, 'FMDD') || '日' AS "BIRTHDAY"
FROM
    oe.customers
WHERE
    date_of_birth BETWEEN TO_DATE('1985-01-01', 'YYYY-MM-DD') 
                      AND TO_DATE('1985-12-31', 'YYYY-MM-DD')
ORDER BY
    id
```


## 解説
### A. EXTRACT
Oracle SQLの **`EXTRACT`** 関数は、日付や日付時刻（Datetime）型のデータから、**「年」「月」「日」「時間」といった特定の要素だけを数値として取り出す**ための関数です。

データをグループ化して「月ごとの集計」を出したい時や、「特定の時間帯」のデータだけを絞り込みたい時に非常に重宝します。

#### 1. 基本的な構文

```sql
EXTRACT(抽出したい要素 FROM 元のデータ)
```

### 指定できる主な要素（フィールド）
* **YEAR**: 年
* **MONTH**: 月
* **DAY**: 日
* **HOUR**: 時（※TIMESTAMP型が必要）
* **MINUTE**: 分（※TIMESTAMP型が必要）
* **SECOND**: 秒（※TIMESTAMP型が必要）


#### 2. 具体的な使い方と例
##### 年・月・日を取り出す（DATE型 / TIMESTAMP型）

もっとも一般的な使い方です。戻り値は **数値（NUMBER型）** になります。

```sql
SELECT
  EXTRACT(YEAR  FROM SYSDATE) AS 今年,
  EXTRACT(MONTH FROM SYSDATE) AS 今月,
  EXTRACT(DAY   FROM SYSDATE) AS 今日
FROM DUAL;
```

##### 時・分・秒を取り出す（注意点あり）
Oracleの標準的な `DATE` 型には時間情報が含まれていますが、**`EXTRACT` で `HOUR` や `MINUTE` を指定する場合、対象は `TIMESTAMP` 型である必要があります。**

```sql
-- ❌ DATE型に対してHOURを使うとエラー（ORA-30076）になる場合がある
-- SELECT EXTRACT(HOUR FROM SYSDATE) FROM DUAL;

-- ⭕ TIMESTAMP型にキャストすればOK
SELECT
  EXTRACT(HOUR   FROM CAST(SYSDATE AS TIMESTAMP)) AS 時,
  EXTRACT(MINUTE FROM CAST(SYSDATE AS TIMESTAMP)) AS 分
FROM DUAL;
```

##### 期間（INTERVAL型）から取り出す

「○日間と○時間」という期間データから、日数だけを抜くこともできます。

```sql
-- 100日と5時間を表すインターバルから「日」を抽出
SELECT EXTRACT(DAY FROM INTERVAL '100 05:00:00' DAY TO SECOND) FROM DUAL;
-- 結果: 100
```
<br>
### B. TO＿DATE
Oracle SQLにおいて、`TO_DATE` は **「文字列（テキスト）」を「日付型（DATE）」に変換する** ための非常に重要な関数です。

データベースの中では、日付は単なる数字や文字ではなく「内部的な日付フォーマット」で管理されています。人間が読み書きする「2026/03/15」といったテキストを、コンピュータが計算可能な「日付オブジェクト」に橋渡しするのがこの関数の役割です。

#### 1. 基本構文

```sql
TO_DATE('文字列', 'フォーマットマスク')
```

* **文字列**: 日付として扱いたいテキスト。
* **フォーマットマスク**: そのテキストが「どの順序で、何を表しているか」を教えるための型紙。

#### 2. よく使うフォーマットマスク一覧

これらを組み合わせて、あらゆる形式のテキストを日付に変換します。

| マスク | 意味 | 例（2026年3月15日 19時の場合） |
| --- | --- | --- |
| **YYYY** | 西暦4桁 | 2026 |
| **MM** | 月（01-12） | 03 |
| **DD** | 日（01-31） | 15 |
| **HH24** | 24時間表記（00-23） | 19 |
| **MI** | 分（00-59） | 45 |
| **SS** | 秒（00-59） | 30 |

> **注意！**: 分は `MM` ではなく **`MI`** です。`MM` は Month（月）として認識されてしまうため、時刻を扱う際は必ず `MI` を使いましょう。

#### 3. なぜ `TO_DATE` が必要なのか？

##### ① 文字列のままだと「大小比較」が正しくできない

文字列として比較すると、辞書順（あいうえお順）になるため、おかしな結果になります。

* **文字列比較**: `'2025/12/31'` $<$ `'2026/01/01'` は正しい。
* **文字列比較の罠**: `'9/30'` $>$ `'10/01'` と判定される（9の方が1より大きいため）。

`TO_DATE` で日付型に変換することで、カレンダー通りの正しい比較が可能になります。

##### ② NLS設定（環境依存）の罠を防ぐ

Oracleには `NLS_DATE_FORMAT` という設定があり、これによってデフォルトの解釈が決まります。

* あるサーバーでは `DD-MON-RR`（15-MAR-26）
* 別のPCでは `YYYY/MM/DD`（2026/03/15）

`TO_DATE(str)` のようにマスクを省略すると、**「自分の環境では動くのに、本番環境ではエラー（ORA-01843: 指定した月が無効です）になる」** という恐怖のバグを生みます。**フォーマットマスクは常に指定するのが鉄則**です。

#### 4. 実践的なサンプル

##### 基本的な変換
```sql
SELECT * FROM sh.sales
WHERE time_id >= TO_DATE('2026-03-15', 'YYYY-MM-DD');
```

##### 時分秒まで含める場合
```sql
SELECT TO_DATE('2026/03/15 19:45:00', 'YYYY/MM/DD HH24:MI:SS') FROM dual;
```
<br>

:::message
### 日付リテラル（ANSI日付リテラル）
**「日付リテラル（ANSI日付リテラル）」** は、`TO_DATE` 関数を使わずに、よりシンプルかつ標準的な形式で日付を指定する方法です。

結論から言うと、**「時間は不要で、決まった形式（YYYY-MM-DD）で書きたい時」** に非常に便利です。

### 日付リテラルの書き方

構文は非常にシンプルです。

```sql
DATE 'YYYY-MM-DD'
```

#### 具体的な例

```sql
-- TO_DATEを使う場合
SELECT * FROM orders WHERE order_date >= TO_DATE('2024-01-01', 'YYYY-MM-DD');

-- 日付リテラルを使う場合
SELECT * FROM orders WHERE order_date >= DATE '2024-01-01';
```
<br>
### DATEリテラルの特徴と注意点
`TO_DATE` と比較して、いくつか守らなければならないルールがあります。
<br>
#### 1. フォーマットは「YYYY-MM-DD」固定

日付リテラルを使う場合、フォーマットを自由に変えることはできません。必ず **西暦4桁-月2桁-日2桁** で書く必要があります。

* ⭕ `DATE '2024-03-15'`
* ❌ `DATE '2024/03/15'` （スラッシュはエラー）
* ❌ `DATE '24-03-15'` （2桁年はエラー）
<br>
#### 2. 時・分・秒を含めることはできない

`DATE` というキーワードを使いますが、日付リテラルで指定できるのは「年月日」までです。時刻（00:00:01など）を指定したい場合は、**TIMESTAMPリテラル** を使う必要があります。

```sql
-- 時刻まで指定したい場合
SELECT TIMESTAMP '2024-03-15 10:30:00' FROM DUAL;
```
<br>
#### 3. 関数ではなく「定数」として扱われる

`TO_DATE` は実行時に文字列を日付に変換する「処理」が走りますが、日付リテラルはデータベースが最初から日付として認識します。そのため、ごく僅かですがパフォーマンス的に有利な場合があります（通常は気にするほどではありません）。

<br>
### どっちを使うべき？（比較表）

| 特徴 | DATEリテラル | TO_DATE関数 |
| --- | --- | --- |
| **書きやすさ** | 短くてスッキリする | 少し長い |
| **フォーマット** | `YYYY-MM-DD` 固定 | 自由自在（スラッシュ等も可） |
| **時刻の指定** | 不可（TIMESTAMPなら可） | 可能 |
| **動的な指定** | 不向き（固定値用） | 変数やカラムを渡せる |
| **標準規格** | ANSI標準（他DBでも通じやすい） | Oracle独自 |

<br>
### 使い分けのガイドライン

* **プログラム内の固定値や、手動でサクッとクエリを投げる時**
* `DATE '2024-01-01'` がおすすめ。読みやすく、タイプ数も少ないです。
<br>
* **画面からの入力値や、特殊な形式の文字列を扱う時**
* `TO_DATE(:input_val, 'YYYY/MM/DD')` を使いましょう。
:::

# 問題7-5（日付の加算） *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、`JOB_ID`が「AD_VP」の従業員に対して、下記の値を取得する。
  - **HIRE_DATE**
  - **HIRE_DATEの２日後**
  - **HIRE_DATEの２ヶ月後**
  - **HIRE_DATEの２年後**

## 期待する結果
| EMPLOYEE_ID | NAME       | HIRE_DATE  | HIRE_DATE + 2DAY | HIRE_DATE + 2MONTH | HIRE_DATE + 2YEAR | 
| ----------- | ---------- | ---------- | ---------------- | ------------------ | ----------------- | 
| 101         | Neena Yang | 2015-09-21 | 2015-09-23       | 2015-11-21         | 2017-09-21        | 
| 102         | Lex Garcia | 2011-01-13 | 2011-01-15       | 2011-03-13         | 2013-01-13        |


## 解答例
```sql
SELECT
    employee_id,
    first_name || ' ' || last_name as name,
    TO_CHAR(hire_date, 'YYYY-MM-DD')                   AS "HIRE_DATE",
    TO_CHAR(hire_date + 2, 'YYYY-MM-DD')               AS "HIRE_DATE + 2DAY",
    TO_CHAR(ADD_MONTHS(hire_date, 2), 'YYYY-MM-DD')    AS "HIRE_DATE + 2MONTH",
    TO_CHAR(ADD_MONTHS(hire_date, 2*12), 'YYYY-MM-DD') AS "HIRE_DATE + 2YEAR"
FROM
    hr.employees
WHERE
    job_id = 'AD_VP';
```

## 解説
**ADD_MONTHS関数**は、指定した日付に「特定の月数」を加算（または減算）した結果を返します。単純な計算だけでなく、**月末の処理に独自の特徴**があるため、そこを押さえておくと実務で非常に役立ちます。

### 1. 基本的な構文
```sql
ADD_MONTHS( 日付, 加算する月数 )
```

* **日付**: 基準となる日付（DATE型など）を指定します。
* **加算する月数**: 足したい月数を整数で指定します。
* **正の数**: 未来の日付（1ヶ月後など）
* **負の数**: 過去の日付（1ヶ月前など）



#### 基本的な例

| SQL文 | 結果の例 | 説明 |
| --- | --- | --- |
| `ADD_MONTHS(SYSDATE, 1)` | 1ヶ月後の今日 | 今日の日付に1ヶ月足します。 |
| `ADD_MONTHS(SYSDATE, -3)` | 3ヶ月前の今日 | 今日の日付から3ヶ月引きます。 |
| `ADD_MONTHS(SYSDATE, 12)` | 1年後の今日 | 12ヶ月足すことで1年後を求められます。 |

### 2. 月末の特殊な挙動

ADD_MONTHS関数には、**「月末の整合性を保つ」** ための特別なルールが2つあります。ここが初心者の方にとって少し意外に感じるポイントです。

#### ルール①：結果の月にその「日」が存在しない場合
加算後の月に、元の「日」が存在しない場合は、**その月の最終日**が返されます。
* 例：**1月31日** の1ヶ月後
* 2月には31日がないため、**2月28日**（閏年なら29日）が返ります。

#### ルール②：元の「日」が月末日の場合
基準となる日付がその月の**最終日**であれば、結果も必ず**その月の最終日**になります。
* 例：**2月28日（平年の末日）** の1ヶ月後
* 28日を指定していますが、「2月の末日」と判断されるため、結果は「3月の末日」である **3月31日** になります。


### 3. 具体的な挙動まとめ表

| 基準日 | 月数 | 結果 | 理由 |
| --- | --- | --- | --- |
| 2024/01/15 | 1 | 2024/02/15 | 通常の加算 |
| 2024/01/31 | 1 | 2024/02/29 | 2月31日はないので、2月末（閏年）を返す |
| 2023/01/31 | 1 | 2023/02/28 | 2月31日はないので、2月末（平年）を返す |
| **2024/02/29** | 1 | **2024/03/31** | **「月末」から「月末」への加算になる** |
| 2024/03/31 | -1 | 2024/02/29 | 過去への減算も同様（月末なら月末へ） |

### 4. よくある質問
* **Q. 小数点（1.5ヶ月など）を指定するとどうなりますか？**
* A. 小数点以下は切り捨てられ、整数として処理されます。`1.5` を指定しても `1` と同じ結果になります。


* **Q. 日にちだけを足したい場合は？**
* A. 日単位の加算は `SYSDATE + 7`（7日後）のように、直接数値を足し算するだけでOKです。

> **💡 ワンポイントアドバイス**
> 請求処理や契約更新など、「毎月の月末」を扱う処理では、このADD_MONTHSの「月末なら月末を返す」という性質が非常に重宝されます。

# 問題7-6（期間の算出） *Lv2*
### HRスキーマ`JOB_HISTORY`テーブルの`START_DATE`と`END_DATE`の期間を「○○年○○ヶ月」の形式で取得してください。（順不同）

## 期待する結果
| EMPLOYEE_ID | JOB_ID     | 期間      | 
| ----------- | ---------- | --------- | 
| 102         | IT_PROG    | 5年6ヶ月  | 
| 101         | AC_ACCOUNT | 4年1ヶ月  | 
| 101         | AC_MGR     | 3年4ヶ月  | 
| 201         | MK_REP     | 3年10ヶ月 | 
| 114         | ST_CLERK   | 1年9ヶ月  | 
| 122         | ST_CLERK   | 0年11ヶ月 | 
| 200         | AD_ASST    | 5年9ヶ月  | 
| 176         | SA_REP     | 0年9ヶ月  | 
| 176         | SA_MAN     | 0年11ヶ月 | 
| 200         | AC_ACCOUNT | 4年5ヶ月  | 

## 解答例
```sql
SELECT
    employee_id,
    job_id,
    TRUNC(MONTHS_BETWEEN(end_date, start_date) / 12) || '年' || 
      MOD(TRUNC(MONTHS_BETWEEN(end_date, start_date)), 12) || 'ヶ月' 
                                                           AS "期間"
FROM
    hr.job_history
```

## 解説
Oracleデータベースで、2つの日付の間の「月数」を計算する際に使われるのが、**MONTHS_BETWEEN関数**になります。

この関数は、単に日数を30で割るような計算ではなく、カレンダーに基づいた月数を返します。特に **「整数になる条件」** を知っておくと、勤続年数や契約期間の計算などで非常に重宝します。

### 1. 基本的な構文

```sql
MONTHS_BETWEEN( 日付1, 日付2 )
```

* **結果**: 「日付1 － 日付2」の結果を**月数**で返します。
* **日付1の方が未来の場合**: 正の数（例：`2`）が返ります。
* **日付2の方が未来の場合**: 負の数（例：`-2`）が返ります。

### 2. 結果が「整数」になる2つのパターン

MONTHS_BETWEENは、以下のいずれかの場合に必ず **整数（端数なし）** を返します。

#### ① 日にちが同じ場合

月が違っても、日が一致していれば整数になります。

* 例：2024/01/15 と 2024/02/15 の間 → **`1`**

#### ② どちらも「月末日」の場合

日が違っていても、それぞれの月の最終日であれば整数になります。ここが非常に便利なポイントです。

* 例：2024/02/29（末日） と 2024/03/31（末日） の間 → **`1`**

### 3. 「端数」が出る場合の計算ルール

上記以外のパターンの場合、結果は小数（端数）を含みます。
Oracleは内部的に **「1ヶ月＝31日」** として計算を行い、不足している日数分を小数として算出します。

| 比較する日付（日付1, 日付2） | 結果 | 解説 |
| --- | --- | --- |
| **2024/02/01, 2024/01/01** | **`1`** | 日にちが同じなので整数 |
| **2024/03/31, 2024/02/29** | **`1`** | どちらも月末なので整数 |
| **2024/01/31, 2024/01/01** | **`0.9677...`** | 1ヶ月に満たないため、31日分で計算された小数 |
| **2024/01/01, 2024/02/01** | **`-1`** | 日付1の方が過去なのでマイナス |

### 4. よく使われる実務での例

#### 例A：勤続月数を計算する

入社日（`hire_date`）から今日までの月数を計算し、端数を切り捨てて表示する場合：

```sql
SELECT 
    last_name, 
    TRUNC(MONTHS_BETWEEN(SYSDATE, hire_date)) AS 勤続月数
FROM 
    employees;
```

#### 例B：年齢を計算する
月数を12で割ることで、年齢（または年数）を出すことができます。

```sql
SELECT 
    TRUNC(MONTHS_BETWEEN(SYSDATE, birth_date) / 12) AS 年齢
FROM 
    customers;
```

# 問題7-8（日時の範囲指定） *Lv2*
### COスキーマ`ORDERS`テーブルの`ORDER_TMS`が2021年6月であるデータを取得してください。対象は`CUSTOMER_ID`が350から359までとしてください。（順不同）

## 期待する結果
| CUSTOMER_ID | ORDER_TMS                   | 
| ----------- | --------------------------- | 
| 351         | 2021-06-03T16:04:46.024146Z | 
| 354         | 2021-06-08T04:17:12.071275Z | 
| 350         | 2021-06-13T18:10:25.420066Z | 
| 352         | 2021-06-30T18:09:40.698324Z | 

## 解答例
```sql
SELECT
    customer_id,
    order_tms
FROM
    co.orders
WHERE
    customer_id BETWEEN 350 AND 359
    AND order_tms >= TIMESTAMP '2021-06-01 00:00:00'
    AND order_tms  < TIMESTAMP '2021-07-01 00:00:00'
```

## 解説
TIMESTAMP型で日付範囲を指定する場合、**「比較演算子（`>=` と `<`）を組み合わせる方法」** が最も適切で、バグが発生しにくい推奨される方法です。

### 1. 最も推奨される書き方：比較演算子の組み合わせ（解答例）
例えば、「2024年3月1日のデータすべて」を取得したい場合、以下のように書くのがベストプラクティスです。

```sql
SELECT *
FROM orders
WHERE order_timestamp >= TIMESTAMP '2024-03-01 00:00:00'
  AND order_timestamp <  TIMESTAMP '2024-03-02 00:00:00';
```

#### なぜこれが「適切」なのか？
* **精度の問題を回避:** TIMESTAMP型は「1秒未満（ミリ秒やナノ秒）」の値を持ちます。
* **境界値の扱い:** `BETWEEN` を使うと、翌日の 00:00:00.000... というジャストの瞬間のデータが含まれてしまう可能性があります。`<`（未満）を使うことで、**「3月1日の終わりギリギリまで」** を確実に指定できます。


### 2. `BETWEEN` を使う際の注意点

`BETWEEN` は「以上、かつ、以下」という意味です。

```sql
-- あまり推奨されない例
WHERE order_timestamp BETWEEN TIMESTAMP '2024-03-01 00:00:00' 
                          AND TIMESTAMP '2024-03-01 23:59:59';
```

### 注意点：
* **取りこぼしのリスク:** `23:59:59.500` のような、秒未満のデータが存在する場合、上記のクエリではそのデータが抽出されません。
* **精度の指定が困難:** TIMESTAMPの精度に合わせて `.999999` と書くのは現実的ではなく、可読性も下がります。

## 解説
SQLにおける日付の「トリム（切り捨て）」は、**`TRUNC` 関数** を指します。
一言でいうと、**「時計の針を強制的に巻き戻して、特定の区切り（月初や年初など）に揃える」** 操作のことです。ハサミで日付の細かい部分を切り落とすイメージです。

### 1. なぜ「切り捨て」が必要なのか？
データベースの日付データ（DATE型やTIMESTAMP型）は、目に見えなくても裏側では「時・分・秒」まで持っています。

* **問題**: 「2023-10-25 10:30:00」と「2023-10-25 18:45:00」は、人間にとっては同じ「10月25日」ですが、**SQLでは「別物」として扱われます。**
* **解決**: `TRUNC` を使って時分秒を切り捨て「2023-10-25 00:00:00」に統一することで、集計や比較ができるようになります。


### 2. よく使う切り捨てパターン
`TRUNC(日付, '書式')` の形で使います。

| 書式（フォーマット） | 切り捨ての結果 | 用途 |
| :--- | :--- | :--- |
| **指定なし (or 'DD')** | その日の **00:00:00** | 「日別」の集計 |
| **'MM'** | その月の **1日 00:00:00** | 「月別」の集計（売上レポートなど） |
| **'Q'** | その四半期の **初日** | 「四半期」のレポート |
| **'YYYY' (or 'YY')** | その年の **1月1日** | 「年次」の推移分析 |
| **'IW'** | その週の **月曜日** | 「週次」の進捗管理 |

### 3. `TRUNC` と `TO_CHAR` の決定的な違い

ここが初心者が一番迷うポイントです。

* **`TO_CHAR(date, 'YYYYMM')`**:
    * 日付を「202310」という **「文字列」** に変える。
    * 見た目を整えるのには良いが、**日付としての計算（1ヶ月足すなど）ができなくなる。**
* **`TRUNC(date, 'MM')`**:
    * 日付を「2023-10-01」という **「日付型（DATE）」** のまま保つ。
    * **そのまま日付順に並び替え（ORDER BY）ができ、日付計算も続けられる。**

> [!TIP]
> **「計算や並び替えをするなら TRUNC、画面に表示する最終形態なら TO_CHAR」** と使い分けるのがプロの技です。

### 4. 実務での活用例：月別売上集計

```sql
SELECT
    TRUNC(order_tms, 'MM') AS order_month, -- すべて「月初」に揃える
    SUM(amount)            AS total_sales
FROM
    orders
GROUP BY
    TRUNC(order_tms, 'MM') -- 月初ごとにグループ化
ORDER BY
    order_month;           -- 日付型なので、正しく時系列で並ぶ
```
