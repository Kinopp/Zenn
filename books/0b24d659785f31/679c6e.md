# 問題5-1（文字列の結合） *Lv1*
### HRスキーマ`EMPLOYEE`テーブルより`MANAGER_ID`が101の従業員の`FIRST_NAME`と`LAST_NAME`を結合した「NAME」を取得してください。FIRST_NAMEとLAST_NAMEの間に半角スペースを入れてください。（順不同）
  
## 期待する結果
| EMPLOYEE_ID | NAME            | 
| ----------- | --------------- | 
| 108         | Nancy Greenberg | 
| 200         | Jennifer Whalen | 
| 203         | Susan Mavris    | 
| 204         | Hermann Baer    | 
| 205         | Shelley Higgins |  


## 解答例
```sql:例1：「||」演算子を使用
SELECT
    employee_id,
    first_name || ' ' || last_name AS NAME
FROM
    hr.employees
WHERE
    manager_id = 101
```

```sql:例2：CONCAT 関数を使用
SELECT
    employee_id,
    CONCAT(
        CONCAT(first_name, ' '),
        last_name
    ) AS NAME
FROM
    hr.employees
WHERE
    manager_id = 101
```

## 解説
文字列を結合して「氏名」を作成する、実務でも非常によく使う演習です。
Oracle SQLには複数の結合方法がありますが、今回の2つの例には **「書きやすさ」と「制限」** という大きな違いがあります。

それぞれの解説と、現場でどちらを選ぶべきかの判断基準をまとめました。

### 例1：`||`（パイプ）演算子を使用
```sql
first_name || ' ' || last_name
```
* **特徴**: 記号を使って直感的に文字列をつなげます。
* **メリット**: **3つ以上の要素（名、スペース、姓など）を一度に繋げられる**ため、非常にスッキリと書けます。

### 例2：`CONCAT` 関数を使用
```sql
CONCAT(CONCAT(first_name, ' '), last_name)
```
* **特徴**: 関数を使って文字列を結合します。
* **制限（重要）**: Oracleの `CONCAT` 関数は **「2つの引数」しか受け取れません。**
* **仕組み**: 3つ以上の要素を繋ぐには、例のように関数の中にさらに関数を入れる「ネスト（入れ子）」構造にする必要があり、少し読みづらくなります。


### 実践的な注意点

#### 1. 現場では `||` 演算子が圧倒的に主流
実務のSQLで `CONCAT` 関数が使われることは、Oracleにおいては稀です。
* **理由**: `||` の方が視認性が高く、要素が増えても `||` を書き足すだけで済むからです。
* **他DBとの違い**: MySQLなどは `CONCAT` で複数の引数を取れますが、Oracleは「2つまで」という頑固な仕様があるため、`||` を使うのが一般的です。

#### 2. 「NULL」を結合したときの挙動（Oracleの優しさ）
ここはOracle SQLの大きな特徴です。
* **Oracleの場合**: `文字列 || NULL` は **「元の文字列」** になります。（例：`'Steven' || NULL` は `'Steven'`）
* **他のDB（SQL ServerやPostgreSQLなど）**: `文字列 + NULL` は **`NULL`** に消えてしまうことが多いです。
Oracleはこの点において非常に寛容で、データが一部欠けていても氏名が消滅しないというメリットがあります。


## 比較まとめ

| 項目 | パイプ演算子 (例1) | `CONCAT` 関数 (例2) |
| :--- | :--- | :--- |
| **結合できる数** | **制限なし（いくつでもOK）** | **2つまで（ネストが必要）** |
| **可読性（読みやすさ）** | ◎ 非常に高い | △ 複雑になると読みづらい |
| **ポータビリティ** | △ DBによって記号が異なる | ○ 標準SQLに近い（が引数制限がある） |
| **推奨度（Oracle）** | **◎ 第一選択** | △ 積極的には使わない |

# 問題5-2（文字列の分割＜文字数で分割＞） *Lv1*
### HRスキーマ`EMPLOYEE`テーブルより、`MANAGER_ID`が101のデータに対し、`1.`で始まる`PHONE_NUMBER`（ピリオドで区切られた1桁-３桁-３桁-４桁の文字列）を4つに分割してください。当問題では、「桁数」で分割するようにしてください。（順不同）
  
## 期待する結果
| EMPLOYEE_ID | PHONE_NUMBER   | PN1 | PN2 | PN3 | PN4 | 
| ----------- | -------------- | --- | --- | --- | --- | 
| 200         | 1.515.555.0165 | 1   | 515 | 555 | 0165 | 
| 203         | 1.515.555.0168 | 1   | 515 | 555 | 0168 | 
| 204         | 1.515.555.0169 | 1   | 515 | 555 | 0169 | 
| 205         | 1.515.555.0170 | 1   | 515 | 555 | 0170 | 
| 108         | 1.515.555.0108 | 1   | 515 | 555 | 0108 | 


## 解答例
```sql
SELECT
    employee_id,
    phone_number,
    SUBSTR(phone_number, 1, 1) AS PN1,
    SUBSTR(phone_number, 3, 3) AS PN2,
    SUBSTR(phone_number, 7, 3) AS PN3,
    SUBSTR(phone_number, 11, 4) AS PN4
FROM
    hr.employees
WHERE
    manager_id = 101
AND phone_number LIKE '1.%'
```

## 解説
今回は、文字列の中から特定の場所を指定して抜き出す **`SUBSTR` 関数** の演習です。
データの「整形」や、電話番号・郵便番号・品番などの「コードの一部」を抽出する際に、実務で最も多用される関数の一つです。

このクエリーがどのように文字を数えているのか、解説します。


### 1. `SUBSTR` 関数の仕組み
`SUBSTR(文字列, 開始位置, [文字数])` という形式で記述します。
Oracle SQLの大きな特徴は、**「1文字目を 1 と数える」** 点です（多くのプログラミング言語は 0 から始まりますが、SQLは 1 からです）。

今回のクエリーを分解するとこうなります：

* **`PN1`: `SUBSTR(phone_number, 1, 1)`**
    * 1文字目から「1文字」取得（例：'1'）
* **`PN2`: `SUBSTR(phone_number, 3, 3)`**
    * 3文字目から「3文字」取得（例：'515'）
* **`PN3`: `SUBSTR(phone_number, 7, 3)`**
    * 7文字目から「3文字」取得（例：'123'）
* **`PN4`: `SUBSTR(phone_number, 11, 4)`**
    * 11文字目から「4文字」取得（例：'4567'）



### 2. `WHERE` 句のポイント
`phone_number LIKE '1.%'`
* 最初の文字が **'1'** で、2文字目が **'.' (ドット)** であるデータを抽出しています。
* つまり、このクエリーは「国番号が 1 で始まる特定のフォーマットの電話番号」を想定した切り出しを行っています。

### 実践的な注意点

#### 1. 「区切り文字（ドット）」も1文字と数える
`SUBSTR` は、数字だけでなくドット（.）やハイフン（-）もすべて「1文字」としてカウントします。
* 今回の `PN2` が3文字目から始まっているのは、2文字目の「.」を読み飛ばしているからです。
* **罠**: もしデータの中に `1.123...` ではなく `1-123...` と `1.1.1...` のようにフォーマットが混在している場合、固定の位置指定（マジックナンバー）で切り出すと、結果にドットが混ざってしまうことがあります。

### 2. フォーマット変更に弱い
この書き方は、**「データの長さが常に一定であること」** が前提です。
* もし電話番号が `1.12.456.789` のように桁数が変わると、抽出結果がズレて使い物にならなくなります。
* **対策**: 桁数がバラバラな場合は、ドットの位置を自動で探す **`INSTR` 関数** と組み合わせて使うのがプロの技です。

### 3. 文字数（第3引数）を省略した場合
`SUBSTR(phone_number, 11)` のように最後の引数を書かなかった場合、**「11文字目から最後まですべて」** を抜き出します。末尾の文字数が不定な場合は、あえて省略するのも手です。

### 比較まとめ

| 項目 | `SUBSTR` (今回の方法) | `REGEXP_SUBSTR` (正規表現) |
| :--- | :--- | :--- |
| **手軽さ** | ◎ 非常に簡単 | △ 習熟が必要 |
| **柔軟性** | × 1文字でもズレると失敗する | ◎ 複雑なパターンに対応可能 |
| **処理速度** | ○ 速い | △ わずかに重い |

<br>
:::message
### **「マジックナンバー（固定の数字）に注意」**
実務で `SUBSTR(..., 7, 3)` と書くときは、コメントに「7文字目は市外局番の開始位置」のように理由を添えておくと、後から読む人が助かります。数字だけだと、なぜ 7 なのかが分からなくなるからです。
:::

# 問題5-3（文字列の置換１）  *Lv2*
### HRスキーマ`DEPARTMENTS`テーブルより`DEPARTMENT_ID`が100より小さいデータについて、AをXに置換してください（大文字のみ置換）。（順不同）
  
## 期待する結果
| DEPARTMENT_NAME  | DEPARTMENT_NAME_A_TO_X | 
| ---------------- | ---------------------- | 
| Administration   | Xdministration         | 
| Marketing        | Marketing              | 
| Purchasing       | Purchasing             | 
| Human Resources  | Human Resources        | 
| Shipping         | Shipping               | 
| IT               | IT                     | 
| Public Relations | Public Relations       | 
| Sales            | Sales                  | 
| Executive        | Executive              | 

## 解答例
```sql
SELECT
    department_name,
    REPLACE(department_name, 'A', 'X') AS department_name_a_to_x
FROM
    hr.departments
WHERE
    department_id < 100
```

## 解説
今回は、文字列の中にある特定の文字を別の文字に置き換える **`REPLACE` 関数** の演習です。

データの「クレンジング（表記揺れの修正）」や、特定の記号を消去したいときなど、実務でも非常に登場回数が多い関数です。このクエリーの仕組みと、Oracle SQLならではの注意点を解説します。


### 1. `REPLACE` 関数の仕組み
`REPLACE(対象列, 探す文字列, 置き換える文字列)` という形で記述します。

* **今回の処理**: `department_name` の中にある **大文字の 'A'** をすべて **大文字の 'X'** に変換します。
* **動作例**:
    * `Administration` → `Xdministration`
    * `Marketing` → `Marketing`
    * `IT` → `IT` （'A' が含まれない場合は何も起きない）


### 実践的な注意点

#### 1. 大文字・小文字を厳密に区別する
ここが最も重要なポイントです。Oracleの `REPLACE` 関数は、**大文字と小文字を別物として扱います。**
* `REPLACE('Apple', 'A', 'X')` → `Xpple` （大文字は変わる）
* `REPLACE('Apple', 'p', 'x')` → `Axxle` （小文字は変わる）
* もし「大文字も小文字も両方変えたい」場合は、`REPLACE` を2重（入れ子）にするか、正規表現関数を使う必要があります。

#### 2. 「すべて」の文字が置き換わる
`REPLACE` は、見つけた対象文字を**1つ残らず**置き換えます。
* もし「最初の1つだけ変えたい」といった細かい制御が必要な場合は、`REPLACE` ではなく `REGEXP_REPLACE`（正規表現置換）などのより高度な関数を使います。

#### 3. 文字を「消去」したい時にも使える
第3引数を省略するか、空文字（''）を指定すると、文字を消すことができます。
* 例：`REPLACE(phone_number, '-', '')`
    * これによって「090-1234-5678」を「09012345678」のように整形できます。

# 問題5-4（文字列の置換２）  *Lv2*
### HRスキーマ`DEPARTMENTS`テーブルより`DEPARTMENT_ID`が100より小さいデータについて、AをXに置換してください（大文字小文字ともに置換）。（順不同）

  
## 期待する結果
| DEPARTMENT_NAME  | DEPARTMENT_NAME_A_TO_X | 
| ---------------- | ---------------------- | 
| Administration   | Xdministrxtion         | 
| Marketing        | Mxrketing              | 
| Purchasing       | Purchxsing             | 
| Human Resources  | Humxn Resources        | 
| Shipping         | Shipping               | 
| IT               | IT                     | 
| Public Relations | Public Relxtions       | 
| Sales            | Sxles                  | 
| Executive        | Executive              | 

## 解答例
```sql
SELECT
    department_name,
    TRANSLATE(department_name, 'Aa,', 'Xx') AS department_name_a_to_x
FROM
    hr.departments
WHERE
    department_id < 100
```

## 解説
前回の `REPLACE` 関数に続き、今回は似ているようで全く性質が異なる **`TRANSLATE` 関数** です。

ここを正確に理解できていると、大量のデータを一括でクリーニングする際のスピードと柔軟性が格段に上がります。初心者の方が最も混乱しやすい「置換のルール」を中心に解説します。

### 1. `TRANSLATE` 関数の仕組み
`TRANSLATE(文字列, 置換前文字リスト, 置換後文字リスト)` という形式で、**「1文字ずつ」** 対応させて変換します。

今回の設定 `'Aa,'` を `'Xx'` に変換するルールを分解すると以下のようになります：

* **`A`** は **`X`** に変換される
* **`a`** は **`x`** に変換される
* **`,` (カンマ)** は 対応する文字がないため **削除** される



### 2. 実行例（シミュレーション）
* `Administration`
    * 最初の `A` → `X`
    * 途中の `a` → `x`
    * 結果：`Xdministrxtion`
* `Sales, Marketing`（もしカンマがあったら）
    * `a` → `x`, `,` → 削除
    * 結果：`Sxles Mxrketing`


### 実践的な注意点

#### 1. `REPLACE` との決定的な違い
ここが最大のポイントです。
* **`REPLACE`**: 「カタマリ（文字列）」で探します。
* **`TRANSLATE`**: 「1文字単位」でバラバラに探します。

> **例：'ABC' を 'Z' に変えたい場合**
> * `REPLACE('ABCDEFADBDCD', 'ABC', 'Z')` → "ABC" という並びがある時だけ "Z" になる。→ [ZDEFADBDCD]
> * `TRANSLATE('ABCDEFADBDCD', 'ABC', 'Z')` → "A" も "B" も "C" も、見つけ次第すべて変換（または削除）される。→ [ZDEFZDDD]


#### 2. 「文字の削除」という強力なテクニック
今回の例のカンマのように、**「置換前リストにあって、置換後リストにない文字」は消える** という性質があります。
これを利用して、データから不要な記号を一気に掃除することができます。
* 例：`TRANSLATE(tel, '0123456789-()', '0123456789')`
    * これで、電話番号からハイフンやカッコだけを抜き取ることができます。

#### 3. NULLの扱いに注意（Oracleの罠）
置換後文字リストを完全に「空（''）」にしてしまうと、Oracleでは **結果全体がNULL** になってしまいます。
* × `TRANSLATE(name, 'abc', '')` → 結果はすべて空（NULL）になる
* ○ `TRANSLATE(name, '#abc', '#')` → 1文字ダミー（#など）を置くことで、a, b, c だけを消すのが定石です。

### 比較まとめ

| 項目 | `REPLACE` | `TRANSLATE` |
| :--- | :--- | :--- |
| **置換の単位** | 文字列（カタマリ） | 1文字ずつ（一対一） |
| **得意なこと** | 特定の単語の書き換え | 複数文字の同時変換・削除 |
| **対応文字数** | 置換前と後で違ってOK | 原則、対応する位置で決まる |
| **処理速度** | 普通 | **非常に高速**（大量置換に強い） |

<br>
::: message
### **「暗号化やフォーマット変換の味方」**
実務では、全角英数を半角に直したり、特定の伏せ字（マスキング）を作ったりする時に `TRANSLATE` は真価を発揮します。`REPLACE` を何重にも入れ子にするくらいなら、`TRANSLATE` 1回で済ませられないか検討するのが「SQL上級者」への近道です。
:::

# 問題5-5（大文字・小文字変換）   *Lv2*
### COスキーマ`CUSTOMERS`テーブルより、`FULL_NAME`と下記の通り編集した値を比較して、一致しないものを取得してください。（順不同）
`EMAIL_ADDRESS`の
- 「@internalmail」を削除
- 「.」を半角スペースに置換
- スペースで分割した文字列をそれぞれ先頭文字を大文字にする
  
## 期待する結果
| CUSTOMER_ID | EMAIL_ADDRESS                   | FULL_NAME             | EMAIL_NAME         | 
| ----------- | ------------------------------- | --------------------- | ------------------ | 
| 123         | erda.sheer@internalmail         | erda Sheer            | Erda Sheer         | 
| 194         | norman.s.lobregat@internalmail  | Norman Simon Lobregat | Norman S Lobregat  | 
| 195         | tim.d.blasingim@internalmail    | Tim Dominic Blasingim | Tim D Blasingim    | 
| 227         | son.mclagan@internalmail        | Son McLagan           | Son Mclagan        | 
| 251         | dannie.mccoin@internalmail      | Dannie McCoin         | Dannie Mccoin      | 
| 262         | kendrick.mcmurtrey@internalmail | Kendrick McMurtrey    | Kendrick Mcmurtrey | 

## 解答例
```sql
SELECT
    CUSTOMER_ID,
    EMAIL_ADDRESS,
    FULL_NAME,
    INITCAP(replace(replace( EMAIL_ADDRESS ,'@internalmail', ''  ), '.', ' ')) as email_name
FROM
    CO.CUSTOMERS
where NOT(FULL_NAME = 
    INITCAP(replace(replace( EMAIL_ADDRESS ,'@internalmail', ''  ), '.', ' ')))
order by customer_id
```

## 解説
今回の演習は、複数の関数を組み合わせて（ネストさせて）データを加工する、より実戦的な **「データクレンジング（洗浄）」** のテクニックです。

メールアドレスから名前を推測し、登録されている氏名と一致しない「不整合なデータ」をあぶり出すという、非常に面白いクエリーです。

### 1. 関数の「入れ子（ネスト）」を解読する
このクエリーの心臓部は `INITCAP(replace(replace(...)))` です。SQLでは、**内側の関数から順番に**処理されます。

1.  **`replace(EMAIL_ADDRESS, '@internalmail', '')`**
    * まず、ドメイン部分（`@internalmail`）を消去します。
    * 例: `john.doe@internalmail` → `john.doe`
2.  **`replace(..., '.', ' ')`**
    * 次に、残った文字列のドット（`.`）を半角スペースに置き換えます。
    * 例: `john.doe` → `john doe`
3.  **`INITCAP(...)`**
    * 最後に、各単語の先頭を大文字、それ以外を小文字に整えます。
    * 例: `john doe` → `John Doe`

### 2. WHERE句のロジック：不整合の抽出
`WHERE NOT (FULL_NAME = ...)`
* 「元の `FULL_NAME`」と「メールから生成した名前」を比較し、**一致しない（NOT）** 行だけを抽出しています。
* これにより、結婚で苗字が変わったのにメールアドレスが古いままの人や、入力ミスがある人を特定できます。

### 実践的な注意点

#### 1. `INITCAP` の挙動に注意
`INITCAP` は非常に便利ですが、**「英字以外の文字（スペースや記号）」の直後の文字をすべて大文字にする**という性質があります。
* `o'brian` → `O'Brian` （これは嬉しい挙動）
* `mcdonald` → `Mcdonald` （`McD...` にしたい場合は、この関数だけでは不十分）
実務では、特殊な綴りに対応できないケースがあることを覚えておきましょう。

#### 2. パフォーマンスへの影響
`WHERE` 句に関数をたくさん書くと、データベースは全行に対してその計算を行わなければならず、処理が重くなる（インデックスが使えなくなる）ことがあります。
大量のデータを扱う場合は、加工済みの値を保存しておく列（計算列）を作るなどの設計的な工夫が検討されます。

:::message
### **「関数のネストは、玉ねぎの皮を剥くように読む」**
複雑な式に出会ったら、一番内側のカッコから順に「何が起きているか」をメモしていくのが解読のコツです。今回のクエリーのように、結果を `FULL_NAME` と比較することで、データの品質（質）をチェックできるのは、非常に価値のあるスキルですよ！
:::

# 問題5-6（文字列のパディング）  *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、下記の通り編集した値を取得してください。（順不同）
* **`EMPLOYEE_ID`を6桁にする。6桁に満たない場合、前ゼロ埋めして表示する。**
* **`EMAIL`を10桁にする。10桁に満たない場合、右スペースを「X」で埋めて表示する。**

## 期待する結果
| EMP_ID6 | EMAIL10    | 
| ------- | ---------- | 
| 000201  | MMARTINEXX | 
| 000202  | PDAVISXXXX | 
| 000203  | SJACOBSXXX | 
| 000204  | HBROWNXXXX | 
| 000205  | SHIGGINSXX | 
| 000206  | WGIETZXXXX | 

## 解答例
```sql
SELECT
    LPAD(employee_id, 6, '0') as "EMP_ID6",
    RPAD(email, 10, 'X') as "EMAIL10"
FROM
    hr.employees
WHERE
    employee_id > 200
```

## 解説
今回の演習は、データの「見た目」を整える **パディング処理（埋め込み）** です。
`LPAD` と `RPAD` は、帳票出力や固定長ファイルの作成、あるいはIDの桁数を揃える際など、実務の「最後の仕上げ」で非常によく使われる関数です。

クエリーの仕組みと、知っておかないと怖い「切り捨て」の仕様について解説します。

### 1. LPAD（左パディング）
`LPAD(employee_id, 6, '0')`
* **意味**: `employee_id` が **6文字** になるまで、**左側**に `'0'` を埋めます。
* **用途**: いわゆる「ゼロ埋め（ゼロパディング）」です。
    * 例：`201` → `000201`

### 2. RPAD（右パディング）
`RPAD(email, 10, 'X')`
* **意味**: `email` が **10文字** になるまで、**右側**に `'X'` を埋めます。
* **用途**: 文字列を一定の長さに揃えたい時や、簡易的なマスキングに使われます。
    * 例：`SKING` → `SKINGXXXXX`


### 実践的な注意点：ここがハマりどころ！

#### 1. 「切り捨て」の恐怖
ここが最も重要な仕様です。もし元の文字列が、指定した長さよりも**長かった場合**、どうなるでしょうか？
* **答え**: 指定した長さで **切り捨て（カット）** されます。
* 例：`RPAD('ALEXANDER', 5, 'X')` → `ALEXA`
* 「足りない分を埋める」だけでなく「長さを強制的に固定する」動きをするため、名前やメールアドレスなどの可変長データを扱う際は、データが欠落しないよう十分な長さを指定する必要があります。

#### 2. 数値から文字列への「暗黙の型変換」
今回の `employee_id` は数値型ですが、`LPAD` は文字列を扱う関数です。
* Oracleは内部で自動的に数値を文字列に変換して処理してくれますが、本来は `LPAD(TO_CHAR(employee_id), 6, '0')` と書くのが、データ型を意識した「丁寧なSQL」です。

#### 3. スペースでのパディング（デフォルト）
第3引数（埋める文字）を省略すると、**半角スペース**で埋められます。
* `RPAD(last_name, 20)`
* これを実行すると、名前の後ろがスペースで埋まり、等幅フォントで表示した際に縦のラインが綺麗に揃うようになります。


### 比較まとめ

| 関数 | 埋める方向 | 主な用途 |
| :--- | :--- | :--- |
| **`LPAD`** | **左** (Left) | IDの桁揃え（0001など）、右寄せ表示 |
| **`RPAD`** | **右** (Right) | 固定長データの作成、左寄せ表示、伏せ字 |

### COスキーマ`SHIPMENTS`テーブルより、`SHIPMENT_ID`が150から159のデータに対して、下記の通り編集した値を取得してください。（順不同）
* **`DELIVERY_ADDRESS`の先頭にある「Chatfield,」をトリム（除去）する**
* **`DELIVERY_ADDRESS`の末尾にある「USA」をトリム（除去）する。**

## 期待する結果
| SHIPMENT_ID | DELIVERY_ADDRESS           | L_TRIM                     | R_TRIM                 | 
| ----------- | -------------------------- | -------------------------- | ---------------------- | 
| 150         | Boca Raton, FL 33433 USA   | Boca Raton, FL 33433 USA   | Boca Raton, FL 33433   | 
| 151         | Rutland, OH 45775 USA      | Rutland, OH 45775 USA      | Rutland, OH 45775      | 
| 153         | Lakeland, FL 33803 USA     | Lakeland, FL 33803 USA     | Lakeland, FL 33803     | 
| 154         | Falls Church, VA 22043 USA | Falls Church, VA 22043 USA | Falls Church, VA 22043 | 
| 155         | Chatfield, OH 44825 USA    | OH 44825 USA               | Chatfield, OH 44825    | 
| 156         | Chatfield, OH 44825 USA    | OH 44825 USA               | Chatfield, OH 44825    | 
| 157         | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017     | 
| 158         | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017     | 
| 159         | Fonda, NY 12068 USA        | Fonda, NY 12068 USA        | Fonda, NY 12068        |

## 解答例
```sql
SELECT
    shipment_id,
    delivery_address,
    LTRIM(delivery_address, 'Chatfield,') AS "L_TRIM",
    RTRIM(delivery_address, 'USA')        AS "R_TRIM"
FROM
    co.shipments
WHERE
    shipment_id BETWEEN 150 AND 159
```

## 解説
今回の演習は、文字列の端にある不要な文字を削ぎ落とす **`LTRIM`（左トリム）** と **`RTRIM`（右トリム）** です。

一見、「特定の単語を消している」ように見えますが、実はここに**初心者の方が最もハマりやすい仕様**が隠されています。実務でバグを出さないための「超重要ポイント」を解説します。


### 1. 関数の基本的な動き
* **`LTRIM(文字列, [削除キャラクタセット])`**: 文字列の **左側（先頭）** から、指定した文字が含まれている限り削除し続けます。
* **`RTRIM(文字列, [削除キャラクタセット])`**: 文字列の **右側（末尾）** から、指定した文字が含まれている限り削除し続けます。

### 2. 今回の具体的な処理
* **`LTRIM(..., 'Chatfield,')`**: 左端にある「C」「h」「a」「t」「f」「i」「e」「l」「d」「,」のいずれかに該当する文字をすべて消します。
* **`RTRIM(..., 'USA')`**: 右端にある「U」「S」「A」のいずれかに該当する文字をすべて消します。


### 実務での注意点：最大の落とし穴

#### 「単語」ではなく「文字の集合」として扱われる
ここが一番の注意点です。Oracleの `LTRIM/RTRIM` は、第2引数を「一つの単語」として見るのではなく、**「一文字一文字のリスト」**として扱います。

> **例：`RTRIM('USA_USA', 'USA')` の結果はどうなる？**
> * 期待：`'USA_'` （右側の "USA" という単語が消える）
> * **現実：`'USA_'`** （これはOK）
> 
> **例：`RTRIM('ABBA', 'BA')` の結果はどうなる？**
> * 期待：`'AB'`？
> * **現実：`''` (空文字)** >   右端から見て 'A' も 'B' もリストに含まれているため、全部削られてしまいます。

#### 1. スペース削除がデフォルト
第2引数を省略すると、**半角スペース**を削除します。
実務で最も多いのは `RTRIM(column)` という書き方で、データの後ろに付いてしまった余計なスペースを掃除するのに使われます。

#### 2. 大文字・小文字は区別される
`RTRIM(address, 'usa')` と書いても、データが `'USA'`（大文字）であれば一文字も削除されません。

#### 3. 文字列の途中の文字は消せない
あくまで「端」から削る関数です。
`'Chatfield, USA'` というデータに対して、途中のカンマだけを消したい場合は、以前学習した **`REPLACE`** を使う必要があります。

## 比較まとめ

| 関数 | 削除する場所 | 性質 | 推奨シーン |
| :--- | :--- | :--- | :--- |
| **`LTRIM`** | 左端のみ | 指定した文字群を削る | 先頭のゼロ(0)や記号の除去 |
| **`RTRIM`** | 右端のみ | 指定した文字群を削る | 末尾のスペースや改行の除去 |
| **`TRIM`** | 両端 | 基本は1文字のみ指定可 | 両端の空白を一気に消す時 |

# 問題5-8（文字列の長さ）  *Lv2*
### OEスキーマ`PRODUCT_INFORMATION`の`PRODUCT_NAME`が24文字のデータを取得してください。（順不同）

## 期待する結果
| NAME_LENGTH              | 
| ------------------------ | 
| Laptop 128/12/56/v90/110 | 
| Plasma Monitor 10/LE/VGA | 
| Manual - Vision Tools2.0 | 
| Business Cards Box - 250 | 
| Business Cards - 1000/2L | 

## 解答例
```sql
SELECT
    product_name AS name_length
FROM
    oe.product_information
WHERE
    LENGTH(product_name) = 24
```

## 解説
今回は、文字列の長さを測る **`LENGTH` 関数** の演習です。

特定の文字数のデータを探したり、入力文字数のバリデーション（検証）をしたりと、実務のあらゆる場面で「物差し」として活躍する関数です。シンプルながらも奥が深い、注意すべきポイントを解説します。

### 動作のポイント
* **`LENGTH(product_name)`**: `product_name`（商品名）の文字数をカウントします。
* **`WHERE ... = 24`**: 文字数がちょうど24文字のものだけを抽出します。


### 実践的な注意点

#### 1. 「文字数」と「バイト数」の使い分け
日本国内のシステム開発では、ここが最大の鬼門です。

* **`LENGTH`**: **文字の数**を数えます（「Oracle」も「オラクル」も4文字）。
* **`LENGTHB`**: **バイト数**を数えます。
    * データベースの設定（AL32UTF8など）によりますが、全角文字は1文字で3バイト消費することが多いです。
    * 「合計30バイト以内の商品名」という制限がある場合は、`LENGTHB` を使う必要があります。



#### 2. インデックスへの影響（パフォーマンス）
`WHERE` 句の左辺に関数を書く（`WHERE LENGTH(列) = ...`）と、その列にインデックスが貼ってあっても、**原則としてインデックスが使われなくなります。**
* 100万件のデータがある場合、全件の文字数をその場で計算し直すため、処理が非常に重くなります。
* 大規模なシステムで文字数検索を頻繁に行う場合は、あらかじめ「文字数」を保存しておく専用の列を作るなどの工夫が必要です。

#### 3. 空白文字の存在
「見た目は同じ24文字なのに、検索にヒットしない」というトラブルがよくあります。
* 原因の多くは、データの末尾に目に見えない **半角スペース** が入っていることです。
* より確実に「文字が入っている部分だけ」を測りたい場合は、`LENGTH(RTRIM(product_name))` のように、空白を削る関数を組み合わせて使います。


### 比較まとめ

| 関数 | 単位 | 特徴 |
| :--- | :--- | :--- |
| **`LENGTH`** | 文字数 | 人間が見た時の文字の数。 |
| **`LENGTHB`** | バイト数 | 物理的なデータの重さ。通信や保存容量のチェック用。 |
| **`VSIZE`** | バイト数 | 内部的な格納サイズ。LENGTHBとほぼ同じだがNULLの扱いが異なる。 |

# 問題5-9（NULLの変換）  *Lv2*
### HRスキーマ`DEPARTMENTS`より、`DEPARTMENT_ID`が100から150のデータに対し、`DEPARTMENT_ID`とそれに対応する`MANAGER_ID`を取得してください。ただし、`MANAGER_ID`がNULLの場合は「0」に変換してください。（順不同）

## 期待する結果
| DEPARTMENT_ID | MANAGER_ID | 
| ------------- | ---------- | 
| 100           | 108        | 
| 110           | 205        | 
| 120           | 0          | 
| 130           | 0          | 
| 140           | 0          | 
| 150           | 0          | 

## 解答例
```sql:例1：NVLを使用
SELECT
    department_id,
    NVL(manager_id, 0) AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例2：NVL2を使用
SELECT
    department_id,
    NVL2(manager_id, manager_id ,0)  AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例3：COALESCEを使用
SELECT
    department_id,
    COALESCE(manager_id ,0) AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例4：DECODEを使用
SELECT
    department_id,
    DECODE(manager_id, null, 0, manager_id) AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例5：CASE式を使用
SELECT
    department_id,
    CASE WHEN manager_id IS NULL THEN 0 ELSE manager_id END AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```

## 解説
Oracle SQLにおいて、データがない状態（NULL）をどう扱うかは、バグを防ぎ、見やすいレポートを作るための最重要スキルの一つです。

提示された5つの例は、すべて「マネージャーが不在（NULL）なら 0 を表示する」という同じ結果を返しますが、**実務での使い分け**には明確な理由があります。それぞれの特徴を整理しました。


### 例1：`NVL(manager_id, 0)` 【王道・シンプル】
* **特徴**: Oracle独自の最も有名な関数です。「NULLならこれに変える」という直感的な動きをします。
* **実務**: 最も多用されます。引数が2つだけなので、単純な置き換えに最適です。

### 例2：`NVL2(manager_id, manager_id, 0)` 【多機能】
* **特徴**: `NVL2(対象, NULLじゃない時の値, NULLの時の値)` という3つの引数を取ります。
* **実務**: 「値がある場合とない場合で、全く別の加工をしたい時」に便利です。今回の例のように同じ値を返すだけなら、`NVL` の方がスマートです。

### 例3：`COALESCE(manager_id, 0)` 【標準・柔軟】
* **特徴**: 標準SQL（どのDBでも使える）の関数です。引数をいくつでも並べられ、**「最初に見つかったNULLでない値」** を返します。
* **実務**: `COALESCE(携帯, 自宅, 勤務先, '連絡先なし')` のように、優先順位をつけて値を探す時に真価を発揮します。

### 例4：`DECODE(manager_id, null, 0, manager_id)` 【職人芸】
* **特徴**: Oracle特有の関数で、If-Then-Elseのロジックを1行で書けます。
* **実務**: 古いシステムのコードでよく見かけますが、現在は `CASE` 式に取って代わられつつあります。

### 例5：`CASE式` 【可読性・最強】
* **特徴**: 標準SQLであり、最も読みやすく、複雑な条件にも対応できます。
* **実務**: 複雑な分岐が必要な場合はこれ一択です。



### 実践的な注意点：エンジニアが意識すること

#### 1. データ型を一致させる（超重要）
すべてのNULL関連関数において、**「元の列の型」と「置き換える値の型」は一致させる**のが鉄則です。
* **OK**: `NVL(manager_id, 0)` （数値型 ⇄ 数値型）
* **NG**: `NVL(manager_id, 'なし')` （数値型 ⇄ 文字列型 → エラーまたは予期せぬ型変換）
> 文字列で「なし」と出したい場合は、`NVL(TO_CHAR(manager_id), 'なし')` と明示的に変換しましょう。

#### 2. パフォーマンスの微妙な違い
`NVL` と `COALESCE` には、内部的な「評価タイミング」の違いがあります。
* **`NVL`**: 第2引数が計算式の場合、NULLであってもなくても計算を実行します。
* **`COALESCE`**: 必要になるまで（第1引数がNULLであると判明するまで）第2引数を計算しません（短絡評価）。
> 第2引数に重い関数を入れる場合は、`COALESCE` の方が効率的になることがあります。

### 比較まとめ

| 関数・式 | 種類 | 推奨されるシーン | 備考 |
| :--- | :--- | :--- | :--- |
| **`NVL`** | Oracle独自 | **単純なNULL置換** | 最も一般的 |
| **`NVL2`** | Oracle独自 | NULL時/非NULL時で処理を分けたい時 | 引数が3つ |
| **`COALESCE`** | **標準SQL** | 複数の候補から値を探したい時 | 汎用性が高い |
| **`DECODE`** | Oracle独自 | 短い等価比較の分岐 | 読みづらくなることも |
| **`CASE`** | **標準SQL** | **複雑な条件分岐** | 可読性が最も高い |

> **「結局どれを使えばいい？」**
> 迷ったら、**単純な置き換えなら `NVL`、複数の候補があるなら `COALESCE`、複雑なロジックなら `CASE`** を選ぶのが、現代のOracleエンジニアのスタンダードな作法です。

# 問題5-10（文字列の分割＜正規表現を使用＞） *Lv3*
### OEスキーマ`CUSTOMERS`テーブルの顧客の所得レベル（'INCOME_LEVEL'）を、「RANK」「LOWER」「UPPER」の3つの項目に切り分けて抽出してください。（順不同）
- **`CUSTOMER_ID` が 150 ～ 159 の範囲に該当する顧客を対象とします。**
- **元のデータ形式が「<RANK>: <LOWER> - <UPPER>」となっているものを、それぞれの項目へ分割します。**
- **文字列の内容に応じて、LOWER（下限）とUPPER（上限）には以下のように値を設定します。**

  | 元の表記パターン | RANK | LOWER（下限） | UPPER（上限） |
  | --- | --- | --- | --- |
  | **通常形式**<br>`A: 1000 - 2000` | A | 1000 | 2000 |
  | **Below（～未満）**<br>`A: Below 1000` | A | **`<`** | 1000 |
  | **and above（～以上）**<br>`A: 1000 and above` | A | 1000 | **`>=`** |


## 期待する結果
| CUSTOMER_ID | INCOME_LEVEL         | RANK | LOWER   | UPPER   | 
| ----------- | -------------------- | ---- | ------- | ------- | 
| 150         | D: 70,000 - 89,999   | D    | 70,000  | 89,999  | 
| 151         | F: 110,000 - 129,999 | F    | 110,000 | 129,999 | 
| 152         | A: Below 30,000      | A    | <       | 30000   | 
| 153         | I: 170,000 - 189,999 | I    | 170,000 | 189,999 | 
| 154         | L: 300,000 and above | L    | 300000  | >=      | 
| 155         | E: 90,000 - 109,999  | E    | 90,000  | 109,999 | 
| 156         | F: 110,000 - 129,999 | F    | 110,000 | 129,999 | 
| 157         | G: 130,000 - 149,999 | G    | 130,000 | 149,999 | 
| 158         | J: 190,000 - 249,999 | J    | 190,000 | 249,999 | 
| 159         | G: 130,000 - 149,999 | G    | 130,000 | 149,999 | 

## 解答例
```sql
SELECT
    customer_id,
    income_level,
    -- 1番目の項目（レベル記号など）
    TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 1)) AS rank,
    -- 2番目の項目（下限値など）
    CASE
        WHEN ( REGEXP_LIKE ( income_level,
                             'Below' ) ) THEN
            '<'
        WHEN ( REGEXP_LIKE ( income_level,
                             'above' ) ) THEN
            REGEXP_REPLACE(
                TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 2)),
                '[^0-9]',
                ''
            )
        ELSE
            TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 2))
    END                                               AS "LOWER",

    -- 3番目の項目（上限値など）
    CASE
        WHEN ( REGEXP_LIKE ( income_level,
                             'Below' ) ) THEN
            REGEXP_REPLACE(
                TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 2)),
                '[^0-9]',
                ''
            )
        WHEN ( REGEXP_LIKE ( income_level,
                             'above' ) ) THEN
            '>='
        ELSE
            TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 3))
    END                                               AS "UPPER"
FROM
    oe.customers
WHERE customer_id BETWEEN  150 AND 159
```

## 解説
今回の演習は、**「正規表現（Regular Expression）」** を駆使した非常に高度な内容です。

実務では、一つのカラムに「A: 10000 - 20000」のように複数の情報が混ざって保存されている（非正規化された）データを、分析のためにバラバラに分解する「データクレンジング」の工程でこのテクニックが真価を発揮します。

複雑なロジックを解きほぐして解説します。

### 1. `REGEXP_SUBSTR` による分割
```sql
REGEXP_SUBSTR(income_level, '[^:-]+', 1, 1)
```
* **パターン `[^:-]+`**: 「`:`（コロン）でも `-`（ハイフン）でもない文字」が1つ以上続く塊を指します。
* これにより、記号を「区切り文字」として扱い、文字列を分割しています。
    * 4番目の引数（1, 2, 3...）を変えることで、1番目の塊（ランク）、2番目の塊（数値など）を自在に抜き出しています。

### 2. `REGEXP_REPLACE` による数字の抽出
```sql
REGEXP_REPLACE(..., '[^0-9]', '')
```
* **パターン `[^0-9]`**: 「0から9の数字**ではない**文字」をすべて空文字（`''`）に置き換えます。
* これにより、文字列の中に混じっている「$」や「,（カンマ）」、あるいは「Below」といった文字を削ぎ落とし、**純粋な数字だけ**を抽出しています。

### 3. `CASE` 式による文脈判断
`income_level` の中身が「〜以下（Below）」なのか「〜以上（above）」なのかを `REGEXP_LIKE` で判定し、出力する記号や数値を切り替えています。
* **Below の場合**: 下限（LOWER）を `<` とし、上限（UPPER）に抽出した数値をセット。
* **above の場合**: 下限（LOWER）に数値をセットし、上限（UPPER）を `>=` と設定。

### 実践的な注意点

#### 1. 正規表現は「諸刃の剣」
正規表現は非常に強力ですが、通常の `SUBSTR` や `REPLACE` に比べて**CPU負荷が高い**という側面があります。
* 数百万件のデータに対して複雑な正規表現を連発すると、検索速度が著しく低下することがあります。
* 固定長データや単純な区切りであれば、通常の文字列関数の組み合わせで書けないか一度検討するのがプロの視点です。

#### 2. フォーマットの「揺れ」に弱い
今回の正規表現 `[^:-]+` は、区切り文字が `:` か `-` であることを前提としています。
* もしデータの中に `A; 10000 / 20000` のように、セミコロンやスラッシュで区切られたイレギュラーな行が混じると、正しく分割できずに「空振り」してしまいます。
* 実務では、事前に `DISTINCT` などを使って、**パターンの種類がすべて網羅できているか**を確認することが重要です。

#### 3. 非正規化データへの警鐘
そもそも、一つのカラムに複数の意味（ランク、下限、上限）を持たせるのは、データベース設計としては「あまり良くない状態」です。
* このようなクエリーを頻繁に書く必要がある場合、システム改修のタイミングで「ランク」「下限値」「上限値」を**別のカラムに分けて保存する**ように設計変更を提案するのが、エンジニアとしての本質的な解決策になります。

### 変換パターンのまとめ

| 元のデータ例 | rank | LOWER | UPPER | 解説 |
| :--- | :--- | :--- | :--- | :--- |
| `A: 10000 - 20000` | A | 10000 | 20000 | 通常の範囲指定 |
| `B: Below 5000` | B | < | 5000 | Belowを検知して置換 |
| `C: 50000 and above` | C | 50000 | >= | aboveを検知して置換 |

# 【完全版】問題5-11（文字列のマスキング） *Lv3*
### ERスキーマ`EMPLOYEE`テーブルより`DEPARTMENT_ID`=100の従業員のFIRST_NAME、LAST_NAME、EMAIL、PHONE_NUMBERを取得する。
- **ただし、以下のルールに従いマスキングをすること。**
  - **先頭３文字がアルファベットまたは数字の場合、先頭３文字を２文字ずらす**
  - **2文字ずらした際、アルファベットまたは数字の最後を超える場合、先頭に戻る**
    - （例１）**aBz**D → **cDb**D
    - （例２）**089**123 → **201**123 

## 期待する結果
| EMPLOYEE_ID | FIRST_NAME  | LAST_NAME | EMAIL    | PHONE_NUMBER   | 
| ----------- | ----------- | --------- | -------- | -------------- | 
| 108         | Pcpcy       | Itwenberg | PITUENBE | 3.715.555.0108 | 
| 109         | Fcpiel      | Hcxiet    | FHCVIET  | 3.715.555.0109 | 
| 110         | Lqjn        | Ejgn      | LEJEN    | 3.715.555.0110 | 
| 111         | Kuoael      | Uekarra   | KUEIARRA | 3.715.555.0111 | 
| 112         | Lque Manuel | Wtoan     | LOWRMAN  | 3.715.555.0112 | 
| 113         | Nwks        | Rqrp      | NRQPP    | 3.715.555.0113 | 

## 解答例
```sql
WITH
    FUNCTION masking_value (
        p_value IN VARCHAR2
    ) RETURN VARCHAR2
        DETERMINISTIC
    IS
    BEGIN
        RETURN
            CASE
                WHEN REGEXP_LIKE ( SUBSTR(p_value, 1, 3),
                                   '[a-zA-Z0-9]' )THEN
                    TRANSLATE(
                        SUBSTR(p_value, 1, 3),
                        'abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789',
                        'cdefghijklmnopqrstuvwxyzabCDEFGHIJKLMNOPQRSTUVWXYZAB2345678901'
                    )
                    || SUBSTR(p_value,
                              4,
                              LENGTH(p_value))
                ELSE
                    p_value
            END;
    END;

SELECT
    employee_id,
    masking_value(first_name)   AS first_name,
    masking_value(last_name)    AS last_name,
    masking_value(email)        AS email,
    masking_value(phone_number) AS phone_number
FROM
    hr.employees
WHERE
    department_id = 100
```

## 解説
今回は、**「SQL文の中にPL/SQL関数を埋め込む」** という、Oracle 12cから導入された非常に高度な機能（PL/SQL in WITH Clause）が登場しました。

この書き方をマスターしていると、データベースに永続的な関数を作成する権限がなくても、その場限りの複雑な加工ロジックをSQLだけで完結させることができます。特に「個人情報の秘匿（マスキング）」という実務で極めて重要なテーマを扱っています。

クエリーの核心部分を深掘りして解説します。

### 1. `WITH FUNCTION` 句（インライン関数）
通常、関数を作るには `CREATE FUNCTION` 命令が必要ですが、この構文を使うと **「このSELECT文の間だけ有効な関数」** を定義できます。
* **メリット**: スクリプトのポータビリティ（持ち運びやすさ）が上がります。他の環境でも、関数を事前に作成することなく実行できるからです。

### 2. マスキングのロジック
この `MASKING_VALUE` 関数は、一種の「簡易シーザー暗号」のような動きをしています。
* **`SUBSTR(P_VALUE, 1, 3)`**: 文字列の先頭3文字だけを加工対象にしています。
* **`TRANSLATE` による置換**: 
    * `a` → `c`, `b` → `d` ... と、アルファベットと数字を「2文字分ずらす」マッピングを行っています。
    * これにより、`SKING` が `UKING` のように、元の値を推測しにくく加工されます。
* **`|| SUBSTR(..., 4, ...)`**: 4文字目以降は無加工で結合しています。


### 3. `DETERMINISTIC` キーワードの重要性
関数の定義にある `deterministic` は、パフォーマンスに直結するプロの指定です。
* **意味**: 「入力が同じなら、必ず同じ結果を返す」ことを宣言します。
* **効果**: Oracleが計算結果をキャッシュ（保存）してくれるため、同じ名前が何度も出てくるような巨大なテーブルでも、2回目以降の計算をスキップして高速化できます。


### 実践的な注意点：セキュリティと運用

#### 1. PII（個人識別情報）の保護
実務で本番環境のデータをテスト環境に持っていく際、このようなマスキングは必須です。
* **注意**: 今回の `TRANSLATE` による加工は「可逆的（ルールを知っていれば元に戻せる）」です。本格的なセキュリティ要件では、ハッシュ化（SHA-2など）や、完全にランダムな値に置き換える手法が取られます。

#### 2. 文字列の長さに注意
`TRANSLATE` は「1文字を1文字」に置き換えるため、文字列の長さは変わりません。
* しかし、加工後の文字列がシステムのバリデーション（桁数チェックなど）に引っかからないか注意が必要です。

# 【完全版】問題5-12（文字列中の特定文字のカウント） *Lv3*
### OEスキーマ`CUSTOMERS`テーブル、`CUST_EMAIL`より文字列「`XI`」が１つ以上含まれるレコードを取得してください。（順不同）


## 期待する結果
| CUST_EMAIL                             | CNT | 
| -------------------------------------- | --- | 
| Sean.Stockwell@PYRRHULOXIA.EXAMPLE.COM | 1   | 
| Mary.Collins@PYRRHULOXIA.EXAMPLE.COM   | 1   | 
| Goldie.Slater@PYRRHULOXIA.EXAMPLE.COM  | 1   | 
| Kelly.Quinlan@PYRRHULOXIA.EXAMPLE.COM  | 1   | 
| Bryan.Huston@PYRRHULOXIA.EXAMPLE.COM   | 1   | 

## 解答例
```sql:例1:LENGTH と REPLACE により算出
WITH co_num AS (
    SELECT
        cust_email,
        -- (元の長さ - 置換後の長さ) / 探したい単語の長さ
        (LENGTH(cust_email) - LENGTH(REPLACE(cust_email, 'XI', '')))/LENGTH('XI') cnt
    FROM
        oe.customers
)
SELECT
    *
FROM
    co_num
WHERE
    cnt > 0
```
```sql:例2:REGEXP_COUNTを使用
WITH co_num AS (
    SELECT
        cust_email,
        REGEXP_COUNT(cust_email, 'XI') cnt
    FROM
        oe.customers
)
SELECT
    *
FROM
    co_num
WHERE
    cnt > 0
```

## 解説
文字列の中に特定の単語が「何回登場するか」を数える、非常にテクニックが必要な演習です。

Oracle 11g以前から使われてきた　**「知恵の輪」のような数え方**と、現代的な　**「正規表現」によるスマートな数え方**。この2つを並べて比較します。

それぞれのロジックの面白さと、実務での注意点を解説します。


### 例1：`LENGTH` と `REPLACE` による算出（数学的アプローチ）
これは、専用の関数がない環境でも使える「知恵の輪」のようなロジックです。

1.  **全体の長さを測る**: `LENGTH(cust_email)`
2.  **特定の文字を消す**: `REPLACE(cust_email, 'XI', '')` で、数えたい単語をすべて消去します。
3.  **減った分を計算する**: 「元の長さ - 消した後の長さ」を計算します。
4.  **単語数に変換する**: 消えた「文字数」を、1単語あたりの長さ（今回の 'XI' なら 2）で割ることで、「何個消えたか」が導き出されます。

$$\text{出現回数} = \frac{\text{元の文字数} - \text{置換後の文字数}}{\text{検索対象の文字数}}$$


### 例2：`REGEXP_COUNT` を使用（機能的アプローチ）
Oracle 11gから導入された、正規表現を使って出現回数をカウントする直球の関数です。
* **メリット**: ロジックがシンプルで、コードの意図が誰にでもすぐ伝わります。
* **柔軟性**: 今回は `'XI'` という固定文字ですが、`'X[0-9]'`（Xの後に数字が続くもの）のように、複雑なパターンを数えることも可能です。

### 実践的なアドバイス

#### 1. 処理速度の使い分け
実は、単純な固定文字列（今回のような 'XI'）を大量に処理する場合、**例1の方がわずかに高速**な場合があります。正規表現エンジンは強力な反面、内部的な処理が重いためです。ただし、数万件程度のデータであれば、読みやすさ重視で `REGEXP_COUNT` を選ぶのが現代のスタンダードです。

#### 2. 重なり（オーバーラップ）の挙動
「'AAAA' の中に 'AA' は何個あるか？」というケースで差が出ることがあります。
* **例1の方法**: `REPLACE` すると `AA` が一気に消えるため、2個とカウントされます。
* **例2の方法**: デフォルトでは「重なっていない部分」を数えるため 2個になりますが、正規表現の指定次第で挙動を制御できる柔軟性があります。

#### 3. 大文字・小文字の無視
`REGEXP_COUNT` の第4引数に `'i'` を指定すると、大文字の 'XI' も小文字の 'xi' もまとめて数えることができます。
```sql
REGEXP_COUNT(cust_email, 'XI', 1, 'i') -- 大文字小文字を区別しない
```

| **柔軟性** | × 固定文字列のみ | **◎ パターン指定が可能** |
| **互換性** | ◎ 古いDBでも動く | △ Oracle 11g以降 |
| **推奨シーン** | 極限のパフォーマンス追求 | **一般的な開発・分析業務** |
