# 問題5-1（文字列の結合） *Lv1*
### HRスキーマ`EMPLOYEE`テーブルより`MANAGER_ID`が101の従業員の`FIRST_NAME`と`LAST_NAME`を結合した「NAME」を取得してください。FIRST_NAMEとLAST_NAMEの間に半角スペースを入れてください。（順不同）
  
## 期待する結果
| EMPLOYEE_ID | NAME            | 
| ----------- | --------------- | 
| 108         | Nancy Greenberg | 
| 200         | Jennifer Whalen | 
| 203         | Susan Mavris    | 
| 204         | Hermann Baer    | 
| 205         | Shelley Higgins |  


## 解答例
```sql:例1：「||」演算子を使用
SELECT
    employee_id,
    first_name || ' ' || last_name AS "NAME"
FROM
    hr.employees
WHERE
    manager_id = 101
```

```sql:例2：CONCAT 関数を使用
SELECT
    employee_id,
    CONCAT(
        CONCAT(first_name, ' '),
        last_name
    ) AS "NAME"
FROM
    hr.employees
WHERE
    manager_id = 101
```

## 解説
Oracle SQLで文字列を結合（連結）する方法は、主に2つあります。もっとも一般的なのは **`||`（ダブルパイプ）演算子** を使う方法です。

### 1. 基本の「||」演算子
Oracleで文字列をくっつけるもっとも標準的で、もっとも便利な方法です。

* **使いかた**: `文字列1 || 文字列2`
* **メリット**: いくつでも繋げることができ、読みやすいです。

```sql
SELECT 
    FIRST_NAME || ' ' || LAST_NAME AS フルネーム
FROM 
    HR.EMPLOYEES;
-- 結果例: 'Steven King'
```

### 2. CONCAT 関数
関数を使って結合することも可能ですが、**「引数を2つしか取れない」** という制約があるため、実務ではあまり使われません。

* **使いかた**: `CONCAT(文字列1, 文字列2)`
* **3つ以上繋ぐ場合**: `CONCAT(CONCAT(A, B), C)` のように入れ子にする必要があり、非常に読みづらくなります。

```sql
-- 2つならOK
SELECT CONCAT('Hello', 'World') FROM dual;

-- 3つ以上は不便（|| を使うべき）
SELECT CONCAT(CONCAT('A', 'B'), 'C') FROM dual;
```

### 3. Oracle特有の注意点：NULLの扱い
多くのDB（SQL ServerやPostgreSQLなど）では、「文字列 + NULL」の結果は **NULL** になってしまいますが、**OracleではNULLを「空文字」として扱って結合してくれます。**

* **他DB**: `'Hello' + NULL` → `NULL` （全てが消える！）
* **Oracle**: `'Hello' || NULL` → `'Hello'` （そのまま残る！）

> **ポイント**
> 住所の結合などで「ビル名がNULL」の場合でも、Oracleならエラーやデータ消失を気にせず `住所 || ビル名` と書くだけで綺麗に繋がります。

# 問題5-2（文字列の分割＜文字数で分割＞） *Lv1*
### HRスキーマ`EMPLOYEE`テーブルより`PHONE_NUMBER`（ピリオドで区切られた３桁-３桁-４桁の文字列）を３つに分割してください。当問題では、「桁数」で分割するようにしてください。（順不同）
  
## 期待する結果
| EMPLOYEE_ID | PHONE_NUMBER | PN1 | PN2 | PN3  | 
| ----------- | ------------ | --- | --- | ---- | 
| 108         | 515.124.4569 | 515 | 124 | 4569 | 
| 200         | 515.123.4444 | 515 | 123 | 4444 | 
| 203         | 515.123.7777 | 515 | 123 | 7777 | 
| 204         | 515.123.8888 | 515 | 123 | 8888 | 
| 205         | 515.123.8080 | 515 | 123 | 8080 | 


## 解答例
```sql
SELECT
    employee_id,
    phone_number,
    SUBSTR(phone_number, 1, 3) AS "PN1",
    SUBSTR(phone_number, 5, 3) AS "PN2",
    SUBSTR(phone_number, 9, 4) AS "PN3"
FROM
    hr.employees
WHERE
    manager_id = 101
```

## 解説
**`SUBSTR`**（サブストリング）関数は、文字列の中から　**「指定した位置」から「指定した文字数」だけを抜き出す**ための関数です。

Oracleを使用していたら、トップレベルで頻繁に使用する関数の一つであると思います。
絶対に抑えておくべきものです。


### 1. 基本的な書き方

```sql
SUBSTR(対象の文字列, 開始位置, [切り出す文字数])
```

* **開始位置**: どこから切り出し始めるか。
    * **1** から数え始めます。
    * **0** を指定しても、多くのDB（Oracleなど）では **1** と見なされます。
    * **負の数** を指定すると、**「後ろから数えて」** 切り出します（超便利！）。
* **切り出す文字数**（省略可）: 何文字分取るか。
    * 省略すると、開始位置から **最後まで** 全てを切り出します。

### 2. 具体的な使用例
文字列 `'ORACLE-SQL'` を使って、挙動を比較してみましょう。

| SQL文 | 結果 | 説明 |
| :--- | :--- | :--- |
| `SUBSTR('ORACLE-SQL', 1, 3)` | `'ORA'` | 1文字目から3文字分。 |
| `SUBSTR('ORACLE-SQL', 8)` | `'SQL'` | 8文字目から最後まで（長さ省略）。 |
| `SUBSTR('ORACLE-SQL', -3)` | `'SQL'` | **後ろから数えて** 3文字目から最後まで。 |
| `SUBSTR('ORACLE-SQL', -7, 6)` | `'ORACLE'` | 後ろから7文字目（O）から6文字分。 |


### 3. 実務で知っておくべき「3つの注意点」

#### ① 日本語（全角文字）の扱い
Oracleなどの場合、`SUBSTR` は「文字数」で数えます。
* `SUBSTR('あいうえお', 1, 2)` → `'あい'` （2文字）

ただし、**「バイト数」**で切り出したい場合は、**`SUBSTRB`** 関数を使う必要があります。全角文字を扱う際にバイト数で切ってしまうと、文字化け（泣き別れ）が発生することがあるので、通常は `SUBSTR` を使うのが安全です。

#### ② 他のDBでの名称の違い
「SUBSTR」という名前はOracleやPostgreSQLで一般的ですが、他のDBでは名前が異なることがあります。機能はほぼ同じです。
* **MySQL / PostgreSQL**: `SUBSTRING()`
* **SQL Server**: `SUBSTRING()`

#### ③ 範囲外を指定した場合
* 開始位置が文字列の長さより大きい場合（例：5文字のデータに対して「10文字目から」指定）：**NULL**（または空文字）が返ります。
* 文字数に `0` や負の数を指定した場合：基本的には **NULL** が返ります。

# 問題5-3（文字列の置換１）  *Lv2*
### HRスキーマ`DEPARTMENTS`テーブルより`DEPARTMENT_ID`が100より小さいデータについて、AをXに置換してください（大文字のみ置換）。（順不同）
  
## 期待する結果
| DEPARTMENT_NAME  | DEPARTMENT_NAME_A_TO_X | 
| ---------------- | ---------------------- | 
| Administration   | Xdministration         | 
| Marketing        | Marketing              | 
| Purchasing       | Purchasing             | 
| Human Resources  | Human Resources        | 
| Shipping         | Shipping               | 
| IT               | IT                     | 
| Public Relations | Public Relations       | 
| Sales            | Sales                  | 
| Executive        | Executive              | 

## 解答例
```sql
SELECT
    department_name,
    REPLACE(department_name, 'A', 'X') AS department_name_a_to_x
FROM
    hr.departments
WHERE
    department_id < 100
```

## 解説
**`REPLACE`** 関数は、文字列の中にある「指定した文字列」を「別の文字列」に**そっくりそのまま置き換える**ための関数です。

### 1. 基本的な書き方

```sql
REPLACE(対象の文字列, 検索文字列, [置換文字列])
```

* **検索文字列**: 置き換えたい（探したい）文字を指定します。
* **置換文字列**（省略可）: 新しく入れたい文字を指定します。
    * 省略（または空文字を指定）した場合は、**検索文字列が削除**されます。

### 具体的な使用例
| SQL文 | 結果 | 説明 |
| :--- | :--- | :--- |
| `REPLACE('ABC-123', '-', '_')` | `'ABC_123'` | ハイフンをアンダーバーに置換 |
| `REPLACE('03-1234-5678', '-')` | `'0312345678'` | ハイフンを削除（空に置換） |
| `REPLACE('東京都港区', '東京都', '')` | `'港区'` | 特定の接頭辞を削除 |


### 2. 実務での注意点

#### ① 大文字・小文字を厳密に区別する
`REPLACE` は検索文字列の**大文字と小文字を区別**します。
* `REPLACE('Oracle SQL', 'oracle', 'ORACLE')` → **置換されません。**
* 対策：大文字小文字を無視したい場合は、事前に `UPPER` 関数で揃えるか、正規表現版の `REGEXP_REPLACE` を使います。

#### ② すべての箇所が置換される
1箇所だけでなく、見つかった**すべてのパターン**が置換されます。
* `REPLACE('2026/03/17', '/', '-')` → `'2026-03-17'`
* もし「最初の1箇所目だけ変えたい」という場合は、`REPLACE` ではなく正規表現関数などを使う必要があります。

#### ③ NULL の扱い
* 対象の文字列が **NULL** の場合：結果は **NULL** です。
* 検索文字列が **NULL** の場合：置換は行われず、**元の文字列**がそのまま返ります（Oracleの場合）。


### 3. 応用編：改行コードの削除
画面から入力されたデータに含まれる改行（CRLFなど）を消して1行にしたい時によく使います。

```sql
-- 改行（CHR(10)やCHR(13)）を空文字に置換して削除
SELECT REPLACE(REPLACE(remarks, CHR(13), ''), CHR(10), '') 
FROM table_name;
```

# 問題5-4（文字列の置換２）  *Lv2*
### HRスキーマ`DEPARTMENTS`テーブルより`DEPARTMENT_ID`が100より小さいデータについて、AをXに置換してください（大文字小文字ともに置換）。（順不同）

  
## 期待する結果
| DEPARTMENT_NAME  | DEPARTMENT_NAME_A_TO_X | 
| ---------------- | ---------------------- | 
| Administration   | Xdministrxtion         | 
| Marketing        | Mxrketing              | 
| Purchasing       | Purchxsing             | 
| Human Resources  | Humxn Resources        | 
| Shipping         | Shipping               | 
| IT               | IT                     | 
| Public Relations | Public Relxtions       | 
| Sales            | Sxles                  | 
| Executive        | Executive              | 

## 解答例
```sql
SELECT
    department_name,
    TRANSLATE(department_name, 'Aa,', 'Xx') AS department_name_a_to_x
FROM
    hr.departments
WHERE
    department_id < 100
```

## 解説
**`TRANSLATE`** 関数は、一言でいうと**「1文字ずつの置き換え（マッピング）」**を行う関数です。
`REPLACE` 関数が「特定の単語（塊）」を置き換えるのに対し、`TRANSLATE` は「aは1に、bは2に、cは3に…」といった具合に、文字を1対1で変換していくのが特徴です。

### 1. 基本的な書き方

```sql
TRANSLATE(対象の文字列, 検索文字リスト, 置換文字リスト)
```

* **検索文字リスト**: 置き換えたい文字を並べます。
* **置換文字リスト**: 検索文字リストの**同じ位置にある文字**に置き換わります。

### 具体的な使用例
| SQL文 | 結果 | 説明 |
| :--- | :--- | :--- |
| `TRANSLATE('ABCDE', 'ACE', '135')` | `'1B3D5'` | A→1, C→3, E→5 に変換 |
| `TRANSLATE('2026-03-17', '0123456789', 'abcdefghij')` | `'cagf-ad-bh'` | 数字を対応するアルファベットに置換 |


### 2. REPLACE との決定的な違い

ここを混同するとバグの原因になります。

* **`REPLACE`**: 検索文字列を「一つの塊」として探します。
* **`TRANSLATE`**: 検索文字列を「バラバラの一文字ずつ」として扱います。

> **例：`'apple'` に対して `'ap'` を `'X'` に変えたい場合**
> * `REPLACE('apple', 'ap', 'X')` → **`'Xple'`** （「ap」という並びがある所だけ置換）
> * `TRANSLATE('apple', 'ap', 'X')` → **`'XXle'`** （「a」も「p」も「X」に置換）


### 3. 知っておくべき注意点と「Oracleの癖」

#### ① 「削除」のルール
`検索文字リスト` にある文字が、`置換文字リスト` の対応する位置に文字がない場合、その文字は**削除**されます。

* `TRANSLATE('A-B-C', 'ABC-', 'abc')` → **`'a b c'`**
  （A→a, B→b, C→c に変換されますが、ハイフン「-」に対応する文字が4番目にないため、ハイフンは削除されます）

#### ② Oracleでの「空文字」問題
Oracleでは、`置換文字リスト` を**空（NULLまたは''）にすると、結果全体が NULL になってしまいます。**

* `TRANSLATE('012345', '123', '')` → **結果は NULL** （123が消えるのではなく、全てが消滅します）

> **回避策（文字を消したい時）**:
> ダミーの文字を先頭に付けます。
> `TRANSLATE(target, '#123', '#')`
> こうすると、「#」は「#」のまま、1, 2, 3 は対応する文字がないので削除されます。

#### ③ 重複する文字
`検索文字リスト` に同じ文字を2回書いた場合、**最初に書いた方の設定**が優先されます。
* `TRANSLATE('A', 'AA', '12')` → **`'1'`** （2番目の '2' は無視されます）


### 4. どんな時に使うと便利？
* **全角から半角への変換**: 「アイウエオ」を一気に「ｱｲｳｴｵ」に変える。
* **特定の文字種の一括削除**: 文字列から数字だけを全部消したい、記号だけを全部消したい。
* **暗号化・難読化**: 文字を特定のルールで別の文字にずらす。

#### パフォーマンスのメリット
複数の文字を置換したい時、`REPLACE` を何重にもネスト（入れ子）にするよりも、`TRANSLATE` 1回で書く方が **SQLの実行速度が圧倒的に速い** です。

```sql
-- ❌ 遅くて読みにくい
REPLACE(REPLACE(REPLACE(col, 'A', '1'), 'B', '2'), 'C', '3')

-- ✅ 速くてスッキリ
TRANSLATE(col, 'ABC', '123')
```

https://www.shift-the-oracle.com/sql/functions/translate.html

# 問題5-5（大文字・小文字変換）   *Lv2*
### COスキーマ`CUSTOMERS`テーブルより、`FULL_NAME`と下記の通り編集した値を比較して、一致しないものを取得してください。（順不同）
`EMAIL_ADDRESS`の
- 「@internalmail」を削除
- 「.」を半角スペースに置換
- スペースで分割した文字列をそれぞれ先頭文字を大文字にする
  
## 期待する結果
| CUSTOMER_ID | EMAIL_ADDRESS                   | FULL_NAME             | EMAIL_NAME         | 
| ----------- | ------------------------------- | --------------------- | ------------------ | 
| 123         | erda.sheer@internalmail         | erda Sheer            | Erda Sheer         | 
| 194         | norman.s.lobregat@internalmail  | Norman Simon Lobregat | Norman S Lobregat  | 
| 195         | tim.d.blasingim@internalmail    | Tim Dominic Blasingim | Tim D Blasingim    | 
| 227         | son.mclagan@internalmail        | Son McLagan           | Son Mclagan        | 
| 251         | dannie.mccoin@internalmail      | Dannie McCoin         | Dannie Mccoin      | 
| 262         | kendrick.mcmurtrey@internalmail | Kendrick McMurtrey    | Kendrick Mcmurtrey | 

## 解答例
```sql
SELECT
    CUSTOMER_ID,
    EMAIL_ADDRESS,
    FULL_NAME,
    INITCAP(replace(replace( EMAIL_ADDRESS ,'@internalmail', ''  ), '.', ' ')) as email_name
FROM
    CO.CUSTOMERS
where NOT(FULL_NAME = 
    INITCAP(replace(replace( EMAIL_ADDRESS ,'@internalmail', ''  ), '.', ' ')))
order by customer_id
```

## 解説
SQLで英字の「大文字・小文字」を制御する3つの基本関数はデータの見た目を整えるだけでなく、**「検索漏れを防ぐ」**　という実務上の重要な役割を担っています。

### 1. 各関数の役割まとめ

まずは、それぞれの変換結果を比較してみましょう。

| 関数 | 役割 | 変換例 (`'sQL is FUN'`) |
| :--- | :--- | :--- |
| **`UPPER`** | すべて**大文字**にする | `'SQL IS FUN'` |
| **`LOWER`** | すべて**小文字**にする | `'sql is fun'` |
| **`INITCAP`** | **単語の先頭だけ大文字**にする | `'Sql Is Fun'` |


### 2. 各関数の詳細解説

#### ① `UPPER` / `LOWER` （大文字・小文字）
これらは主に **「大文字小文字を区別しない検索」** を実現するために使われます。
SQLでは `'Smith'` と `'SMITH'` は別物として扱われるため、ユーザーがどう入力してもヒットするように変換を挟むのが定石です。

```sql
-- ユーザーがどう入力しても、SMITHとして検索する
SELECT * FROM employees WHERE UPPER(last_name) = 'SMITH';
```

#### ② `INITCAP` （イニシャル・キャピタル）
名前やタイトルのように、各単語の1文字目を大文字にし、2文字目以降を小文字に変換します。
* **ポイント**: スペースだけでなく、記号（ハイフン、ピリオドなど）も単語の区切りとして認識します。
* `INITCAP('mary-ann')` → `'Mary-Ann'`


### 3. 知っておくべき注意点（落とし穴）

#### ① 検索パフォーマンスの低下（最重要！）
`WHERE` 句の左辺（カラム側）に関数を使うと、**インデックスが使われなくなり、検索が劇的に遅くなる**ことがあります。

* **❌ 遅い例**: `WHERE UPPER(last_name) = 'SMITH'`
    （全ての行を一度大文字に変換してから比べるため、フルスキャンが発生します）
* **✅ 対策**: 大文字小文字を無視した検索を頻繁に行うなら、**「関数索引（Functional Index）」**を作成するか、最初から大文字でデータを保存する運用を検討します。

#### ② `INITCAP` の予期せぬ挙動
`INITCAP` は「2文字目以降を強制的に小文字にする」という性質があります。
* `INITCAP('iPhone')` → **`'Iphone'`**
このように、途中にある大文字（キャメルケース）が消されてしまうため、商品名などの固有名詞に使う際は注意が必要です。

#### ③ 全角文字（日本語）への影響
OracleなどのDBでは、全角の英字（ＡＢＣ）も変換対象になりますが、漢字やひらがなには影響を与えません（エラーにもなりません）。
* `UPPER('ｓｑｌ漢字')` → `'ＳＱＬ漢字'`

# 問題5-6（文字列のパディング）  *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、下記の通り編集した値を取得してください。（順不同）
* **`EMPLOYEE_ID`を6桁にする。6桁に満たない場合、前ゼロ埋めして表示する。**
* **`EMAIL`を10桁にする。10桁に満たない場合、右スペースを「X」で埋めて表示する。**

## 期待する結果
| EMP_ID6 | EMAIL10    | 
| ------- | ---------- | 
| 000201  | MMARTINEXX | 
| 000202  | PDAVISXXXX | 
| 000203  | SJACOBSXXX | 
| 000204  | HBROWNXXXX | 
| 000205  | SHIGGINSXX | 
| 000206  | WGIETZXXXX | 

## 解答例
```sql
SELECT
    LPAD(employee_id, 6, '0') as "EMP_ID6",
    RPAD(email, 10, 'X') as "EMAIL10"
FROM
    hr.employees
WHERE
    employee_id > 200
```

## 解説
SQLにおいて、文字列の「パディング（穴埋め）」は、データの見た目を整えたり、システムの仕様に合わせて桁数を固定したりするために欠かせない操作です。
主に **`LPAD`**（左パディング）と **`RPAD`**（右パディング）の2つの関数が使われます。

### 1. LPAD と RPAD の基本

#### LPAD (Left Pad)
文字列の**左側**を指定した文字で埋めます。主に「数字の先頭を0で埋める（ゼロパディング）」際によく使われます。

* **構文**: `LPAD(元の文字列, 全体の長さ, [埋める文字])`
* **例**: `LPAD('123', 5, '0')` → `'00123'`

#### RPAD (Right Pad)
文字列の**右側**を指定した文字で埋めます。固定長ファイルの作成や、簡易的なグラフ表示などに使われます。

* **構文**: `RPAD(元の文字列, 全体の長さ, [埋める文字])`
* **例**: `RPAD('SQL', 5, '*')` → `'SQL**'`


### 2. 実務での活用シーン
#### ① ゼロパディング（IDの桁揃え）
社員番号や商品コードなどを、常に5桁や10桁で表示したい場合です。
```sql
SELECT LPAD(employee_id, 5, '0') FROM employees;
-- 123 -> '00123'
```

#### ② マスキング処理
クレジットカード番号や電話番号の一部を隠す際にも応用できます。
```sql
SELECT RPAD(SUBSTR(card_no, 1, 4), 16, '*') FROM payments;
-- '1234567812345678' -> '1234************'
```

#### ③ 固定長データの作成
銀行とのデータ連携など、各項目のバイト数が厳格に決まっているファイル（全銀フォーマットなど）を作成する際に必須となります。


### 3. 注意すべき「3つの落とし穴」

#### ① 「切り捨て」の罠
ここが最大の注意点です。指定した「全体の長さ」が**元の文字列よりも短い場合、文字列は切り捨てられます。**

* `LPAD('OracleSQL', 5, '0')` → **`'Oracl'`**

「足りない時だけ埋めてくれる」のではなく、**「強制的にその長さに収める」** 関数であるため、予期せぬデータ欠損を招くことがあります。

#### ② 全角文字とバイト数の問題
日本語（全角文字）を扱う場合、DBの設定によって「文字数」で数えるか「バイト数」で数えるかが異なります。
* **文字数単位**: `LPAD('あ', 3, ' ')` → `'  あ'`（3文字分）
* **バイト数単位**: 全角1文字を2バイトと数える設定だと、レイアウトがガタガタに崩れる原因になります。
> ※ Oracleでは `LPAD` は文字数、`LPADB` はバイト数といった使い分けが必要な場合があります。

#### ③ 埋める文字（第3引数）の省略
第3引数を省略した場合、通常は **「半角スペース」** で埋められます。
* `LPAD('ABC', 5)` → `'  ABC'`
しかし、意図せずスペースが入ると、後続のプログラムで比較処理（`= 'ABC'`）をする際にヒットしなくなるなどのトラブルに繋がります。

# 問題5-7（文字列のトリミング）  *Lv2*
### COスキーマ`SHIPMENTS`テーブルより、下記の通り編集した値を取得してください。（順不同）
* **`DELIVERY_ADDRESS`の先頭にある「Chatfield,」をトリム（除去）する**
* **`DELIVERY_ADDRESS`の末尾にある「USA」をトリム（除去）する。**

## 期待する結果
| SHIPMENT_ID | DELIVERY_ADDRESS           | L_TRIM                     | R_TRIM                 | 
| ----------- | -------------------------- | -------------------------- | ---------------------- | 
| 150         | Boca Raton, FL 33433 USA   | Boca Raton, FL 33433 USA   | Boca Raton, FL 33433   | 
| 151         | Rutland, OH 45775 USA      | Rutland, OH 45775 USA      | Rutland, OH 45775      | 
| 153         | Lakeland, FL 33803 USA     | Lakeland, FL 33803 USA     | Lakeland, FL 33803     | 
| 154         | Falls Church, VA 22043 USA | Falls Church, VA 22043 USA | Falls Church, VA 22043 | 
| 155         | Chatfield, OH 44825 USA    | OH 44825 USA               | Chatfield, OH 44825    | 
| 156         | Chatfield, OH 44825 USA    | OH 44825 USA               | Chatfield, OH 44825    | 
| 157         | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017     | 
| 158         | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017 USA     | Dalmatia, PA 17017     | 
| 159         | Fonda, NY 12068 USA        | Fonda, NY 12068 USA        | Fonda, NY 12068        |

## 解答例
```sql
SELECT
    shipment_id,
    delivery_address,
    LTRIM(delivery_address, 'Chatfield,') AS "L_TRIM",
    RTRIM(delivery_address, 'USA')        AS "R_TRIM"
FROM
    co.shipments
WHERE
    shipment_id BETWEEN 150 AND 159
```

## 解説
パディングで「文字を埋める」のとは逆に、不要な余白を削ぎ落としてデータを綺麗にするのが **`TRIM`** 一族の役割です。
ユーザーが入力したデータには、目に見えない「末尾のスペース」などが混じることが多く、これが原因で検索がヒットしないというトラブルが多発します。そんな時の救世主がこれらの関数です。

### 1. 基本的な役割と違い

それぞれの関数が「どこ」を掃除するかを整理しましょう。

| 関数 | 掃除する場所 | 例：`'  SQL  '` の場合 | 結果 |
| :--- | :--- | :--- | :--- |
| **`LTRIM`** | **左側** (Leading) のみ | `LTRIM('  SQL  ')` | `'SQL  '` |
| **`RTRIM`** | **右側** (Trailing) のみ | `RTRIM('  SQL  ')` | `'  SQL'` |
| **`TRIM`** | **両端** (Both) | `TRIM('  SQL  ')` | `'SQL'` |


### 2. 実務での活用シーン

#### ① 検索の空振りを防ぐ
ユーザーがフォーム入力時に誤って入れたスペースを無視して検索できます。
```sql
-- ユーザーが 'SMITH ' (後ろにスペース) と入力してもヒットさせる
SELECT * FROM employees WHERE RTRIM(last_name) = 'SMITH';
```

#### ② 固定長データの加工
システム連携などで「10桁固定（足りない分はスペース埋め）」になっているデータから、純粋な値だけを取り出したい時に `RTRIM` を多用します。


### 3. スペース以外の「特定の文字」を消す方法
実はこれらの関数、スペース以外も消すことができます。

* **`LTRIM(文字列, 消したい文字セット)`**
* **`RTRIM(文字列, 消したい文字セット)`**

```sql
-- 先頭の「0」をすべて消したい
SELECT LTRIM('000123', '0') FROM dual; -- 結果: '123'

-- 両端の「#」や「!」を消したい
SELECT TRIM(BOTH '#' FROM '##IMPORTANT##') FROM dual; -- 結果: 'IMPORTANT'
```

### 4. 知っておくべき注意点（落とし穴）

#### ① 「文字の間のスペース」は消せない
`TRIM` はあくまで「両端」が担当範囲です。文字列の真ん中にあるスペースを消したい場合は、以前紹介した **`REPLACE`** 関数を使う必要があります。



> **例：**
> `TRIM(' S Q L ')` → `'S Q L'` （真ん中のスペースは残る）
> `REPLACE(' S Q L ', ' ', '')` → `'SQL'` （すべて消える）

#### ② 全角スペースの扱い
多くの環境で、`TRIM` がデフォルトで消すのは「半角スペース」のみです。**「全角スペース」** は別の文字として扱われるため、残ってしまいます。
* **対策**: 全角スペースも消したい場合は `REPLACE(col, '　', '')` を併用するか、`LTRIM(col, ' 　')` のように第2引数に全角スペースを含めます。

#### ③ インデックスへの影響
`UPPER` などの時と同様、`WHERE` 句の左辺で `RTRIM(column)` と書くと、インデックスが効かなくなり検索が遅くなる可能性があります。データ登録時にあらかじめ `TRIM` してから保存しておくのが、パフォーマンス的には最も健全です。

# 問題5-8（文字列の長さ）  *Lv2*
### OEスキーマ`PRODUCT_INFORMATION`の`PRODUCT_NAME`が24文字のデータを取得してください。（順不同）

## 期待する結果
| PRODUCT_NAME             | 
| ------------------------ | 
| Plasma Monitor 10/LE/VGA | 
| Laptop 128/12/56/v90/110 | 
| Manual - Vision Tools2.0 | 
| Business Cards Box - 250 | 
| Business Cards - 1000/2L | 

## 解答例
```sql
SELECT
    product_name AS len
FROM
    oe.product_information
WHERE
    LENGTH(product_name) = 24
```

## 解説
SQLにおいて **`LENGTH`** 関数は、文字列の「長さ」を測るための基本中の基本となる関数です。
しかし、一見単純そうに見えて、**「文字数なのか、バイト数なのか」「NULLはどうなるのか」**　といった、データベースの種類や設定によって結果が変わる落とし穴がいくつか存在します。

### 1. 基本的な使い方
指定した文字列が「何文字あるか」を数値で返します。

```sql
SELECT LENGTH('Oracle') FROM dual; -- 結果: 6
SELECT LENGTH('SQLは楽しい') FROM dual; -- 結果: 6 (「は」や「楽」も1文字)
```

### 2. 似て非なる関数との違い（Oracleを例に）
特に日本語などの全角文字を扱う場合、どの「長さ」を知りたいかによって関数を使い分ける必要があります。

| 関数 | 単位 | 特徴 |
| :--- | :--- | :--- |
| **`LENGTH`** | **文字数** | 見たままの文字数を数える。 |
| **`LENGTHB`** | **バイト数** | メモリ上のサイズ。UTF-8なら全角1文字=3バイトなど。 |
| **`LENGTHC`** | **完全な文字数** | サロゲートペア（特殊な漢字や絵文字）を正しく1文字と数える。 |

### 3. 注意すべき「4つの落とし穴」

#### ① NULL と 空文字（''）の挙動
Oracleでは、**空文字はNULLと見なされる**ため、`LENGTH('')` の結果は `0` ではなく **`NULL`** になります。

> **注意:** > MySQLやPostgreSQLでは `LENGTH('')` は `0` を返します。お使いのDBがどちらのタイプか（空文字とNULLを区別するか）によって、計算式の結果が変わるため非常に重要です。

#### ② CHAR型（固定長）の空白
カラムの型が `CHAR(10)` の場合、データが `'ABC'` だけであっても、後ろにスペースが埋められて保存されます。
* `LENGTH` をかけると、この末尾のスペースまで数えて `10` が返ってくることがあります。
* **対策:** 純粋な文字数を知りたい場合は、`LENGTH(RTRIM(column))` のようにスペースを除去してから測ります。

#### ③ 特殊な文字（サロゲートペア）
「𠮷（つちよし）」や「絵文字（😊）」などは、内部的に2文字分のデータ（サロゲートペア）として保持されることがあります。
* 普通の `LENGTH` だと、これらを **「2」** と数えてしまう場合があります。
* 厳密に見た目通りの数を数えたい場合は、`LENGTHC` を検討してください。

#### ④ パフォーマンス
`WHERE` 句で `LENGTH(column) > 100` のように使うと、全ての行に対して計算が発生するため、大量のデータがある場合は検索が遅くなります。

### 4. 実務での活用例

#### データバリデーション（桁数チェック）
「パスワードは8文字以上」といったチェックをSQL側で行う場合。
```sql
SELECT * FROM users WHERE LENGTH(password) < 8;
```

#### データの分布調査
「どのくらいの長さのコメントが多く投稿されているか」を分析する場合。
```sql
SELECT LENGTH(comment_text), COUNT(*) 
FROM reviews 
GROUP BY LENGTH(comment_text);
```

# 問題5-9（NULLの変換）  *Lv2*
### HRスキーマ`DEPARTMENTS`より、DEPARTMENT_IDとそれに対応するMANAGER_IDを取得してください。ただし、MANAGER_IDがNULLの場合は「0」に変換してください。（順不同）

## 期待する結果
| DEPARTMENT_ID | MANAGER_ID | 
| ------------- | ---------- | 
| 100           | 108        | 
| 110           | 205        | 
| 120           | 0          | 
| 130           | 0          | 
| 140           | 0          | 
| 150           | 0          | 

## 解答例
```sql:例1：NVLを使用
SELECT
    department_id,
    NVL(manager_id, 0) AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例2：NVL2を使用
SELECT
    department_id,
    NVL2(manager_id, manager_id ,0)  AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例3：COALESCEを使用
SELECT
    department_id,
    COALESCE(manager_id ,0) AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例4：DECODEを使用
SELECT
    department_id,
    DECODE(manager_id, null, 0, manager_id) AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```
```sql:例5：CASE式を使用
SELECT
    department_id,
    CASE WHEN manager_id IS NULL THEN 0 ELSE manager_id END AS "MANAGER_ID"
FROM
    hr.departments
WHERE
    department_id BETWEEN 100 AND 150
```

## 解説
Oracleにおいて「NULL」は値が存在しない（不明な）状態を指しますが、これを放置すると計算結果がすべてNULLになるなど、扱いにくい「DB界の幽霊」のような存在です。
これを実在する値（0や'なし'など）に置き換えるための主な関数は下記の通りです。


### 1. NULLを置き換える主要な関数

#### ① NVL(値, 置き換え値)
最も基本的で、Oracleエンジニアが一番よく使う関数です。
* **役割**: 第1引数がNULLなら、第2引数を返します。
* **例**: `NVL(salary, 0)` → 給与がNULLなら0円にする。

#### ② NVL2(値, NULLでない時の値, NULLの時の値)
「NULLかどうか」で結果を完全に切り分けたい時に使います。
* **役割**: 3つの引数を取り、1番目が「NULL以外」なら2番目、「NULL」なら3番目を返します。
* **例**: `NVL2(comm, '歩合あり', '歩合なし')`

#### ③ COALESCE(値1, 値2, ..., 値N)
標準SQL準拠の関数で、複数の候補から「最初に見つかったNULLでない値」を採用します。
* **役割**: 引数を左から順にチェックし、最初にNULLでないものがあればそれを返します。
* **例**: `COALESCE(携帯番号, 自宅番号, '連絡先なし')`


### 2. その他の手法（条件分岐）

#### DECODE関数
Oracle独自の「If-Then-Else」関数です。NULLの判定も可能です。
* **例**: `DECODE(column, NULL, 0, column)`
    * 第1引数がNULLなら0、そうでなければそのままの値を返す。

#### CASE式
標準SQLで、より複雑な条件と一緒にNULLを扱いたい時に使います。
* **例**: 
  ```sql
  CASE WHEN score IS NULL THEN '未受験' ELSE TO_CHAR(score) END
  ```


### 3. 比較まとめ表

| 関数 | 規格 | 特徴 | おすすめの用途 |
| :--- | :--- | :--- | :--- |
| **`NVL`** | Oracle独自 | シンプルで高速 | 単一カラムのNULL埋め |
| **`NVL2`** | Oracle独自 | 3つの引数で条件分岐 | 表示文言を切り替えたい時 |
| **`COALESCE`** | 標準SQL | 複数の引数を取れる | 優先順位に従って値を探す時 |
| **`DECODE`** | Oracle独自 | 関数形式で簡潔 | 短い条件分岐 |
| **`CASE`** | 標準SQL | 可読性が高く柔軟 | 複雑なロジックが必要な時 |

### 4. 注意点と落とし穴

#### ① データ型の一致
**元の値と置き換え値のデータ型を一致させる**必要があります。
* ❌ `NVL(salary, '未設定')` -- salaryが数値ならエラー
* ✅ `NVL(TO_CHAR(salary), '未設定')` -- 型を合わせればOK

#### ② 評価のタイミング（ショートサーキット）
* **`NVL`**: 第1引数がNULLであってもなくても、**第2引数の中身も一度計算（評価）されます**。第2引数に重い関数を入れると無駄に負荷がかかることがあります。
* **`COALESCE`**: NULLでない値が見つかった時点で**以降の引数は評価しません**（ショートサーキット）。効率的です。

#### ③ 空文字の扱い
Oracle特有の挙動として、**「空文字（''）はNULLとして扱われる」** ことを忘れないでください。
* `NVL('', 'A')` → 結果は `'A'` になります。

# 問題5-10（文字列の分割＜正規表現を使用＞） *Lv3*
### OEスキーマ`CUSTOMERS`テーブルの顧客の所得レベル（'INCOME_LEVEL'）を、「RANK」「LOWER」「UPPER」の3つの項目に切り分けて抽出してください。（順不同）
- **`CUSTOMER_ID` が 150 ～ 159 の範囲に該当する顧客を対象とします。**
- **元のデータ形式が「<RANK>: <LOWER> - <UPPER>」となっているものを、それぞれの項目へ分割します。**
- **文字列の内容に応じて、LOWER（下限）とUPPER（上限）には以下のように値を設定します。**

  | 元の表記パターン | RANK | LOWER（下限） | UPPER（上限） |
  | --- | --- | --- | --- |
  | **通常形式**<br>`A: 1000 - 2000` | A | 1000 | 2000 |
  | **Below（～未満）**<br>`A: Below 1000` | A | **`<`** | 1000 |
  | **and above（～以上）**<br>`A: 1000 and above` | A | 1000 | **`>=`** |


## 期待する結果
| CUSTOMER_ID | INCOME_LEVEL         | RANK | LOWER   | UPPER   | 
| ----------- | -------------------- | ---- | ------- | ------- | 
| 150         | D: 70,000 - 89,999   | D    | 70,000  | 89,999  | 
| 151         | F: 110,000 - 129,999 | F    | 110,000 | 129,999 | 
| 152         | A: Below 30,000      | A    | <       | 30000   | 
| 153         | I: 170,000 - 189,999 | I    | 170,000 | 189,999 | 
| 154         | L: 300,000 and above | L    | 300000  | >=      | 
| 155         | E: 90,000 - 109,999  | E    | 90,000  | 109,999 | 
| 156         | F: 110,000 - 129,999 | F    | 110,000 | 129,999 | 
| 157         | G: 130,000 - 149,999 | G    | 130,000 | 149,999 | 
| 158         | J: 190,000 - 249,999 | J    | 190,000 | 249,999 | 
| 159         | G: 130,000 - 149,999 | G    | 130,000 | 149,999 | 

## 解答例
```sql
SELECT
    customer_id,
    income_level,
    -- 1番目の項目（レベル記号など）
    TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 1)) AS rank,
    -- 2番目の項目（下限値など）
    CASE
        WHEN ( REGEXP_LIKE ( income_level,
                             'Below' ) ) THEN
            '<'
        WHEN ( REGEXP_LIKE ( income_level,
                             'above' ) ) THEN
            REGEXP_REPLACE(
                TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 2)),
                '[^0-9]',
                ''
            )
        ELSE
            TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 2))
    END                                               AS "LOWER",

    -- 3番目の項目（上限値など）
    CASE
        WHEN ( REGEXP_LIKE ( income_level,
                             'Below' ) ) THEN
            REGEXP_REPLACE(
                TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 2)),
                '[^0-9]',
                ''
            )
        WHEN ( REGEXP_LIKE ( income_level,
                             'above' ) ) THEN
            '>='
        ELSE
            TRIM(REGEXP_SUBSTR(income_level, '[^:-]+', 1, 3))
    END                                               AS "UPPER"
FROM
    oe.customers
WHERE customer_id BETWEEN  150 AND 159
```

## 解説
SQLにおける正規表現（Regular Expression）を使いこなせると、複雑な文字列操作が1行のクエリで魔法のように片付きます。
通常の `LIKE` や `SUBSTR` が「定規」だとしたら、これらは **「変幻自在のレーザーカッター」** です。

### 1. REGEXP_LIKE ：「パターンに合うか？」を判定する
`LIKE` 演算子の強化版です。主に `WHERE` 句で条件として使われます。

* **役割**: 指定したパターンに一致するデータがあるかどうかを `TRUE/FALSE` で返します。
* **ここが凄い**: 「数字だけで構成されている」「大文字で始まる」といった複雑な条件が1つで書けます。

```sql
-- 「AまたはBで始まる名前」を探す
SELECT first_name FROM employees
WHERE REGEXP_LIKE(first_name, '^[AB]');
```

### 2. REGEXP_SUBSTR ：「一部を切り出す」
`SUBSTR` 関数の強化版です。

* **役割**: パターンに一致した部分文字列を抜き出します。
* **ここが凄い**: 「カンマ区切りの3番目」や「メールアドレスのドメイン部分」など、位置が特定しづらい部分をスマートに抜けます。

```sql
-- メールアドレスからドメイン (@以降) を抜く
SELECT REGEXP_SUBSTR(email, '@[^.]+') FROM users;
-- 'user@google.com' -> '@google'
```

### 3. REGEXP_REPLACE ：「別の文字に置き換える」
`REPLACE` 関数の強化版です。

* **役割**: パターンに一致した部分を、別の文字列に置換します。
* **ここが凄い**: 「余計な記号を全部消す」だけでなく、**「後方参照」** を使って「0312345678」を「03-1234-5678」のように整形し直すことができます。

```sql
-- 数字の塊をハイフンで区切る（後方参照 \1, \2 の活用）
SELECT REGEXP_REPLACE('0312345678', '([0-9]{2})([0-9]{4})([0-9]{4})', '\1-\2-\3') FROM dual;
-- 結果: '03-1234-5678'
```

### 4. 注意点
#### ① パフォーマンスの代償
正規表現は非常に強力ですが、**CPU負荷が通常の関数より高い**です。
* `LIKE 'ABC%'` で済むなら `REGEXP_LIKE` は使わないのがプロの鉄則。
* 大量（数百万件〜）のデータに対して全件スキャンで正規表現をかけると、目に見えて処理が重くなります。

#### ② 方言（DBMSによる違い）
正規表現の書き方（エンジン）はデータベースによって微妙に異なります。
* **Oracle**: POSIX標準 + 一部の独自拡張。
* **PostgreSQL**: Perl互換（PCRE）に近い、非常に強力なもの。
* **MySQL/MariaDB**: バージョンによって使えるメタ文字が異なります。

#### ③ 「欲張りマッチ（Greedy Match）」に注意
例えば `'.*'` （任意の文字の繰り返し）と書くと、文字列の最後まで一気に飲み込んでしまいます。意図せず長い範囲がヒットしてしまうことが多いため、`'?'` を使った非欲張りマッチや、`'[^,]+'`（カンマ以外の繰り返し）のように **「何を含まないか」** を指定するのがコツです。

# 【完全版】問題5-11（文字列のマスキング） *Lv3*
### ERスキーマ`EMPLOYEE`テーブルより`DEPARTMENT_ID`=100の従業員のFIRST_NAME、LAST_NAME、EMAIL、PHONE_NUMBERを取得する。
- **ただし、以下のルールに従いマスキングをすること。**
  - **先頭３文字がアルファベットまたは数字の場合、先頭３文字を２文字ずらす**
  - **2文字ずらした際、アルファベットまたは数字の最後を超える場合、先頭に戻る**
    - （例１）**aBz**D → **cDb**D
    - （例２）**089**123 → **201**123

| 	110	| 	Lqjn	| 	Ejgn	| 	LEJEN	| 	737.124.4269	|

## 解答例
```sql
with
    function MASKING_VALUE (
        P_VALUE in varchar2
    ) return varchar2
        deterministic
    is
    begin
        RETURN
            case
                when REGEXP_LIKE ( SUBSTR(P_VALUE, 1, 3),
                                   '[a-z|A-Z|0-9]' )then
                    translate(
                        substr(p_value, 1, 3),
                        'abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789',
                        'cdefghijklmnopqrstuvwxyzabCDEFGHIJKLMNOPQRSTUVWXYZAB2345678901'
                    )
                    || substr(p_value,
                              4,
                              length(p_value))
                ELSE
                    p_value
            END;
    END;

SELECT
    employee_id,
    masking_value(first_name)   AS first_name,
    masking_value(last_name)    AS last_name,
    masking_value(email)        AS email,
    masking_value(phone_number) AS phone_number
FROM
    hr.employees
WHERE
    department_id = 100;
```

## 解説
Oracle（12c以降）では、`WITH`句の中で**一時的なPL/SQLファンクション（またはプロシージャ）を定義**することができます。
通常、関数を使うには事前に `CREATE FUNCTION` でデータベースに保存しておく必要がありますが、この機能を使えば **「そのSQL文の中だけで有効な使い捨ての関数」** を作成できます。

### 1. 基本的な構文
`WITH` の直後に `FUNCTION` キーワードを使って定義し、最後にメインの `SELECT` 文を書きます。

```sql
WITH
  -- 1. ファンクションの定義
  FUNCTION get_tax_price(price NUMBER) RETURN NUMBER IS
  BEGIN
    RETURN price * 1.1; -- 10%の税込計算
  END;
-- 2. メインのSELECT文
SELECT 
    last_name, 
    salary, 
    get_tax_price(salary) as salary_with_tax
FROM 
    hr.employees
WHERE 
    rownum <= 5;
/
```
<br>

### 2. なぜこの機能を使うのか？（メリット）

#### ① 実行速度の向上（コンテキストスイッチの削減）
通常、SQLからPL/SQL関数を呼び出すと、**「SQLエンジン」と「PL/SQLエンジン」の間で処理の切り替え（コンテキストスイッチ）** が発生し、これがオーバーヘッドになります。
`WITH`句で定義した関数は、SQLの一部としてインライン化されやすいため、通常の関数呼び出しよりも高速に動作する場合があります。

#### ② 権限がない環境でもロジックが組める
開発現場ではファンクションを定義し、使いたい場面が多くありますが、データベース自体にファンクションを登録する必要があるため、気軽に登録できません。
データベースに `CREATE FUNCTION` する権限（作成権限）がないユーザーでも、自分だけのロジックをSQL内に閉じ込めて実行できます。

#### ③ 「使い捨て」ロジックの整理
そのSQLで1回しか使わないような特殊な変換ロジックを、わざわざデータベースの共通資産として登録せずに済みます。管理の手間が省け、コードのポータビリティ（持ち運びやすさ）も上がります。

#### ④ DRY（Don't Repeat Yourself）原則の実現
今回の問題のような、そのクエリ固有の複雑なロジックに最適。「共通化するほどではないが、このクエリ内では何度も使う」ようなロジックを１か所にまとめることができます。

### 3. 実務での活用例
複雑な判定ロジックをSQLの `CASE` 式で書くと読みにくくなりますが、ファンクションにすると非常にスッキリします。

```sql
WITH
  FUNCTION get_grade(sal NUMBER) RETURN VARCHAR2 IS
  BEGIN
    IF sal >= 15000 THEN RETURN 'S';
    ELSIF sal >= 10000 THEN RETURN 'A';
    ELSE RETURN 'B';
    END IF;
  END;
SELECT 
    last_name, 
    salary, 
    get_grade(salary) as grade
FROM 
    hr.employees;
/
```
### 4. 注意点
1. **文末のスラッシュ（/）**:
SQL*PlusやSQL Developerなどのツールで実行する場合、PL/SQLブロックが含まれるため、文末に `/` が必要な場合があります。
2. **スコープ（有効範囲）**:
そのSQL文が終われば消えてしまいます。他のSQLで使い回すことはできません。
3. **読みやすさ**:
便利ですが、あまりに長いロジックを書き込むとSQL全体の見通しが悪くなるため、適度なバランスが重要です。

# 【完全版】問題5-12（文字列中の特定文字のカウント） *Lv3*
### OEスキーマ`CUSTOMERS`テーブル、`CUST_EMAIL`より文字列「`XI`」が１つ以上含まれるレコードを取得してください。（順不同）


## 期待する結果
| CUST_EMAIL                             | CNT | 
| -------------------------------------- | --- | 
| Sean.Stockwell@PYRRHULOXIA.EXAMPLE.COM | 1   | 
| Mary.Collins@PYRRHULOXIA.EXAMPLE.COM   | 1   | 
| Goldie.Slater@PYRRHULOXIA.EXAMPLE.COM  | 1   | 
| Kelly.Quinlan@PYRRHULOXIA.EXAMPLE.COM  | 1   | 
| Bryan.Huston@PYRRHULOXIA.EXAMPLE.COM   | 1   | 

## 解答例
```sql:LENGTH と REPLACE により算出
WITH co_num AS (
    SELECT
        cust_email,
        -- (元の長さ - 置換後の長さ) / 探したい単語の長さ
        (LENGTH(cust_email) - LENGTH(REPLACE(cust_email, 'XI', '')))/LENGTH('XI') cnt
    FROM
        oe.customers
)
SELECT
    *
FROM
    co_num
WHERE
    cnt > 0
```
```sql:REGEXP_COUNTを使用
WITH co_num AS (
    SELECT
        cust_email,
        REGEXP_COUNT(cust_email, 'XI') cnt
    FROM
        oe.customers
)
SELECT
    *
FROM
    co_num
WHERE
    cnt > 0
```

## 解説
「特定の文字が何個含まれているか」を数える専用の関数は（一部を除いて）標準SQLにはありません。しかし、**`LENGTH`** と **`REPLACE`** を組み合わせた「引き算」のパズルで解くことができます。

### 1. 基本的な考え方（アルゴリズム）

ロジックは非常にシンプルです。

1.  元の文字列の長さを測る。
2.  その文字列から、**数えたい文字だけを消去（REPLACE）** する。
3.  消去した後の文字列の長さを測る。
4.  **「元の長さ」ー「消去後の長さ」** が、消えた文字の数（出現回数）になる。

### 2. SQLの実装例
例えば、`'A,B,C,D'` という文字列の中にカンマ（`,`）がいくつあるか数える場合：

```sql
SELECT 
    LENGTH('A,B,C,D') - LENGTH(REPLACE('A,B,C,D', ',', '')) AS comma_count
FROM dual;
```

* 元の長さ：7
* カンマを消した後の長さ（`'ABCD'`）：4
* 結果：$7 - 4 = 3$（カンマは3個！）

### 3. 応用編：2文字以上の単語を数える場合
当問題のように、「特定の単語（例：`'SQL'`）」が何回出てくるかを数えるときは、最後に**その単語の長さで割る**必要があります。

$$\frac{\text{元の長さ} - \text{置換後の長さ}}{\text{探したい単語の長さ}}$$

```sql
-- 'SQL'が何回出現するか
SELECT 
    (LENGTH(text_col) - LENGTH(REPLACE(text_col, 'SQL', ''))) / LENGTH('SQL')
FROM my_table;
```

### 4. 注意点と回避策

#### ① NULL の罠
対象の文字列が NULL だったり、探したい文字が一つも含まれていなかったりすると、結果が不安定になることがあります。`COALESCE`（または `NVL`）で保護すると安全です。

```sql
SELECT 
    COALESCE(LENGTH(col), 0) - COALESCE(LENGTH(REPLACE(col, ',', '')), 0)
FROM table;
```

#### ② 大文字・小文字の区別
`REPLACE` は大文字と小文字を厳密に区別します。
「'sql' も 'SQL' もまとめて数えたい」場合は、事前に全体を `UPPER` で揃えてから計算します。

#### ③ Oracle 11g以降なら専用関数がある
Oracle 11g 以降を使っているなら、正規表現を使ったもっと簡単な関数があります。

```sql
-- 第2引数で指定した文字（,）の出現回数を直接返す
SELECT REGEXP_COUNT('A,B,C,D', ',') FROM dual;
```
