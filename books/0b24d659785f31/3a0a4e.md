# 前提知識
Oracle SQLの `APPROX` 関数群は、一言でいえば **「100点の正確さを捨てる代わりに、爆速で結果を出す」** ための集計ツールです。
数千万行、数億行といったビッグデータを扱う際、正確な値を出すためにシステムが唸りを上げて数分待たされる状況を、わずかな誤差（通常1〜2%程度）を許容することで数秒に短縮してくれます。
![](https://storage.googleapis.com/zenn-user-upload/47a0078a8601-20260320.png)

## 1. 主要なAPPROX関数とその役割

### ① APPROX_COUNT_DISTINCT
「重複を除いた件数（ユニーク数）」を推定します。
* **通常の関数**: `COUNT(DISTINCT column)`
* **用途**: 巨大なログから「今日の訪問者数（UU）」をざっくり把握したい時。

### ② APPROX_SUM / APPROX_AVG
数値の合計や平均を推定します。
* **通常の関数**: `SUM(column)`, `AVG(column)`
* **用途**: 天文学的な数のセンサーデータやトラフィック量の総計を、リソースを抑えて集計したい時。

### ③ APPROX_RANK
「だいたい何位か」という順位を高速に計算します。
* **通常の関数**: `RANK() OVER (ORDER BY ...)`
* **用途**: 数億の商品の中から「売れ筋トップ10」を、メモリを節約しながら特定したい時。

## 2. メリット

1.  **圧倒的なスピード**: データをソートしたり、巨大なハッシュテーブルをメモリ（PGA）上に作ったりしないため、処理が非常に軽いです。
2.  **リソースの節約**: データベースサーバーのメモリ消費を劇的に抑えられます。
3.  **「傾向」の把握に最適**: ビジネスの意思決定において、「1,000,001人」か「999,998人」かは重要でないことが多いですが、「今すぐ数字が出る」ことは非常に重要です。

## 3. 注意点

非常に便利ですが、**「適材適所」** が求められる関数です。

### 1. 誤差がある（精度は97〜99%程度）
内部的に「HyperLogLog」などの数学的アルゴリズムを使用しているため、結果はあくまで推定値です。

> **財務・経理処理には絶対に使わない**
> 1円のズレも許されない決算報告や請求処理には、通常の `SUM` や `COUNT` を使用してください。

### 2. 結果がわずかに変動する可能性がある（非決定的）
同じデータに対して実行しても、並列処理の状況などにより、結果がごく僅かに（誤差の範囲内で）変わることがあります。

### 3. バージョンの制限
多くの機能は **Oracle 12c Release 2 (12.2)** 以降で導入されました。`APPROX_SUM` や `APPROX_RANK` などはさらに新しいバージョン（18cや19c）からのものもあります。

## 4. 既存のクエリを書き換えずに使う方法

「コードを直すのは面倒だけど、このセッションだけ高速化したい」という時は、以下のコマンドが魔法のように効きます。

```sql
-- セッション中、すべての COUNT(DISTINCT) を APPROX版に自動変換する
ALTER SESSION SET APPROX_FOR_COUNT_DISTINCT = TRUE;

-- セッション中、集計や順位付けを自動的に APPROX版に変換する
ALTER SESSION SET APPROX_FOR_AGGREGATION = TRUE;
```

## まとめ

| 項目 | 通常の集計関数 | APPROX関数群 |
| :--- | :--- | :--- |
| **正確性** | 100%（絶対） | 98%〜99%（推定） |
| **速度** | データ量に比例して遅くなる | **常に高速** |
| **メモリ使用量** | 多い（PGAを消費） | **極めて少ない** |
| **推奨シーン** | 決算、請求、正確な集計 | **ダッシュボード、傾向分析、ビッグデータ** |

# 問題18-1（おおよその数）  *Lv2*
### SHスキーマ`SALES`テーブルに存在する`cust_id`のおおよその件数を取得してください（重複は排除してください）。

## 期待する結果
| APPROX_CUSTOMER_COUNT | 
| --------------------- | 
| 7014                  | 

※ この値でない可能性あり

## 解答例
```sql
SELECT 
    APPROX_COUNT_DISTINCT(cust_id) AS approx_customer_count
FROM 
    sh.sales;
```

## 解説
Oracle SQLの **`APPROX_COUNT_DISTINCT`** は、巨大なデータ集合の中から「一意な値の数（ユニーク数）」を**高速に推定**するための関数です。
通常の `COUNT(DISTINCT ...)` がすべての値をメモリ上で比較して正確な数字を出すのに対し、これは数学的なアルゴリズム（HyperLogLog）を用いて、わずかな誤差を許容する代わりに劇的なスピードアップを実現します。

### 1. 基本的な使い方
```sql
SELECT 
    APPROX_COUNT_DISTINCT(cust_id) AS approx_customer_count
FROM 
    sh.sales;
```

#### 実行結果のイメージ
もし正確なユニーク数が `1,000,000` だった場合、この関数は `998,540` のような「極めて近い値」を返します。

#### 近似値でなく、正確な値を算出する場合
```sql
SELECT 
    COUNT(DISTINCT cust_id) AS customer_count
FROM 
    sh.sales;
```


### 2. なぜこれを使うのか？
「正確さ」よりも「速さ」や「リソース節約」が優先されるシーンで真価を発揮します。

* **圧倒的なスピード**: 数億〜数千億行のデータでも、ソート処理を行わないため非常に高速です。
* **メモリ（PGA）の節約**: メモリ消費量を劇的に抑えられるため、システム全体の負荷を下げられます。
* **大規模分析に最適**: 「だいたい何人くらいのユーザーが来たか」という傾向を把握するだけなら、正確な1人単位の数字は不要なことが多いためです。

### 3. 注意点と制限事項
非常に強力ですが、以下の「落とし穴」を理解して使う必要があります。

#### ① 100%の正確性は保証されない
通常、誤差は **0.9%〜2%程度** に収まるように設計されています。
> [!CAUTION]
> **経理・監査には不向き**
> 1円、1人単位の正確さが求められる法定報告書や請求処理には、絶対に使用しないでください。

#### ② 結果がわずかに変動する可能性がある
同じデータに対して実行しても、並列処理の状況などにより、結果がごく僅かに（誤差の範囲内で）変動することがあります。

#### ③ インデックスが使われない
この関数は全件スキャンを前提としたアルゴリズムであるため、Bツリーインデックスなどは活用されません（ただし、それでも通常版より速いことがほとんどです）。

### 4. 既存のSQLを「自動で」高速化する方法

コードを書き換えるのが面倒な場合、セッションの設定を変えるだけで、既存の `COUNT(DISTINCT)` をすべて内部的に `APPROX_COUNT_DISTINCT` として実行させることができます。

```sql
-- この設定以降、COUNT(DISTINCT) が自動的に APPROX版に置き換わる
ALTER SESSION SET APPROX_FOR_COUNT_DISTINCT = TRUE;

-- 普通に書くだけで高速（近似）処理される
SELECT COUNT(DISTINCT cust_id) FROM sh.sales;
```

### まとめ：使い分けのガイドライン

| 項目 | COUNT(DISTINCT) | APPROX_COUNT_DISTINCT |
| :--- | :--- | :--- |
| **正確性** | 100%（絶対） | 98%〜99%程度（推定） |
| **速度** | データ量に比例して遅くなる | 常に高速 |
| **メモリ使用量** | 多い | 非常に少ない |
| **推奨シーン** | 決算、請求、正確な集計 | ダッシュボード、傾向分析、ビッグデータ |

「100点の正確さのために10分待つか、98点の正確さのために1秒待つか」。この賢い選択肢を与えてくれるのが `APPROX` 系の魅力です。

# 【完全版】問題18-2（おおよそのランキング）　　*Lv3*
### SHスキーマ`SALES`テーブルにより、`AMOUNT_SOLD`のおおよその合計値と、おおよそのランキングTOP10を取得してください。（順位の大きい順に表示）。

## 期待する結果
| PROD_ID | TOTAL_SALES      | RANKING | 
| ------- | ---------------- | ------- | 
| 26      | 2572944.13000026 | 10      | 
| 28      | 3543725.89000017 | 9       | 
| 29      | 3845387.37999989 | 8       | 
| 21      | 5498727.80999996 | 7       | 
| 15      | 5635963.08000012 | 6       | 
| 13      | 6312268.40000003 | 5       | 
| 20      | 6691996.80999932 | 4       | 
| 14      | 7189171.77000041 | 3       | 
| 17      | 8314815.40000011 | 2       | 
| 18      | 15011642.5200011 | 1       | 

## 解答例
```sql
SELECT
    PROD_ID,
    APPROX_SUM(AMOUNT_SOLD) AS TOTAL_SALES,
    APPROX_RANK(
        ORDER BY
            APPROX_SUM(AMOUNT_SOLD) DESC
    )                       AS RANKING
FROM
    SH.SALES
GROUP BY
    PROD_ID
HAVING
    APPROX_RANK(
        ORDER BY
            APPROX_SUM(AMOUNT_SOLD) DESC
    ) <= 10;
```

# 【完全版】問題18-2（おおよそのランキング）　　*Lv3*
### SHスキーマ`SALES`テーブルにより、`AMOUNT_SOLD`のおおよその合計値と、おおよそのランキングTOP10を取得してください。（順位の大きい順に表示）。

## 期待する結果
| PROD_ID | TOTAL_SALES      | RANKING | 
| ------- | ---------------- | ------- | 
| 26      | 2572944.13000026 | 10      | 
| 28      | 3543725.89000017 | 9       | 
| 29      | 3845387.37999989 | 8       | 
| 21      | 5498727.80999996 | 7       | 
| 15      | 5635963.08000012 | 6       | 
| 13      | 6312268.40000003 | 5       | 
| 20      | 6691996.80999932 | 4       | 
| 14      | 7189171.77000041 | 3       | 
| 17      | 8314815.40000011 | 2       | 
| 18      | 15011642.5200011 | 1       | 

## 解答例
```sql
SELECT
    PROD_ID,
    APPROX_SUM(AMOUNT_SOLD) AS TOTAL_SALES,
    APPROX_RANK(
        ORDER BY
            APPROX_SUM(AMOUNT_SOLD) DESC
    )                       AS RANKING
FROM
    SH.SALES
GROUP BY
    PROD_ID
HAVING
    APPROX_RANK(
        ORDER BY
            APPROX_SUM(AMOUNT_SOLD) DESC
    ) <= 10;
```


## 解説
Oracle SQLの **`APPROX_RANK`** は、膨大なデータセットに対して「おおよその順位」を高速に計算するための関数です。
通常の `RANK` 関数が全データを厳密にソート（並べ替え）して順位を決めるのに対し、`APPROX_RANK` はメモリ消費を抑え、ソート処理を簡略化することで、**数千万〜数億行のデータから「上位N件」を特定するスピードを劇的に向上**させます。

### 1. 基本的な構文
`APPROX_RANK` は単体ではなく、必ず **`APPROX_SUM`** または **`APPROX_COUNT_DISTINCT`** と組み合わせて使用します。
**`APPROX_SUM`** は、文字通り数値の「合計」を近似値で高速に計算する関数です。

```sql
APPROX_RANK (
    [ PARTITION BY 項目 ] 
    ORDER BY APPROX_SUM(集計項目) | APPROX_COUNT_DISTINCT(集計項目) DESC
)
```

#### 近似値でなく、正確な値を算出する場合
```sql
SELECT
    *
FROM
    (
        SELECT
            prod_id,
        -- ① 売上合計の近似値を算出
            total_sales,
            RANK()
            OVER(
                ORDER BY
                    total_sales DESC
            ) AS ranking
        FROM
            (
                SELECT
                    prod_id,
                    SUM(amount_sold) AS total_sales
                FROM
                    sh.sales
                GROUP BY
                    prod_id
            )
    )
WHERE
    ranking <= 10 -- ③ 上位10位までを表示
ORDER BY
    ranking DESC
```


### 2. 注意点と制限

#### ① 僅差のデータにおける入れ替わり
「1位と2位」のように差がはっきりしていれば正確に出ますが、「99位と100位」のように**値が極めて近い場合、順位が入れ替わる可能性**があります。

> 「上位10名に豪華賞品」といった、1位のズレも許されない厳密なビジネスルールには使用しないでください。

#### ② HAVINGが必須
単一のクエリ内で `APPROX_RANK` を使用する場合、`HAVING` 句での絞り込みは「必須」です。`ORA-62237: APPROX_RANK関数はここでは使用できません` というエラーが発生します。

`APPROX_RANK` は、全データをソートしません。その代わりに、**メモリの中に「上位N件分」だけの作業スペース**を確保して、データを流し込みながら順位を入れ替えていきます。

* `HAVING ... <= 10` と書くことで、初めて Oracle は「あ、メモリに 10 個分の席を用意すればいいんだな」と判断できます。
* もし `HAVING` がないと、「何番目までキープすればいいか」が決まらないため、近似アルゴリズムが成立しなくなってしまうのです。
