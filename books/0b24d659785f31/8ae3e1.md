# 問題3-1（全データ取得）  *Lv1*
### **HRスキーマ`REGIONS`テーブルの全てのデータを取得してください。（順不同）**

## 期待する結果
| REGION_ID | REGION_NAME | 
| --------- | ----------- | 
| 10        | Europe      | 
| 20        | Americas    | 
| 30        | Asia        | 
| 40        | Oceania     | 
| 50        | Africa      | 

## 解答例
```sql:例1：「*」を使用
SELECT
    *
FROM
    hr.regions
```
```sql:例2：全てのカラムを指定
SELECT
    region_id,
    region_name
FROM
    hr.regions
```

## 解説
例1、例2、2つのクエリーは、どちらも「全てのデータを取り出す」という目的は同じですが、**実務レベルではその意味合いが大きく異なります。**


### 例1：アスタリスク（*）による全カラム選択
* **内容**: `hr.regions`テーブルに定義されている **全ての列（カラム）** を、定義された順番通りに取得します。
* **用途**: 主に、テーブルの中にどんなデータが入っているか「とりあえず確認したい」という**アドホック（一時的）な調査**に向いています。

### 例2：カラム名指定による選択
* **内容**: 取得したい**列の名前を明示的**に指定して取得します。
* **用途**: **アプリケーション開発や定型レポートの作成**など、本番環境で動かすプログラムにおいては、こちらの書き方が「鉄則」です。

### 実践的な注意点と使い分け
実務でどちらを使うべきか、以下の観点から比較してみましょう。

#### 1. パフォーマンスの差
例1の `SELECT *` は、必要のないデータまでネットワーク経由で転送するため、無駄な負荷がかかります。
* テーブルに100個の列があり、必要なのが2個だけだった場合、残りの98個分のデータ転送が**パフォーマンスを低下**させます。
* Oracle Database内部でも「どの列があるか」を解析するステップが増えるため、明示的な指定の方が効率的です。

#### 2. 変更に対する強さ（保守性）
ここが最も重要なポイントです。
* **例1の危険性**: もし後からテーブルに新しい列が追加されたり、列の順番が変わったりすると、プログラム側で「1番目の列はID、2番目は名前」と決め打ちしていた場合に**エラーや予期せぬ挙動**を引き起こします。
* **例2の安全性**: 列が追加されても、指定した列だけは確実に取得できるため、プログラムが壊れるリスクを最小限に抑えられます。

### 比較まとめ

| 比較項目 | `SELECT *` (例1) | カラム名指定 (例2) |
| :--- | :--- | :--- |
| **可読性** | 何を取得しているか不明 | 取得内容が一目瞭然 |
| **実行速度** | 遅くなる可能性がある | 最適化しやすい |
| **保守性** | テーブル変更に弱い | **テーブル変更に強い** |
| **推奨シーン** | 手動でのデータ確認 | **開発・本番コード** |

Oracle FreeSQLにはこのスタイルで記述するサポート機能があります（一部て修正が必要かもしれません）。

# 問題3-2（必要なデータの取得） *Lv1*
### **HRスキーマ`COUNTRIES`テーブルのうち`REGION_ID`が20のデータを取得し、`COUNTRY_ID`および`COUNTRY_NAME`を表示してください。（順不同）**

## 期待する結果
| COUNTRY_ID | COUNTRY_NAME             | 
| ---------- | ------------------------ | 
| AR         | Argentina                | 
| BR         | Brazil                   | 
| CA         | Canada                   | 
| MX         | Mexico                   | 
| US         | United States of America | 


## 解答例
```sql
SELECT
    country_id,
    country_name
FROM
    hr.countries
WHERE
    region_id = 20
```

## 解説
このクエリーは、テーブルから特定の条件に合致するデータのみを抽出する **「フィルタリング（選択）」** の基本形です。

### 動作の流れ
1. **`FROM hr.countries`**: `hr` スキーマの `countries`（国）テーブルを参照します。
2. **`WHERE region_id = 20`**: 全データの中から、`region_id` 列の値が `20` である行だけを絞り込みます。
3. **`SELECT country_id, country_name`**: 絞り込まれた行の中から、IDと国名の2カラムだけを表示します。


### 実践的な注意点

#### 1. データ型への意識（暗黙の型変換）
この例では `region_id = 20` と数値を指定していますが、もしこの列が「文字列型（VARCHAR2など）」だった場合は注意が必要です。
* `WHERE column = 20`（数値として比較）
* `WHERE column = '20'`（文字列として比較）

Oracleは親切なので、型が違っても自動で変換して計算してくれることがありますが（暗黙の型変換）、これを多用すると**大きな落とし穴**にはまってしまう事があります（下記参照）。「数値には数値、文字列には引用符（'）」を徹底しましょう。

#### 2. 文字列の比較は「大文字・小文字」を区別する
Oracle SQLでは、**データとしての文字列は大文字と小文字を厳密に区別**します。
* `WHERE country_name = 'Japan'` → ヒットする
* `WHERE country_name = 'JAPAN'` → ヒットしない
※命令文（SELECTなど）は区別されませんが、中身のデータは別物として扱われる点に注意してください。


:::message
### **暗黙の型変換に潜む「大きな落とし穴」**
「動くから大丈夫」と多用するのは禁物です。特に注意すべきなのが パフォーマンスへの影響 です。暗黙の型変換が行われると、本来設定されている **「インデックス（索引）」が有効に機能しなくなる**ことがあります。その結果、データの検索速度が極端に落ち(※)、システム全体の大きな遅延や障害を招く恐れがあります。
この現象はエラーとして表示されないため、原因に気づきにくく、後から修正するのが非常に大変です。 **「型は最初から正しく合わせる」** のが、安全で高速なSQLを書くための基本です。
(※)データ量によっては、数百倍〜数千倍遅くなることもあります。
:::

# 問題3-3（別名） *Lv1*
### **HRスキーマ`COUNTRIES`テーブルのうち`REGION_ID`が2のデータを取得し、`COUNTRY_ID`および`COUNTRY_NAME`を表示してください。（順不同）**
### **その際、列名を以下のように変更し、表示してください。**
* 　**COUNTRY_ID　　　→ 　ID**
* **COUNTRY_NAME　→ 　NAME**

## 期待する結果
| ID         | NAME                     | 
| ---------- | ------------------------ | 
| AR         | Argentina                | 
| BR         | Brazil                   | 
| CA         | Canada                   | 
| MX         | Mexico                   | 
| US         | United States of America | 


## 解答例
```sql
SELECT
    country_id as "ID",
    country_name as "NAME"
FROM
    hr.countries
WHERE
    region_id = 20
```

## 解説
今回のコードは、カラム名に「別名（エイリアス）」をつける手法がメインテーマです。

### 列別名（エイリアス）の設定
`country_id as "ID"` のように記述することで、検索結果のヘッダー（列名）を元のテーブル定義とは異なる名前に置き換えています。
* **`as` 句**: 別名を指定するキーワードです（Oracleでは `as` を省略して `country_id "ID"` と書くことも可能ですが、可読性のために `as` を付けるのが一般的です）。
* **用途**: 
    * 複雑な計算式（例：`salary * 12`）に `as "Annual Salary"` と名前を付けて分かりやすくする。
    * 日本語のヘッダーを表示させる（例：`as "国ID"`）。

### 実践的な注意点：二重引用符（" "）の罠
Oracle SQLにおいて、ダブルクォーテーションで囲むかどうかには大きな意味があります。

#### 1. 大文字・小文字の固定
* **囲まない場合**: `as ID` と書くと、Oracle内部で自動的に**大文字**に変換され、結果は `ID` と表示されます。
* **囲む場合**: `as "Id"` と書くと、**入力した通り**に表示されます。
    * ※今回の例 `as "ID"` は、囲まなくても大文字になりますが、明示的に指定することで「この列名で固定する」という意思表示になります。

#### 2. 別名が使えない場所（SQLの実行順序）
ここが初心者が最もハマりやすいポイントです。**`SELECT` で付けた別名は、同じクエリーの `WHERE` 句では使えません。**

* **NG例**:
  ```sql
  SELECT country_id as "ID"
  FROM hr.countries
  WHERE "ID" = 10  -- エラーになります！
  ```
* **理由**: SQLは内部的に `FROM` → `WHERE` → `SELECT` の順で処理されるため、`WHERE` が動いている時点では、まだ `ID` という別名が存在していないからです。
* **例外**: `ORDER BY`（並び替え）句は最後に実行されるため、別名を使用できます。

#### 3. 特殊文字や空白
別名にスペースを入れたい場合は、必ず `" "` で囲む必要があります。
* `as "Country Code"` （○ 有効）
* `as Country Code` （× エラー）

---

## 比較まとめ

| 書き方 | 実行結果のヘッダー | 性質 |
| :--- | :--- | :--- |
| `country_id` | `COUNTRY_ID` | デフォルト（大文字） |
| `country_id as id` | `ID` | 自動で大文字に変換される |
| `country_id as "id"` | `id` | **小文字が維持される** |
| `country_id as "国コード"` | `国コード` | 日本語などが使用可能 |


<br>
:::message
### システム開発では " " を使わないのが無難？
結論から言うと、**「システム開発の現場では、識別子（テーブル名や列名）を " " で囲むのは避けるのが鉄則」** です。
実務でこれをやると**負の遺産**になりかねません。なぜエンジニアたちがこれほどまでに `" "` を嫌うのか、その裏にある「3つの地獄」を解説します。

<br>
#### 1. 大文字・小文字の「固定」地獄

Oracle（や多くの標準SQL）では、引用符で囲まない名前は内部的に**すべて大文字**として扱われます。

* **囲まない場合**: `SELECT user_id` も `SELECT USER_ID` も、DB側は「あ、`USER_ID` のことね」と解釈してくれます。
* **" " で囲んだ場合**: `"UserId"` と定義すると、**以後、一字一句（大文字小文字まで）完全に一致させないとエラー**になります。

**現場で起こる悲劇：**
誰かが一人でもテーブル作成時に `"UserId"` と書いてしまうと、他の開発者が `SELECT user_id ...` と書くたびに「そんな列はありません」と怒られ、デバッグに時間を無駄にさせることになります。

<br>
#### 2. プログラムコード内の「エスケープ」地獄

これが実務上の最大のデメリットかもしれません。JavaやPython、PHPなどのプログラム内でSQLを書く際、SQL文自体を文字列（`" "` や `' '`）で囲みますよね。

もしDB側に `" "` が必要な列名があると、コードはこうなります：

* **普通の書き方**:
    `sql = "SELECT user_id FROM users"`
* **" " が必要な書き方**:
    `sql = "SELECT \"UserId\" FROM \"Users\""` （言語によっては `\"` や `""` でエスケープが必要）

すべてのクエリーにこれが入ると、**視認性が最悪**になります。タイピングミスも増えますし、パッと見て何が書いてあるか分からなくなるため、バグの温床になります。

<br>
#### 3. 予約語との衝突を「許してしまう」危うさ

本来、SQLには使ってはいけない「予約語（`SELECT`, `ORDER`, `GROUP` など）」があります。

* `CREATE TABLE my_table (order NUMBER)` → **エラー（orderは予約語だから）**
* `CREATE TABLE my_table ("ORDER" NUMBER)` → **作成できてしまう**

「あ、予約語だけど `" "` で囲めば使えるじゃん！」と安易に使うと、その後の開発で **「この ORDER は命令の ORDER なのか、列名の ORDER なのか？」** と解析エンジンや人間が混乱する原因になります。

<br>
#### まとめ：なぜ「使わないのが無難」なのか

| 項目 | `" "` を使わない（標準） | `" "` を使う（非推奨） |
| :--- | :--- | :--- |
| **柔軟性** | 大文字・小文字を気にしなくて良い | 厳密に一致させる必要がある |
| **開発効率** | エスケープ不要でスッキリ書ける | エスケープだらけで読みづらい |
| **ツール親和性** | どのツールでも問題なく動く | ツールによって挙動が変わるリスクあり |
| **設計の質** | 予約語を避ける良い設計になる | 悪い設計（予約語使用）を許容してしまう |

<br>
#### 当演習の解答例では、SQLのルールをより深く理解していただくための練習として、意図的に `" "` を使って記述しているものがあります。実際のシステム開発現場での推奨ルールとは一部異なる箇所がありますので、その点をご理解いただいた上で学習を進めていただければ幸いです。

:::

# 問題3-4（並び替え） *Lv1*
### **OEスキーマ`CUSTOMERS`テーブルのうち、`NLS_TERRITORY`が「CHINA」であるデータに対し、**
   *　**`CUST_FIRST_NAME` の昇順**
   *　**`CUST_LAST_NAME` の降順**
### **で並び替え表示してください**


## 期待する結果
| CUSTOMER_ID | CUST_FIRST_NAME | CUST_LAST_NAME | 
| ----------- | --------------- | -------------- | 
| 981         | Daniel          | Gueney         | 
| 308         | Glenda          | Dunaway        | 
| 309         | Glenda          | Bates          | 
| 360         | Helmut          | Capshaw        | 


## 解答例
```sql:例1：カラムを指定する
SELECT
    customer_id,
    cust_first_name,
    cust_last_name
FROM
    oe.customers
WHERE
    nls_territory = 'CHINA'
ORDER BY
    cust_first_name ASC,
    cust_last_name DESC
```
```sql:例2：列順を指定する
SELECT
    customer_id,
    cust_first_name,
    cust_last_name
FROM
    oe.customers
WHERE
    nls_territory = 'CHINA'
ORDER BY
    2, 3 DESC
```

## 解説
今回の演習では、データの並び替えを行う `ORDER BY` 句が登場しました。
この2つのクエリーは結果こそ同じになりますが、**「保守性（メンテナンスのしやすさ）」** という点で大きな違いがあります。

### 例1：カラム名による明示的な指定
```sql
ORDER BY cust_first_name ASC, cust_last_name DESC
```
* **内容**: 名前（First Name）を **昇順（ASC: 小さい順）** で並べ、同じ名前の人がいた場合は苗字（Last Name）を **降順（DESC: 大きい順）** で並べます。
* **ポイント**: 何を基準に並べているかが一目瞭然です。

### 例2：列番号（位置）による指定
```sql
ORDER BY 2, 3 DESC
```
* **内容**: `SELECT` 句で書いた順番の **2番目のカラム（cust_first_name）** と **3番目のカラム（cust_last_name）** を基準に並べ替えます。
* **ポイント**: `ASC` は省略可能（デフォルト）なので、`2` は昇順になります。


### 実践的な注意点と使い分け

#### 1. 列番号指定（例2）は「諸刃の剣」
列番号で指定する方法は、タイピング量が減って楽ですが、実務のシステム開発では**原則として非推奨**とされることが多いです。

* **理由**: 後で `SELECT` 句の間に新しいカラムを追加したり、順番を入れ替えたりした瞬間、**意図しない順番でソートされるバグ**に変わってしまうからです。
* **使い道**: 自分がその場限りの調査で SQL を投げる「使い捨てのクエリー」なら非常に便利です。

#### 2. ASC と DESC の挙動（Oracle特有の挙動）
* **NULLの扱い**: Oracleでは、昇順（ASC）のとき **NULLは最後** に、降順（DESC）のとき **NULLは最初** に来ます。
    * もし「常にNULLを最後にしたい」場合は、`ORDER BY 2 DESC NULLS LAST` のように書くテクニックもあります。

#### 3. パフォーマンスへの影響
並び替えは、データベースにとって非常に「重い」処理です。
* 数百万件のデータを `ORDER BY` すると、処理に時間がかかりシステムが重くなる原因になります。
* 頻繁に並び替えを行うカラム（今回の例なら氏名など）には、あらかじめ **「インデックス」** を作成しておくのが、プロの設計です。

### 比較まとめ

| 比較項目 | カラム名指定 (例1) | 列番号指定 (例2) |
| :--- | :--- | :--- |
| **可読性** | 非常に高い | 低い（SELECT句を確認しに行く必要がある） |
| **変更への強さ** | 強い（列順が変わってもOK） | **非常に弱い（列順変更でバグる）** |
| **推奨シーン** | **プログラム、定型レポート** | 一時的なデータ調査、アドホックな集計 |

<br>
:::message

### **「ASC」の使い分け基準**
SQLを書く際、「ASCは省略できるのになぜ存在するのか？」「いつ書くべきなのか？」という疑問を抱く方は少なくありません。

明確な正解が決まっているわけではありませんが、実務においてコードの美しさと読みやすさを両立させるための、ひとつの判断基準をご紹介します。
<br>
#### 1. すべての条件が「昇順」の場合

この場合は、**ASCを省略します。**
デフォルトの挙動が昇順であるため、あえて記述しないことでコードを冗長（無駄に長い状態）にせず、スッキリと見せることができます。

<br>
#### 2. 「昇順」と「降順」が混在する場合

この場合は、**ASCを省略せずに明示します。**
「ここは昇順、あそこは降順」と意図がはっきり伝わるようになり、コードの**可読性（読みやすさ）** が向上するためです。他の人がコードを見た際にも、設定ミスではないことが一目で分かります。
:::

# 問題3-5（重複削除） *Lv1*
### **SHスキーマ`COUNTRIES`テーブルより`COUNTRY_REGION`を取得してください。その際、重複は排除するようにしてください。（順不同）**

## 期待する結果
| COUNTRY_REGION | 
| -------------- | 
| Americas       | 
| Europe         | 
| Asia           | 
| Oceania        | 
| Africa         | 
| Middle East    | 

## 解答例
```sql:例1：DISTINCTを使用
SELECT DISTINCT
    country_region
FROM
    sh.countries
```
```sql:例2：GROUP BYを使用（参考）
SELECT
    country_region
FROM
    sh.countries
GROUP BY
    country_region
```

## 解説
今回の演習は、データの「重複を削る」手法ですね。`DISTINCT` と `GROUP BY` は、「結局どっちを使えばいいの？」と迷いやすいポイントです（`GROUP BY`は「8章 集計関数」で本格的に学習します）。

Oracle SQLにおけるそれぞれの役割と、現場での使い分けを解説します。

### 例1：`DISTINCT` を使用
* **役割**: 結果セットから**重複する行を完全に取り除き**、ユニーク（一意）な値だけを表示します。
* **ニュアンス**: 「どんな種類があるのか、リストをパッと見たい」という時に使います。
* **対象**: `SELECT` の直後に記述し、指定したすべての列の組み合わせで重複を判断します。

### 例2：`GROUP BY` を使用（参考）
* **役割**: 指定した列の値が同じものを**ひとまとめ（グループ化）**にします。
* **ニュアンス**: 「地域ごとに集計（合計や平均など）をするための準備」という色が強いです。
* **対象**: `FROM` 句の後に記述し、グループ化する基準の列を指定します。

### 実践的な注意点と使い分け

#### 1. 実行結果は同じだが「目的」が違う
今回の例のように、単一のカラムの重複を除くだけなら、得られる結果は全く同じです。しかし、SQLを読む人（または未来の自分）に伝わるメッセージが異なります。

* **「種類を知りたいだけ」** なら、直感的にわかりやすい **`DISTINCT`** を使うのが正解です。
* **「この後、各地域のデータ件数（COUNT）などを出す予定がある」**なら、**`GROUP BY`** を使います。

#### 2. パフォーマンスの差（Oracleの場合）
古いデータベースでは `GROUP BY` の方が速いと言われることもありましたが、最新の Oracle Database では最適化（オプティマイザ）が非常に賢いため、**この程度のクエリーで速度差が出ることはまずありません。** 速度よりも「コードの意図が伝わるか」を優先しましょう。

#### 3. `NULL` の扱い
どちらの方法も、`NULL`（空データ）を一つの値として扱います。
* 重複した `NULL` がたくさんあっても、結果には **1つの `NULL` 行** として表示されます。

#### 4. 複数カラムを指定した場合の挙動
ここが間違いやすいポイントです。
```sql
SELECT DISTINCT department_id, job_id FROM employees
```
この場合、「部署IDだけ」でも「職種IDだけ」でもなく、**「部署IDと職種IDの組み合わせ」** がユニークなものだけが抽出されます。

### 比較まとめ

| 項目 | `DISTINCT` | `GROUP BY` |
| :--- | :--- | :--- |
| **主な目的** | 重複行の排除（リスト作成） | データのグループ化（集計準備） |
| **記述場所** | `SELECT` の直後 | `FROM` や `WHERE` の後 |
| **集計関数** | 基本的には併用しない | `SUM`, `COUNT` などと併用する |
| **可読性** | 「一意な値を見たい」意図が明確 | 「何かの単位でまとめたい」意図が明確 |


<br>
:::message
### とりあえず `DISTINCT`
「とりあえず `DISTINCT`」は、多くのエンジニアが通る **「禁断の果実」** です。よく分からないけどデータが重複しているので、とりあえず`DISTINCT`で重複をなくしちゃおうという行為です。一見すると結果が綺麗になるため、問題が解決したように見えてしまいますが、実は裏で深刻な副作用を引き起こしていることが多々あります。
私は開発プロジェクトで、誰かが書いたSQLスクリプトに`DISTINCT`があると、ちょっと警戒してしまいます…

なぜ「とりあえず `DISTINCT`」が危険なのか、その理由を具体的に解説します。
<br>
### 1. 根本原因：意図しない「1対多」の結合
もっとも多いのが、結合条件の不足によって **「1行のはずが10行、100行に増えている」** ケースです。

※「結合」は後の章で出てきますので、ここでは感覚で理解できれば大丈夫です。
簡単にいうと複数のテーブルからデータを抽出することをいいます。


<br>
#### 具体例
「顧客名の一覧を出したい」と思い、顧客（`CUSTOMERS`）と販売履歴（`SALES`）を結合したとします。

```sql
-- 悪い例：DISTINCTで無理やり消す
SELECT DISTINCT c.cust_id, c.cust_last_name
FROM sh.customers c
JOIN sh.sales s ON c.cust_id = s.cust_id;
```

* **何が起きているか**: 1人の顧客が1,000回購入していれば、`JOIN` 直後の内部データではその顧客の行が**1,000行に増殖**しています。
* **DISTINCTの役割**: その増殖した1,000行を、後から無理やり1行にギュッと凝縮して「なかったこと」にしています。
<br>

### 2. 結合キーの不足に気づけない
例えば「有効な契約期間内のデータだけ出したい」のに、期間の条件（`BETWEEN`）を書き忘れて過去の全データと結合され、重複が発生したとします。

* `DISTINCT` を使うと、その「書き忘れ」という**バグ自体が隠蔽されてしまいます。**
* 本来なら「データが重複している ＝ 結合条件かフィルタが足りない」という**アラート（警告）** として受け取るべき事象を、自ら消し去っていることになります。

<br>

### 結論：`DISTINCT` は「最後の手段」

`DISTINCT` を使っていいのは、**「論理的にどうしても重複が発生する構造であり、かつそれが仕様である」** と断言できるときだけです。

もし「なぜか増えたから消す」という動機で `DISTINCT` を使いたくなったら、それは **「SQLがあなたに『結合条件か設計が間違っているよ！』と教えてくれているサイン」** だと捉えてみてください。
