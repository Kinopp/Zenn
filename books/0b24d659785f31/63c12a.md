# 前提知識
**PIVOT（ピボット）**と**UNPIVOT（アンピボット）** は、データの「向き」をガラッと変える魔法のような操作です。

## 1. PIVOT / UNPIVOTとは？
一言で言うと、**「行と列の入れ替え」** です。

* **PIVOT（行 → 列）：** 縦に並んだデータを横に広げて、集計レポート（クロス集計）を作る。
* **UNPIVOT（列 → 行）：** 横に並んだデータを縦に並べ直して、コンピュータが処理しやすい「生データ」に戻す。

  ![](https://storage.googleapis.com/zenn-user-upload/0f814b0d0961-20260323.png)

## 2. PIVOT：報告書作成の強い味方

例えば、`HR.EMPLOYEES` スキーマで「どの部署に、どの職種の人が何人いるか」を一覧表にしたい場合、普通に `GROUP BY` すると結果は縦長になります。これを **「横長の見やすい表」** にするのがPIVOTです。

## 3. UNPIVOT：データの「お片付け」
実務では、以下のように「1月, 2月, 3月...」と列が分かれた「人間には見やすいけれど、SQLでは集計しにくい表」に出会うことがあります。

| 商品名 | 1月売上 | 2月売上 | 3月売上 |
| :--- | :--- | :--- | :--- |
| リンゴ | 100 | 150 | 120 |

これをUNPIVOTすると、以下のようになります。

| 商品名 | 月 | 売上 |
| :--- | :--- | :--- |
| リンゴ | 1月 | 100 |
| リンゴ | 2月 | 150 |
| リンゴ | 3月 | 120 |

**「列名をデータとして扱いたい」** ときはUNPIVOTの出番です。

## 期待する結果
| CUST_ID | ITEM           | VALUE                                                                                                     | 
| ------- | -------------- | --------------------------------------------------------------------------------------------------------- | 
| 100005  | AFFINITY_CARD  | 1                                                                                                         | 
| 100005  | BASEBALL       | 0                                                                                                         | 
| 100005  | COMMENTS       | Affinity card is a great idea. But your store is still too expensive. I am tired of your lousy junk mail. |

| 100005  | OCCUPATION     | Crafts                                                                                                    | 
| 100005  | SOCCER         | 1                                                                                                         |

| 100005  | UNKNOWN        | 0                                                                                                         | 
| 100005  | YRS_RESIDENCE  | 5                                                                                                         | 

## 解答例
```sql
WITH sd AS ( -- unpivotするためにこのwith句で型を統一する
    SELECT
        cust_id,
        education,
        occupation,
        household_size,
        to_char(yrs_residence) AS yrs_residence,
        to_char(affinity_card) AS affinity_card,
        to_char(cricket)       AS cricket,
        to_char(baseball)      AS baseball,
        to_char(tennis)        AS tennis,
        to_char(soccer)        AS soccer,
        to_char(golf)          AS golf,
        to_char(unknown)       AS unknown,
        to_char(misc)          AS misc,
        comments
    FROM
        sh.supplementary_demographics
    WHERE
        cust_id = 100005
)
SELECT
    cust_id,
    item,
    value
FROM
    sd UNPIVOT ( value
        FOR item
    IN ( education AS 'EDUCATION',
         occupation AS 'OCCUPATION',
         household_size AS 'HOUSEHOLD_SIZE',
         yrs_residence AS 'YRS_RESIDENCE',
         affinity_card AS 'AFFINITY_CARD',
         cricket AS 'CRICKET',
         baseball AS 'BASEBALL',
         tennis AS 'TENNIS',
         soccer AS 'SOCCER',
         golf AS 'GOLF',
         unknown AS 'UNKNOWN',
         misc AS 'MISC',
         comments AS 'COMMENTS' ) )
ORDER BY
    cust_id,
    item
```

## 解説
このクエリを一言でいうと、**「横長（ワイド）なテーブルを、縦長（ロング）なリスト形式に変換する」** という処理を行っています。SQLの世界ではこれを **UNPIVOT（アンピボット）** と呼びます。
Excelの表で「項目が列（横）に並んでいる」状態を、データベースで扱いやすい「項目名と値のペア（縦）」に変換するイメージです。

### 1. UNPIVOTの仕組み
通常、テーブルは「1人＝1行」で、属性（住所、電話番号、趣味など）がカラムとして横に並びます。これを **「ワイド形式（横持ち）」** と呼びます。
UNPIVOTは、これを **「ロング形式（縦持ち）」** に変換します。

#### 基本的な構文
```sql
SELECT * FROM テーブル名
UNPIVOT (
    データ値のカラム名  -- (例: value)
    FOR 項目名のカラム名 -- (例: item_name)
    IN (カラム1, カラム2, カラム3...) -- 縦に並べたいカラムを指定
);
```

### 2. 知っておくべき「3つの注意点」
UNPIVOTは非常に便利ですが、実務で使う際に必ずと言っていいほど直面する「落とし穴」があります。

#### ① データ型を「完全に」一致させる必要がある
これが最も多いエラーの原因です。
`IN` 句で指定するすべてのカラムは、**同じデータ型**でなければなりません。
* **NG**: `氏名(VARCHAR2)` と `年齢(NUMBER)` を同時にUNPIVOTしようとするとエラーになります。
* **対策**: 先ほどのクエリにあったように、`TO_CHAR` や `CAST` を使って、あらかじめすべてのカラムを文字列型などに統一しておく必要があります。

#### ② デフォルトでは「NULL」が消滅する
UNPIVOTの標準的な動作（`EXCLUDE NULLS`）では、**値がNULLのカラムは結果から除外されます。**
* **困るシーン**: 「アンケートで未回答の項目も含めてリスト化したい」とき。
* **対策**: `UNPIVOT INCLUDE NULLS` と明示的に記述することで、NULLの行も維持できます。

#### ③ 行数が「掛け算」で増える
100万行のテーブルに対し、10個のカラムをUNPIVOTすると、結果は**1,000万行**になります。
* **リスク**: メモリ消費や処理時間が大幅に増えるため、大規模なテーブルに対して行う場合は、あらかじめ `WHERE` 句で対象を絞り込んでおくことが鉄則です。

### 3. いつ使うのが正解？

* **BIツール・グラフ作成の前処理**: 
  「月」が横に並んでいる売上表を、1つの「日付」軸にまとめたいとき。
* **属性の検索**: 
  「趣味1」「趣味2」「趣味3」…と別れているカラムから、特定の趣味を持つ人を一括で探したいとき。
* **画面表示の効率化**: 
  スマホアプリのプロフィール画面のように、項目を上から下にずらっと並べるAPIを作るとき。

### 4. クエリの構造と役割（解答例）

#### ① WITH句（sd）：データ型の統一
ここがこのクエリの「隠れたファインプレー」です。
`UNPIVOT` を行う際、新しく作られる `value` カラムには、元の複数のカラムの値が混ざって入ることになります。
* `education` は文字列（VARCHAR2）
* `yrs_residence` は数値（NUMBER）
このように型がバラバラだとエラーになってしまうため、**`to_char` を使ってあらかじめ全ての項目を文字列型に揃えています。** これを忘れると `UNPIVOT` は成功しません。

#### ② UNPIVOT句：縦持ちへの変換
ここで魔法が起きます。
* **`value`**: 各項目の「中身（値）」が入る新しいカラム名です。
* **`FOR item`**: 「どのカラムから来たか」という「項目名」が入る新しいカラム名です。
* **`IN (...)`**: ここで指定したカラムたちが、縦に並び変わります。`AS 'EDUCATION'` のように書くことで、出力される項目名をきれいに整えています。

# 【完全版】問題13-2（行→列の入れ替え1） *Lv3*
### COスキーマ`ORDERS`テーブルの注文に対して、年単位の総額を算出してください。1注文に対する金額は`ORDER_ITEMS`テーブルの`UNIT_PRICE × QUANTITY`で計算してください。年単位に1列とするようにしてください。

## 期待する結果
| Y2021     | Y2022    | 
| --------- | -------- | 
| 222438.11 | 86160.42 | 

## 解答例
```sql
SELECT
    *
FROM
    (
        WITH order_total AS (
            SELECT
                order_id,
                SUM(unit_price * quantity) AS total
            FROM
                co.order_items
            GROUP BY
                order_id
        )
        SELECT
            extract(YEAR FROM o.order_tms)  AS yr,
            SUM(ot.total)                   ttl
        FROM
            order_total ot
            LEFT JOIN co.orders   o ON ot.order_id = o.order_id
        GROUP BY
            extract(YEAR FROM o.order_tms)
    ) PIVOT (
        SUM(ttl)
        FOR yr
        IN ( 2021 AS y2021, 2022 AS y2022 )
    )
```

## 解説
**PIVOT（ピボット）**は、UNPIVOTとは真逆の操作です。一言でいうと、**「縦に並んだ複数の行を、横一列のカラム（列）に集約して、クロス集計表を作る」** 機能です。
Excelの「ピボットテーブル」をSQLのクエリ内で完結させるイメージです。

### 1. PIVOTの仕組み
「日付、商品名、売上」という縦長のリスト（ロング形式）を、「日付」を軸にして「商品Aの売上、商品Bの売上...」と横に広げる（ワイド形式）のがPIVOTの役割です。

#### 基本的な構文
```sql
SELECT * FROM (
    SELECT 月, 商品名, 売上 FROM 売上テーブル
)
PIVOT (
    SUM(売上)                     -- ① 集計したい値（必ず集計関数が必要）
    FOR 商品名 IN ('A' AS 商品A, 'B' AS 商品B) -- ② 横に並べたい項目
);
```

### 2. 知っておくべき「3つの注意点」
PIVOTはレポート作成の強力な武器ですが、UNPIVOTよりも制約が厳しいため注意が必要です。

#### ① 「集計関数」が必須
PIVOTは単に行を入れ替えるだけでなく、**「集約」** を行います。
* **ポイント**: `SUM`（合計）、`AVG`（平均）、`COUNT`（件数）などの集計関数を必ず指定する必要があります。
* **理由**: 同じ「商品A・1月」のデータが複数行あった場合、それらを1つのセルにまとめるためのルールが必要だからです。

#### ② 横に並べる項目を「決め打ち」する必要がある
これが最も不便に感じる点かもしれません。
* **制約**: `IN` 句の中に、**「どの値をカラム名にするか」を具体的に書かなければなりません。**
* **弱点**: 商品が100種類あるなら100個書く必要があります。新商品が増えたらクエリを書き直さなければなりません。
> [!NOTE]
> 自動で動的にカラムを増やしたい場合は「動的SQL」を組むなどの高度なテクニックが必要になります。

#### ③ 暗黙的な GROUP BY に注意
PIVOT句に含まれていないカラムは、すべて自動的に **`GROUP BY` の対象（軸）**になります。
* **リスク**: もし元のテーブルに「不要な細かいカラム（例えば秒単位のタイムスタンプなど）」が残っていると、行がうまくまとまらず、スカスカの表になってしまいます。
* **対策**: `PIVOT` に渡す前に、あらかじめ `SELECT` 文（インラインビューやWITH句）で、**必要なカラムだけに絞り込んでおく**のが鉄則です。

### 3. UNPIVOT と PIVOT の比較

| 項目 | UNPIVOT（アンピボット） | PIVOT（ピボット） |
| :--- | :--- | :--- |
| **変化** | 横長 → 縦長 | 縦長 → 横長 |
| **主な用途** | データの正規化・検索用 | レポート作成・クロス集計 |
| **集計関数** | 不要（分解するだけ） | **必須**（集約するため） |
| **行数の変化** | **増える** | **減る** |

### 4. クエリの構造と役割（解答例）
このクエリは、**「注文ごとの合計金額を算出し、それを年次で集計した上で、最終的に年ごとの値を横並び（カラム）に変換する」** という、実務的なレポート作成の構成になっています。

特に、**「複雑な集計（SUM）を行ってから、さらに横持ち（PIVOT）に変える」** という流れを組みこんでいます。

#### クエリの3段階構造
このクエリは、大きく分けて以下の3つのステップでデータを加工しています。

##### ① 注文ごとの合計を計算（WITH句：`order_total`）
まず、商品明細テーブル（`order_items`）から、単価 × 数量を計算し、`order_id` ごとに合算しています。
* **ポイント**: ここで明細データを「注文単位」の一行に凝縮しています。

##### ② 年次集計への集約（中の SELECT文）
次に、計算した注文合計を注文親テーブル（`orders`）と結合し、注文日時（`order_tms`）から「年」を抽出して集計します。
* **結果のイメージ**:
  
  | YR   | TTL       | 
  | ---- | --------- | 
  | 2021 | 222438.11 | 
  | 2022 | 86160.42  | 

##### ③ 横並びへの変換（外側の PIVOT句）
最後に、縦に並んだ「年」の行を、横のカラムに変換します。
* **最終結果のイメージ**:
  
  | Y2021     | Y2022    | 
  | --------- | -------- | 
  | 222438.11 | 86160.42 |

#### ポイント
##### 1. 集計の二段構え
このクエリでは `SUM` を2回使っています。
1. `order_items` 内での「1注文あたりの合計」
2. PIVOT内での「1年あたりの合計」
PIVOTは必ず集計関数を必要とするため、このように**「先に集計した結果を、さらにPIVOTで集計（SUMのSUM）する」**形は、クロス集計の定番パターンです。

##### 2. 比較が圧倒的にしやすい
2021年と2022年が横に並ぶため、Excelのダッシュボードやレポートに出力した際、**「前年比の計算」や「並べての比較」が非常にスムーズ**になります。

##### 3. データの「削ぎ落とし」
PIVOTに渡す前のサブクエリ内で `yr` と `ttl` 以外の不要なカラムをすべて排除しています。これにより、PIVOTが「どの単位で行をまとめればいいか」を迷わず、1行に綺麗に収まる結果が得られます。

# 【完全版】問題13-3（行→列の入れ替え2） *Lv3*
### COスキーマ`ORDERS`テーブルの注文に対して、月単位の総額を算出してください。1注文に対する金額は`ORDER_ITEMS`テーブルの`UNIT_PRICE × QUANTITY`で計算してください。年単位に1列とするようにしてください。（月の昇順で表示）


## 期待する結果
| MT  | Y2021    | Y2022    | 
| --- | -------- | -------- | 
| 1   |          | 33323.64 | 
| 2   | 3256.22  | 24081.63 | 
| 3   | 11405.42 | 23269.17 | 
| 4   | 18317.39 | 5485.98  | 
| 5   | 18254.94 |          | 
| 6   | 20500.44 |          | 
| 7   | 19884.46 |          | 
| 8   | 23353.01 |          | 
| 9   | 26362.46 |          | 
| 10  | 25922.88 |          | 
| 11  | 25412.36 |          | 
| 12  | 29768.53 |          | 

## 解答例
```sql
SELECT
    *
FROM
    (
        WITH order_total AS (
            SELECT
                order_id,
                SUM(unit_price * quantity) AS total
            FROM
                co.order_items
            GROUP BY
                order_id
        )
        SELECT
            extract(YEAR FROM o.order_tms)  AS yr,
            extract(MONTH FROM o.order_tms) AS mt,
            SUM(ot.total)                   ttl
        FROM
            order_total ot
            LEFT JOIN co.orders   o ON ot.order_id = o.order_id
        GROUP BY
            extract(YEAR FROM o.order_tms),
            extract(MONTH FROM o.order_tms)
    ) PIVOT (
        SUM(ttl)
        FOR yr
        IN ( 2021 AS y2021, 2022 AS y2022 )
    )
ORDER BY
    mt
```

## 解説
このクエリは、**「月ごとの売上を、2021年と2022年で横並びにして比較する」** ための、非常に実用的で高度なレポート作成クエリです。
上記、問題13-2のクエリに「月（`mt`）」の要素が加わったことで、**「前年同月比」を視覚的に一瞬で把握できる形**になっています。


### 1. クエリの構造と役割（解答例）
このクエリは、データを3つのステップで整形しています。

#### ① 注文ごとの合計を算出（`WITH order_total`）
まず、明細レベルのデータを「1注文＝1行」の合計金額（`total`）にまとめます。

#### ② 年月ごとの集計（中の SELECT文）
ここで「年（`yr`）」と「月（`mt`）」を切り出し、年月単位の合計売上（`ttl`）を計算します。
* **結果のイメージ**:
  
  | YR   | MT  | TTL      | 
  | ---- | --- | -------- | 
  | 2021 | 2   | 3256.22  | 
  | 2021 | 3   | 11405.42 | 
  | 2021 | 4   | 18317.39 | 
  | 2021 | 5   | 18254.94 | 
  | ... | ... | ... |

#### ③ 月を軸にした年次比較（外側の PIVOT句）
ここがこのクエリのハイライトです。
* **`FOR yr IN (2021, 2022)`**: 「年」を横のカラムに展開します。
* **`mt`（月）の扱い**: PIVOT句に含まれていない `mt` は、自動的に **「行」を特定する軸** になります。
* **最終結果のイメージ**:
  
  | MT  | Y2021    | Y2022    | 
  | --- | -------- | -------- | 
  | 1   |          | 33323.64 | 
  | 2   | 3256.22  | 24081.63 | 
  | 3   | 11405.42 | 23269.17 | 
  | 4   | 18317.39 | 5485.98  | 
  | 5   | 18254.94 |          | 
  | ... | ... | ... |

### 2. ポイント
#### 1. 「前年同月比」が一目瞭然
月が縦に1〜12まで並び、その横に各年の数字が来るため、「去年の5月と今年の5月でどれくらい差があるか」という分析が非常にしやすくなります。

#### 2. PIVOTの「暗黙のグループ化」の活用
PIVOT内に `mt` を含めないことで、`mt` ごとに1行にまとめるよう指示しています。もしここで `mt` を含め忘れると、年ごとの合計が1行出るだけになってしまいます。この「残すべき軸（今回は月）」をサブクエリに含めておくのが、PIVOTを使いこなすコツです。

#### 3. データの欠損に強い
`LEFT JOIN` を使っているため、仮に特定の月に注文がなくても、カレンダー的な月データがあれば行として維持されます（ただし、このクエリでは `order_total` 起点なので、注文が全くない月はリストから消えます。完全に12ヶ月出したい場合は `Partitioned Outer Join` との組み合わせが最強です）。

## 解答例
```sql
SELECT
    *
FROM
    (
        SELECT
            education,
            household_size
        FROM
            sh.supplementary_demographics
    ) PIVOT (
        COUNT(*)
        FOR household_size
        IN ( '1' AS size_1, '2' AS size_2, '3' AS size_3, '4-8' AS size_4_to_8, '9+' AS size_9_plus )
    )
ORDER BY
    education;
```

## 解説
このクエリは、**「学歴（education）」と「世帯人数（household_size）」の相関関係を一目で把握するための「クロス集計表（マトリクス）」を作成する**ものです。
顧客属性データを分析して、「高学歴な人は世帯人数が多い傾向にあるか？」といった仮説を検証する際などに非常に役立つテンプレート的な書き方です。

### 1. クエリの構造と役割
このクエリは、縦に並んだ生データを「横に広げる」ことで、統計レポートのような形式に変換しています。

#### ① サブクエリ（データの絞り込み）
```sql
SELECT education, household_size FROM sh.supplementary_demographics
```
ここで、分析に**必要な2つのカラムだけ**に絞り込んでいます。PIVOTを使う際は、余計なカラム（例えば顧客IDなど）が混ざっていると、行が細かく分かれすぎて集計がうまくいかないため、この「事前の絞り込み」が非常に重要です。

#### ② PIVOT句（クロス集計の核）
* **`COUNT(*)`**: 各セルの値を「件数」で埋めるよう指示しています。
* **`FOR household_size IN (...)`**: 本来は縦に並んでいる「1」「2」「3」といった世帯人数の値を、横のカラム名（`size_1`, `size_2`...）に変換しています。
* **行の軸（education）**: PIVOT句に含まれていない `education` が、自動的に「縦の行」の軸として選ばれます。


### 2. ポイント

#### 1. カテゴリデータの可視化
「4-8人」や「9人以上」といった、元のデータにある**文字列のカテゴリ**をそのままカラム名として定義しています。これにより、単なる数値集計以上の「意味のあるレポート」になります。

#### 2. シンプルなマトリクス形式
最終的な出力結果は以下のようになります。
| EDUCATION | SIZE_1 | SIZE_2 | SIZE_3 | ... |
| :--- | :--- | :--- | :--- | :--- |
| Bachelors | 120 | 450 | 300 | ... |
| Masters | 45 | 200 | 150 | ... |

このように、**「教育レベル × 世帯規模」の分布**が一瞬でわかるのが最大のメリットです。

#### 3. データの「削ぎ落とし」の重要性
もしサブクエリを使わずに `SELECT * FROM sh.supplementary...` に対して PIVOT をかけると、他の属性（年収や居住地など）によって行がバラバラになり、クロス集計表が完成しません。**「軸にするもの以外はサブクエリで捨てる」** のがPIVOTの鉄則です。
