## 解答例
```sql:JSON_VALUEを使用
SELECT
    product_id,
    JSON_VALUE(product_details, '$.colour' RETURNING VARCHAR2(20))             AS COLOUR,
    JSON_VALUE(product_details, '$.gender' RETURNING VARCHAR2(20))             AS GENDER,
    JSON_VALUE(product_details, '$.brand' RETURNING VARCHAR2(20))              AS BRAND,
    JSON_VALUE(product_details, '$.reviews[0].rating' RETURNING NUMBER(2))  AS RATING,
    JSON_VALUE(product_details, '$.reviews[0].review' RETURNING VARCHAR2(1000)) AS REVIEW
FROM
    CO.PRODUCTS
WHERE
    JSON_VALUE(product_details, '$.gender' RETURNING VARCHAR2(100)) = q'[Girl's]'
```
```sql:ドット記法
SELECT
    p.product_id,
    p.product_details.colour,
    p.product_details.gender,
    p.product_details.brand,
    p.product_details.reviews[0].rating AS rating,
    p.product_details.reviews[0].review AS review
    
FROM
    co.products p
WHERE 
    p.product_details.gender = q'[Girl's]'
```

## 解説
### 1. JSON_VALUE 関数（例1）
`JSON_VALUE` は、JSONドキュメントの中から **「特定の値を1つ（スカラ値）」** 取り出し、SQLのデータ型（NUMBERやVARCHAR2など）に変換して返す関数です。

### 基本構文
```sql
SELECT JSON_VALUE(po_details, '$.Customer' RETURNING VARCHAR2(100)) 
FROM j_purchaseorder;
```

* **第1引数:** JSONデータが入っている列名。
* **第2引数:** パス式（`$` はルート、`.` は階層を辿る）。
* **RETURNING句:** 戻り値の型を指定できます（省略時はVARCHAR2(4000)）。


### 2. ドット記法（例2）
JavaScriptやPythonのように、テーブルの別名を使って直感的にJSONの階層を辿る書き方です。

### 基本構文
```sql
SELECT p.po_details.Customer
FROM j_purchaseorder p; -- 「p」という別名（エイリアス）が必須！
```

### 3. どちらを使うべき？ 徹底比較

| 比較項目 | JSON_VALUE 関数 | ドット記法 |
| :--- | :--- | :--- |
| **記述量** | 長くなる | **非常に短い** |
| **型指定** | `RETURNING` で明示的に指定可能 | **常に文字列**（またはJSON）として返る |
| **柔軟性** | エラー時の挙動などを細かく制御可能 | シンプルな取得のみ |
| **テーブル別名** | 不要（あっても良い） | **必須** |
| **主な用途** | 型を厳密にしたい計算やWHERE句 | データのサクッとした確認 |

### 4. 初心者が陥りやすい「3つの注意点」

#### ① ドット記法には「テーブル別名」が絶対に必要
SQLの標準的なカラムと区別するため、ドット記法を使うときは必ずテーブルに別名を付け、それを通す必要があります。
* ❌ `SELECT po_details.Customer FROM ...`
* ⭕ `SELECT t.po_details.Customer FROM j_purchaseorder t`

#### ② 戻り値のデータ型に注意
ドット記法で数値を取り出しても、SQL上は「文字列」として扱われることがあります。
* `WHERE p.json_col.age > 20` と書いた際、`age` が文字列として比較されると意図しない結果になる場合があります。数値を確実に数値として扱いたい場合は `JSON_VALUE(... RETURNING NUMBER)` を使うのが安全です。

### ③ 「単一の値」か「配列」か
`JSON_VALUE` はあくまで **1つの値** を取るためのものです。
JSONの中に `{"tags": ["SQL", "DB"]}` のような配列があり、その配列丸ごとや複数の値を取りたい場合は、`JSON_TABLE` を使う必要があります。


### 5. （参考）Oracle 23ai での進化：JSON型
23aiでは、内部的に最適化されたバイナリ形式の **JSON型** が導入されています。
* **高速:** 毎回パース（解析）しないため、検索が速い。
* **Boolean対応:** `true` / `false` をそのまま扱える。
* **柔軟:** `JSON_VALUE` を使わなくても、ドット記法でのパフォーマンスが大幅に向上しています。

```sql
-- 23aiなら、これだけでBooleanの判定もスッキリ！
SELECT * FROM employees_json
WHERE info.is_active = true;
```

# 問題17-2（階層構造を持つJSONデータの取り扱い） *Lv3*
**COスキーマの`PRODUCTS`テーブルにある`PRODUCT_DETAILS`はBLOB型でJSON形式のデータが格納されています。下記のJSONデータのキーに対する値を取得してください。`PRODUCT_ID`が23のデータを対象としてください。**
- **キー：colour**
- **キー：gender**
- **キー：brand**
- **キー：reviewsの全てのrating**
- **キー：reviewsの全てのreview**



## 期待する結果
| PRODUCT_ID | COLOUR | GENDER | BRAND    | RATING | REVIEW                                                                        | 
| ---------- | ------ | ------ | -------- | ------ | ----------------------------------------------------------------------------- | 
| 23         | blue   | Men's  | ADORNICA | 3      | Laboris elit pariatur labore duis fugiat ad in nulla tempor.                  | 
| 23         | blue   | Men's  | ADORNICA | 5      | Voluptate minim officia id excepteur.                                         | 
| 23         | blue   | Men's  | ADORNICA | 8      | Eiusmod cupidatat et nisi ipsum non aliqua.                                   | 
| 23         | blue   | Men's  | ADORNICA | 1      | Aute veniam irure dolor laborum esse ut ex tempor non velit.                  | 
| 23         | blue   | Men's  | ADORNICA | 2      | Nostrud nostrud mollit incididunt et tempor excepteur sit ut id exercitation. | 
| 23         | blue   | Men's  | ADORNICA | 6      | Labore tempor cillum laborum commodo et sit est qui aute enim id aute.        | 

## 解答例
```sql
SELECT
    product_id,
    colour,
    gender,
    brand,
    rating,
    review
FROM
    co.products,
    JSON_TABLE ( product_details
            COLUMNS (
                colour VARCHAR ( 20 ) PATH '$.colour',
                gender VARCHAR ( 20 ) PATH '$.gender',
                brand VARCHAR ( 20 ) PATH '$.brand',
                NESTED PATH '$.reviews[*]'
                    COLUMNS (
                        rating NUMBER ( 2 ) PATH '$.rating',
                        review VARCHAR2 ( 1000 ) PATH '$.review'
                    )
            )
        )
    AS jt
where product_id = 23
```

## 解説
`JSON_TABLE` は、JSON という「階層構造を持つデータ」を、SQL で扱いやすい「行と列のテーブル形式」に変換する非常に強力な関数です。
その中でも `NESTED PATH` は、JSON の中にある **「配列（Array）」をバラして、複数の行として出力したい時** に欠かせない機能です。

### 1. JSON_TABLE と NESTED PATH の視覚的イメージ

例えば、1人のユーザーが複数の「趣味（hobbies）」を持っているデータを想像してください。

) and on the right, it shows two rows in a table (Row 1: Alice, Reading; Row 2: Alice, Coding), highlighting how NESTED PATH flattens the array.]

このように、親のデータ（名前など）を保持したまま、配列の中身を **「縦に展開（フラット化）」** するのが `NESTED PATH` の役割です。

### 2. 基本的な書き方

Oracle や MySQL 8.0+ などで使われる標準的な書き方は以下の通りです。

```sql
SELECT *
FROM JSON_TABLE(
  '[{"name": "田中", "hobbies": ["読書", "登山"]}, {"name": "鈴木", "hobbies": ["映画"]}]',
  '$[*]' COLUMNS (
    user_name VARCHAR2(20) PATH '$.name',
    -- 配列部分を NESTED PATH で展開
    NESTED PATH '$.hobbies[*]' COLUMNS (
      hobby_name VARCHAR2(20) PATH '$'
    )
  )
) jt;
```

**実行結果イメージ:**
| user_name | hobby_name |
| :--- | :--- |
| 田中 | 読書 |
| 田中 | 登山 |
| 鈴木 | 映画 |


### 3. 注意点と陥りやすいポイント
`NESTED PATH` を使う際には、いくつか特有の挙動に注意が必要です。

#### ① 親データのリピート（直積のような動き）
上記の例のように、配列が展開されると、その配列の外側にあるデータ（`user_name`）は**同じ値が繰り返し表示**されます。集計関数（`SUM` や `COUNT`）をそのまま使うと、重複して計算してしまう可能性があるため注意してください。

#### ② NULL の扱い（ON EMPTY / ON ERROR）
JSON のパスが存在しない場合や、データが空の場合の挙動を定義しておくのが安全です。
* `NULL ON EMPTY`: データがない場合は NULL を入れる（デフォルト）。
* `ERROR ON ERROR`: 変換に失敗したらクエリごとエラーにする。
これを明示しないと、なぜか特定の行が表示されないといったトラブル時に原因究明が難しくなります。

#### ③ 兄弟関係にある複数の配列
1つの `JSON_TABLE` 内に、並列する（ネストしていない）複数の `NESTED PATH` を書くと、データベース製品によっては **「デカルト積（すべての組み合わせ）」** が発生し、意図しない膨大な行数が出力されることがあります。
> **対策:** 複数の配列をバラしたい場合は、1つずつ `JSON_TABLE` を当てるか、データの構造をよく確認してください。

#### ④ データ型の定義
`COLUMNS` 句で指定するデータ型（`VARCHAR2`, `NUMBER` など）は、SQL 側の型です。JSON 側のデータがその型に収まらない（例：長い文字列が `VARCHAR2(10)` に入らない）場合、デフォルトでは **NULL に化けるかエラーになる** ため、サイズ設計は余裕を持って行いましょう。
