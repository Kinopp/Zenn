## 概要
**Bridge（ブリッジ）パターン**は、**「機能の追加」と「実装の追加」を切り離して、それぞれを独立して拡張できるようにする**ためのパターンです。
「クラスの継承が爆発して収拾がつかない！」という事態を救うための架け橋（Bridge）となります。
通常、機能を追加しようとすると「継承」を使いがちですが、そうすると「機能 × 実装」の分だけクラスが増えてしまいます。Bridgeパターンでは、これらを **「抽象（何をするか）」と「実装（どうやるか）」の2つの階層に分離** し、それらを橋渡し（委譲）でつなぎます。

## イメージ
【効果】
継承には「機能」の拡張と抽象メソッドの「実装」という２つのパターンがある。その２つのパターンの階層を明確に分けることで変更に対して互いに影響を与えないような実装を行える。
![](https://storage.googleapis.com/zenn-user-upload/68e272980fc4-20260406.png =580x)



## メリット
### 1. クラスの爆発を防げる
例えば「丸、四角」という図形があり、「赤、青」で塗りたい場合、継承だと「赤い丸」「青い丸」「赤い四角」「青い四角」と4つのクラスが必要です。ここに「三角」を追加するとさらに増えます。
Bridgeなら「図形（機能）」と「色（実装）」を分けるので、**足し算の数（2+2）だけクラスを作れば済む**ようになります。

### 2. 修正が他に影響しない
「実装」の部分を書き換えても、それを利用する「抽象」側のコード（クライアント側）を修正したり再コンパイルしたりする必要がありません。**「修正の波及」を最小限に抑えられます。**

### 3. 実行時に「実装」を切り替えられる
継承はコンパイル時に構造が決まってしまいますが、Bridgeは「委譲（持っているものを使う）」なので、プログラムの実行中に「今はWindows用の実装を使う」「次はMac用」といった具合に、中身を動的に差し替えることが可能です。


## デメリット
### 1. 設計が複雑になり、コード量が増える
単純な継承で済むようなケースでも、わざわざインターフェースを定義して橋渡しを作るため、初動のコード量が増えます。「やりすぎ（オーバーエンジニアリング）」になりやすいパターンでもあります。

### 2. 直感的に理解しにくい
「機能の階層」と「実装の階層」という2つのピラミッドが並び立つ構造は、初心者にとって「今どっちのクラスをいじっているのか？」という混乱を招きやすいです。


## 適用パターン

| 状況 | Bridgeを使うべき？ |
| :--- | :--- |
| **機能も実装も、将来的にどんどん増えそう** | **最適** |
| **OSやプラットフォーム依存の処理を隠したい** | **最適** |
| **一度作ったら構造が変わることはない** | **継承で十分（不向き）** |

> **例え話で理解：リモコンとテレビ**
> 
> **リモコン（抽象）** には「電源ボタン」や「音量ボタン」という「機能」があります。
> 一方、**テレビ本体（実装）** には「ソニー製」「パナソニック製」などの「具体的な仕組み」があります。
> リモコンというインターフェースを共通化しておけば、中身のテレビがどこのメーカーに変わっても、私たちは「ボタンを押す」という操作（機能）を変えずに済みます。

## サンプルソース


```Java:Abstraction.java
public class Abstraction {
    private final Implementor implementor;

    public Abstraction(Implementor implementor) {
        this.implementor = implementor;
    }
    
    public void method1(){
        implementor.implMethodA();
    }
    public void method2(){
        implementor.implMethodB();
    } 
}
```
```Java:Implementor.java
public interface Implementor {
    public void implMethodA();
    public void implMethodB();
}
```
```Java:ConcreteImplementor.java
public class ConcreteImplementor implements Implementor{
    @Override
    public void implMethodA() {
        System.out.println("実装メソッドAです");
    }

    @Override
    public void implMethodB() {
        System.out.println("実装メソッドBです");
    }
}
```
```Java:RefinedAbstraction.java
public class RefinedAbstraction extends Abstraction{
    public RefinedAbstraction(Implementor implementor) {
        super(implementor);
    }

    public void refineMethodA() {
        method1();
        System.out.println("機能メソッドAです");
    }

    public void refineMethodB() {
        method2();
        System.out.println("機能メソッドBです");
    }
}
```
```Java:Client.java
public class Client {
    public static void main(String... args){
        RefinedAbstraction abstraction = new RefinedAbstraction(new ConcreteImplementor());
        abstraction.refineMethodA();
        abstraction.refineMethodB();
    }
}
```


### 実行結果
```
実装メソッドAです
機能メソッドAです
実装メソッドBです
機能メソッドBです
```
