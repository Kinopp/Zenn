## 概要
**Visitor（ビジター）パターン**は、**「データ構造」と「処理」を切り離す** ためのパターンです。
「モノ（データ）」の中に「やりたいこと（処理）」を書き込むのではなく、外部から「訪問者（Visitor）」がやってきて処理を代行するイメージです。

Visitorパターンを理解するには、**「健康診断」** を想像するのが分かりやすいです。

* **データ構造（Element）：** 教室で待っている「生徒」たち。
* **処理（Visitor）：** 学校にやってくる「お医者さん」。

生徒（データ）は自分の体（構造）を持っていますが、自分で診察（処理）はしません。
1.  まず、**内科医（Visitor A）** が教室を回り、全員を診察します。
2.  次に、**歯科医（Visitor B）** がやってきて、同じ生徒たちを診察します。

ここで重要なのは、**「新しい検査（眼科検診など）が増えても、生徒（データ構造）は自分自身を改造する必要がない」** ということです。ただ、新しいお医者さんを受け入れるだけで済みます。

### 【解説】
### 1. Element（左側のピンク枠）：データ構造自体
**Compositeパターン** の木構造（LeafとComposite）が登場しています。
* **`accept(Visitor)`：** これが「玄関」です。訪問者（Visitor）を中に入れるためのメソッドです。
* **役割：** 「私はLeafです、どうぞ」「私はCompositeです、どうぞ」と、自分が何者であるかを訪問者に伝えるだけで、具体的な計算などは自分では行いません。

### 2. Visitor（右側のピンク枠）：データ構造にさせたい処理
ここが「専門家（点検員）」の集まりです。
* **`visit(Leaf)` / `visit(Composite)`：** 対象がLeafだった場合、Compositeだった場合の具体的な処理（名前を取る、サイズを計算するなど）をここに書きます。
* **役割：** データ構造の「外側」に処理を追い出すことで、**データ構造を汚さずに新しい機能（例：Elementの集計、書き出しなど）を追加**できます。

### 3. 連係の流れ（オレンジの注釈）
1. Clientが「サイズの計算をしたい！」と思い、**サイズ取得Visitor**を作ります。
2. そのVisitorをElementの `accept()` に投げ込みます。
3. Element側で `visitor.visit(this)` を実行し、自分自身をVisitorに差し出します。
4. Visitorがその中身を見て、ガリガリと計算を行います。

### まとめ
この図のキモは、**「データ構造（Element）」と「アルゴリズム（Visitor）」の依存関係をひっくり返している**点にあります。
画像の下の方にある「オーバーロード」という言葉の通り、訪問者が相手を見て「あ、君はLeafだね。じゃあこの処理をするよ」と判断することで、データ構造側に `if-else` 文を書かなくて済むようになります。


## メリット

### 1. 新しい「処理」の追加が簡単
データ構造側のクラス（Element）を一切修正することなく、新しいVisitorクラスを作るだけで機能を追加できます。
* **例：** 「ファイル一覧」というデータ構造に対して、「検索機能」の次に「容量計算機能」を付けたくなったら、新しいVisitorを作るだけで済みます。既存の安定したコードを触らずに機能拡張できる（**開放閉鎖の原則**）のが最大の強みです。

### 2. 関連するロジックを1箇所に集約できる
バラバラのクラスに散らばっていた「似たような処理」を、一つのVisitorクラスにまとめられます。これにより、「この機能のロジックはここにある」という見通しが良くなり、コードの保守性が上がります。

### 3. 状態を保持しながら巡回できる
Visitorは各要素を順番に回る際、自分の中に「計算の途中経過」などを保持できます。例えば、フォルダ構造を回りながら「ファイルサイズの合計」をカウントするといった処理がスマートに記述できます。



## デメリット

### 1. 新しい「データ（要素）」の追加に極めて弱い
Visitorパターンの最大の弱点です。データ側のクラス（Element）の種類が1つ増えると、**すべてのVisitorクラスにその新要素用のメソッドを追加**しなければなりません。
* **例：** 「ファイル」「フォルダ」を扱うシステムに「ショートカット」という要素を追加すると、検索Visitor、容量計算Visitor、削除Visitor……すべてに修正が必要になります。

### 2. カプセル化を壊しやすい
Visitorが外から処理を行うため、本来は隠しておきたいデータクラスの内部情報を、Visitorがアクセスできるように公開（getterの作成など）せざるを得ない場合があります。

### 3. 構造が直感的ではない（ダブルディスパッチ）
「データがVisitorを受け入れ(accept)、Visitorがデータを処理する(visit)」という、キャッチボールのような **二重呼び出し（ダブルディスパッチ）** という仕組みを使います。慣れていないと、処理の流れを追うのが少し難しく感じることがあります。


## 活用パターン

| 項目 | 向いているケース | 向いていないケース |
| :--- | :--- | :--- |
| **データ構造** | 種類がほぼ固定されている | 頻繁に新しい種類が増える |
| **処理内容** | どんどん新しい機能を追加したい | 処理の内容が決まりきっている |

**「データ構造はガッチリ固まっているけれど、やりたいことは後からどんどん増えそうだな」** という場面でこそ、Visitorパターンはその真価を発揮します。



## サンプルソース

```Java:Element.java
public interface Element {
    public void accept(Visitor visitor);
}
```
```Java:Component.java
import java.util.Iterator;

public abstract class Component implements Element{
    //Leafはadd出来ない為、デフォルトは例外とする。Nodeはaddメソッドをオーバーライドする
    public void add(Component component) throws Exception{
        throw new Exception();
    }
    
    public Iterator iterator() throws Exception {
        throw new Exception();
    }
    
    public abstract String getName();
}
```
```Java:Leaf.java
public class Leaf extends Component{

    private final String name;
    
    public Leaf(String name) {
        this.name = name;
    }

    @Override
    public String getName() {
        return name;
    }

    @Override
    public void accept(Visitor visitor) {
        visitor.visit(this);
    }
}
```
```Java:Composite.java
import java.util.ArrayList;
import java.util.Iterator;
import java.util.List;

public class Composite extends Component{

    private final String name;
    private final List<Component> components = new ArrayList<>();

    public Composite(String name) {
        this.name = name;
    }
    
    @Override
    public Iterator iterator() {
        return components.iterator();
    }
    
    @Override
    public void add(Component component){
        components.add(component);
    }
    
    @Override
    public String getName() {
        return name;
    }

    @Override
    public void accept(Visitor visitor) {
        visitor.visit(this);
    }
}
```
```Java:Visitor.java
public interface Visitor {
    public void visit(Leaf leaf);
    public void visit(Composite composite);
}
```
```Java:ListVisitor.java
import java.util.Iterator;

public class ListVisitor implements Visitor{

    @Override
    public void visit(Leaf leaf) {
        System.out.println(leaf.getName());
    }

    @Override
    public void visit(Composite composite) {
        System.out.println(composite.getName());
        // 配下のComponent及びLeafを全て表示
        Iterator<Component> iterator = composite.iterator();
        while (iterator.hasNext()){
            Component comp = iterator.next();
            comp.accept(this);
        }
    }
}
```

```Java:Client.java
public class Client {
    public static void main(String... args) throws Exception{
        //rootディレクトリ作成
        Component rootdir = new Composite("rootdir");
        //tmpディレクトリ作成し、file1,file2を作成
        Component tmpdir = new Composite("tmpdir");
        Leaf file1 = new Leaf("file1");
        Leaf file2 = new Leaf("file2");
        tmpdir.add(file1);
        tmpdir.add(file2);
        //rootディレクトリの直下にtmpディレクトリを置く
        rootdir.add(tmpdir);
        
        //usrディレクトリ作成し、file3,file4を作成
        Component usrdir = new Composite("usrdir");
        Leaf file3 = new Leaf("file3");
        Leaf file4 = new Leaf("file4");
        usrdir.add(file3);
        usrdir.add(file4);
        //rootディレクトリの直下にusrディレクトリを置く
        rootdir.add(usrdir);
        
        //ListVisitorを渡してファイル・ディレクトリの一覧を表示
        rootdir.accept(new ListVisitor());
    }
}

```
