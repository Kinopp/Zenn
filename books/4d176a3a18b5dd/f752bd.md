## 概要
**Composite（コンポジット）パターン**は、**「『単品』と『セット（詰め合わせ）』を、区別せずに同じように扱う」** ためのパターンです。ファイルシステム（ファイルとフォルダの関係）をイメージすると非常に分かりやすいです。

最も身近な例は、パソコンの **「ファイルシステム」** です。

* **ファイル（単体）：** 中身があるデータ。それ以上中には何も入れられません。
* **フォルダ（複合体）：** ファイルや、さらに別のフォルダを入れることができます。

ここで重要なのは、**「サイズを表示する」** という操作です。
あなたが「サイズを知りたい」と思ったとき、相手がファイルであってもフォルダであっても、右クリックして「プロパティ」を見ればサイズがわかりますよね。フォルダの場合は、中に入っているすべてのファイルとサブフォルダの合計を自動で計算してくれます。

このように、**「中身が1つなのか、たくさん入っている箱なのかを意識せずに、同じ命令（メソッド）を送れる」** ようにするのがCompositeパターンの本質です。

### 【解説】
### 1. Componentクラス（共通の顔）
画像内のオレンジの枠「Componentクラス」に注目してください。
これが、ファイルとフォルダの **「共通の親（インターフェース）」** になります。
画像左下のメモにある通り、**「ファイルなのかディレクトリ（フォルダ）なのかの差異を意識せず」** に済むのは、このComponentという共通の型があるからです。

### 2. 再帰呼び出し（マトリョーシカの連鎖）
中央と右側の赤い矢印「再帰呼び出し」がこのパターンの核心です。
* フォルダ（Component）の中に、また別のフォルダ（Component）が入る。
* その中にもまたフォルダが入る…。
これを繰り返すことで、**どんなに深い階層構造になっても、一番上のフォルダを操作するだけで、中身すべてに命令を伝えることができる**ようになります。

### 3. Client（利用側）のメリット
Clientは、目の前のオブジェクトが「ただのファイル」なのか「100個のファイルが入った巨大なフォルダ」なのかを判別する `if` 文を書く必要がありません。
ただ「Componentさん、処理をお願いします」と一言頼むだけで済むのです。

### まとめ
この図のキモは、**「入れ子構造をフラットに捉える」**　という考え方です。
画像の中で「同一視」という言葉が強調されていますが、これは　**「1個のもの」と「1個以上の集まり」を同じように扱える魔法**のようなもの。これによって、ディレクトリをまるごとコピーするような処理が、非常に短いコードで書けるようになります。



## メリット
### 1. 「単品」と「セット」を全く同じコードで操作できる
使う側（クライアント）は、相手が「1つのデータ」なのか「それらを束ねたグループ」なのかを気にする必要がありません。
* **例:** フォルダの中にある「全てのファイルの合計サイズ」を知りたいとき、中身がファイルでもサブフォルダでも、ただ「サイズを教えて」と命令（再帰的な呼び出し）を出すだけで完結します。

### 2. 木構造（ツリー構造）を自然に表現できる
階層構造をそのままクラス構成に落とし込めるため、ファイルシステム、組織図、XML/HTMLの構造、GUIのウィンドウとボタンの関係などを非常に直感的に実装できます。

### 3. 新しい種類の部品を簡単に追加できる
新しい「単品（Leaf）」や、新しいルールの「セット（Composite）」を追加しても、それを利用する側のコードを修正する必要がありません（**開放閉鎖の原則**）。



## デメリット
### 1. 設計を「汎用的」にしすぎるリスクがある
「単品」と「セット」を同じインターフェースで扱うため、本来は「セット」にしかできない操作（「子要素を追加する」など）を、「単品」側にも持たせなければならないことがあります。
* **例:** ファイル（単品）に対して「中身に別のファイルを詰め込む」という、意味のないメソッドが定義されてしまうことがあります。

### 2. 特定の部品だけを制限するのが難しい
「このフォルダには画像ファイルしか入れられないようにしたい」といった細かい制限をかけようとすると、実行時に型をチェックするコードが必要になり、パターンのメリットである「一貫性」が薄れてしまいます。

### 3. 再帰によるデバッグの難しさ
構造が非常に深くなると、処理が何層にもわたって再帰的に呼び出されるため、エラーが起きたときに「今、どの階層のどの部品で止まったのか」を追いかけるのが大変になることがあります。

## 適用パターン
| 項目 | 向いているケース |
| :--- | :--- |
| **データの形** | 階層構造（親子関係）がある |
| **操作感** | 全体をまとめて処理したり、個別に処理したりを混ぜて行いたい |
| **複雑さ** | 再帰的なデータ構造をシンプルに管理したい |

> **例え話で理解：お弁当箱**
> 
> お弁当（Composite）の中には、「唐揚げ（Leaf）」もあれば、「小さなおかず入れ（別のComposite）」が入っていることもあります。
> 食べる人は、「唐揚げ」を食べるのも「おかず入れの中身」を食べるのも、同じ「食べる」というアクションだけで済みます。中身が単品かセットかをいちいち意識しなくていいのが、このパターンの心地よさです。

### ちょっと高度なポイント：「透明性」vs「安全性」
Compositeパターンには2つの派生があります。
* **透明性重視:** 全てのメソッドを共通インターフェースに置く（使い勝手は最高だが、単品に不要なメソッドができる）。
* **安全性重視:** 「追加・削除」などのメソッドはセット専用クラスにだけ置く（単品に無駄なメソッドはないが、使う側で「これはセットか？」の確認が必要になる）。

実際の開発では、どちらのメリットを優先するかで設計が分かれます。


## サンプルソース
```Java:Component.java
public abstract class Component {
    //Leafはadd出来ない為、デフォルトは例外とする。Nodeはaddメソッドをオーバーライドする
    public void add(Component composite) throws Exception{
        throw new Exception();
    }
    public abstract String getName();
    public abstract void printList();
}
```
```Java:Composite.java
import java.util.ArrayList;
import java.util.List;

public class Composite extends Component{
    private final String name;
    private final List<Component> components = new ArrayList<>();

    public Composite(String name) {
        this.name = name;
    }
    
    @Override
    public void add(Component component){
        components.add(component);
    }
    
    @Override
    public String getName() {
        return name;
    }

    @Override
    public void printList() {
        System.out.println(this.getName());
        // 配下のComponent及びLeafを全て表示
        for(Component component: components){
            component.printList();
        }
    }
}
```
```Java:Leaf.java
public class Leaf extends Component{
    private final String name;
    
    public Leaf(String name) {
        this.name = name;
    }

    @Override
    public String getName() {
        return name;
    }

    @Override
    public void printList() {
        System.out.println(this.getName());
    }
}
```
```Java:Client.java
public class Client {
    public static void main(String... args){
        //rootディレクトリ作成
        Composite rootdir = new Composite("rootdir");
        //tmpディレクトリ作成し、file1,file2を作成
        Composite tmpdir = new Composite("tmpdir");
        Leaf file1 = new Leaf("file1");
        Leaf file2 = new Leaf("file2");
        tmpdir.add(file1);
        tmpdir.add(file2);
        //rootディレクトリの直下にtmpディレクトリを置く
        rootdir.add(tmpdir);
        
        //usrディレクトリ作成し、file3,file4を作成
        Composite usrdir = new Composite("usrdir");
        Leaf file3 = new Leaf("file3");
        Leaf file4 = new Leaf("file4");
        usrdir.add(file3);
        usrdir.add(file4);
        //rootディレクトリの直下にusrディレクトリを置く
        rootdir.add(usrdir);
        
        //rootディレクトリの配下のNode,Leafを全て表示
        rootdir.printList();
    }
}
```
