## 概要
**Decorator（デコレーター）パターン**は、**「中身を書き換えずに、外側に『飾り』をつけて機能をどんどん追加する」** パターンです。マトリョーシカのように、オブジェクトを別のオブジェクトで包んでいくイメージです。

もっとも分かりやすい例えは、アイスクリームの注文です。

1.  **ベース（本体）：** バニラアイス
2.  **トッピングA（飾り）：** チョコソースをかける
3.  **トッピングB（飾り）：** ナッツをまぶす

このとき、トッピングAもトッピングBも、**「アイスクリームの一種」として振る舞う**のがポイントです。チョコをかけた後でも、それは依然として「食べられるアイス」ですよね。このように、中身を別のオブジェクトで包んでも、外側からの見え方（インターフェース）を変えずに機能だけを盛っていくのがDecoratorパターンの真髄です。

### 【解説】
### 1. 継承（is-a）による拡張（画像上段：×）
「文字列」を継承して「色付き」を作り、さらにそれを継承して「枠付き」を作る……という流れです。
* **問題点:** 「色はいらないけど枠だけ欲しい」と思った時、また別の継承ルートを作らなければなりません。
* **クラスの爆発:** 機能が増えるたびに、組み合わせの数だけクラスが増えてしまい、管理不能になります（これを「静的で拡張性が乏しい」と表現しています）。

### 2. 委譲（has-a）による拡張（画像下段：○）
ここがDecoratorパターンの独壇場です。
* **Componentクラス:** 文字列、色付け、枠囲み、すべてを「同じ仲間（型）」として扱います。
* **再帰呼び出し:** 「枠囲みクラス」が内部に「色付けクラス」を抱え、さらにその中が「文字列」…という具合に、**実行時に動的に包んでいきます。**
* **柔軟性:** 「枠だけ」「色だけ」「色＋枠＋影」など、**組み合わせが自由自在**になります。

### まとめ
この図のキモは、**「is-a（継承）から has-a（構成・委譲）へのパラダイムシフト」** にあります。
画像の中で「Componentクラスとして同一視する」という言葉がありますが、これこそがミソです。外側から包んでいる「飾り」も、中身と同じインターフェースを持っているからこそ、Clientは何も気にせず「HELLO」を表示（メソッド呼び出し）できるのです。


## メリット
### 1. 「継承」を使わずに機能を拡張できる
通常、機能を追加しようとすると「継承」を使いがちですが、継承はコンパイル時に構造が固まってしまいます。Decoratorなら、**実行時に（プログラムが動いている最中に）**「今はこれとこれを追加しよう」と動的に機能を盛り付けることができます。

### 2. クラスの爆発を防げる
例えば「トッピング付きのコーヒー」を作る場合、継承だと「ミルクコーヒー」「砂糖コーヒー」「ミルク砂糖コーヒー」「ホイップコーヒー」……と、組み合わせの数だけクラスが必要になります。
Decoratorなら、**「ミルク」「砂糖」「ホイップ」という3つの飾りクラス**を作るだけで、どんな組み合わせも自由自在に作れます。

### 3. 単一責任の原則を守りやすい
「基本の処理」と「追加の機能（ログ出力、キャッシュ、暗号化など）」を別々のクラスに分けられます。それぞれのクラスが1つのことだけに集中できるため、コードがスッキリします。



## デメリット
### 1. 小さなオブジェクトが大量に生成される
「飾り」を増やすたびに、それを包む新しいインスタンスが作られます。似たような小さなオブジェクトがメモリ上にたくさん並ぶことになり、慣れていない人がデバッグ（中身を確認）しようとすると「今どの層にいるんだ？」と混乱しがちです。

### 2. 構造の特定が難しい
外側から見ると、最終的に何層包まれているのか、どの順番で包まれているのかが分かりにくいです。また、一番内側の「中身（ConcreteComponent）」を直接操作したいときに、厚い皮に包まれていてアクセスしにくいという問題も起こります。

### 3. 「包む順番」に依存することがある
「暗号化した後に圧縮する」のと「圧縮した後に暗号化する」のでは結果が変わります。Decoratorを組み合わせる際、その順番を正しく管理する必要があり、ミスをするとバグの原因になります。

## 適用パターン
| 項目 | 向いているケース |
| :--- | :--- |
| **拡張のタイミング** | 実行時にコロコロと機能を変えたいとき |
| **組み合わせの数** | 機能の組み合わせが膨大になるとき |
| **既存コードの保護** | 元のクラスを一切いじらずに機能を追加したいとき |

> **例え話で理解：アイスクリームのトッピング**
> 
> ベースの「バニラアイス（基本クラス）」があります。
> そこに「チョコ（飾り1）」をかけ、「ナッツ（飾り2）」をまぶします。
> 食べる人は「バニラアイス」を食べているつもりですが、実際には「ナッツで包まれた、チョコで包まれた、バニラアイス」を食べています。この **「トッピングを自由に変えられる柔軟性」** こそがDecoratorの魅力です。もし、この柔軟性がなければ、最初から「チョコかけナッツまぶしのバニラアイスXX円」というように、あらかじめ設定しなければなりません。



## サンプルソース

```Java:Component.java
public abstract class Component {
    public abstract void display();
}
```
```Java:Decorator.java
public abstract class Decorator extends Component{
    protected Component component;

    public Decorator(Component component) {
        this.component = component;
    }  
}
```
```Java:ConcreteComponent.java
public class ConcreteComponent extends Component{
    private final String string;
    
    public ConcreteComponent(String string) {
        this.string = string;
    }
    
    @Override
    public void display() {
        System.out.println(string);
    }
}
```
```Java:ConcreteDecorator1.java
public class ConcreteDecorator1 extends Decorator{
    public ConcreteDecorator1(Component component) {
        super(component);
    }

    @Override
    public void display() {
        System.out.println("---------------");
        super.component.display();
        System.out.println("---------------");
    }
}
```
```Java:ConcreteDecorator2.java
public class ConcreteDecorator2 extends Decorator{
    public ConcreteDecorator2(Component component) {
        super(component);
    }

    @Override
    public void display() {
        System.out.println("***************");
        super.component.display();
        System.out.println("***************");
    }
}
```
```Java:Client.java
public class Client {
    public static void main(String... args){
        Component component1 = new ConcreteComponent("Hello Java");
        Component component2 = new ConcreteDecorator1(component1);
        Component component3 = new ConcreteDecorator2(component2);
        component3.display();
    }
}
```
