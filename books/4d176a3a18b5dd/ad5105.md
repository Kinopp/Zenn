## 概要
**Template Method（テンプレートメソッド）パターン**を一言でいうと、**「全体の流れ（手順）は決まっているけれど、中身の詳細は後で決める」** という仕組みです。

料理のレシピに例えると、
「カレーを作る」というレシピ（テンプレート）には、「具材を切る」「炒める」「煮込む」「味付けをする」という**手順**が決まっています。でも、「何の具材を入れるか」や「ルーは甘口か辛口か」という**詳細**は、作る人（サブクラス）に任されています。

### 【解説】
### 1. スーパークラス（司令塔）
左側の薄い黄色のボックスです。ここには `templateMethod()` という「指揮者（オーケストラのイラスト）」がいます。
* **templateMethod():** 処理の全体的な「流れ」をコントロールします。
* **空のメソッド（A, B, C）:** 枠組みだけ用意されていて、中身は空っぽです。「ここは何かしらの処理が必要だよ」という予約席のようなものです。

### 2. サブクラス（現場作業員）
右側の薄い緑のボックスです。ここでは実際に「作業員」が動いています。
* **継承:** スーパークラスの「流れ」を引き継ぎます。
* **具体化:** 親が空にしていた「A, B, C」という枠の中に、自分たちの具体的な処理（図の中の丸や三角の形）を詰め込みます。

### 3. Client（利用側）
一番上の青いボックスです。
利用者は、サブクラスAかB、どちらを使うか選んで呼び出すだけです。**「どういう順番で処理が動くか」を意識する必要はありません。** 呼び出せば、親が決めた通りの順番で、子が用意した中身が実行されます。

### まとめ
このパターンの真髄は、左側のピンクの箱 `templateMethod()` の中に **「複雑な共通処理」を閉じ込めている** 点にあります。
サブクラスは、自分たちにしかできない「ユニークな部分（A, B, C）」だけに集中すればいい。まさに **「役割分担の黄金比」** を形にしたのが、このTemplate Methodパターンです。


## メリット

### 1. コードの重複をなくせる（再利用性）
「煮込む」「炒める」といった共通の処理を親クラス（スーパークラス）に一度だけ書けば、どんなカレーを作るときもそのコードを使い回せます。同じことを何度も書かなくて済むので、修正も一箇所で済みます。

### 2. アルゴリズムの「骨組み」を守れる
全体の流れ（手順）は親クラスでがっちりガードされています。サブクラスが勝手に「煮込む前に味付けをする」といった手順の入れ替えをできないように制限できるため、バグが入りにくくなります。

### 3. 「穴埋め」だけで機能が作れる
サブクラスを作る人は、親クラスが用意した「空欄（抽象メソッド）」を埋めるだけで新しい機能を追加できます。全体の複雑な処理を知らなくても、自分の担当部分だけを実装すればいいので楽ちんです。



## デメリット

### 1. クラスの数が増えて管理が大変
ちょっとした違いを作るだけでも、そのたびに新しいサブクラスを作る必要があります。「甘口カレー」「辛口カレー」「シーフードカレー」……と、クラスがどんどん増えて全体像が把握しにくくなることがあります。

### 2. 継承のルールに縛られる
Template Methodは「継承（extends）」を使います。Javaなどの言語では、クラスは1つしか継承できないというルールがあるため、このパターンを使うと「他のクラスを継承したくてもできない！」という不自由さが生まれることがあります。

### 3. 親クラスと子クラスが「ズブズブ」になる
親クラスの「手順」が少し変わると、すべての子クラスに影響が出ます。親と子の関係が密接（**密結合**）になりすぎるため、大きなシステムの修正では「一箇所変えたら全部壊れた！」というリスクが伴います。


## 適用パターン

| おすすめの用途 | 避けるべきケース |
| :--- | :--- |
| **複数の処理で「流れ」がほぼ同じとき** | 処理の手順がバラバラで統一感がないとき |
| **フレームワークやライブラリを作るとき** | 継承を使わずに「部品（委譲）」で解決したいとき |
| **将来、特定の手順だけ変えた版が増えそうなとき** | シンプルな処理で、クラスを分けるまでもないとき |


## サンプルソース
```Java:AbstractClass.java
public abstract class AbstractClass {
    
    // 抽象メソッドを利用して処理内容を記載
    public void templateMethod(){
        System.out.println("----- start -----");
        method1();   // 具体的な処理はわからないが、method1を行う
        for (int i=0; i<5; i++){
            method2();   // 具体的な処理はわからないが、method2を行う
        }
        method3();   // 具体的な処理はわからないが、method3を行う
        System.out.println("-----  end  -----");
    }
    
    // 具体的な処理はサブクラスに記載
    protected abstract void method1();
    protected abstract void method2();
    protected abstract void method3();
    
}
```
```Java:ConcreteClass.java
public class ConcreteClass extends AbstractClass{
    @Override
    protected void method1() {
        System.out.println("method1です");
    }
    @Override
    protected void method2() {
        System.out.println("method2です");
    }
    @Override
    protected void method3() {
        System.out.println("method3です");
    }
}
```

```java:Client.java
public class Client {
    public static void main(String... args){
        AbstractClass clazz = new ConcreteClass();
        clazz.templateMethod();
    }
}
```
