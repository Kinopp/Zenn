## 概要
**Proxy（プロキシ）パターン**を一言でいうと、**「本物のオブジェクトの前に『代理人』を立てて、アクセスをコントロールする」** パターンです。「プロキシ」は日本語で「代理」を意味します。

わかりやすい例えは、**「忙しい社長と、そのスケジュールを管理する秘書」** の関係です。

* **あなた（クライアント）：** 社長に会ってハンコをもらいたい。
* **社長（本物のオブジェクト）：** 実際にハンコを押す権限を持っているが、とても忙しい。
* **秘書（代理人/プロキシ）：** 社長への窓口。

あなたが「社長に会いたい」と言ったとき、秘書は以下のような動きをします。
1.  **アクセス制限：** 「アポイントはありますか？」と確認する。
2.  **キャッシュ：** 「その件なら、以前社長が『OK』と言っていたので私が判を押しておきます」と代行する。
3.  **遅延実行：** 「今は不在なので、戻ってきたら伝えておきます」と調整する。

このように、**「本物を直接触らせず、代理人が状況に応じて適切に振る舞う」** のがProxyパターンです。

### 【解説】
### 1. 同一のインターフェース（Subject）
画像右上の青いボックスに「本人、代理人ともに同一のインターフェースを持つ」とありますね。
これが非常に重要です。Client（利用側）からすれば、相手が「本人」か「代理人」かを見分ける必要がなく、**同じ操作（メソッド呼び出し）で扱える**ようになっています。

### 2. 代理人が「自分で処理する」場合（メソッド1, 2）
画像の中央、Proxyの「メソッド1」「メソッド2」は自分で「処理」を行っています。
「本人を出すまでもない簡単な用件」や「すでに知っている情報（キャッシュ）」であれば、代理人がその場でパッと返事をしてしまいます。これにより、システム全体のスピードが上がります。

### 3. 代理人が「本人に頼む」場合（メソッド3）
「メソッド3」の赤い矢印に注目してください。
これは代理人では手に負えない、重たい処理や重要な判断です。この時初めて、代理人は**本人（RealSubject）をインスタンス化（生成）** したり、処理を丸投げしたりします。

### まとめ
この図のキモは、**「Clientは本人の存在を意識しなくていい」** という点と、**「代理人が賢く立ち回ることで、本人の負担を減らしている」** という点です。
プログラミングの世界では、ネットワーク越しにデータを取ってくる際（リモートプロキシ）や、メモリを節約したい時によく使われる、非常にスマートなパターンなんですよ。


## メリット
### 1. 「重い」処理をギリギリまで遅らせられる（仮想プロキシ）
画像データや巨大なファイルの読み込みなど、作成に時間がかかるオブジェクトがある場合、最初は軽い「代理人」だけを作っておき、本当に必要になった瞬間に初めて本物を作る（遅延初期化）ことができます。

### 2. セキュリティやアクセス制限をかけられる（保護プロキシ）
本物のオブジェクトを操作させる前に、代理人が「あなたにその権限はありますか？」とチェックを入れることができます。本物のクラスの中身を汚さずに、アクセス制御ロジックを分離できます。

### 3. ネットワーク越しでも「隣にいる」ように扱える（遠隔プロキシ）
実際には別のサーバーにあるオブジェクトでも、手元にある「代理人」を操作している感覚で利用できます。通信の複雑なコードを代理人の中に隠蔽できるのが強みです。

### 4. 付加機能を後付けできる
実行ログを録ったり、一度計算した結果をキャッシュ（保存）して使い回したりといった、本筋とは関係ないけれど必要な処理を代理人に任せることができます。



## デメリット
### 1. レスポンスがわずかに遅れる
代理人を一枚挟むため、直接本物を叩くよりも「中継」の分だけオーバーヘッドが発生します。特にネットワーク経由などの場合、この設計がボトルネックになる可能性もあります。

### 2. コードの複雑化
「インターフェース」「本物のクラス」「代理人クラス」と、クラスの数が増えます。単純なアクセスで十分な場所に導入すると、かえって構造が読みづらくなることがあります。

### 3. 予期せぬ挙動に繋がる
利用者は「本物」を触っているつもりでも、実は「代理人」が裏でキャッシュを返していたり、エラーを握りつぶしていたりすることがあります。デバッグ時に「なぜか最新のデータが反映されない！」と混乱する原因になることも。


## サンプルソース


```Java:Subject.java
public interface Subject {
    public void request1();
    public void request2();
    public void request3();
}
```
```Java:RealSubject.java
public class RealSubject implements Subject{
    @Override
    public void request1() {
        System.out.println("実際の主体のリクエスト１です");
    }

    @Override
    public void request2() {
        System.out.println("実際の主体のリクエスト２です");
    }

    @Override
    public void request3() {
        System.out.println("実際の主体のリクエスト３です");
    }
}
```
```Java:Proxy.java
public class Proxy implements Subject{
    private RealSubject realSubject;
    
    @Override
    public void request1() {
        System.out.println("代理のリクエスト１です");
    }

    @Override
    public void request2() {
        System.out.println("代理のリクエスト２です");
    }

    @Override
    public void request3() {
        createRealSubject();
        //Proxyが判断し必要な場合のみRealSubjectを使用する。
        realSubject.request3();
    }
    
    private void createRealSubject(){
        //RealSubjectのインスタンスが存在しない場合のみインスタンス化する。
        if (realSubject == null){
            realSubject = new RealSubject();
        }
    }
}
```
```Java:Client.java
public class Client {
    public static void main(String... args){
        Subject subject = new Proxy();
        subject.request1();
        subject.request2();
        subject.request3();
    }
}
```


### 実行結果
```
代理のリクエスト１です
代理のリクエスト２です
実際の主体のリクエスト３です
```
