## 概要
**Strategy（ストラテジー）パターン**を一言でいうと、**「作戦（アルゴリズム）を切り替え可能にする」** 仕組みのことです。

たとえるなら、**「ゲームの装備品」** や **「ネットショップの支払い方法」** のようなイメージです。
ショップのレジ（本体）は変えずに、支払う人が「クレジットカードで作戦A」「PayPayで作戦B」と、その場に合った「作戦」をカチャッと差し替えて使えるようにするのがこのパターンの特徴です。

#### 戦国武将が兵隊に陣形を指示し、戦うことを例として取り上げました。
* **Clientクラス**：兵隊に指示をする戦国武将（大将）のイメージです。
* **Contextクラス**：兵隊のイメージです。どの陣形で戦うかの指示を受け取り、指示に従い戦います。
* **Strategyインタフェース**：「陣形」を表します。具体的な陣形ではなく、概念です。
* **鶴翼の陣クラス**：具体的な陣形「鶴翼の陣」です。陣形の一種なので、Strategyインタフェースを実装しています。
* **魚鱗の陣クラス、長蛇の陣クラス**：鶴翼の陣クラスと同様です。

#### Strategyパターンの真髄：具体的な「陣形」を知らない強み

`Client`（指揮官）は `Context`（実戦部隊）に対し、「今回はこの陣形で戦うぞ！」と具体的なインスタンスを手渡します。しかし、このパターンの最大の特徴は、**`Context` 側が「どの陣形を受け取ったか」を一切意識していない**点にあります。

`Context` にとって、それはあくまで「何らかの陣形（インターフェース）」でしかありません。実際のソースコードを覗いても、`Context` 内には「鶴翼の陣」や「魚鱗の陣」といった具体的なクラス名は一切登場しないのです。

##### この「無関心」が生む3つのメリット

* **戦略の自由な差し替え**
    `Context` を修正することなく、状況に応じて「長蛇の陣」から「魚鱗の陣」へと、その場で戦略をスイッチできます。
* **影響範囲の限定**
    新しい陣形を追加しても、`Context` のコードを書き換える必要がないため、既存のシステムを壊すリスクがありません。
* **役割の明確な分離**
    「作戦を選ぶ人（`Client`）」と「作戦を実行する人（`Context`）」の責任が完全に分かれるため、コードの見通しが劇的に良くなります。



## メリット

### 1. 「if-else」や「switch」の地獄から抜け出せる
「もし支払い方法がAなら……、Bなら……」と大きなクラスの中に大量の分岐を書くと、コードが読みづらく、修正も怖くなります。Strategyを使えば、各作戦を独立したクラスに分けるため、メインのコードが驚くほどスッキリします。

### 2. 新しい「作戦」を増やすのが簡単
新しい支払い方法（例：仮想通貨払い）を追加したいとき、既存のコードをいじる必要はありません。新しい「仮想通貨クラス」を作るだけで済みます。他の部分を壊す心配がないので、安心して拡張できます。

### 3. 実行中に「作戦」を切り替えられる
プログラムが動いている最中に、「今は急いでいるから最短ルート検索」「今は景色を見たいから観光ルート検索」といった具合に、ユーザーの選択に合わせて中身のロジックを自由に入れ替えられます。



## デメリット

### 1. クラスの数がどんどん増える
どんなに小さな作戦でも、それぞれを別々のクラスにする必要があります。作戦が10個あれば、クラスも10個増えるため、プロジェクト全体が少し複雑に見えてしまうことがあります。

### 2. 使う側（Client）が「作戦」を知っている必要がある
「どの作戦を使うか」を決めるのは、多くの場合、それを利用する側のプログラムです。「今はどの作戦が最適か？」を利用者が理解して選ばなければならないため、利用側の負担が増えることがあります。

### 3. データを受け渡すのが少し面倒
すべての作戦で共通の「窓口（インターフェース）」を使わなければなりません。ある作戦ではデータが1つで済むのに、別の作戦では3つ必要……という場合、それらをどうやって共通の窓口でやり取りするか、設計に工夫が必要になります。

## 適用パターン

| おすすめの用途 | 避けるべきケース |
| :--- | :--- |
| **似たような目的の「やり方」が複数あるとき** | 処理の仕方が1つしかなく、今後も増えないとき |
| **複雑な条件分岐（if文）が多発しているとき** | 処理が非常にシンプルで、分けるメリットが薄いとき |
| **アルゴリズムを状況に応じて切り替えたいとき** | クラスを増やすコストが、コードの綺麗さを上回るとき |



## サンプルソース
```Java:Context.java
public class Context {
    // ContextクラスはStrategyを持つ。持つのはクラスではなくインタフェース
    // 大事なのは、Contextクラスは具体的にどのStrategyかを意識させないこと！
    private final Strategy strategy;
    
    public Context(Strategy strategy) {   
        this.strategy = strategy;
    }

    public void execute(){
        System.out.println("これから戦略を実行します！");
        
        // 何かしらの戦略のexecuteメソッドを実行する。
        strategy.execute();
        
        System.out.println("戦略を実行しました！");
    }
}
```

```Java:Strategy.java
public interface Strategy {
    public void execute();
}
```

```Java:ConcreteStrategy1.java
public class ConcreteStrategy1 implements Strategy {
    @Override
    public void execute() {
        System.out.println("Strategy1を実行します");
    }
}
```

```Java:ConcreteStrategy2.java
public class ConcreteStrategy2 implements Strategy {
    @Override
    public void execute() {
        System.out.println("Strategy2を実行します");
    }
}
```

```Java:Client.java
public class Client {
    public static void main(String... args) throws Exception{
        Context context1 = new Context(new ConcreteStrategy1()); // 戦略1を渡す
        context1.execute();
        
        System.out.println("----------------------------------------------------");
        
        Context context2 = new Context(new ConcreteStrategy2());  //戦略2を渡す
        context2.execute();
    }
}
```
