## 概要
**Factory Method（ファクトリーメソッド）パターン**は、**「インスタンス（モノ）の作り方を、子クラスにお任せする」** 仕組みのことです。
たとえるなら、**「全国展開しているラーメンチェーン店」** のようなイメージです。
本部は「ラーメンを作る」という手順（メソッド）だけを決めておき、実際に「醤油ラーメンを作るのか」「味噌ラーメンを作るのか」という具体的な中身は、各店舗（子クラス）が決める……という役割分担をします。

### 【解説】
普通、何かを作るときは `new 自動車()` と直接書けば済みます。しかし、それだと「自動車」という名前にコードが縛られてしまいます。

Factory Methodを使うと、利用側（Client）は **「何の製品が来るか」を詳しく知らなくても、「工場にお願いすれば、とにかく製品が手に入る」** という状態を作れます。

### 1. スーパークラス同士の「約束」
画像上部の青いボックス（抽象的な工場・製品）に注目してください。
* **抽象的な工場:** 「うちの系列の工場は、必ず `createProduct()` で製品を作ります」と宣言します。
* **抽象的な製品:** 「この工場で作られるものは、すべて『製品』として扱います」という共通の型です。
ここで「作り方の手順（枠組み）」だけが決まります。

### 2. サブクラスでの「実作業」
画像下部の赤いボックス（自動車・ジュース）が現場です。
* **自動車工場**は、`createProduct()` の中で `new 自動車()` をします。
* **ジュース工場**は、`createProduct()` の中で `new ジュース()` をします。
このように、**「具体的に何を `new` するか」という責任を、親から子（サブクラス）へ押し付けている**のがポイントです。

### 3. Client（利用側）のメリット
Clientは、「自動車工場」を「工場（スーパークラス）」という型として扱います。
「この工場が何を作っているかは詳しく知らないけれど、`createProduct()` を呼べば何かしら便利な製品をくれるはずだ」という風に、**具体的な中身と切り離して（疎結合に）考えることができる**ようになります。

### まとめ
この図の最も重要なメッセージは、**「工場（Factory）」と「製品（Product）」の継承関係がパラレル（並行）になっている**点です。
「新しい製品を作りたいなら、新しい工場も用意してね」というルールを明確にすることで、プログラムがぐちゃぐちゃになるのを防いでいます。


## メリット
### 1. 使う側（Client）が具体的な中身を知らなくていい
これがいちばんの利点です。使う側は「何か製品（Product）を作って！」と頼むだけで済みます。中身が「最新モデル」に変わっても、頼み方を変える必要がないので、プログラムの修正が最小限で済みます。

### 2. 新しい「製品」を増やすのが簡単（拡張性）
新しい製品（例：塩ラーメン）を追加したいとき、既存のコード（醤油や味噌のコード）をいじる必要はありません。新しい「塩ラーメン工場クラス」を作るだけでOKです。これを**「オープン・クローズドの原則（拡張に開いていて、修正に閉じている）」**と呼びます。

### 3. 生成ロジックを一箇所にまとめられる
「インスタンスを作る時に、この初期設定をして、あの値をチェックして……」といった面倒な準備を、工場（Factory）の中に隠しておけます。あちこちでバラバラに `new` するよりも、管理がずっと楽になります。



## デメリット
### 1. クラスの数が倍増する
製品（Product）を1つ増やすたびに、それを作るための工場（Creator）の子クラスもセットで作らなければなりません。「製品クラス」と「工場クラス」がペアで増えていくので、管理するファイルの数が多くなりがちです。

### 2. コードが少し複雑（回りくどく）なる
単純に `new ClassA()` と書けば済むところを、わざわざインターフェースを定義して、工場クラスを作って……と準備が必要になります。小規模なプログラムだと「かえって面倒くさい！」と感じることもあります。

### 3. 継承に依存している
Factory Methodは「継承（extends）」を使います。子クラスを作らなければならないため、設計が固定されやすく、柔軟性が失われる場合もあります。

## 適用パターン

| おすすめの用途 | 避けるべきケース |
| :--- | :--- |
| **作るモノの種類が、今後どんどん増えそうなとき** | 作るモノが一生「これ1つ」と決まっているとき |
| **作り方が複雑で、使う側にその手順を見せたくないとき** | 直感的に `new` したほうが誰が見ても分かりやすいとき |
| **フレームワーク側で、利用者に具体的なクラスを作らせたいとき** | クラスを増やすコストが、メリットを上回るとき |


## サンプルソース


```Java:Factory.java
public abstract class Factory { 
    public final Product create(){
        Product product = factoryMethod();
        return product;
    }
    // 何の製品を作るかはサブクラスに任せる
    protected abstract Product factoryMethod();
}
```
```Java:ConcreteFactory.java
public class ConcreteFactory extends Factory{
    // ConcreteFactoryはConcreteProductを生成することをここに実装
    @Override
    protected Product factoryMethod() {
        return new ConcreteProduct();
    }
}
```
```Java:Product.java
public interface Product {
    public void use();
}
```
```Java:ConcreteProduct.java
public class ConcreteProduct implements Product{
    @Override
    public void use() {
        System.out.println("ConcreteProductを使用します");
    }
}
```
```Java:Client.java
public class Client {
    public static void main(String... args){
        //使用する側に何の工場かは指定してもらう必要があるが、
        //何の製品かは意識させたくない。
        Factory factory = new ConcreteFactory();
        Product product = factory.create();
        //上記の代わりに以下の実装でも同様の処理となる。
        //ただし、利用者側が工場が作る製品を意識する必要が出てくる。
        //Product product = new ConcreteProduct();
        
        product.use();
    }
}
```
