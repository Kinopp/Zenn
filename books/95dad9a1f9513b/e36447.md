# 前提知識
副問い合わせは、SQLの中に別のSQLをネストさせる（入れ子にする）手法です。一言で言うと **「クエリの結果を、別のクエリの材料として使う」** 高度なテクニックです。

![](https://storage.googleapis.com/zenn-user-upload/af358dba335c-20260322.png)
## 1. 副問い合わせとは？
通常、SQLで「給与が平均以上の人」を探そうとすると、以下の2ステップが必要です。
1.  平均給与を計算する（例：50万円）
2.  その値（50万円）を使って比較する

副問い合わせを使えば、この2つを1つのSQLで完結できます。
> **「未知の値（まだ計算してみないとわからない値）」** を使って検索条件を作りたいときに、副問い合わせが真価を発揮します。

## 2. 「非相関」と「相関」の違い

### ① 非相関副問い合わせ（独立型）
副問い合わせが単独で実行できるタイプです。
* **例：** 全体の平均給与より高い人を出す。
* **動き：** 最初に副問い合わせが1回だけ実行され、その結果が外側に渡されます。

### ② 相関副問い合わせ（連動型）
外側のクエリ（メインクエリ）の1行ごとに、中身を計算し直すタイプです。
* **例：** 「自分が所属する部署」の平均給与より高い人を出す。
* **動き：** 外側のテーブルのデータを使って、中身を1行ずつ評価します。

# 問題9-1（副問い合わせの基本） *Lv2*
### SHスキーマ`SALES`テーブルより、2019年1月1日から2019年3月31日までの合計売上個数（`QUANTITY_SOLD`の合計）と合計売上金額（`AMOUNT_SOLD`の合計）を取得してください。（順不同）


## 期待する結果
| CUST_ID | NAME            | SUM_QUANTITY_SOLD | SUM_AMOUNT_SOLD | 
| ------- | --------------- | ----------------- | --------------- | 
| 12783   | Rose Lehman     | 80                | 33676.07        | 
| 12539   | Lucas Gilboy    | 72                | 9647.59         | 
| 25244   | Rufus Rokus     | 74                | 9710.03         | 
| 2760    | Luana Crisp     | 72                | 9652.08         | 
| 5696    | Sabrina Pack    | 71                | 9595.62         | 
| 2791    | Madra Leary     | 74                | 5194.77         | 
| 9038    | Marvel Bakerman | 72                | 45103.24        | 
| 3720    | Michael Lincoln | 73                | 9675.79         | 
| 1255    | Sally Filbert   | 74                | 9710.03         | 

## 解答例
```sql:例1：インラインビューを使用
SELECT
    c.cust_id,
    c.cust_first_name || ' ' || cust_last_name AS name,
    sa.sum_quantity_sold,
    sa.sum_amount_sold
FROM
    sh.customers c
    INNER JOIN (
        SELECT
            cust_id,
            SUM(quantity_sold) AS sum_quantity_sold,
            SUM(amount_sold)   AS sum_amount_sold
        FROM
            sh.sales
        WHERE
            time_id BETWEEN DATE '2019-01-01' 
                        AND DATE '2019-03-31'
        GROUP BY
            cust_id
    ) sa ON c.cust_id = sa.cust_id
WHERE
    sa.sum_quantity_sold > 70
```
```sql:例2：WITH句を使用
WITH sa AS (
    SELECT
        cust_id,
        SUM(quantity_sold) AS sum_quantity_sold,
        SUM(amount_sold)   AS sum_amount_sold
    FROM
        sh.sales
    WHERE
        time_id BETWEEN DATE '2019-01-01' 
                    AND DATE '2019-03-31'
    GROUP BY
        cust_id
)
SELECT
    c.cust_id,
    c.cust_first_name || ' ' || cust_last_name AS name,
    sa.sum_quantity_sold,
    sa.sum_amount_sold
FROM
    sh.customers c
    INNER JOIN sa 
       ON c.cust_id = sa.cust_id
WHERE
    sa.sum_quantity_sold > 70
```

## 解説
副問い合わせ（サブクエリ）を一言で表現すると、**「クエリの中に入れ子（マトリョーシカ状態）になった別のクエリ」**　のことです。

### 1. SELECT文の「インプット」と「アウトプット」
SQLの基本的な特徴として、SELECT文は以下のサイクルで動いています。
* **インプット**：表（テーブル）を受け取る
* **アウトプット**：結果を「表（テーブル）」として出力する

この性質があるため、**「SELECTした結果（表）」に対して、さらに別のSELECT文を実行したり、結合したりすること**が可能です。これが副問い合わせの根本的な仕組みです。

図で表現すると、
* **インプット**：左の緑色の４テーブル
* **アウトプット**：右の最終結果（FINAL OUTPUT）
  
**仮想的な途中結果(オレンジ色)** を得るために使用するのが副問い合わせのイメージです。

![](https://storage.googleapis.com/zenn-user-upload/d94611cd4aeb-20260318.png)

当問題では、`SALES`テーブルと`CUSTOMERS`テーブルを直接結合するのではなく、副問い合わせにより`SALES`テーブルで集計を行い、仮想的な途中結果を作り、その結果と`CUSTOMERS`テーブルを結合しています。直接結合してしまうと、直接集計に関係ない項目（`CUST_FIRST_NAME`と`CUST_LAST_NAME`）も集計キーにしないと実現できなくなります。
```sql:直接結合する求め方（初心者のやり方)
SELECT
    c.cust_id,
    c.cust_first_name || ' ' || cust_last_name     AS name,
    SUM(sa.quantity_sold) AS sum_quantity_sold,
    SUM(sa.amount_sold)   AS sum_amount_sold
FROM
    sh.customers c
    INNER JOIN sh.sales sa 
       ON c.cust_id = sa.cust_id
WHERE
    sa.time_id BETWEEN DATE '2019-01-01' 
                   AND DATE '2019-03-31'
GROUP BY
    c.cust_id,
    c.cust_first_name,  -- 本来不要な集計キー
    c.cust_last_name    -- 本来不要な集計キー
HAVING
    SUM(sa.quantity_sold) > 70
```


### 2. 「2段構え」の処理を1つにまとめる
副問い合わせを活用すると、以下のような2ステップの処理を、たった1つのSQLで完結させることができます。
1.  **ステップ1**：あるデータを知るために、まず別のデータを確認する（内側のクエリ）
2.  **ステップ2**：その結果を「検索条件」として使い、本命のデータを取得する（外側のクエリ）

### 3. SQLマスターへの入り口
このテクニックを自在に使いこなせるようになることは、**「SQLマスター」への第一歩**といっても過言ではありません。
副問い合わせを習得することで、単純なデータ抽出だけでなく、実務における非常に複雑な業務要件もSQLだけでスマートに表現できるようになります。

### 4. with句の利用
SQLにおける**WITH句**は、一言で言うと **「複雑なクエリに一時的な名前をつけて、使い回せるようにする機能」** です。
正式には「共通テーブル式（CTE: Common Table Expression）」と呼ばれますが、実務では「WITH句（ウィズく）」と呼ばれることが一般的です。

#### ① WITH句のイメージ
複雑な料理を作る前に、あらかじめ「下ごしらえした材料」に名前をつけて置いておくようなイメージです。
通常、SQLは下から上、あるいは中から外へと読み解く必要がありますが、WITH句を使うと**「まずこれを用意して、次にそれを使って……」**と、上から下へ論理的に記述できるようになります。

#### ② 基本的な書き方
```sql
WITH テンポラリ名 AS (
    -- ここにサブクエリ（下ごしらえ）を書く
    SELECT department_id, AVG(salary) as avg_sal
    FROM employees
    GROUP BY department_id
)
-- メインのクエリでその名前をテーブルのように使う
SELECT *
FROM テンポラリ名
WHERE avg_sal > 5000;
```

#### ③ WITH句を使う3つのメリット

- ####  クエリが読みやすくなる（可読性）

  サブクエリを「入れ子（マトリョーシカ）」にする必要がないため、クエリ全体の構造がスッキリします。後から見返したときに「この部分は〇〇の集計だな」と一目で分かります。

- #### 同じ処理を再利用できる
  一つのSQLの中で、同じ集計結果を何度も参照したい場合、WITH句で一度定義してしまえば、何度でも呼び出すことができます。

- #### デバッグ（修正）が楽になる

  複雑なロジックがWITH句ごとに切り分けられているため、「どこで計算が間違っているか」の特定や修正が非常にスムーズになります。

#### ④ サブクエリとの違い
「やりたいこと」はサブクエリ（副問い合わせ）と同じですが、見た目と管理のしやすさが大きく違います。

* **サブクエリ**: メインの文の中に「埋め込まれる」ので、深くなると読みづらい。
* **WITH句**: メインの文の前に「定義する」ので、整理整頓されている。


:::message
### **副問い合わせの理解は、SQL初心者脱却への第一歩**

SQLを自在に操るための鍵は、単に構文を覚えることではなく、**「思考のプロセス」**　を切り替えることにあります。

#### 1. 「表（テーブル）から表を作る」という右脳思考
SQLの真髄は、**「ある表を加工して、また別の新しい表を作り出す」** というプロセスの繰り返しにあります。この「塊（集合）」を直感的に捉える感覚こそが、いわゆる **「右脳思考」** です。副問い合わせは、このプロセスを具現化するための最も重要なツールと言えます。

#### 2. 「一発で解決しよう」という罠を避ける
SQLを難しく感じてしまう原因の多くは、**「一度のクエリで、一気に最終的な答えを出そうとする」** ことにあると考えています。
* **初心者の傾向**: 複雑な条件を一つの塊に詰め込もうとして、途中で混乱してしまう。
* **脱却への道**: 副問い合わせを活用し、「まずはこの中間データ（表）を作ろう」と段階を踏むことで、一つ一つのステップがシンプルになります。
  
#### 3. ゴールから逆算する「バックワード思考」
複雑な問題にぶつかった時は、以下の手順で考えてみてください。

1.  **ゴールを確認**: 最終的にどのようなデータが欲しいか？
2.  **逆算する**: その結果を得るために、「その一歩手前」でどんなデータ（中間テーブル）があれば助かるか？
3.  **組み立てる**: その「一歩手前のデータ」を副問い合わせで作る。

このように、**「ゴールに辿り着くためのパーツ」を副問い合わせで作っていく**という発想を持つだけで、SQLのハードルはぐっと下がります。

# 問題9-2（単一行サブクエリ） *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、給与（`salary`）の平均値よりも大きい人を取得してください。取得対象は入社日（`HIRE_DATE`）が2018年1月1日以降の人。（給与額の降順）

## 期待する結果
| EMPLOYEE_NAME  | SALARY | 
| -------------- | ------ | 
| Eleni Zlotkey  | 10500  | 
| Mattea Marvins | 7200   | 
| David Lee      | 6800   | 

## 解答例
```sql
SELECT
    first_name  || ' ' || last_name AS employee_name,
    salary
FROM
    hr.employees
WHERE
    salary > (
        SELECT
            AVG(salary)
        FROM
            hr.employees
        )
    AND hire_date >= DATE'2018-01-01'
ORDER BY
    salary DESC;
```

## 解説
カッコ内の副問い合わせが先に実行され、例えば「6400」という平均給与が算出されます。その後、メインのクエリが「給与が6400より大きい社員」を抽出します。
副問い合わせ（サブクエリ）の中でも、もっとも基本的でよく使われる **「単一行サブクエリ」** について、その仕組みと実務での注意点は下記の通りです。

### 1. 単一行サブクエリとは？
単一行サブクエリとは、内側のクエリ（サブクエリ）が返す結果が **「1行だけ」** であるもののことを指します。通常は「1つの値（スカラ値）」を返すため、メインのクエリで `=` や `>` などの比較演算子と一緒に使うことができます。

イメージとしては、**「変数に一度値を代入してから、それを使って比較する」** という処理を1つのSQLで行っている状態です。

#### 具体的な使用例
「アベル（Abel）さんよりも給与が高い従業員」を探したい場合、まずアベルさんの給与を知る必要があります。

```sql
SELECT last_name, salary
FROM employees
WHERE salary > (SELECT salary FROM employees WHERE last_name = 'Abel');
```
このカッコ内の `(SELECT salary ...)` が「単一行サブクエリ」です。

### 2. 使用できる演算子
単一行サブクエリは結果が1つだけなので、以下の**単一行比較演算子**を使用します。

* **`=`** （等しい）
* **`>`** （より大きい）
* **`>=`** （以上）
* **`<`** （より小さい）
* **`<=`** （以下）
* **`<>`** または **`!=`** （等しくない）

### 3. 知っておくべき注意点（エラーの回避策）

ここが実務で非常に重要なポイントです。

#### ① 「複数行」返ってくるとエラーになる
もっとも多いエラーがこれです。サブクエリの結果が2行以上返ってくると、`=` などの単一行演算子では処理できず、以下のエラーが発生します。
> **ORA-01427: 単一行副問合せにより2行以上の行が戻されました**

* **対策**: サブクエリの `WHERE` 句で主キー（IDなど）を指定して必ず1行になるようにするか、複数行返る可能性がある場合は `IN` 演算子（多重行サブクエリ用）に変更する必要があります。

#### ② 結果が「0行（NULL）」の場合
もしサブクエリの条件に合うデータがなく、結果が0行だった場合、サブクエリは `NULL` を返します。
* **挙動**: `WHERE salary > NULL` のような比較になり、結果として**メインのクエリも1行も表示されません。**
* **注意**: エラーにはなりませんが、「データがあるはずなのに何も出ない」という時は、サブクエリが空（NULL）になっていないかを確認しましょう。

#### ③ グループ関数との組み合わせ
「平均給与（AVG）より高い人」のように、グループ関数を使う場合は必ず単一行サブクエリになります（集計結果は1つにまとまるため）。
```sql
-- 平均給与より高い人を抽出
WHERE salary > (SELECT AVG(salary) FROM employees)
```

# 問題9-3（複数行サブクエリ） 　*Lv2*
### HRスキーマ`EMPLOYEES`テーブルより、社員を取得してください。取得対象は`DEPARTMENTS`テーブル`DEPARTMENT_NAME`が「IT」である部門に所属する人。（順不同）

## 期待する結果
| EMPLOYEE_NAME   | JOB_ID  | DEPARTMENT_ID | 
| --------------- | ------- | ------------- | 
| Alexander James | IT_PROG | 60            | 
| Bruce Miller    | IT_PROG | 60            | 
| David Williams  | IT_PROG | 60            | 
| Valli Jackson   | IT_PROG | 60            | 
| Diana Nguyen    | IT_PROG | 60            | 

## 解答例
```sql:例1：INを使用
SELECT
    first_name || ' ' || last_name AS employee_name,
    job_id,
    department_id
FROM
    hr.employees
WHERE
    department_id IN (
        SELECT
            department_id
        FROM
            hr.departments
        WHERE
            department_name = 'IT'
    );
```
```sql:例2：EXISTSを使用
SELECT
    first_name || ' ' || last_name AS employee_name,
    job_id,
    department_id
FROM
    hr.employees e
WHERE
    EXISTS (
        SELECT
            'X'
        FROM
            hr.departments d
        WHERE
            e.department_id = d.department_id
            AND department_name = 'IT'
    );
```

## 解説
複数行サブクエリは、単一行サブクエリとは使い勝手が異なり、専用の演算子が必要になる点がポイントです。

### 1. 複数行サブクエリとは
サブクエリ（内側のSQL）を実行した結果、**2行以上のデータが返ってくるもの**を指します。
例えば、「IT部門の従業員IDをすべて取得する」というサブクエリを書いた場合、該当者が複数いれば、その結果は「103, 104, 105...」というリストになります。これを外側のクエリで扱うには、専用の書き方が必要です。

### 2. よく使われる演算子
複数行サブクエリでは、 `=` や `<` といった通常の比較演算子の代わりに、以下の演算子を使用します。

| 演算子 | 意味 | 解説 |
| :--- | :--- | :--- |
| **`IN`** | いずれかと一致 | リスト内の値のどれか1つでも一致すればOK。最もよく使われます。 |
| **`ANY`** | いずれかと比較 | `> ANY (リスト)` と書くと、「リスト内の最小値よりも大きい」という意味になります。 |
| **`ALL`** | すべてと比較 | `> ALL (リスト)` と書くと、「リスト内の最大値よりも大きい」という意味になります。 |
| **`EXISTS`** | 存在するかどうか | サブクエリの結果が1行でも存在すれば真となります。 |

### 3. 初心者が陥りやすいポイント
複数行サブクエリを扱う際、特に注意すべき点が2つあります。

#### ① 単一行演算子（=, <, >）を使うとエラーになる
サブクエリの結果が複数行なのに、外側で `=` を使ってしまうと、実行時に以下のようなエラーが発生します。
> `ORA-01427: single-row subquery returns more than one row`
> （単一行副問合せにより2行以上の行が戻されました）

「今は1行しかデータがないから `=` で動いている」というクエリでも、**将来的にデータが増えて2行以上返る可能性がある場所**には、最初から `IN` を使っておくのが安全です。

#### ② 「NOT IN」と「NULL」の罠
これが最も恐ろしい注意点です。**サブクエリの結果の中に一つでも `NULL` が含まれている場合、`NOT IN` は一行も結果を返しません。**

* **理由**: SQLの論理では「NULLと等しくない」という比較が「不明」となり、全体の結果が判定できなくなるためです。
* **対策**: サブクエリ内で `WHERE 列名 IS NOT NULL` を追加してNULLを除外するか、`NOT EXISTS` を使用するようにしましょう。

# 問題9-4（差分の抽出） *Lv2*
### SHスキーマ`CUSTOMERS`テーブルより、2021年に一度も購入しなかった顧客（完全離脱者）を特定してください。具体的には`SALES`テーブル`time_id`に2021年のデータを持たない人が対象となります。出力対象は`CUST_YEAR_OF_BIRTH`が1924の人。（順不同）

## 期待する結果
| CUST_ID | CUST_GENDER | CUST_YEAR_OF_BIRTH | 
| ------- | ----------- | ------------------ | 
| 102011  | M           | 1924               | 
| 102258  | F           | 1924               | 
| 103921  | M           | 1924               | 
| 102555  | M           | 1924               | 
| 101256  | M           | 1924               |  

## 解答例
```sql:例1：INを使用
SELECT
    cust_id,
    cust_gender,
    cust_year_of_birth
FROM
    sh.customers
WHERE
    cust_id NOT IN (
        SELECT
            cust_id
        FROM
            sh.sales
        WHERE
            time_id BETWEEN DATE'2021-01-01' AND DATE'2021-12-31'
    )
    AND cust_year_of_birth = 1924;
```
```sql:例2：EXISTSを使用
SELECT
    cust_id,
    cust_gender,
    cust_year_of_birth
FROM
    sh.customers
WHERE
    NOT EXISTS (
        SELECT
            'X'
        FROM
            sh.sales
        WHERE
                sh.sales.cust_id = sh.customers.cust_id
            AND time_id BETWEEN DATE'2021-01-01' AND DATE'2021-12-31'
    )
        AND cust_year_of_birth = 1924;
```
## 解説
SQLで「特定のデータが存在するか」を確認する際に使われる **`IN`** と **`EXISTS`** は、似ているようでいて実は「得意分野」や「落とし穴」が異なります。

### 1. IN 演算子（値のリストと比較）
`IN` は、特定の列の値が「指定したリスト（またはサブクエリの結果）の中にあるか」をチェックします。

* **イメージ**: 「この箱（リスト）の中に、私の探しているカード（値）は入っていますか？」と聞くような感覚です。
![](https://storage.googleapis.com/zenn-user-upload/11fde255334b-20260319.png)
  
* **特徴**:
  * **OR展開** : `x IN (a,b,c)` は数学的に `x=a OR x=b OR x=c` と等価。
  * **テンポラリテーブルの生成** ： サブクエリの場合、エンジンは結果セットをメモリ上の「作業用テーブル」として構築してから左辺と比較することが多い。
  * **重複の無駄** ： リスト内に重複があっても、エンジンは評価を続ける場合がある（最適化次第）。

### 2. EXISTS 演算子（行の存在を確認）
`EXISTS` は、サブクエリの中で「条件に合う行が1行でも見つかるか」をチェックします。

* **イメージ** : 「向こうの部屋（サブクエリ）に、条件に合う人は一人でも居ますか？」と聞き、一人見つかった瞬間に「はい、居ました！」と答えて処理を終える（打ち切る）感覚です。
![](https://storage.googleapis.com/zenn-user-upload/73e82724564f-20260319.png)

* **特徴** :
  * **ショートサーキット（即時終了）**：条件に合致する行を1件でも見つけた瞬間、検索を打ち切り`TRUE`を返す。全件スキャンを回避。
  * **`SELECT`句は無視される** ： `EXISTS`は行の「存在」のみを評価する。そのため、`SELECT 1/0` のようなエラーになる式を書いても、評価されないためクエリは正常に動く。

### 3. どちらを使うべき？（使い分けの基準）

一般的には、以下のような基準で使い分けます。

| 比較項目 | IN が得意なケース | EXISTS が得意なケース |
| :--- | :--- | :--- |
| **データの量** | サブクエリの結果（リスト）が**小さい**場合 | 外側のテーブルが小さく、サブクエリ側のテーブルが**大きい**場合 |
| **NULLの扱い** | **要注意。** サブクエリにNULLがあると結果が変わる | NULLの影響を受けにくい |
| **直感的な理解** | 非常に分かりやすい | 少し慣れが必要（相関サブクエリのため） |
| **パフォーマンス** | サブクエリの結果が小さい場合 | サブクエリ側のテーブルが巨大な場合 |

- 昔は「`EXISTS`の方が速い」と言われていましたが、最近のOracleなどの賢いオプティマイザは、`IN` を内部で `EXISTS` や `JOIN` に書き換えて最適化してくれるため、**速度差が出ないことも多い** です。
- 実践的には、上記特徴を踏まえ、`IN`を使う強い理由がない限り、`EXISTS`を使用するのが無難であると思います。

### 4. 使うときの注意点
実務で最も大きなトラブルに繋がりやすいのが、**「否定形（NOT）」と「NULL」** の組み合わせです。

#### ① NOT IN の罠（結果が0件になる恐怖）
もしサブクエリの結果の中に、**たった一つでも `NULL` が含まれている場合、`NOT IN` は一行も結果を返しません。**

* **理由**: SQLのルール上、`NULL` との比較は「不明」となり、全体として「一致しないとも、一致するとも言えない」状態になるためです。
* **対策**: `NOT NULL`制約が付与されていない項目など、`NULL`になる可能性が少しでもある項目に対し、`NOT IN` を使うときは、必ずサブクエリ側で `WHERE 列名 IS NOT NULL` を入れるか、**`NOT EXISTS`** に書き換えましょう。

#### ② EXISTS の SELECT 句
`EXISTS` のサブクエリ内で `SELECT *` と書いても `SELECT 1` と書いても、実はパフォーマンスに差はありません。なぜなら `EXISTS` は「行があるかないか」しか見ておらず、列の中身は見ていないからです。
慣習的には `SELECT 1` と書くことが多いです。`SELECT 'X'`と書く人もいます（私は`'X'`派です）。理由はよくわかりません。「存在する（eXists）」の「X」から取ったという説が有力らしいですが、Oracleの世界では、`DUAL`表を`SELECT`すると、`'X'`が返ってくることから、私はダミー文字が`'X'`であるからだと思っていました。

# 問題9-6（最新の履歴取得） *Lv2*
### SHスキーマ`SALES`テーブルより、`cust_id`=2の顧客が最後に購入した日の、販売詳細を特定してください。（順不同）


## 期待する結果
| CUST_ID | TIME_ID              | PROD_ID | AMOUNT_SOLD | 
| ------- | -------------------- | ------- | ----------- | 
| 2       | 2022-09-05T00:00:00Z | 18      | 1499.06     | 
| 2       | 2022-09-05T00:00:00Z | 18      | 1491.56     | 
| 2       | 2022-09-05T00:00:00Z | 18      | 1204.26     | 


## 解答例
```sql:例1：MAXを使用
SELECT 
    s.cust_id, 
    s.time_id, 
    s.prod_id, 
    s.amount_sold
FROM sh.sales s
WHERE 1=1
  AND s.time_id = (
    SELECT MAX(time_id) 
      FROM sh.sales sub 
     WHERE sub.cust_id = s.cust_id
  )
AND s.cust_id = 2;
```
```sql:例2：ALLを使用
SELECT 
    s.cust_id, 
    s.time_id, 
    s.prod_id, 
    s.amount_sold
FROM sh.sales s
WHERE 1=1
  AND s.time_id >= ALL(
    SELECT time_id 
      FROM sh.sales sub 
     WHERE sub.cust_id = s.cust_id
  )
  AND s.cust_id = 2;
```

## 解説
### 1. ALL演算子とは？
`ALL` は、比較演算子（`>`, `<`, `=`, `!=` など）と組み合わせて使い、**「リスト内（またはサブクエリの結果）のすべての値に対して条件を満たしているか」**を判定します。

* **`> ALL (リスト)`**：リスト内の**最大値よりも大きい**（全員に勝っている）
* **`< ALL (リスト)`**：リスト内の**最小値よりも小さい**（誰よりも低い）
* **`!= ALL (リスト)`**：リスト内の誰とも一致しない（`NOT IN` と同じ意味）

### 2. 基本的なクエリ例
「部門30の**誰よりも**高い給与をもらっている社員」を探す場合：

```sql
SELECT ENAME, SAL
FROM EMP
WHERE SAL > ALL (SELECT SAL FROM EMP WHERE DEPTNO = 30);
```
※ 内部的には「部門30の最高給与（MAX）」と比較しているのと似た動きになります。


### 3. 初心者が陥りやすい2つの注意点
#### ① リストに「NULL」が含まれると、結果は常に「空」
これが最大の落とし穴です。
もしサブクエリの結果の中に一つでも `NULL` が混じっていると、`ALL` の結果は決して「真（TRUE）」になりません。

* **理由：** `SAL > NULL` は「不明（UNKNOWN）」と判定されるため、全体の `AND` 条件が成立しなくなるからです。
* **対策：** サブクエリ内で `WHERE SAL IS NOT NULL` と書くか、`NVL` 関数で NULL を除外する必要があります。

#### ② 空のリスト（0行）の場合、結果は「すべて真」になる
意外かもしれませんが、サブクエリが**1行も返さなかった場合、`ALL` は常に TRUE** を返します。
* 「誰もいないグループの誰よりも高いか？」という問いに対し、SQLは「条件に違反する人がいないので、OK」と判断するのです。これは直感に反することが多いため、注意が必要です。


### 4. 実務的なアドバイス：MAX/MIN への書き換え
上級者向けのテクニックとして、`ALL` を使わずに済む場合は `MAX` や `MIN` を使う方が、読みやすく、NULLの扱いも制御しやすいため推奨されることがあります。

* **ALLを使用：** `WHERE SAL > ALL (SELECT SAL FROM EMP WHERE DEPTNO = 30)`
* **MAXを使用：** `WHERE SAL > (SELECT MAX(SAL) FROM EMP WHERE DEPTNO = 30)`
  * ※ `MAX` の場合、1行もなければ結果は NULL になり、比較結果は FALSE になるので、`ALL` より直感的です。

# 問題9-7（スカラ・サブクエリ） *Lv2*
### HRスキーマ`EMPLOYEES`テーブルより`NAME`、`SALARY`を取得し、さらに`DEPARTMENTS`テーブルより、該当する`DEPARTMENT_NAME`を取得してください。対象は`EMPLOYEE_ID`が110より小さいデータとしてください。（順不同）

## 期待する結果
| EMPLOYEE_ID | NAME            | SALARY | DEPT_NAME | 
| ----------- | --------------- | ------ | --------- | 
| 100         | Steven King     | 24000  | Executive | 
| 101         | Neena Yang      | 17000  | Executive | 
| 102         | Lex Garcia      | 17000  | Executive | 
| 103         | Alexander James | 9000   | IT        | 
| 104         | Bruce Miller    | 6000   | IT        | 
| 105         | David Williams  | 4800   | IT        | 
| 106         | Valli Jackson   | 4800   | IT        | 
| 107         | Diana Nguyen    | 4200   | IT        | 
| 108         | Nancy Gruenberg | 12008  | Finance   | 
| 109         | Daniel Faviet   | 9000   | Finance   | 

## 解答例
```sql:例1：スカラ・サブクエリを使用
SELECT
    e.employee_id,
    e.first_name || ' ' || e.last_name AS name,
    e.salary,
    -- ここで部署テーブル(departments)から部署名を1つだけSELECTする
    (SELECT d.department_name 
     FROM hr.departments d 
     WHERE d.department_id = e.department_id) AS dept_name
FROM
    hr.employees e
WHERE
    e.employee_id < 110
```
```sql:例2：結合を使用
SELECT
    e.employee_id,
    e.first_name || ' ' || e.last_name AS name,
    e.salary,
    d.department_name AS dept_name
FROM
    hr.employees e
    INNER JOIN hr.departments d
       ON d.department_id = e.department_id
WHERE
    e.employee_id < 110
```


## 解説
複数のテーブルから値を取得する場合、一般的には結合を行いますが、結合を行わずに、SELECT文の項目（カラム）の中で、別のSELECT文を実行して値を取得する方法を、**「スカラ・サブクエリ（Scalar Subquery）」** と呼びます。

### 1. 守らなければならない「鉄則」
SELECT句の中でSELECTを行う場合、以下のルールを破るとエラー（`ORA-01427` など）になります。

1.  **「1列」だけ返すこと**: 
    `(SELECT col1, col2 ...)` のように複数の列を返すことはできません。
2.  **「1行」だけ返すこと**: 
    サブクエリの結果が2行以上になると、どの値を表示していいか分からずエラーになります。必ず1つの値（スカラ値）になるように `WHERE` 句や集計関数で絞り込む必要があります。

### 2. `JOIN` とどっちがいいの？
`JOIN`（結合）については後の章で登場する概念なので、ここではなんとなく理解するぐらいでと止めておき、`JOIN`（結合）を学んでから再度参照いただけると良いかと思います。

| 項目 | スカラ・サブクエリ | JOIN（結合） |
| :--- | :--- | :--- |
| **可読性** | シンプルで、何を取得しているか直感的にわかる | 結合条件が増えると複雑に見える |
| **パフォーマンス** | 大量データ（数万件〜）では処理が重くなりやすい | データベースが最適化しやすく、一般的に高速 |
| **用途** | 全体平均との比較や、補助的な名称取得に最適 | メインのデータ収集や、複数の項目が必要な時に最適 |

> **使い分けのコツ**
> 基本は **`JOIN`** を使い、「どうしても全体の平均値と並べたい」ときや「わざわざ結合するほどでもない、ちょっとした名称を1つ出したい」ときに **スカラ・サブクエリ** を使うのが、スマートなSQLの書き方です。
