# 問題3-1（全データ取得）  *Lv1*
### **HRスキーマ`REGIONS`テーブルの全てのデータを取得してください。（順不同）**

## 期待する結果
| REGION_ID | REGION_NAME | 
| --------- | ----------- | 
| 10        | Europe      | 
| 20        | Americas    | 
| 30        | Asia        | 
| 40        | Oceania     | 
| 50        | Africa      | 

## 解答例
```sql:例1：「*」を使用
SELECT
    *
FROM
    hr.regions
```
```sql:例2：全てのカラムを指定
SELECT
    region_id,
    region_name
FROM
    hr.regions
```

## 解説
テーブルから全データを取得するには
- 「*」を指定する（例1）
- 全カラムを指定する（例2）
  
の２通りがあります。

アプリケーションの一部として使用する場合は、原則、**「*」は使用せず、必要はカラムのみを指定**することをお勧めします。

主な理由は以下の通りです。
### 1. **将来の仕様変更に対するリスク回避**
将来テーブルが拡張された場合、想定外の動作をする可能性があります。

### 2. **パフォーマンス**
不要で大きなデータを取り扱うことで、無駄な資源を使い、パフォーマンスに悪影響を与える可能性があります。

 「*」を使用しても問題ない場合は主に以下の場合であると考えます。
#### ① **テスト等で使用する自身の管理するSQLスクリプト**
アプリケーションの一部としてでなく、自身が管理するテストツールとしてのSQLスクリプトであれば、「*」を使用することで、効率よく開発することが可能になります。

#### ② **仕様として「漏れなく全データを取り扱う」場合**
仕様として「漏れなく全データを取り扱う」のであれば、「*」を使うことで、カラムの指定漏れなど、余計なバクを発生することがなくなります。

上記を踏まえ、状況に応じて、どちらを使うかは適切に判断しましょう。

<br>
:::message
### **SQLスクリプトを大文字で書くか小文字で書くか？**

SQLの文法自体は、大文字・小文字を区別しません。推奨されるスタイルとしては、可読性の観点から **「キーワードは大文字、テーブル名やカラム名は小文字」** で記述するのが良いと思います。当演習でもこのスタイルで記述いたします。
Oracle FreeSQLにはこのスタイルで記述するサポート機能があります（一部て修正が必要かもしれません）。
![](https://storage.googleapis.com/zenn-user-upload/e30a61c4d003-20260309.png)
:::

# 問題3-2（必要なデータの取得） *Lv1*
### **HRスキーマ`COUNTRIES`テーブルのうち`REGION_ID`が20のデータを取得し、`COUNTRY_ID`および`COUNTRY_NAME`を表示してください。（順不同）**

## 期待する結果
| COUNTRY_ID | COUNTRY_NAME             | 
| ---------- | ------------------------ | 
| AR         | Argentina                | 
| BR         | Brazil                   | 
| CA         | Canada                   | 
| MX         | Mexico                   | 
| US         | United States of America | 


## 解答例
```sql
SELECT
    country_id,
    country_name
FROM
    hr.countries
WHERE
    region_id = 20
```

## 解説
### 1. SELECT文を構成する「3つの基本要素」

SQLでデータを抽出する際、最も基本となるのが以下の3つの要素です。これらを組み合わせることで、必要なデータを取り出すことができます。

* **SELECT** ：「**どの列（データ）**」を表示するかを指定します。
* **FROM** ：「**どのテーブル（箱）**」からデータを取り出すかを指定します。
* **WHERE** ：「**どのような条件**」でデータを絞り込むかを指定します。

### 2. 文字列と数値の書き分けルール

SQLでは、データの種類（型）によって書き方が決まっています。

* **文字列**：シングルクォーテーション（`' '`）で囲みます。
  - 例：`'AB'`、`'東京都'`
* **数値**：何も囲まず、そのまま記述します。
  - 例：`123`、`5000`

### 3. 「暗黙の型変換」について

今回の例では、`region_id` という列が **NUMBER型（数値）** であるため、以下解答例のように記述するのが正解です。

> `WHERE region_id = 20`

しかし、SQLでは `WHERE region_id = '20'` のように数値を `' '` で囲んで「文字列」として取り扱っても、エラーにならず正常に動作することがあります。

これは、データベース側が「これは数値として比較したいのだな」と気を利かせて、自動的に型を変換してくれるためです。これを **「暗黙の型変換」** と呼びます。

便利に思える機能ですが、以下の理由から**データ型に合わせて正しく記述すること**を強くお勧めします。



:::message
### **暗黙の型変換に潜む「大きな落とし穴」**
「動くから大丈夫」と多用するのは禁物です。特に注意すべきなのが パフォーマンスへの影響 です。暗黙の型変換が行われると、本来設定されている **「インデックス（索引）」が有効に機能しなくなる**ことがあります。その結果、データの検索速度が極端に落ち(※)、システム全体の大きな遅延や障害を招く恐れがあります。
この現象はエラーとして表示されないため、原因に気づきにくく、後から修正するのが非常に大変です。 **「型は最初から正しく合わせる」** のが、安全で高速なSQLを書くための基本です。
(※)データ量によっては、数百倍〜数千倍遅くなることもあります。
:::

# 問題3-3（別名） *Lv1*
### **HRスキーマ`COUNTRIES`テーブルのうち`REGION_ID`が2のデータを取得し、`COUNTRY_ID`および`COUNTRY_NAME`を表示してください。（順不同）**
### **その際、列名を以下のように変更し、表示してください。**
* 　**COUNTRY_ID　　　→ 　ID**
* **COUNTRY_NAME　→ 　NAME**

## 期待する結果
| ID         | NAME                     | 
| ---------- | ------------------------ | 
| AR         | Argentina                | 
| BR         | Brazil                   | 
| CA         | Canada                   | 
| MX         | Mexico                   | 
| US         | United States of America | 


## 解答例
```sql
SELECT
    country_id as "ID",
    country_name as "NAME"
FROM
    hr.countries
WHERE
    region_id = 20
```

## 解説
### SQLでの列名への別名（エイリアス）の付け方
抽出したデータの見出しを分かりやすくするために、列には「別名」を付けることができます。記述のルールと、エラーを防ぐためのポイントを解説します。

#### 1. 「AS」キーワードの使用
列名の後に **`AS <別名>`** と記述することで、任意の名前を付けることができます。

* 例：`country_id AS ID`
* なお、この **`AS` は省略することも可能** です。

#### 2. ダブルクォーテーション（""）の活用

別名を指定する際、基本的にはダブルクォーテーション（`""`）で囲むことをおすすめします。
* **なぜ囲むのか？**
通常、スペースや記号（`@` など）、キーワード（`SELECT` など）を名前に使用できません。しかし、`""` で囲むことで、これらを含む名前もエラーなく使用できるようになります。
* **おすすめの習慣**
特殊文字がない場合でも、常に `""` で囲む癖をつけておくと、予期せぬエラーを防げるため安心です。

#### 具体的な記述例（OKとNGのパターン）

| 判定 | 記述例 | 解説 |
| --- | --- | --- |
| **⭕️** | `country_id AS ID` | 「ID」は通常の文字列なので、`""` なしでも通ります。 |
| **⭕️** | `country_id ID` | `AS` を省略した書き方です。これでも正しく動作します。 |
| **❌** | `country_name AS N@ME` | `@` はSQLが認識できない記号のため、エラーになります。 |
| **⭕️** | `country_name AS "N@ME"` | `""` で囲むことで、`@` を含む名前も使用可能になります。 |

# 問題3-4（並び替え） *Lv1*
### **OEスキーマ`CUSTOMERS`テーブルのうち、`NLS_TERRITORY`が「CHINA」であるデータに対し、**
   *　**`CUST_FIRST_NAME` の昇順**
   *　**`CUST_LAST_NAME` の降順**
### **で並び替え表示してください**


## 期待する結果
| CUSTOMER_ID | CUST_FIRST_NAME | CUST_LAST_NAME | 
| ----------- | --------------- | -------------- | 
| 981         | Daniel          | Gueney         | 
| 308         | Glenda          | Dunaway        | 
| 309         | Glenda          | Bates          | 
| 360         | Helmut          | Capshaw        | 


## 解答例
```sql:例1：カラムを指定する
SELECT
    customer_id,
    cust_first_name,
    cust_last_name
FROM
    oe.customers
WHERE
    nls_territory = 'CHINA'
ORDER BY
    cust_first_name ASC,
    cust_last_name DESC
```
```sql:例2：列順を指定する
SELECT
    customer_id,
    cust_first_name,
    cust_last_name
FROM
    oe.customers
WHERE
    nls_territory = 'CHINA'
ORDER BY
    2, 3 DESC
```

## 解説
抽出したデータを並び替える（ソートする）際は、**ORDER BY句**を使用します。基本的なルールは以下の通りです。

### 1. 昇順と降順の指定

データの並び順は、キーワードを使って指定します。
* **昇順（小さい順）**: `ASC` を指定します。なお、デフォルト（初期設定）が昇順のため、`ASC` は**省略が可能**です。
* **降順（大きい順）**: `DESC` を指定します。

### 2. 複数のソート条件
ソート条件を複数設定したい場合は、各条件を **カンマ（,）** で区切って記述します。

### 3. 列番号による指定
列名だけでなく、抽出データの **列の順番（左からの番号）** で指定することも可能です。
例えば、抽出する列が以下の順番である場合を考えます。

| 番号 | 列名 |
| --- | --- |
| 1番目 | `customer_id` |
| 2番目 | `cust_first_name` |
| 3番目 | `cust_last_name` |

このとき、**「2, 3 DESC」** と指定すると、以下のような並び順になります。

* まず、2番目の列（`cust_first_name`）で**昇順**に並び替える（ASCは省略）。
* 次に、3番目の列（`cust_last_name`）で**降順**に並び替える。

:::message

### **「ASC」の使い分け基準**
SQLを書く際、「ASCは省略できるのになぜ存在するのか？」「いつ書くべきなのか？」という疑問を抱く方は少なくありません。

明確な正解が決まっているわけではありませんが、実務においてコードの美しさと読みやすさを両立させるための、ひとつの判断基準をご紹介します。
<br>
#### 1. すべての条件が「昇順」の場合

この場合は、**ASCを省略します。**
デフォルトの挙動が昇順であるため、あえて記述しないことでコードを冗長（無駄に長い状態）にせず、スッキリと見せることができます。

<br>
#### 2. 「昇順」と「降順」が混在する場合

この場合は、**ASCを省略せずに明示します。**
「ここは昇順、あそこは降順」と意図がはっきり伝わるようになり、コードの**可読性（読みやすさ）** が向上するためです。他の人がコードを見た際にも、設定ミスではないことが一目で分かります。
:::

# 問題3-5（重複削除） *Lv1*
### **SHスキーマ`COUNTRIES`テーブルより`COUNTRY_REGION`を取得してください。その際、重複は排除するようにしてください。（順不同）**

## 期待する結果
| COUNTRY_REGION | 
| -------------- | 
| Americas       | 
| Europe         | 
| Asia           | 
| Oceania        | 
| Africa         | 
| Middle East    | 

## 解答例
```sql:例1：DISTINCTを使用
SELECT DISTINCT
    country_region
FROM
    sh.countries
```
```sql:例2：GROUP BYを使用（参考）
SELECT
    country_region
FROM
    sh.countries
GROUP BY
    country_region
```

## 解説
**`DISTINCT`** は、SQLの結果セットから**重複した行を排除し、ユニーク（一意）な行のみを返す**ためのキーワードです。
`DISTINCT` は `SELECT` の直後に記述し、**「選択されたすべてのカラムの組み合わせ」** が重複しないように制御します。

> **注意点**:
> `SELECT DISTINCT col1, col2` と書いた場合、「col1のみ」ではなく **「col1とcol2のペア」** がユニークなものが抽出されます。特定の1列だけを重複排除して、他の列はそのまま出す、という使い方はできません。


:::message
### とりあえず `DISTINCT`
「とりあえず `DISTINCT`」は、多くのエンジニアが通る **「禁断の果実」** です。よく分からないけどデータが重複しているので、とりあえず`DISTINCT`で重複をなくしちゃおうという行為です。一見すると結果が綺麗になるため、問題が解決したように見えてしまいますが、実は裏で深刻な副作用を引き起こしていることが多々あります。
私は開発プロジェクトで、誰かが書いたSQLスクリプトに`DISTINCT`があると、ちょっと警戒してしまいます…

なぜ「とりあえず `DISTINCT`」が危険なのか、その理由を具体的に解説します。
<br>
### 1. 根本原因：意図しない「1対多」の結合
もっとも多いのが、結合条件の不足によって **「1行のはずが10行、100行に増えている」** ケースです。

※「結合」は後の章で出てきますので、ここでは感覚で理解できれば大丈夫です。
簡単にいうと複数のテーブルからデータを抽出することをいいます。


<br>
#### 具体例
「顧客名の一覧を出したい」と思い、顧客（`CUSTOMERS`）と販売履歴（`SALES`）を結合したとします。

```sql
-- 悪い例：DISTINCTで無理やり消す
SELECT DISTINCT c.cust_id, c.cust_last_name
FROM sh.customers c
JOIN sh.sales s ON c.cust_id = s.cust_id;
```

* **何が起きているか**: 1人の顧客が1,000回購入していれば、`JOIN` 直後の内部データではその顧客の行が**1,000行に増殖**しています。
* **DISTINCTの役割**: その増殖した1,000行を、後から無理やり1行にギュッと凝縮して「なかったこと」にしています。
<br>

### 2. 結合キーの不足に気づけない
例えば「有効な契約期間内のデータだけ出したい」のに、期間の条件（`BETWEEN`）を書き忘れて過去の全データと結合され、重複が発生したとします。

* `DISTINCT` を使うと、その「書き忘れ」という**バグ自体が隠蔽されてしまいます。**
* 本来なら「データが重複している ＝ 結合条件かフィルタが足りない」という**アラート（警告）** として受け取るべき事象を、自ら消し去っていることになります。

<br>

### 結論：`DISTINCT` は「最後の手段」

`DISTINCT` を使っていいのは、**「論理的にどうしても重複が発生する構造であり、かつそれが仕様である」** と断言できるときだけです。

もし「なぜか増えたから消す」という動機で `DISTINCT` を使いたくなったら、それは **「SQLがあなたに『結合条件か設計が間違っているよ！』と教えてくれているサイン」** だと捉えてみてください。
:::
